# Задание_№3

```java
class User {
 int id;
 String name;
}

HashMap<User, String> map = new HashMap();

// В какой ситуации будет утерян элемент внутри map?
```

---
Элемент будет утерян в **HashMap**, если изменить поля объекта **User**, которые участвуют в вычислении `hashCode()`, после того как объект уже положен в мапу.

### Классический пример:
```java
HashMap<User, String> map = new HashMap<>();
User user = new User();
user.id = 1;
user.name = "John";

map.put(user, "some value");  // кладем объект
// Меняем поле, которое влияет на hashCode
user.id = 2;  // или user.name = "Mike"
// Потеряли доступ к значению!
map.get(user);  // вернет null
map.get(new User(1, "John"));  // тоже null
```

### Почему так происходит?
1. **HashMap** при `put()` вычисляет хеш объекта и определяет **бакет** (корзину), куда положить элемент    
2. При изменении полей объекта его **hashCode** меняется    
3. При попытке `get()` вычисляется **новый** хеш, и HashMap ищет элемент **в другом бакете**    
4. Старый бакет помнит про объект, но доступа к нему уже нет - элемент **потерян** (утечка памяти)    

### Визуализация:
```java
// Было: user.hashCode() = 123 → бакет #3
map.put(user, "value");  
// Стало: user.hashCode() = 456 → бакет #7
map.get(user);  // ищем в бакете #7, там пусто
// Объект остался в бакете #3, но мы не можем к нему обратиться
```
### Как избежать?
1. **Не мутировать** ключи после добавления в HashMap    
2. Использовать **immutable** объекты как ключи (как String, Integer)    
3. Если свой класс - сделать его **immutable** (поля `final`, без сеттеров)
