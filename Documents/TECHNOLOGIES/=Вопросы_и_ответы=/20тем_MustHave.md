# Место № 20 // **Record и Generics** // 29 интервью из 75
## 1. **Чем record отличается от обычного класса?**
- Неизменяемый (immutable) по умолчанию
- Автоматически генерирует: `equals()`, `hashCode()`, `toString()`, геттеры
- Все поля — `private final`
- Нельзя наследоваться от другого класса (кроме `java.lang.Record`)
- Нельзя быть абстрактным
- Запрещено менять состояние после создания (нет сеттеров)
- Очень компактный синтаксис
> Коротко: **record** — это «*value class*» / «*data carrier*» с минимумом бойлерплейта.

## 2. **Ограничения record** <br>(наследование, поля, конструктор)    
- **Наследование** — нельзя *extends* другой класс (только *implicit* от *Record*)
- **Реализация интерфейсов** — можно
- **Поля** — только те, что в заголовке *record* (добавлять обычные поля нельзя)
- **Конструкторы** —
	- есть канонический (с параметрами всех компонент)
    - можно добавлять дополнительные конструкторы
    - можно переопределять канонический (compact constructor)
- Нельзя объявлять не-final поля
- Нельзя добавлять методы, которые меняют состояние компонент

## 3. **Wildcards: extends и super**<br>(**PECS** — Producer Extends, Consumer Super)    
```java
List<? extends Number> producer  → можно только читать (get → Number)
List<? super   Integer> consumer → можно только писать (add → Integer или ниже)
```

| Wildcard        | Где <br>обычно используется        | Что <br>можно делать | Что нельзя                            |
| :-------------- | :--------------------------------- | :------------------- | :------------------------------------ |
| `? extends T`   | Producer <br>(возвращает элементы) | get → T              | add (кроме null <br>в редких случаях) |
| `? super T`     | Consumer <br>(принимает элементы)  | add → T              | get → только Object                   |
| `? (unbounded)` | Не знаем тип вообще                | get → Object         | add → почти ничего                    |

## 4. **Type Erasure** — что это и как проявляется?    
- На этапе компиляции вся информация о generic-типах сохраняется
- После компиляции (в байт-коде) `generic`-типы стираются → остаются `raw`-типы
- `List<String>` → `List` в runtime
- Проблемы, которые из-за этого возникают:
   - Нельзя `new T()`
   - Нельзя `new T[]`
   - Нельзя `instanceof List<String>` *(только raw или unbounded wildcard)*
   - Нельзя создать массив generic-типов `T[] arr = new T[10]`;
- Решения: 
 **1. Использовать `ArrayList<T>` вместо массива** (рекомендуемый способ)
		  `List<T> list = new ArrayList<>();`
 **2. Создать массив через `Array.newInstance()` (рефлексия)**
		 ` T[] arr = (T[]) Array.newInstance(clazz, size);`
 **3. Создать массив `Object[]` и приводить при получении**
		  `Object[] arr = new Object[10];`
		  `T element = (T) arr[0];`
 **4. Передать класс в конструктор (Type Token)**
```java
public class MyClass<T> {
	private T[] arr;	
	public MyClass(Class<T> clazz, int size) {
		arr = (T[]) Array.newInstance(clazz, size);
	}
}
```
 **5. Использовать несоздаваемый тип в `ArrayList.toArray(T[])`**
`T[] arr = list.toArray((T[]) Array.newInstance(clazz, list.size()));

## 5. **Bounded Type Parameters** — что это?
```java
<T extends Number & Comparable<T>>   // верхняя граница (upper bound)
<T super   Number>                   // нижняя граница (lower bound) — редко
```
- `<T extends A & B & C>` — T должен реализовывать/наследоваться от всех
- Часто используется в утилитных методах, коллекциях, компараторах
- Пример:
```java
public static <T extends Comparable<? super T>> T max(T a, T b) { ... }
```

# Место № 19 // **CompletableFuture и асинхронность** // 32 интервью из 75

##  6. **Future** vs **CompletableFuture** — главные отличия

| Характеристика                     | Future (Java 5)                     | CompletableFuture (Java 8+)                                  |
| ---------------------------------- | ----------------------------------- | ------------------------------------------------------------ |
| Можно вручную завершить            | Нет                                 | Да (complete(), completeExceptionally())                     |
| Композиция / chaining              | Нет (только get())                  | Да — thenApply, thenCompose, thenCombine, handle и т.д.      |
| Обработка исключений в цепочке     | Только в get() (блокирует)          | Встроено: exceptionally, handle, whenComplete                |
| Параллельное выполнение нескольких | Нужно вручную через ExecutorService | Готово: allOf(), anyOf(), thenCombine()                      |
| Неблокирующие continuation         | Нет                                 | Да — почти все методы async-варианты (thenApplyAsync и т.д.) |
| CompletionStage                    | Нет                                 | Да — можно строить реактивные пайплайны                      |
Коротко: **Future** — это просто «чек на результат позже», **CompletableFuture** — мощный инструмент для построения асинхронных пайплайнов.

## 7 **Как комбинировать несколько CompletableFuture**
В `CompletableFuture` есть несколько способов комбинирования:
### 1️⃣ Параллельное ожидание всех
```java
CompletableFuture.allOf(f1, f2, f3)
```
- Ждёт завершения всех    
- Не возвращает результаты напрямую    
- Используется для параллельных задач    
### 2️⃣ Ожидание первого завершённого
```java
CompletableFuture.anyOf(f1, f2)
```
- Возвращает результат первого завершившегося    
- Удобно для race-стратегий    
### 3️⃣ Комбинация двух результатов
```java
f1.thenCombine(f2, (r1, r2) -> ...)
```
- Оба должны завершиться успешно    
- Объединяет результаты    
### 4️⃣ Зависимая цепочка (flatMap)
```java
f1.thenCompose(result -> anotherFuture(result))
```
- Один future зависит от результата другого    
- Аналог flatMap    
### 5️⃣ Обработка ошибок
```java
exceptionally(...)  
handle(...)  
whenComplete(...)
```
- Позволяют управлять исключениями    
## 🧠 Как объяснить на собеседовании
- `allOf` — параллельные независимые задачи    
- `thenCombine` — объединить результаты    
- `thenCompose` — зависимая асинхронная цепочка    
- `anyOf` — взять первый результат

## 8. **Обработка исключений в асинхронной цепочке**
✅ Краткий, чёткий ответ (middle)
В `CompletableFuture` есть **3** основных способа обработки ошибок:
### 1️⃣ `exceptionally(fn)` — fallback при ошибке
cf.exceptionally(ex -> "fallback");
- Вызывается **только если было исключение**    
- Возвращает заменяющее значение    
- Удобен для простого fallback    
> 📌 Самый часто используемый.
### 2️⃣ `handle(biFn)` — универсальный способ
cf.handle((res, ex) -> ex != null ? fallback : res + " ok");
- Вызывается **всегда**    
- Можно изменить результат    
- Видит и `res`, и `ex`    
> 📌 Самый гибкий вариант.
### 3️⃣ `whenComplete(biConsumer)` — side-effect
cf.whenComplete((res, ex) -> log.error("Ошибка!", ex));
- Вызывается всегда    
- **Не меняет результат**    
- Подходит для логирования    
> 📌 Аналог `finally`.
### ⚠ Важно понимать
- Если исключение не обработать → оно пробросится при `get()` / `join()`    
- `exceptionally` и `handle` позволяют продолжить цепочку    
- `join()` → кидает `CompletionException` (unchecked)    
- `get()` → кидает `ExecutionException` (checked)    
### 🧠 Как сказать на собеседовании
> "Для fallback использую `exceptionally`,  
> для универсальной обработки — `handle`,  
> для логирования — `whenComplete`.  
> Если ошибку не перехватить — она всплывает в `join()` или `get()`."

## 9. **Параллельные запросы** (самые частые паттерны)
```Java
// 1. Ждём все (самый популярный)
CompletableFuture.allOf(f1, f2, f3)
    .thenRun(() -> { /* все готовы */ })
    .join();

// 2. Собираем результаты (часто со Stream)
List<String> results = Stream.of(f1, f2, f3)
    .map(CompletableFuture::join)
    .toList();

// 3. Первый победивший (race)
CompletableFuture.anyOf(f1, f2, f3)
    .thenAccept(winner -> { ... });

// 4. Комбинируем два независимых результата
f1.thenCombine(f2, (a, b) -> a + b);

// 5. Зависимая последовательность (flatMap-стиль)
f1.thenCompose(res -> callNextService(res));
```
Для большого количества → часто используют allOf + массив / список + join() / Stream.

✅ Краткий ответ (уровень middle)
Основные паттерны параллельных вызовов через `CompletableFuture`:

### 1️⃣ Fan-out / Fan-in (самый частый)
Параллельно отправляем несколько запросов → ждём все → объединяем.
```java
CompletableFuture.allOf(f1, f2, f3)
```
✔ Максимальная параллельность  
✔ Используется в микросервисах  
⚠ Нужно отдельно получить результаты
> 📌 Типичный кейс: агрегатор API.

### 2️⃣ thenCombine — объединение двух результатов
```java
f1.thenCombine(f2, (r1, r2) -> combine(r1, r2));
```
✔ Когда оба результата нужны  
✔ Чище, чем allOf для 2 future

### 3️⃣ thenCompose — зависимый вызов
```java
f1.thenCompose(res -> callAnotherService(res));
```
✔ Второй запрос зависит от первого  
✔ Асинхронный flatMap

### 4️⃣ anyOf — race pattern
```java
CompletableFuture.anyOf(f1, f2);

```
✔ Берём первый ответ  
✔ Используется для fallback / fastest-wins

### 5️⃣ Timeout + fallback
```java
cf.orTimeout(1, SECONDS)  
  .exceptionally(ex -> fallback);
```
✔ Защита от зависаний  
✔ Часто в продакшене

### 🧠 Как сформулировать на собеседовании
> Самый частый паттерн — fan-out/fan-in через `allOf`.  
> Для объединения двух — `thenCombine`.  
> Для зависимых вызовов — `thenCompose`.  
> Для race-логики — `anyOf`.  
> Обязательно добавляем timeout и fallback.

## 10. **Блокирующий vs неблокирующий код**

| Стиль                           | Пример вызова                 | Блокирует поток? | Когда использовать                             | Типичный пул потоков           |
| ------------------------------- | ----------------------------- | ---------------- | ---------------------------------------------- | ------------------------------ |
| Блокирующий                     | future.get()                  | Да               | Тесты, main, batch-задачи                      | —                              |
| Блокирующий, но удобный         | future.join()                 | Да               | То же, но без checked exceptions               | —                              |
| Неблокирующий continuation      | thenApplyAsync(...)           | Нет              | Веб / реактивные приложения                    | ForkJoinPool.commonPool()      |
| Неблокирующий с кастомным пулом | thenApplyAsync(..., executor) | Нет              | Контроль нагрузки, I/O-bound задачи            | Собственный ThreadPoolExecutor |
| Полностью асинхронный пайплайн  | цепочка then... без get/join  | Нет              | Spring WebFlux, микросервисы, высокая нагрузка | —                              |

## Лучшая практика 2025–2026:
- В контроллерах / сервисах — **никогда** не вызывать `get()` /` join()` в основном потоке
- Возвращать `CompletableFuture<T>` клиенту (или `Mono`/`Flux` в реактивном стеке)
- Для I/O-bound → свой пул с фиксированным размером (не commonPool)

# Место № 18 // **ООП** и **SOLID** // 39 интервью из 75

- Как-бы ты отрефакторил, 
- показывают участок кода и "что именно тут нарушено"

## 11. **Примеры нарушения принципов SOLID** (самые частые на собеседованиях)

| Принцип                       | Нарушение (пример)                                                                           | Почему нарушение                                                              |
| ----------------------------- | -------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| **S** — Single Responsibility | UserService делает: сохраняет пользователя, отправляет email, логирует, генерирует PDF-отчёт | Класс меняется по нескольким причинам → трудно тестировать/поддерживать       |
| **O** — Open/Closed           | PaymentProcessor с if (type == "card") ... else if (type == "paypal") ...                    | При добавлении новой оплаты (crypto) — правим существующий класс              |
| **L** — Liskov Substitution   | Square extends Rectangle → setWidth(5) меняет и height                                       | Подстановка Square вместо Rectangle ломает поведение клиента                  |
| **I** — Interface Segregation | Worker implements Workable, Eatable, Sleepable (для человека и робота)                       | Робот вынужден реализовывать eat() → бросает UnsupportedOperationException    |
| **D** — Dependency Inversion  | OrderService { private MySQLRepo repo = new MySQLRepo(); }                                   | Жёсткая зависимость от конкретной реализации → трудно менять БД / тестировать |

## 12. **Как отрефакторить плохой код** (типичные паттерны)
- **SRP нарушение** → выносим ответственность в отдельные классы/сервисы 
  Плохо: `UserManager` (save + email + log) 
  Хорошо: `UserRepository`, `NotificationService`, `AuditLogger` → инжектим через конструктор
- **OCP нарушение** → Strategy / Plugin / Enum + Factory 
  Плохо: switch по типу 
  Хорошо: `interface PaymentStrategy { void pay(); }` → `CreditCardStrategy`, `PayPalStrategy` → `PaymentProcessor` принимает стратегию
- **LSP нарушение** → убираем наследование, переходим на композицию или отдельные интерфейсы 
  Плохо: Bird { fly() } → Penguin extends Bird 
  Хорошо: interface Flyable, Penguin implements Swimmable (без Flyable)
- **ISP нарушение** → разбиваем жирный интерфейс 
  Плохо: `Machine { start(), stop(), print(), scan() }` 
  Хорошо: `Printable`, `Scannable`, `Startable` — класс реализует только нужное
- **DIP нарушение** → Dependency Injection + интерфейсы 
  Плохо: new внутри класса 
  Хорошо: `@Autowired` / конструктор с `Repository repo` (интерфейс)
## 13. **Зачем вообще нужен SOLID?**
- Код легче **поддерживать** (меньше багов при изменениях)
- Легче **тестировать** (моки, изоляция)
- Легче **расширять** (новые фичи без правки старого кода)
- Уменьшает **сцепление** → система гибче, переиспользование выше
- Упрощает **понимание** кода новыми разработчиками
- Помогает избежать **технического долга** в долгоживущих проектах
> Коротко: **SOLID** — это не догма, а набор инструментов, чтобы код не превратился в спагетти через 2–3 года.

## 14. **Композиция vs наследование** — когда что использовать

| Критерий                      | Наследование (is-a)                     | Композиция (has-a)                      |
| ----------------------------- | --------------------------------------- | --------------------------------------- |
| Связь                         | Жёсткая, статическая                    | Свободная, динамическая                 |
| Гибкость                      | Низкая (нельзя поменять суперкласс)     | Высокая (можно менять/подменять части)  |
| Переиспользование             | Только через иерархию                   | Любое поведение через делегирование     |
| Проблемы                      | Хрупкая иерархия, LSP часто ломается    | Больше кода (делегирование)             |
| Когда предпочитать            | Полиморфизм + «является» (Animal → Dog) | Почти всегда (особенно в бизнес-логике) |
| Рекомендация (Effective Java) | Favor composition over inheritance      | —                                       |

#### Правило 2025–2026:
- Используй **наследование** только для **полиморфизма** и когда LSP гарантированно выполняется.
- Во всех остальных случаях — **композиция** + интерфейсы (Strategy, Decorator, Delegate).

## 15. **Абстракции и зависимости**
- **Зависеть от абстракций, а не от конкретики** (DIP)
- Высокоуровневые модули не должны зависеть от низкоуровневых — оба зависят от абстракций
- Пример: 
  Плохо: `ReportGenerator` зависит от `PdfLibrary` 
  Хорошо: `ReportGenerator` зависит от `ReportRenderer` (интерфейс) → `PdfRenderer`, `HtmlRenderer`

Цель: инверсия управления → легко менять реализации, тестировать, расширять.

# Место № 17 // **Микросервисы** // 43 интервью из 75
- В основном сводится к обсуждению паттернов проектирования

## 16. Saga
**Паттерн Saga** — это способ управления распределенными транзакциями (данными) в микросервисной архитектуре, где нет глобального ACID-транзакций.
- **Суть:** Одна бизнес-операция разбивается на несколько локальных транзакций в разных сервисах. Каждая локальная транзакция публикует событие, которое триггерит следующую транзакцию.    
- **Компенсация:** Если один из шагов завершается неудачей, Saga запускает серию _компенсирующих транзакций_ (rollback), которые отменяют изменения, сделанные предыдущими шагами.    
- **Реализации:**    
    - _Хореография (Choreography):_ Сервисы общаются через события (например, Kafka), нет центрального координатора.        
    - _Оркестрация (Orchestration):_ Центральный класс (оркестратор) говорит сервисам, что делать.        

## 17. Circuit Breaker (Автоматический выключатель)
**Паттерн для обеспечения отказоустойчивости.** Предотвращает каскадное падение сервисов, когда один из сервисов недоступен или работает медленно.
- **Как работает (как в электротехнике):**    
    1. _Closed (Замкнут):_ Запросы проходят к сервису. Если ошибки накапливаются (или таймауты), счетчик открывается.        
    2. _Open (Разомкнут):_ Запросы _не_ отправляются в проблемный сервис, а мгновенно падают с ошибкой (fallback). Даем сервису время восстановиться.        
    3. _Half-Open (Полуоткрыт):_ Через некоторое время пропускается несколько запросов для проверки. Если они успешны — выключатель закрывается, если нет — снова открывается.
> **Пример:** Resilience4j, Hystrix.

## 18. Service Discovery (Обнаружение сервисов)
**Механизм, позволяющий сервисам находить друг друга по сети**, не используя жестко прописанные IP-адреса и порты (особенно актуально в динамичных средах типа Kubernetes).
- **Компоненты:**    
    - _Service Registry:_ База данных с адресами всех доступных экземпляров сервисов.        
    - _Provider:_ Сервис, который при старте регистрируется в реестре, а при остановке — дерегистрируется.        
    - _Consumer:_ Сервис, который хочет обратиться к другому сервису. Он спрашивает адрес у реестра.        
- **Варианты:**    
    - _Client-Side Discovery:_ Клиент сам обращается к реестру (например, Eureka) и балансирует нагрузку (Ribbon).        
    - _Server-Side Discovery:_ Клиент идет через балансировщик (например, Kubernetes + Kube-Proxy), который сам скрывает логику обнаружения.        

## 19. API Gateway
**Это единая точка входа (шлюз) для всех клиентов (фронтенд, мобильные приложения).** Является реализацией паттерна "Вратарь" (Gateway/Edge Server).
- **Основные функции:**    
    - **Маршрутизация:** Перенаправляет запросы клиентов к нужным микросервисам (например, `/users` -> user-service, `/orders` -> order-service).    
    - **Агрегация:** Может собрать данные из нескольких сервисов в один ответ (чтобы клиент не делал 5 запросов).        
    - **Кросс-функциональность:** Аутентификация/JWT-проверка, логирование, rate limiting (ограничение запросов), SSL termination.
> **Примеры:** Spring Cloud Gateway, Zuul, Kong, Nginx.

## 20. Шардирование и масштабирование
**Два разных подхода к увеличению производительности:**
- **Масштабирование:**    
    - _Вертикальное:_ Добавляем ресурсы (CPU, RAM) одной машине. Легко, но имеет предел.        
    - _Горизонтальное:_ Запускаем больше копий (инстансов) сервиса. Сложнее (нужен балансировщик), но надежнее и безграничнее (облака).        
- **Шардирование (Sharding) — это способ горизонтального масштабирования данных.**    
    - _Суть:_ Большая таблица (или база данных) разбивается на более мелкие части (шарды) по какому-то ключу (например, `user_id`).        
    - _Цель:_ Распределить нагрузку на запись и чтение между разными серверами БД.        
    - _Минусы:_ Сложность запросов (нельзя просто сделать JOIN по всем шардам), сложность ребалансировки при добавлении новых шардов.

## 21. N+1 проблема
Это классическая проблема производительности при работе с ORM.
- **Суть:** Когда ты получаешь N родительских записей (например, 10 книг), Hibernate делает **один запрос** для получения книг, а затем для каждой книги делает еще по **одному запросу** (всего N доп. запросов), чтобы подгрузить связанные данные (например, авторов). Итого: 1 (на список) + N (на авторов) = N+1 запросов.    
- **Как обнаружить:** Включить логирование SQL (show_sql) и увидеть лавину селектов в цикле.    
- **Как лечить:**    
    - **JOIN FETCH:** Написать JPQL с `JOIN FETCH entity.relation`.        
    - **@EntityGraph:** Указать граф загрузки для конкретного запроса.        
    - **@BatchSize:** Подгружать связанные сущности пачками (например, по 10 штук за раз).        

## 22. **Lazy** vs **Eager** загрузка (**FetchType**)
Режимы загрузки связанных сущностей (коллекций или объектов).
- **Eager (Жадная):** Связанные данные подгружаются _немедленно_, в том же запросе (через JOIN) или отдельным запросом сразу. 
  **Минус:** Если данные не нужны, мы зря тащим тяжелый граф объектов из БД.    
- **Lazy (Ленивая):** Связанные данные подгружаются _только в момент первого обращения_ к ним (геттеру). 
	  **Плюс:** Экономия ресурсов, если данные не используются. 
	  **Минус:** Нужно следить, чтобы сессия Hibernate (транзакция) была еще открыта в момент обращения (`LazyInitializationException`).    
- _Правило:_ По умолчанию для `@ManyToOne` и `@OneToOne` — EAGER, для `@OneToMany` и `@ManyToMany` — LAZY. На практике часто все явно выставляют в LAZY и контролируют загрузку через запросы.    

## 23. Жизненный цикл Entity (Состояния)
Объект-сущность в Hibernate проходит через 4 состояния:
1. **Transient (Новое/Временное):** Объект создан через `new`, но **не привязан** к сессии Hibernate и не имеет записи в БД.    
2. **Persistent (Управляемое):** Объект привязан к сессии (`persist`, `save`, `get`, `load`). Изменения в этом объекте автоматически синхронизируются с БД в момент flush.  
3. **Detached (Отвязанное):** Объект был Persistent, но сессия закрылась или был вызван `detach`/`evict`. У объекта есть ID в БД, но Hibernate за ним не следит.    
4. **Removed (Удаленное):** Объект помечен на удаление (`remove`), но транзакция еще не закоммичена.    

## 24. Уровни кэшей
Hibernate имеет многоуровневую систему кэширования для уменьшения количества запросов к БД.
- **1-й уровень (L1 — First Level Cache):**    
    - Это кэш **сессии (Session)**. Обязательный и включенный всегда.        
    - Работает в пределах одной транзакции. Если дважды прочитать одну запись по ID в рамках одной сессии, запрос в БД уйдет только один раз.        
- **2-й уровень (L2 — Second Level Cache):**    
    - Это кэш **фабрики сессий (SessionFactory)**. Разделяется между всеми сессиями.  
    - Нужно подключать отдельной библиотекой (EHCache, Redis). Хранит данные, общие для всего приложения.        
    - Требует настройки (`@Cacheable`, `@Cache`).        
- **3-й уровень (Кэш запросов):**    
    - Кэширует результаты JPQL/HQL запросов. Работает поверх L2.        
    - Теряет смысл, если данные часто меняются (требует инвалидации).        

## 25. Dirty Checking (Проверка изменений)
Механизм Hibernate, который автоматически сохраняет в БД изменения, сделанные в Persistent объекте, без явного вызова `update`.
- **Как работает:**    
    1. Когда объект загружается в Persistent состояние, Hibernate снимает с него "слепок" (snapshot) текущего состояния.        
    2. В момент `flush()` (обычно при коммите транзакции) Hibernate сравнивает текущее состояние объекта со снапшотом.        
    3. Если есть различия (поле изменилось), Hibernate сам генерирует и выполняет SQL `UPDATE` только для изменившихся полей.        
- **Пример:**
```java
User user = session.get(User.class, 1L); // persistent
user.setName("Новое имя"); // dirty checking сработает автоматически
session.getTransaction().commit(); // UPDATE выполнится сам
```

## 26. Топики и партиции
**Базовые единицы хранения данных в Kafka.**
- **Топик (Topic):** Это категория или "канал" для сообщений (аналог таблицы в БД). Производители пишут в топик, потребители читают из него.    
- **Партиция (Partition):**    
    - Топик физически разбит на несколько партиций (кусков).        
    - Партиция — это упорядоченный, неизменяемый лог (очередь), куда сообщения дописываются в конец.        
    - Сообщения внутри _одной_ партиции имеют строгий порядок (увеличивающийся offset).        
    - Партиции позволяют _распараллеливать_ обработку: разные потребители могут читать разные партиции одновременно.        
    - Выбор партиции: если ключа нет — round-robin, если есть ключ (key) — хеш ключа определяет партицию (гарантия порядка для одного ключа).        

## 27. Consumer Groups (Группы потребителей)
**Способ масштабирования потребления и обеспечения отказоустойчивости.**
- **Суть:** Несколько потребителей (инстансов приложения) объединяются в группу с общим `group.id`.    
- **Как работает:**    
    - Каждая партиция топика назначается **ровно одному** потребителю внутри группы.        
    - Это гарантирует, что сообщение будет прочитано только одним потребителем из группы (не дублируется внутри группы).        
    - Если потребителей больше, чем партиций, лишние будут простаивать.        
    - Если потребителей меньше, некоторые потребители будут читать из нескольких партиций.        
- **Ребалансировка:** При добавлении/удалении потребителя в группе происходит перераспределение партиций.    

## 28. Гарантии доставки сообщений
Уровни надежности, которые Kafka может гарантировать.
- **At most once (не более одного раза):** Сообщение может потеряться, но не будет обработано повторно. (Producer отправляет и не ждет подтверждения).    
- **At least once (не менее одного раза) — _наиболее частый случай_:** Сообщение точно будет доставлено, но возможны дубли. (Producer ждет `ack` от брокера; если `ack` не пришел, шлет снова -> дубль).    
- **Exactly once (ровно один раз):** Сообщение не теряется и не дублируется.    
    - Достигается комбинацией: идемпотентный продюсер (`enable.idempotence=true`) + транзакции + правильная работа с оффсетами на стороне потребителя.        
    - _Важно:_ "Exactly once" часто подразумевает идемпотентную обработку на стороне потребителя.        

## 29. Offset и commit
Механизм, позволяющий потребителю запоминать, что он уже прочитал.
- **Offset:** Уникальный (в пределах партиции) порядковый номер сообщения. Каждое сообщение в партиции имеет свой offset.    
- **Commit:** Процесс сохранения текущей позиции (какой offset прочитан).    
- **Виды коммитов:**    
    - _Автоматический (`enable.auto.commit=true`):_ Потребитель раз в `auto.commit.interval.ms` сам коммитит последний прочитанный offset. Удобно, но есть риск потерять сообщения или обработать дубли (при краше).   
    - _Ручной:_ Разработчик сам вызывает `commitSync()` или `commitAsync()`. Дает полный контроль.        
        - _At most once:_ Коммитим _до_ обработки сообщения.            
        - _At least once:_ Коммитим _после_ обработки сообщения (стандарт).            

## 30. Репликация
Обеспечение отказоустойчивости данных за счет копирования партиций на несколько брокеров.
- **Суть:** Каждая партиция имеет несколько копий (replicas), распределенных по разным серверам.    
- **Типы реплик:**    
    - **Leader:** Одна из реплик, которая обслуживает все запросы на запись и чтение для данной партиции.        
    - **Follower (In-sync replicas — ISR):** Остальные реплики, которые пассивно копируют данные от лидера.        
- **Фактор репликации (replication.factor):** Сколько всего копий данных хранится (например, 3 означает, что данные переживут потерю 2 брокеров).    
- **Если лидер падает:** Один из ISR-фолловеров автоматически становится новым лидером, и работа продолжается (если настроен `ack=all`).
