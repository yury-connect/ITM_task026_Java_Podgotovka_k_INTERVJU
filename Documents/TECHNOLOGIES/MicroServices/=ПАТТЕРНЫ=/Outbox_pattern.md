# Outbox_pattern
**Outbox Pattern** — паттерн для обеспечения **консистентности** между базой данных и очередью сообщений (или другим внешним сервисом) <u>без использования распределенных транзакций (2PC)</u>.

## Суть
Когда сервис сохраняет данные в БД, он **не отправляет** событие сразу в брокер. Вместо этого он сохраняет событие в **ту же транзакцию** в специальную таблицу `outbox`.

## Как работает (последовательность)	
1. **Транзакция БД:** Сервис пишет бизнес-сущность (например, в таблицу `users`) и запись о событии `UserCreated` в таблицу `outbox`. Это одна атомарная операция.    
2. **Релиайер (Relayer):** Отдельный процесс, сканирует таблицу `outbox`.    
3. **Публикация:** *Релиайер* отправляет сообщение в брокер (*Kafka, RabbitMQ, etc.*).    
4. **Удаление (или отметка):** После успешной публикации и подтверждения доставки запись либо удаляется, либо помечается как обработанная (`processed_at`, `state = 'done'` и т.д.).

## Критически важные нюансы (*PostgreSQL*)	
- **Идемпотентность:** 
	Сообщения могут прийти **дубликатом** (at-least-once семантика). 
	Consumer обязан уметь обрабатывать повторы → уникальный `event_id` / `idempotency_key` в сообщении — must-have.    
- **Порядок сообщений:** 
	Если порядок критичен → сохраняйте в таблице `created_at` / `sequence_id` / `outbox_id` и отправляйте **строго по возрастанию**.
	При поллинге порядок между **разными транзакциями** не гарантируется на 100 %.
	При использовании **логической репликации** порядок сохраняется идеально (порядок коммитов в WAL).
- **Производительность:** 
	Постоянный поллинг (`SELECT … FOR UPDATE SKIP LOCKED`) создаёт ненужную нагрузку на БД, особенно при низкой частоте событий.
	**Рекомендуемый сегодня способ** (2025–2026): **логическая репликация** + `pg_logical_emit_message()` или **Debezium Outbox** / **WAL-ориентированные CDC**-решения.
	Это **push**-модель: БД сама «толкает» события, без polling’а, с минимальной задержкой и сохранением порядка.
- **Идемпотентность публикации:** 
	Релейер должен гарантировать at-least-once и не допускать двойной обработки одной записи несколькими инстансами.
	Классический приём при поллинге:   
```sql
SELECT * FROM outbox
WHERE processed_at IS NULL
ORDER BY id
FOR UPDATE SKIP LOCKED
LIMIT 10;
```

**`KIP LOCKED`** — это директива в PostgreSQL для пропуска заблокированных строк.

## **Суть** 
Обычный `SELECT … FOR UPDATE` будет **ждать** разблокировки строк. `SKIP LOCKED` говорит: «Верни только свободные строки прямо сейчас, заблокированные пропусти».

## Зачем в **Outbox** 
Несколько воркеров/инстансов релейера не дерутся за одну и ту же запись. Каждый получает свой уникальный набор строк и может обрабатывать их параллельно.
	Пример:
```SQL
SELECT id, payload, topic
FROM outbox
WHERE state = 'pending'
ORDER BY created_at, id
FOR UPDATE SKIP LOCKED
LIMIT 50;
```

## Зачем вообще **Outbox**?
- Гарантия: если транзакция закоммитилась → событие **точно будет** отправлено (рано или поздно).
- Если транзакция откатилась → события не будет вообще.
- Нет 2PC / XA-транзакций → проще, надёжнее, лучше производительность.
