**Outbox Pattern** — паттерн для обеспечения **консистентности** между базой данных и очередью сообщений (или другим внешним сервисом) без использования распределенных транзакций (2PC).

### Суть
Когда сервис сохраняет данные в БД, он не отправляет событие сразу в брокер. Вместо этого он сохраняет его в ту же **транзакцию** в специальную таблицу `outbox`.

### Как работает (последовательность)	
1. **Транзакция БД:** Сервис пишет бизнес-сущность в таблицу `users` и запись о событии `UserCreated` в таблицу `outbox`. Это одна атомарная операция.    
2. **Релиайер (Relayer):** Отдельный процесс (`Pooler` или `Publisher`) сканирует таблицу `outbox`.    
3. **Публикация:** *Релиайер* отправляет сообщение в брокер (*Kafka, RabbitMQ*).    
4. **Удаление (или отметка):** После подтверждения доставки запись из `outbox` удаляется или помечается как обработанная.    

### Критически важные нюансы (PostgreSQL)	
- **Идемпотентность:** Сообщения могут быть доставлены дважды (например, если подтверждение от брокера потерялось). *Consumer* на той стороне должен уметь обрабатывать дубликаты (уникальные ключи сообщений).    
- **Порядок сообщений:** Если важен порядок, нужно сохранять в таблице `timestamp` или `sequence_id`, а релайер должен отправлять строго по порядку (часто используя логическую репликацию).    
- **Производительность:** Постоянный поллинг таблицы (`SELECT ... FOR UPDATE SKIP LOCKED`) создает нагрузку. В *PostgreSQL* часто оптимальнее использовать **логическую репликацию** (decode changes from WAL), чтобы "вытягивать" изменения напрямую из журнала транзакций.    
- **Идемпотентность публикации:** Релайер должен гарантировать _at-least-once_ (хотя бы раз) и иметь механизм блокировок (чтобы несколько инстансов релайера не схватили одну запись). `SKIP LOCKED` здесь лучший друг.    

### Зачем?
- Гарантия, что событие _будет_ отправлено (или не будет, если транзакция откатилась).    
- Нет распределенных транзакций (SAGA не нужна на этом этапе).

---
**`KIP LOCKED`** — это директива в PostgreSQL для пропуска заблокированных строк.

**Суть:**  
При использовании `SELECT ... FOR UPDATE` обычный запрос будет **ждать**, пока строки не разблокируются. `SKIP LOCKED` говорит базе: "Пропусти все строки, которые сейчас кто-то блокирует, и верни только те, что свободны прямо сейчас".

**Зачем нужно (Outbox и очереди):**  
В паттерне *Outbox* несколько потоков (воркеров) не должны драться за одно и то же сообщение.  
	Запрос:
```sql
SELECT * FROM outbox ORDER BY id FOR UPDATE SKIP LOCKED LIMIT 10;
```

Каждый воркер получит свои **10 неповторяющихся** строк и обработает их параллельно, не блокируя друг друга.
