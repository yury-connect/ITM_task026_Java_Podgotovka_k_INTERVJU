Spring Bean — это, объект, который управляется IoC-контейнером Spring. 
Чтобы ваш Java-объект стал частью магии Spring, он д.б. зарегистрирован как Bean.

Есть несколько способов зарегистрировать бины:
- **Аннотации** (например, `@Component`, `@Service`, `@Repository`).
- **Java-based конфигурация** с использованием `@Configuration` и `@Bean`.
- **XML-конфигурация** (да, она всё ещё существует, но используется всё реже, ибо зачем, когда есть Java-аннотации?).

# Жизненный цикл Spring **Bean**
## 1. Этап **инициализации** (*Startup*)

- **Создание BeanDefinition**: Spring сканирует конфигурацию (XML, Java-аннотации `@Component`, `@Bean`) и создает «*рецепт*» для создания бина.
  
- **Создание экземпляра (Instantiating)**: Контейнер вызывает конструктор класса, чтобы создать объект.
  
- **Внедрение зависимостей (Populating properties)**: Spring внедряет необходимые зависимости через поля (с `@Autowired`), сеттеры или конструктор.  
  > - **Через конструктор** – рекомендуется для обязательных зависимостей, в комбинации с модификатором **final** обеспечивает неизменяемость объекта.
  > 	- **Через поле** - **`@Autowired`** можно ставить на поле, и Spring внедрит зависимость. Однако этот способ **не рекомендуется**, так как он усложняет тестирование и делает зависимость менее явной. 
  > 	- **Через сеттеры** – исп-ся, если зависимость может изменяться после созд. бина.
  > 	- **Через аннотации `@Value`, `@Qualifier` и др.** – применяется для внедрения простых значений и указания конкретного бина при наличии нескольких кандидатов.
  
- **Aware-интерфейсы**: Если бин реализует специальные интерфейсы, Spring передает ему служебную информацию о контексте (*имя бина, контекст приложения*).  
> 	- `BeanNameAware` — получает **имя бина**
> 	- `BeanFactoryAware` — доступ к **BeanFactory**
 >	- `ApplicationContextAware` — доступ ко **всему контексту Spring**    
>⚠️ Использовать их с осторожностью, т.к. это привязывает код к Spring API и усложняет тестирование.
  
- **BeanPostProcessor *(До инициализации)***: Вызывается метод `postProcessBeforeInitialization` для предварительной обработки (например, <u>создания прокси-объектов</u>).
 > - **Валидация бинов** – проверка корректности данных перед их использованием.
 > - **Изменение свойств** – настройка конфигурации бина в зависимости от профиля приложения (@Profile).
> - **Логирование** – запись информации о бине перед его инициализацией.
  
- **Init-методы** (*Инициализация*): Выполняется пользовательская логика инициализации <u>в следующем порядке</u>:
    1. Метод с аннотацией `@PostConstruct`.
    2. Метод `afterPropertiesSet()` (если реализован интерфейс `InitializingBean`).
    3. Пользовательский `init-method`, указанный в конфигурации.
    Здесь можно проверять настройки бина и подключаться к ресурсам (БД).
    
- **BeanPostProcessor (После инициализации)**: Вызывается метод `postProcessAfterInitialization`. 
 >- **Проксирование бинов** – создание AOP-прокси для аннотаций @Transactional, @Async и подобных.
> - **Кэширование** – автоматическое добавление механизма кэширования для методов.
> - **Изменение бинов** – динамическое добавление новых методов или обертывание логикой безопасности.
> ⚠️ При использовании проксирования бин может подменяться Spring-оберткой (**важно при instanceof**).

## 2. **Использование** (*In Use*)
После завершения всех настроек бин готов к работе и используется приложением. 

## 3. **Уничтожение** (*Shutdown*)
Когда контейнер закрывается, он корректно завершает работу бинов (обычно `singleton`): 

- **Destroy-методы**: Вызываются для освобождения ресурсов (закрытие соединений с БД, потоков и т.д.):
    1. Метод с аннотацией `@PreDestroy`.
    2. Метод `destroy()` (если реализован интерфейс `DisposableBean`).
    3. Пользовательский `destroy-method`, указанный в настройках.

## **`@PostConstruct`** и **`@PreDestroy`**
- **`@PostConstruct`**: выполняет действия после создания бина и установки всех его зависимостей (*например, подключает сервис или загружает данные*).
- **`@PreDestroy`**: срабатывает перед уничтожением бина. Используется для освобождения ресурсов (*например, закрытия соединений*).

Когда Spring видит аннотации `@PostConstruct` и `@PreDestroy`, он гарантирует, что эти методы будут вызваны в соответствующие моменты.
```java
import org.springframework.stereotype.Component; 
import javax.annotation.PostConstruct; 
import javax.annotation.PreDestroy; 

@Component 
public class LifecycleDemoBean { 
	
	@PostConstruct 
	public void init() { 
		System.out.println("Бин инициализирован!"); 
	} 
	
	public void doSomething() { System.out.println("Бин работает!"); } 
	
	@PreDestroy 
	public void cleanup() { 
		System.out.println("Бин уничтожен!"); 
	} 
}
```

## Альтернатива с интерфейсами <br>**`InitializingBean`** и **`DisposableBean`**

Если вы предпочитаете интерфейсы вместо аннотаций (например, для более явного контракта), можно использовать:
- **`InitializingBean`** — метод `afterPropertiesSet()` вызывается после создания бина.
- **`DisposableBean`** — метод `destroy()` вызывается перед уничтожением бина.

```java
import org.springframework.beans.factory.DisposableBean;
import org.springframework.beans.factory.InitializingBean;
import org.springframework.stereotype.Component;

@Component
public class AlternativeLifecycleBean implements InitializingBean, DisposableBean{
    @Override
    public void afterPropertiesSet() throws Exception {
        System.out.println("Объект настроен (InitializingBean)!");
    }

    @Override
    public void destroy() throws Exception {
        System.out.println("Объект уничтожен (DisposableBean)!");
    }
}

```
Чаще используют аннотации `@PostConstruct` и `@PreDestroy` (проще и читабельнее).

## Основные **скоупы**
- **Singleton**: один бин на весь IoC-контейнер.
- **Prototype**: новый объект создаётся каждый раз, когда бин запрашивается.
- **Request**: один экземпляр для каждого HTTP-запроса.
- **Session**: один экземпляр для каждой HTTP-сессии.

Аннотация `@Scope` позволяет управлять жизненным циклом бина.
```java
import org.springframework.context.annotation.Scope;
import org.springframework.stereotype.Component;

@Component
@Scope("prototype") // При каждом использовании -> создаётся новый экземпляр!
public class PrototypeBean {
    public PrototypeBean() {
        System.out.println("Создан новый экземпляр PrototypeBean!");
    }
}
```
