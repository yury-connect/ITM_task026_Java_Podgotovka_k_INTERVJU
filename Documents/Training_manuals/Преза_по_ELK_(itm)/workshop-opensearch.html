Централизованное логирование

Главная идея: все сервисы пишут логи → система собирает их в одно место


  • Сбор логов из всех источников
  • Единое хранилище и индексация
  • Быстрый поиск и фильтрация
  • Дашборды и алерты

Сервисы не думают о том, куда уходят логи. Только stdout.

Что такое OpenSearch

2010 → Elasticsearch создан, лицензия Apache 2.0

2021 → Elastic сменила лицензию на SSPL — нельзя использовать в SaaS

2021 → AWS сделал форк → OpenSearch — снова open-source

  • OpenSearch = Elasticsearch (движок, хранение, поиск)
  • OpenSearch Dashboards = Kibana (Web UI)
  • API практически идентичен — знаете один, знаете оба

Архитектура логирования в Kubernetes

  • Java-сервис (Spring Boot)
    log.info("User created") → log4j2 форматирует в JSON
  • stdout контейнера
    Docker/containerd сохраняет в /var/log/containers/
  • Fluentd DaemonSet
    запущен на каждой ноде, читает логи всех контейнеров
  • OpenSearch
    индексирует документы, хранит
  • OpenSearch Dashboards
    разработчик открывает UI и ищет

Важно: сервисы не отправляют логи напрямую в OpenSearch — только stdout

Путь лога: от кода до интерфейса

Log4j2 в проекте

  • Форматирует логи через JsonTemplateLayout → JSON в stdout
  • Управляет уровнями: ERROR · WARN · INFO · DEBUG
  • Параметры через {} — подставляются только если уровень включён
log.info("User created: userId={}", id);
log.error("User not found: id={}", id, ex);
log.debug("Request payload: {}", body);

На проде обычно INFO и выше · DEBUG включают только при необходимости

Почему JSON-логи?

Plain text лог

2024-01-15 10:23:45 ERROR
[user-service] User not found
id=123 traceId=abc-def
  • Нужен regexp для парсинга
  • Формат может меняться
  • Multiline — проблема

JSON лог

{
  "log.level": "ERROR",
  "message": "User not found",
  "userId": 123,
  "traceId": "abc-def"
}
  • Структурированные данные
  • Парсинг без regexp
  • Поиск по любому полю

Как лог становится документом

Java пишет в stdout:

{
  "timestamp": "2024-01-15T10:23:45Z",
  "log.level": "ERROR",
  "message": "User not found",
  "logger": "UserService",
  "userId": 123
}

Документ в OpenSearch:

{
  "timestamp": "2024-01-15T10:23:45Z",
  "log.level": "ERROR",
  "message": "User not found",
  "logger": "UserService",
  "userId": 123,
  "kubernetes.namespace_name": "itm-platform-dev",
  "kubernetes.container_name": "user-service",
  "kubernetes.pod_name": "user-service-7d9f-xk2p"
}

Fluentd добавляет kubernetes.* поля автоматически — разработчик ничего не делает

Реальные фильтры проекта

  • kubernetes.namespace_name: itm-platform-dev
    → только dev-окружение
  • kubernetes.container_name: user-service
    → только логи user-service
  • log.level: ERROR
    → только ошибки
  • kubernetes.container_name: user-service AND log.level: ERROR
    → ошибки конкретного сервиса

Практика: ваша очередь

  • Найдите все ERROR-логи за последний час
    log.level: ERROR
  • Найдите только логи user-service
    kubernetes.container_name: user-service
  • Найдите ошибки только user-service
    kubernetes.container_name: user-service AND log.level: ERROR
  • ★ Придумайте и выполните свой запрос

Откройте любой найденный документ и изучите все его поля

Качество логов — ответственность разработчика

Плохой лог

log.error("Something went wrong");
log.error("Error: " + e.getMessage());
log.info("done");
  • Нет контекста
  • Конкатенация строк
  • Нет стектрейса

Хороший лог

log.error("User not found: id={}", id, ex);
log.info("User created: userId={}, role={}",
         user.getId(), user.getRole());
  • Есть контекст и ID
  • Параметры через {}
  • Стектрейс передаётся

Мониторинг шире, чем логи

В индустрии принято говорить об Observability — наблюдаемости системы.
Три столпа observability:


Столп Что даёт Инструменты
Logs Что именно произошло и с каким контекстом OpenSearch, ELK
Metrics Числовые показатели системы во времени Prometheus + Grafana
Traces Путь запроса через несколько сервисов Zipkin, Jaeger, Tempo

OpenSearch — это про Logs. Но в реальном проекте используют все три.

Distributed Tracing — трассировка

Проблема

  • Запрос проходит через
    api-gateway → user-service → notification-service
  • Ответ пришёл за 3 секунды — кто виноват?
  • Логи есть в каждом сервисе, но они не связаны

Решение: TraceId

  • Первый сервис генерирует traceId
  • Каждый следующий сервис передаёт его через HTTP-заголовки
  • Все логи по одному запросу имеют один traceId
  • Одним фильтром в OpenSearch находим весь путь запроса

Как выглядит трейс в Zipkin / Jaeger

Waterfall-диаграмма: каждый цветной блок — один сервис, ширина = время. Сразу видно, кто тормозит.

Tracing в Spring Boot 3

Micrometer Tracing — стандарт для Spring Boot 3

<!-- pom.xml -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-brave</artifactId>
</dependency>
<dependency>
    <groupId>io.zipkin.reporter2</groupId>
    <artifactId>zipkin-reporter-brave</artifactId>
</dependency>

  • Micrometer автоматически добавляет traceId и spanId в MDC
  • Они попадают в каждый лог — без дополнительного кода
  • Трейсы отправляются в Zipkin или Jaeger для визуализации

Метрики: Prometheus + Grafana

Prometheus

  • Собирает числовые метрики (pull-модель)
  • Хранит временны́е ряды
  • Язык запросов: PromQL
  • Умеет отправлять алерты

Grafana

  • Визуализирует данные из Prometheus
  • Готовые дашборды для Spring Boot, JVM, K8s
  • Алерты → Slack / PagerDuty
  • Может подключать OpenSearch как источник

Стандартный стек: Micrometer (в коде) → Prometheus (сбор) → Grafana (UI)

Метрики в Spring Boot: минимальная настройка

<!-- pom.xml -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
# application.yml
management:
  endpoints.web.exposure.include: health,metrics,prometheus
  metrics.tags.application: user-service   # тег для фильтрации в Grafana

GET /actuator/prometheus — сотни метрик, Prometheus забирает их сам

Всё вместе: расследование инцидента

  1. Grafana — latency выросла в 14:32, видим аномалию
  2. Zipkin / Jaeger — находим медленный traceId, видим что тормозит payment-service
  3. OpenSearch — фильтруем по traceId, читаем конкретную ошибку с контекстом

Главные выводы

  • В микросервисах нужна централизованная система логирования
  • OpenSearch — open-source форк Elasticsearch, хранит и индексирует логи
  • Цепочка: stdout → Fluentd → OpenSearch → UI
  • JSON-логи через log4j2 JsonTemplateLayout — структурированные, легко искать
  • KQL-фильтры по namespace, container, log.level, traceId — находят любой лог за секунды
  • Качество логов — ответственность разработчика
