# Группа 12: «Странные вопросы» на знание нюансов
(Редко, но метко)

Эти вопросы — настоящий фильтр, который отделяет тех, кто просто использовал Spring, от тех, кто понимает, как он работает под капотом. Отвечаю максимально точно и с аргументацией.

### 114. Почему `@Autowired` нельзя использовать со статическими полями?

**Ответ:** Потому что **статическое поле принадлежит классу, а не экземпляру**, а Spring управляет **экземплярами** бинов через контекст приложения.

**Технические причины:**

1. **Время инициализации:** Статические поля загружаются вместе с классом на этапе загрузки ClassLoader. Spring же создает бины и внедряет зависимости **после** того, как класс уже загружен.
    
2. **Единственный экземпляр:** Статическое поле одно на все экземпляры класса. Если Spring позволил бы внедрить бин в статическое поле, то при наличии нескольких контекстов (например, в тестах) или перезагрузке контекста возникла бы путаница.
    
3. **Контейнерная природа:** Spring хранит свои бины в контейнере, и каждый бин — это экземпляр. Статические поля нарушают эту модель.
    

**Пример, который НЕ работает:**
```java
@Component
public class MyComponent {
    
    @Autowired
    private static MyService myService;  // НЕ БУДЕТ РАБОТАТЬ! myService останется null
    
    public static void doSomething() {
        myService.call();  // NullPointerException
    }
}
```

**Обходные пути (не рекомендуются):**
```java
@Component
public class MyComponent {
    
    private static MyService myService;
    
    @Autowired
    public void setMyService(MyService service) {
        myService = service;  // работает, но антипаттерн
    }
}
```

**Почему это антипаттерн:** Статические поля усложняют тестирование, создают скрытые зависимости и нарушают принципы DI.

### 115. Что произойдет, если заинжектить бин с `prototype` скоупом в `singleton` без прокси?

**Ответ:** Spring внедрит **один и тот же экземпляр** прототипа во все места использования, и этот экземпляр больше никогда не обновится. Прототип ведет себя как синглтон — это классическая ошибка.

**Пример:**
```java
@Component
@Scope("prototype")
public class PrototypeBean {
    private final UUID id = UUID.randomUUID();
    public UUID getId() { return id; }
}
@Component
public class SingletonBean {
    @Autowired
    private PrototypeBean prototypeBean;  // внедряется один раз
    
    public void printId() {
        System.out.println(prototypeBean.getId());  // всегда один и тот же ID
    }
}
```

**Почему так происходит:**

1. Spring создает `SingletonBean` один раз при старте.
    
2. В момент создания Spring находит бин `PrototypeBean` в контексте.
    
3. Контекст создает **один экземпляр** `PrototypeBean` и внедряет его.
    
4. Больше `PrototypeBean` не создается, потому что синглтон уже собран.
    

**Что получили:** Прототип, который не создается заново при каждом обращении — то есть потеряли смысл прототипа.

**Решения проблемы:**

|Решение|Описание|
|---|---|
|**`@Lookup`**|Абстрактный метод, который Spring реализует, возвращая новый прототип|
|**`ObjectFactory` / `Provider`**|Внедрить фабрику и вызывать `getObject()`|
|**`ApplicationContext.getBean()`**|Прямое обращение к контексту (не рекомендуется)|
|**Scoped Proxy**|`@Scope(value = "prototype", proxyMode = ScopedProxyMode.TARGET_CLASS)`|

**Пример с `@Lookup` (рекомендуемый):**
```java
@Component
public class SingletonBean {
    
    @Lookup
    public PrototypeBean getPrototypeBean() {
        return null;  // Spring реализует этот метод
    }
    
    public void printId() {
        System.out.println(getPrototypeBean().getId());  // каждый раз новый ID
    }
}
```

### 116. Можно ли заменить реализацию бина во время выполнения приложения? (Нет, контекст immutable после refresh)

**Ответ:** **Нет**, после того как `ApplicationContext` был обновлен (`refresh()`), бины в нем считаются **неизменяемыми**. Spring контекст не предназначен для динамической замены бинов в рантайме.

**Почему:**

- `ApplicationContext` после `refresh()` становится **read-only** с точки зрения регистрации/замены бинов.
    
- Все синглтоны уже созданы и закешированы.
    
- AOP-прокси уже созданы и ссылаются на оригинальные бины.
    

**Обходные пути (если очень нужно):**

1. **`@RefreshScope` (Spring Cloud)** — уничтожает бин и пересоздает его при изменении конфигурации:
    
```java
@RefreshScope
@Component
public class DynamicBean { ... }
// При изменении config server'а бин будет пересоздан
```

2. **Прокси с делегированием:**
    
```java
@Component
public class DelegatingService {
    private MyService delegate;
    
    public void setDelegate(MyService newDelegate) {
        this.delegate = newDelegate;
    }
    
    public void doWork() {
        delegate.doWork();
    }
}
```

3. **Использование `ObjectProvider` + reload-механизм:**
```java
@Component
public class ReloadableService {
    @Autowired
    private ObjectProvider<MyService> serviceProvider;
    private volatile MyService current;
    
    public void reload() {
        current = serviceProvider.getObject();  // новый экземпляр
    }
}
```

**Вывод:** По умолчанию — нельзя. Это архитектурное решение Spring для обеспечения консистентности и потокобезопасности.

### 117. Как получить все бины определенного интерфейса? (`List<MyInterface>` или `Map<String, MyInterface>`)

**Ответ:** Spring позволяет внедрить **все реализации** определенного интерфейса через `List`, `Map` или `ObjectProvider`.

**Способы получения всех бинов:**

**Способ 1: Внедрение `List<T>` (все бины, реализующие интерфейс):**
```java
@Service
public class NotificationService {
    
    @Autowired
    private List<NotificationSender> senders;  // все бины, реализующие NotificationSender
    
    public void sendAll(String message) {
        senders.forEach(sender -> sender.send(message));
    }
}
```

**Способ 2: Внедрение `Map<String, T>` (имя бина → бин):**
```java
@Service
public class StrategyRouter {
    
    @Autowired
    private Map<String, PaymentStrategy> strategies;  // ключ = имя бина
    
    public void pay(String strategyName, int amount) {
        PaymentStrategy strategy = strategies.get(strategyName);
        strategy.pay(amount);
    }
}
```

**Способ 3: Через `ObjectProvider` (ленивая загрузка):**
```java
@Service
public class LazyRouter {
    
    @Autowired
    private ObjectProvider<ValidationRule> rules;
    
    public void validate() {
        rules.stream().forEach(rule -> rule.validate());  // лениво
    }
}
```

**Способ 4: Через `ApplicationContext` (программно):**
```java
@Component
public class BeanScanner {
    
    @Autowired
    private ApplicationContext context;
    
    public <T> Map<String, T> getBeansOfType(Class<T> type) {
        return context.getBeansOfType(type);
    }
}
```

**Пример с порядком (сортировка):**
```java
public interface Processor {
    int getOrder();
}
@Service
@Order(1)
public class FirstProcessor implements Processor { ... }
@Service
@Order(2)
public class SecondProcessor implements Processor { ... }
@Service
public class ProcessorChain {
    @Autowired
    private List<Processor> processors;  // будут отсортированы по @Order
}
```

**Важный нюанс:** `List<T>` внедряет **все** бины, соответствующие типу, включая прокси. Если нужны только бины из определенного источника, используйте `@Qualifier` или кастомные аннотации.

### 118. Что такое `@Lookup`? (Для инжекта прототипов в синглтоны)

**Ответ:** `@Lookup` — аннотация для **метод-инжекции**, которая заставляет Spring переопределить абстрактный (или конкретный) метод и возвращать новый экземпляр бина при каждом вызове.

**Основное применение:** Получение **нового экземпляра прототипа** внутри синглтон-бина при каждом обращении.

**Синтаксис:**
```java
@Component
public class SingletonBean {
    
    @Lookup
    public PrototypeBean getPrototype() {
        // Spring реализует этот метод автоматически
        return null;  // заглушка, реальный код не выполняется
    }
    
    public void usePrototype() {
        PrototypeBean bean = getPrototype();  // каждый раз новый экземпляр
        bean.doSomething();
    }
}
```

**Как это работает под капотом:**

1. Spring создает **CGLIB-прокси** для класса `SingletonBean`.
    
2. Прокси переопределяет метод, помеченный `@Lookup`.
    
3. В переопределенном методе Spring вызывает `applicationContext.getBean(PrototypeBean.class)`.
    

**Более сложный пример с параметрами:**
```
@Component
public class CommandProcessor {
    
    @Lookup
    public Command createCommand(String commandName) {
        return null;  // Spring передаст аргументы в конструктор бина
    }
}
@Scope("prototype")
@Component
public class Command {
    private final String name;
    
    public Command(String name) {
        this.name = name;
    }
}
```

**Альтернативы `@Lookup`:**

|Способ|Описание|
|---|---|
|`ObjectFactory<PrototypeBean>`|Внедрить фабрику, вызывать `getObject()`|
|`Provider<PrototypeBean>`|JSR-330 аналог|
|`ApplicationContext.getBean()`|Прямое обращение (хуже для тестирования)|
|Scoped Proxy|Создает прокси, который каждый раз создает новый экземпляр|

**Пример с `ObjectFactory`:**
```java
@Component
public class SingletonBean {
    @Autowired
    private ObjectFactory<PrototypeBean> prototypeFactory;
    
    public void use() {
        PrototypeBean bean = prototypeFactory.getObject();  // новый каждый раз
    }
}
```

**Когда использовать `@Lookup`:** Когда нужно передать **аргументы** в конструктор прототипа. Для случаев без аргументов проще использовать `ObjectFactory`.

### 119. Как работает `@RefreshScope` в Spring Cloud? (Уничтожение и пересоздание бина)

**Ответ:** `@RefreshScope` — аннотация Spring Cloud, которая позволяет **обновлять конфигурацию бина** без перезапуска всего приложения. При изменении внешней конфигурации (например, в Config Server) бины с `@RefreshScope` уничтожаются и пересоздаются заново.

**Как работает под капотом:**

1. **Проксирование:** Spring Cloud создает **CGLIB-прокси** для бина вместо самого бина.
    
2. **Кэширование:** Прокси хранит экземпляр бина в **кэше областей (scope cache)**.
    
3. **Пересоздание:** При получении события `RefreshScopeRefreshedEvent` кэш очищается.
    
4. **Ленивое пересоздание:** При следующем обращении к прокси он создает **новый экземпляр** бина с новыми свойствами.
    

**Пример:**
```java
@RefreshScope
@Component
@ConfigurationProperties(prefix = "app")
public class AppConfig {
    private String featureFlag;
    private int timeout;
    // геттеры и сеттеры
}
@RestController
public class ConfigController {
    @Autowired
    private AppConfig appConfig;
    
    @GetMapping("/config")
    public Map<String, Object> getConfig() {
        return Map.of("featureFlag", appConfig.getFeatureFlag(),
                      "timeout", appConfig.getTimeout());
    }
}
```

**Процесс обновления:**
```bash
# 1. Меняем конфигурацию в Config Server
# 2. Отправляем POST запрос на /actuator/refresh
curl -X POST http://localhost:8080/actuator/refresh
# 3. Spring Cloud очищает кэш @RefreshScope бинов
# 4. При следующем GET /config AppConfig пересоздается с новыми значениями
```

**Что происходит при `@RefreshScope`:**
```text
Обычный бин:         @RefreshScope бин:
┌─────────────┐      ┌─────────────────────────┐
│   MyBean    │      │   CGLIB Proxy           │
└─────────────┘      │   ┌─────────────────┐   │
                     │   │   MyBean (cache) │   │
                     │   └─────────────────┘   │
                     └─────────────────────────┘
```

**Ограничения и важные нюансы:**

|Аспект|Описание|
|---|---|
|**Стоимость пересоздания**|Все зависимости бина тоже должны быть `@RefreshScope` или пересоздаваться|
|**Не работает с `@ConfigurationProperties` без `@RefreshScope`**|Сам класс свойств тоже должен быть в refresh scope|
|**Потокобезопасность**|Старые запросы могут использовать старый бин, новые — новый|
|**Не для всех бинов**|Бины, созданные при старте (например, пулы соединений), могут не поддерживать refresh|

**Альтернативы:**

- **`@ConfigurationProperties` + `@RefreshScope`** — для конфигурации
    
- **`Spring Cloud Bus`** — для рассылки refresh события на все инстансы
    
- **Перезагрузка контекста** — более тяжелый вариант
    

**Важный нюанс:** `@RefreshScope` работает **только** в Spring Cloud контексте. В обычном Spring Boot приложении эта аннотация не имеет эффекта.
