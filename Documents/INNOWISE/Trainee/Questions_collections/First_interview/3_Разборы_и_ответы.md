## 1. How to create a **thread**?
###### **Как создать поток (thread) в Java?**

Существует **два основных способа**:
1. **Наследование от класса Thread**
```Java
class MyThread extends Thread {
    public void run() {
        System.out.println("Поток работает");
    }
}
MyThread t = new MyThread();
t.start();   // важно вызывать start(), а не run()!
```

2. **Реализация интерфейса Runnable** (рекоменд-й, особенно с лямбдами Java8+):
```Java
Runnable task = () -> System.out.println("Поток через Runnable!");
Thread t = new Thread(task);
t.start();
```
С Java 8+ возможно использовать **ExecutorService** (из java.util.concurrent) для управления пулом потоков.

**Важно:** Никогда не вызывай `run()` напрямую — это просто метод. `start()` создаёт новый стек и **запускает поток** в JVM.

## 2. What is the difference between **primitive** and **reference** data types?
###### **В чём разница между примитивными и ссылочными типами данных?**

- **Примитивы** (byte, short, int, long, float, double, char, boolean):
    - Хранят **значение** напрямую.
    - Размещаются в **Stack**.
    - Занимают фиксированный объем памяти.
    - Быстрые, нет overhead на объект.
    - Не могут быть null.
    - По умолчанию имеют default (int → 0, boolean → false).
- **Ссылочные типы** (String, Integer, List, любые классы и интерфейсы):
    - Хранят **ссылку** (адрес) на объект в **Heap**.
    - Могут быть null.
    - Имеют методы (toString(), equals() и т.д.).
    - Поддерживают полиморфизм.

**Пример:**` int a = 5;` vs `Integer b = 5;` — второй использует автобоксинг и м.б.=null.

## 3. What can the final keyword be applied to?
###### **К чему можно применить ключевое слово final?**

- **final переменная** — константа, значение нельзя изменить после инициализации.
- **final параметр метода** — внутри метода нельзя переприсвоить.
- **final метод** — нельзя переопределить в наследниках (полезно для безопасности).
- **final класс** — нельзя наследовать (пример: String, Integer).
> **Бонус:** final + static = compile-time константа (экономит память).

## 4. What is **static** and why is it needed?
###### **Что такое static и зачем оно нужно?**

`static` означает «принадлежит **классу**, а не экземпляру».
- static переменные — **общие** для всех объектов (одна копия в памяти).
- static методы — можно вызывать **без создания объекта** (`ClassName.method()`).
- static блок инициализации — выполняется **один раз при загрузке класса**.
- static вложенные классы.

**Зачем нужно:**
- Счётчики объектов, константы (public static final), утилитные методы (Math.random()), фабричные методы. Экономит память и упрощает код.

#### Общая схема хранения
```text
METASPACE (нативная память)
├── Метаданные класса ClassName
│   ├── Имя класса, родитель, интерфейсы
│   ├── Static переменные (примитивы + ссылки)
│   ├── Static методы (байт-код)
│   ├── Static блок инициализации (байт-код <clinit>)
│   ├── Информация о полях и методах
│   └── Static вложенный класс (его метаданные)
│
└── Метаданные StaticNestedClass
    ├── Свои static переменные
    ├── Свои static методы
    └── ...
```

```text
HEAP (куча)
├── Объекты, на которые ссылаются static переменные
├── Объекты, созданные в static методах
└── Экземпляры static вложенных классов
```

```text
STACK (стек потоков)
├── Локальные переменные при вызове static метода
├── Локальные переменные static блока (при выполнении)
└── Адреса возврата и фреймы вызовов
```

## Важные моменты:
1. **Загрузка класса:** Все static компоненты загружаются в Metaspace **в момент первой загрузки класса** JVM (до создания любых объектов).    
2. **Один экземпляр:** Static методы, переменные, блоки и вложенные классы существуют в единственном экземпляре на весь класс (не зависят от объектов).    
3. **`<clinit>` метод:** Компилятор объединяет все static-блоки и инициализации static-переменных в специальный статический инициализатор, который выполняется один раз.    
4. **Время жизни:** Static компоненты живут пока класс загружен в Metaspace (обычно до завершения программы или пока ClassLoader не будет собран GC).    
5. **Исключение:** Если static переменная имеет тип `String` или другой объект, сама переменная (ссылка) в Metaspace, но пул строк (String Pool) и объекты в Heap.

#### Пример для понимания:
```java
public class Example {
    static int count = 10;              // значение 10 в Metaspace
    
    static List<String> list = new ArrayList<>();  // ссылка в Metaspace, объект в Heap
    
    static {                           // байт-код в Metaspace
        int temp = 5;                  // temp в Stack (при выполнении)
        count = temp;
    }
    
    static void method() {             // байт-код в Metaspace
        int local = 42;                // local в Stack (при вызове)
    }
    
    static class Nested {              // метаданные в Metaspace
        static int nestedCount = 7;    // в Metaspace
    }
}
```

**Итог:** Все static-компоненты (переменные, методы, блоки, вложенные классы) хранятся в **Metaspace** (или PermGen до Java 8) в составе метаданных класса. Исключение — сами объекты (если static ссылается на объект) лежат в **Heap**.

## 5. What is a **functional interface**? Name the popular ones
###### **Что такое функциональный интерфейс? Назови популярные.**

**Функциональный интерфейс** — интерфейс с **ровно одним абстрактным методом** (`@FunctionalInterface`).

Популярные из `java.util.function`:
- **`Predicate<T>`** — boolean test(T t) — проверка условия.  
- **`Function<T, R>`** — R apply(T t) — преобразование.  
- **`Consumer<T>`** — void accept(T t) — потребление значения.  
- **`Supplier<T>`** — T get() — поставщик значения.  
- **`UnaryOperator<T>`** и **`BinaryOperator<T>`**.  
> Также **Runnable**, **Callable**, **Comparator**. Используются с лямбдами и Stream API.

## 6. What is a **terminal function**?
###### **Что такое терминальная операция (terminal operation) в Stream API?**

В **Stream API** терминальные операции — это те, которые **завершают** конвейер и запускают обработку. После нее стрим больше нельзя использовать. Они возвращают результат (коллекцию, число, объект) или вызывают побочный эффект.
	Примеры:
- `collect()`, `forEach()`, `reduce()`, `count()`, `anyMatch()`, `findFirst(),` `min()`, `max()`.
> **Важно:** Пока не вызвана терминальная операция — все промежуточные (filter, map) **ленивые** (lazy) и ничего не делают.iate) — возвращают новый Stream (filter, map, flatMap, sorted и т.д.) и ленивые.

## 7. **Map** vs **flatMap** in Stream API
###### **Разница между map и flatMap в Stream API**
- **map(Function)** — преобразует каждый элемент в **один** новый элемент (1 → 1). 
	  Пример: список строк → список их длин.
- **flatMap(Function)** — преобразует каждый элемент в **Stream** (или коллекцию), а потом **«разглаживает»** все эти стримы в один плоский поток (1 → 0..N). 
	  Пример: `List<List<Integer>> → Stream<Integer>`.
> **Когда использовать**:
>- map — простое преобразование.
>- flatMap — когда после преобразования получается вложенная структура (списки списков, Optional, массивы).
```Java
// map
list.stream().map(s -> s.toUpperCase())

// flatMap — когда есть вложенность
listOfLists.stream().flatMap(List::stream)
```

## 8. OOP principles
###### **Принципы ООП (4 основных + иногда SOLID)**

1. **Инкапсуляция** — сокрытие данных и реализация (private поля + getter/setter).
2. **Наследование** — extends / implements (повторное использование кода).
3. **Полиморфизм** — один интерфейс, много реализаций (переопределение методов, перегрузка, интерфейсы).
4. **Абстракция** — скрытие сложных деталей (абстрактные классы и интерфейсы).

> Дополнительно часто спрашивают **SOLID** принципы.
> 	**SOLID** — пять принципов ООП для гибкого и поддерживаемого кода:
>1. **S**ingle responsibility — один класс отвечает за одну задачу.
>2. **O**pen-closed — класс открыт для расширения, закрыт для изменения.
>3. **L**iskov substitution — наследующий класс не ломает поведение родителя.
>4. **I**nterface segregation — интерфейсы узкие и конкретные (нет ненужных методов).
>5. **D**ependency inversion — зависеть от абстракций, а не от конкретных реализаций.

## 9. Contract equals and hashCode
###### **Контракт между equals() и hashCode()**

Если `a.equals(b) == true`, то **обязательно** `a.hashCode() == b.hashCode()`.
Обратное не обязательно (но желательно для производительности): 
	одинаковый hashCode не гарантирует equals().
> **Почему важно**:
>- HashMap, HashSet используют оба метода.
>- Нарушение контракта → объекты «теряются» в коллекции или дублируются.

Правило: если переопределяешь `equals()` — **обязательно** переопределяй `hashCode()` (и наоборот).

## 10. How does inheritance work in Java?
###### **Как работает наследование в Java?**

- Класс может наследовать **только один** класс (одиночное наследование).
- Можно реализовывать **множество интерфейсов**.
- Используется extends и super.
- Конструктор родителя вызывается автоматически (super() по умолчанию).
- Методы переопределяются с аннотацией `@Override`.
> Java не поддерживает множественное наследование классов, чтобы избежать «проблемы ромба».

## 11. What are generic types and why are they needed?
###### **Что такое дженерики (generics) и зачем они нужны?**

**Дженерики** — параметризованные типы (`List<String>, Map<K, V>`).
**Зачем**:
- **Type safety** — проверка типов на этапе компиляции (не будет ClassCastException в рантайме).
- Убирают необходимость явного приведения типов.
- Позволяют писать обобщённый код (один класс/метод для разных типов).
> Пример: `ArrayList<String>` вместо `ArrayList + приведение`.
- С wildcards: `? extends T`, `? super T`.

## 12. Methods of the Object class. Name them and tell why they are needed
###### **Методы класса Object и их назначение**

- toString() — строковое представление объекта (по умолчанию: Class@hash).
  >  **`public String toString()`**
  > 	Формат: `полное_имя_класса@шестнадцатеричный_хэшкод`
- equals(Object) — сравнение по содержимому.
  > **`public boolean equals(Object obj)`**
  > 	Сравнивает ссылки — объекты равны только если это один и тот же объект.
- hashCode() — для хеш-коллекций (Целое число **int** — хэш-код объекта).
  > **`public native int hashCode()`**
   > 	Нативная — типично возвр. адрес объекта в памяти (не гарантируется).
- getClass() — возвращает Class объекта.
  >  **`public final native Class<?> getClass()`**
  > 	 **Возвращает:** Объект `Class`, представ. класс данного объекта. Нативная.
- clone() — поверхностное копирование (нужен Cloneable).
  >  **`protected native Object clone() throws CloneNotSupportedException`**
  > 	 Нативная, создает побитовую копию (поверхностное копирование)
- notify(), 
  >  **`public final native void notify()`**
  > 	 Пробуждает один поток, ожидающий на мониторе этого объекта. 
  > 	 Должен вызываться из синхронизированного контекста.
- notifyAll(), 
  > **`public final native void notifyAll()`**
  > 	Пробуждает все потоки, ожидающие на мониторе этого объекта. 
  > 	Должен вызываться из синхронизированного контекста.
- wait() — для синхронизации потоков.
  > **`public final native void wait(long timeout) throws InterruptedException`**
  > 	Приостанавливает текущий поток на указанное время (миллисекунды) 
  > 	или до уведомления. Должен вызываться из синхронизированного контекста.
- finalize() — устарел (с Java 9), вызывался перед сборкой мусора.
  > **`protected void finalize() throws Throwable`**
  > 	**DEPRECATED** (устарел с Java 9, удален в Java 18+)

| Метод         | Возвращает | Реализация по умолчанию            | Важно                            |
| ------------- | ---------- | ---------------------------------- | -------------------------------- |
| `getClass()`  | `Class<?>` | Нативная, возвращает класс объекта | `final`                          |
| `hashCode()`  | `int`      | Нативная, часто адрес в памяти     | Переопределять с `equals`        |
| `equals()`    | `boolean`  | `this == obj`                      | Переопределять парами            |
| `toString()`  | `String`   | `класс@хэшкод`                     | Переопределять для вывода        |
| `clone()`     | `Object`   | Нативная, shallow copy             | `protected`, требует `Cloneable` |
| `finalize()`  | `void`     | Пустая                             | **DEPRECATED**                   |
| `notify()`    | `void`     | Нативная                           | `final`, из synchronized         |
| `notifyAll()` | `void`     | Нативная                           | `final`, из synchronized         |
| `wait()`      | `void`     | Нативная (перегрузки)              | `final`, из synchronized         |
Все классы неявно наследуют Object.

## 13. The most popular GCs and general algorithm of work

###### **Самые популярные Garbage Collectors и общий алгоритм работы**

В 2026 году популярны:
- **G1 GC** (Garbage-First) — default в большинстве случаев. Разделяет кучу на регионы, собирает мусор по приоритету (сначала регионы с наибольшим мусором). 
  Баланс между throughput и latency.
- **ZGC** (особенно Generational ZGC с Java 21+) — очень низкие паузы (<1 мс), 
  подходит для больших куч (до десятков ТБ).
- **Shenandoah** — низкие паузы, работает параллельно с приложением.
- **Parallel GC** — максимальный throughput, длинные паузы (для batch-задач).
- **Serial GC** — для маленьких приложений или embedded.

**Общий алгоритм**:
1. **Mark** (метка живых объектов) — начиная с GC Roots (стек, статические поля и т.д.).
2. **Sweep** (очистка) или **Compact** (сжатие) — удаление мёртвых объектов и дефрагментация.
3. Generational GC: Young Generation (Eden + Survivor) — копирующий алгоритм (быстрый), Old Generation — mark-sweep-compact.

## 14. What is **try-with-resources** and why is it needed?

###### **Что такое try-with-resources и зачем оно нужно?**

Конструкция, которая **автоматически закрывает** ресурсы (AutoCloseable / Closeable):
```Java
try (FileInputStream fis = new FileInputStream("file.txt");
     BufferedReader br = new BufferedReader(new InputStreamReader(fis))) {
    // работа с ресурсами

```

**Зачем**:
- Гарантирует закрытие ресурсов даже при исключениях.
- Код чище, чем try-finally.
- С Java 7+.

## 15. Int pool and **String pool**. What is it and why is it necessary?

###### **Integer pool и String pool — что это и зачем нужно?**

- **String Pool** (String Intern Pool) — специальная область в куче (PermGen → Metaspace), где хранятся уникальные строки. Литералы "hello" автоматически попадают в пул. `String.intern()` добавляет строку в пул.
- **Integer Pool** (Integer Cache) — кэш объектов Integer от `-128` до `127` (можно менять через `-XX:AutoBoxCacheMax`). При автоупаковке `Integer a = 100`; используется объект из пула.

**Зачем**:
- Экономия памяти.
- Быстрое сравнение через `==` для строк и маленьких чисел.

## 16. **Stack** and **Heap**, what is stored and where

###### **Стек (Stack) и Куча (Heap) — что где хранится**

- **Stack** (стек потоков):
    - Локальные переменные (примитивы и ссылки).
    - Параметры методов.
    - Вызовы методов (stack frames).
    - Быстрый, автоматическое освобождение при выходе из метода.
    - Каждый поток имеет свой стек.
- **Heap** (куча):
    - Все объекты (new Object()).
    - Массивы.
    - Статические переменные (в Metaspace).
    - Управляется Garbage Collector.

**OutOfMemoryError** чаще всего из Heap, 
**StackOverflowError** — из Stack.

## 17. Popular **thread-safe collections**. Name and tell how they work

###### **Популярные thread-safe коллекции**

- **ConcurrentHashMap** — самая популярная. Сегментирована (или с помощью CAS + synchronized), не блокирует весь мап при записи (только 1 корзину), читать без блокировок. Отличная масштабируемость.
- **CopyOnWriteArrayList** / **CopyOnWriteArraySet** — при каждой записи создаётся новая копия внутреннего массива. Итераторы не требуют синхронизации и не кидают `ConcurrentModificationException`. Идеально, когда чтений >> записей (например, конфигурация).
- **ConcurrentLinkedQueue** / **ConcurrentLinkedDeque** — lock-free очередь на CAS.
- **BlockingQueue** реализации: ArrayBlockingQueue (с lock), LinkedBlockingQueue, SynchronousQueue, DelayQueue.
- Collections.synchronizedXXX() — обёртки (устаревший способ, блокируют весь объект).

## 18. **HashMap** algorithm

###### **Алгоритм работы HashMap**

С **Java 8**:
1. Вычисляется `hash = key.hashCode() ^ (hash >>> 16)` (для лучшего распределения).
2. `Индекс = (n - 1) & hash` (где n — размер таблицы, всегда степень 2).
3. Если ячейка пуста — вставляем узел.
4. Если коллизия:
    - До 8 элементов — одно-связный список.
    - При 8+ элементах в бакете список превращается в **красно-чёрное дерево** (TreeNode) — O(log n) вместо O(n).
5. При уменьшении — дерево может вернуться в список (при 6 элементах).
> `Load Factor = 0.75` — при превышении происходит resize (увеличение в 2 раза).
> Время `O(1`) в среднем, `O(log n)` в худшем.

## 19. Differences between **ArrayList** and **LinkedList**. What and when works faster

###### **Разница `ArrayList` vs `LinkedList`. Что и когда быстрее**

| Операция                | ArrayList (на массиве)     | LinkedList (двусвязный список)     |
| ----------------------- | -------------------------- | ---------------------------------- |
| Доступ по индексу get() | O(1) — очень быстро        | O(n) — медленно                    |
| Добавление в конец      | амортизировано O(1)        | O(1)                               |
| Добавление в середину   | O(n) + возможный resize    | O(1) при наличии итератора         |
| Удаление по индексу     | O(n)                       | O(n) (поиск) + O(1)                |
| Итерация                | Быстрее (локальность кэша) | Медленнее                          |
| Память                  | Эффективнее                | Больше (каждый узел хранит ссылки) |

**Когда использовать**:
- **ArrayList** — почти всегда (поиск, последовательный доступ).
- **LinkedList** — когда очень много вставок/удалений в середину и почти нет случайного доступа.

## 20. How do **TreeMap** and **TreeSet** work?

###### **Как работают TreeMap и TreeSet?**

Оба основаны на **красно-чёрном дереве** (self-balancing binary search tree).
- **`TreeMap<K,V>`** — ключи отсортированы по `natural ordering` или `Comparator`. Операции `get`, `put`, `remove` — `O(log n)`.
- **`TreeSet<E>`** — реализация на базе TreeMap, хранит только ключи (значения — dummy).

Плюсы: автоматическая сортировка, поддержка NavigableMap/Set (lower, ceiling, subMap и т.д.). Минусы: медленнее HashMap/HashSet, не thread-safe.
> Требуют `Comparable` или `Comparator`

###### Свойства красно-черного дерева:
1. Каждый узел - **красный** или **черный**    
2. Корень всегда **черный**    
3. Все листья (null) - **черные**

5. Все пути от корня до листьев содержат одинаковое количество **черных** узлов

## 25. What is **serialization** in Java? What do you need to do to serialize an object? What is **serialVersionUID**?

###### **Что такое сериализация в Java?**

**Сериализация** — преобразование объекта в последовательность байт (для передачи по сети, сохранения в файл и т.д.).

**Чтобы сериализовать**:
- Класс должен реализовывать `Serializable` (маркерный интерфейс).
- Все нестатические и непереходные поля должны быть сериализуемыми (или transient).
- Использовать `ObjectOutputStream` / `ObjectInputStream`.

`serialVersionUID` — уникальный идентификатор версии класса. Если не указать явно — генерируется автоматически. При десериализации JVM сравнивает UID объекта и класса. Если они не совпадают, выбрасывается `InvalidClassException`.  
Рекомендуется объявлять явно, чтобы избежать ошибок при изменении структуры класса: `private static final long serialVersionUID = 1L;`

```java
transient // не сериализуем, исключаем поле (пароль, кэш, соединения)
static    // не сериализуется вообще (принадлежит классу)
```

#### **Способы сериализации** в Java:
#### 1. Стандартная (Serializable)
Встроенный механизм Java. Достаточно реализ-ть интерфейс-маркер `Serializable`. 
JVM сама упаковывает объект в бинарный поток. 
Поля, помеченные `transient`, игнорируются. 
> **Минусы:** медленная, только для Java, большой размер данных.
#### 2. Externalizable
Интерфейс для полного ручного контроля. 
Нужно переопределить методы `writeExternal` и `readExternal`. 
> **Плюс:** можно сериализовать только нужные поля в оптимальном формате. 
> **Минус:** всё писать вручную.
#### 3. JSON (библиотеки: Jackson, Gson, JSON-B)
Текстовый формат, понятный человеку. 
Объект превращается в строку с фигурными скобками и ключами. 
> **Плюсы:** кросс-языковой, легко читать и отлаживать. 
> **Минус:** медленнее бинарных форматов.
#### 4. XML (библиотеки: JAXB, XStream)
Текстовый формат с тегами (как HTML). 
> **Плюс:** строгая структура, поддержка схем (XSD). 
> **Минус:** очень громоздкий, много лишних символов.
#### 5. Protocol Buffers (protobuf)
Бинарный формат от Google. 
Сначала описывается схема данных в `.proto` файле, затем генерируется Java-код. 
> **Плюсы:** очень быстрый, компактный, обратно совместимый. 
> **Минус:** нужен отдельный файл схемы.
#### 6. Kryo
Бинарная библиотека, заточенная под Java. 
> **Плюсы:** быстрее стандартной сериализации в несколько раз, поддерживает циклические ссылки. 
> **Минус:** не кросс-языковая.
#### 7. Avro
Бинарный формат от *Apache*. Схема хранится вместе с данными или отдельно. 
> **Плюсы:** хорош для больших данных (Hadoop, Kafka), поддерживает эволюцию схем. 
> **Минус:** сложнее в настройке.
#### 8. MessagePack
Бинарный аналог JSON. Та же структура (ключи-значения), но в бинарном виде. 
> **Плюс:** компактнее JSON, быстрее, при этом кросс-языковой. 
> **Минус:** нечитаемый без декодера.

## 26. What is the **volatile** keyword for?

и запрещает некоторые оптимизации компилятора/JIT.

## 28. **Thread.sleep** vs **Thread.yield** разница

###### **Разница `Thread.sleep()` и `Thread.yield()`**

- **sleep(long millis)** — поток **гарантированно засыпает** минимум на указанное время (может дольше). Переходит в состояние TIMED_WAITING. Освобождает CPU, т.е.  гарантированно не получает процессорное время. Освобождает монитор? **Нет**.
- **yield()** — **рекомендация** /подсказка планировщику, что поток готов уступить процессорное время другим потокам с таким же приоритетом. Не гарантирует ничего. Поток остаётся в состоянии RUNNABLE, может **сразу** продолжить выполнение.

`sleep` — для пауз, 
`yield` — редко используется для тонкой настройки в высоконагруженных системах.

## 29. How do the **wait**, **notify**, **notifyAll** methods work?

###### **Как работают методы wait(), notify(), notifyAll()?**

Все три метода вызываются **только внутри synchronized блока** на одном и том же мониторе.
- **wait()** — поток освобождает монитор и переходит в состояние WAITING 
	  (или TIMED_WAITING). Ждёт, пока другой поток вызовет `notify`.
- **notify()** — пробуждает **один** случайный поток, ожидающий на этом мониторе.
- **notifyAll()** — пробуждает **все** ожидающие потоки.

После пробуждения поток должен снова захватить монитор (поэтому обычно `wait()` в цикле `while`), защита от *spurious wakeup*.

## 30. How does the **Thread.join()** method work?

###### **Как работает метод Thread.join()?**

`join()` заставляет текущий поток **ждать завершения** другого потока.
```Java
Thread t = new Thread(...);
t.start();
t.join();        // текущий поток ждёт, пока t не завершится
t.join(1000);    // ждать максимум 1 секунду
```

Внутри использует `wait()` на мониторе самого потока. 
Когда поток завершается, JVM вызывает `notifyAll()`.
Можно с таймаутом:` join(5000)`. Используется для координации потоков.

## 31. What is AtomicInteger and how does it work?

###### **Что такое `AtomicInteger` и как он работает?**

**AtomicInteger** - это класс из пакета `java.util.concurrent.atomic`, который обеспечивает **атомарные операции** над значением `int` без использования синхронизации (`synchronized`).

| Метод                                   | Что делает                      | Возвращает      |
| --------------------------------------- | ------------------------------- | --------------- |
| `get()`                                 | Читает текущее значение         | текущее         |
| `set(int newValue)`                     | Устанавливает новое             | void            |
| `incrementAndGet()`                     | ++i                             | новое значение  |
| `getAndIncrement()`                     | i++                             | старое значение |
| `decrementAndGet()`                     | --i                             | новое           |
| `addAndGet(int delta)`                  | i += delta                      | новое           |
| `compareAndSet(int expect, int update)` | если i == expect, то i = update | true/false      |
| `getAndSet(int newValue)`               | i = newValue                    | старое значение |

#### Как работает **CAS** (Compare-And-Swap)
**Принцип:**
1. Читаем текущее значение    
2. Вычисляем новое    
3. Пытаемся заменить **только если** текущее не изменилось    
4. Если изменилось — повторяем (spin-loop)

#### Преимущества и недостатки

| Плюсы                                       | Минусы                                         |
| ------------------------------------------- | ---------------------------------------------- |
| Выше производительность, чем `synchronized` | При высокой конкуренции много "пустых" попыток |
| Нет риска deadlock                          | Подходит только для одиночных переменных       |
| Не блокирует потоки (spin-wait)             | Сложнее понять новичкам                        |

#### Когда использовать
✅ **Подходит:**
- Счётчики (хиты, запросы, просмотры)    
- Флаги состояния    
- Простые аккумуляторы    
- Генерация ID (не уникальных, а последовательных)    

❌ **Не подходит:**
- Сложные обновления (зависит от двух и более переменных) — нужна транзакция или `synchronized`    
- Высоконагруженные счётчики (лучше `LongAdder`)    
- Код, где критична гарантия единовременного выполнения блока

#### Альтернативы для разных сценариев

| Сценарий                                       | Инструмент                         |
| ---------------------------------------------- | ---------------------------------- |
| Простой счётчик, редкие коллизии               | `AtomicInteger`                    |
| Много потоков часто обновляют                  | `LongAdder` (лучше масштабируется) |
| Нужна атомарная ссылка на объект               | `AtomicReference`                  |
| Более сложная логика с несколькими переменными | `synchronized` / `ReentrantLock`   |

#### Какие операции **атомарны** в Java?

|Операция|Атомарна?|Примечание|
|---|---|---|
|Чтение/запись `int` (кроме `long`/`double`)|✅ Да|Всегда|
|Чтение/запись `long`/`double` без `volatile`|❌ Нет (до Java 5)|Могут разбиться на 2×32 бита|
|Чтение/запись `volatile long`/`double`|✅ Да|С Java 5+|
|Чтение/запись ссылок (`Object obj`)|✅ Да|Всегда|
|`x++` (инкремент)|❌ Нет|Это 3 операции|
|`x += 1`|❌ Нет|Это 3 операции|
|`x = y`|❌ Нет|Чтение y + запись x|
|`if (x == 0) { x = 1; }`|❌ Нет|Проверка + запись|

#### Механизмы для атомарности

**1. `synchronized`**
```java
synchronized void increment() { counter++; }
```

**2. `java.util.concurrent.atomic`**
```java
AtomicInteger counter = new AtomicInteger();
counter.incrementAndGet();  // атомарно
```

**3. `volatile` + CAS (Compare-And-Swap)**
- Используется внутри **`Atomic*`** классов

#### Классы Atomic*

|Класс|Для чего|
|---|---|
|`AtomicInteger`|`int`|
|`AtomicLong`|`long`|
|`AtomicBoolean`|`boolean`|
|`AtomicReference<V>`|ссылки на объекты|
|`AtomicIntegerArray`|массивы|
|`LongAdder`|высоконагруженные счётчики|

#### Главное правило
> **Чтение и запись простых типов (кроме long/double) — атомарны.**  
> **Всё остальное — нет, если не защищено.**

#### Коротко
- **Атомарно:** чтение, запись (кроме `long`/`double` без `volatile`)
- **Не атомарно:** инкремент, декремент, составные операции, check-then-act
- **Решение:** `Atomic*` классы или `synchronized`

## 33. What are deadlock and livelock?

###### **Что такое deadlock и livelock?**

- **Deadlock (взаимная блокировка)** — два или больше потоков ждут друг друга и никто не может продолжить работу. 
  Классика: `Thread A` держит`lock1` ждёт`lock2`, `Thread B` держит`lock2` ждёт`lock1`
- **Livelock** — потоки активно работают (меняют состояние), но не могут продвинуться к цели. Пример: два человека в узком коридоре постоянно уступают друг другу дорогу и остаются на месте.

Оба — формы зависания, но в livelock потоки не «спят», а тратят CPU.
**Профилактика**: единый порядок захвата локов, таймауты, tryLock().

## 34. Methods for synchronizing access to shared resources between threads. Name them and name the advantages and disadvantages

###### **Методы синхронизации доступа к общим ресурсам между потоками. Назовите их и укажите преимущества и недостатки**

1. **synchronized** (ключ-е слово) — простой, но грубый (низкая производительность).
2. **ReentrantLock** (java.util.concurrent) — более гибкий (tryLock, fair/unfair, multiple conditions условия).
3. **Atomic переменные** (CAS) — высокая производительность при низкой конкуренции.
4. **Concurrent коллекции** — (ConcurrentHashMap и др.) встроенная безопасность.
5. **volatile** — только видимость (не для сложных операций).
6. **Semaphore**, **CountDownLatch**, **CyclicBarrier**, **Phaser**, **Exchanger**.
7. **ReadWriteLock** — разделение чтение/запись.
8. **Executors + Future/CompletableFuture**.

**Преимущества/недостатки** зависят от сценария: 
> - synchronized — просто, но может снижать производительность; 
> - Lock — гибче, но сложнее в использовании.

## 35. What is the synchronized keyword used for and what are its features?

###### **Для чего используется `synchronized` и какие у него особенности?**

**`synchronized`** - это ключевое слово, используемое для обеспечения **потокобезопасности** (thread-safety) путем организации **мониторной синхронизации** доступа к критическим секциям кода. // ... что блок кода или метод будет выполнен только одним потоком одновременно (монитор).

**Особенности**:
1. Работает на основе монитора объекта (intrinsic lock).    
2. В Java каждый объект имеет монитор.    
3. Может быть применен к:    
    - _Нестатическому методу_ — блокировка на экземпляре `this`.        
    - _Статическому методу_ — блокировка на объекте класса `Class<...>`.        
    - _Блоку кода_ — позволяет указать объект блокировки.        
4. Гарантирует не только атомарность, но и **видимость** изменений (happens-before).

**Особенности**:

- Reentrant (повторно входимый) — один поток может повторно захватить тот же монитор.
- При исключении монитор автоматически освобождается.
- Не поддерживает fairness по умолчанию (может быть starvation).
- С Java 5+ часто лучше использовать ReentrantLock.

#### **ReentrantLock** 
`ReentrantLock` — класс из `java.util.concurrent.locks`, реализующий **ручное управление блокировкой** с возможностью повторного входа (реентерабельностью).

#### Главные особенности

| Характеристика        | Описание                                                    |
| --------------------- | ----------------------------------------------------------- |
| **Ручной контроль**   | Явно вызываем `lock()` и `unlock()`                         |
| **Реентерабельность** | Один поток может захватить блокировку несколько раз         |
| **Счётчик**           | Хранит, сколько раз захвачена. Освобождается столько же раз |
| **Справедливость**    | Можно включить режим "первый пришёл — первый получил"       |

#### Отличие от `synchronized`

| Аспект         | `synchronized`                      | `ReentrantLock`                          |
| -------------- | ----------------------------------- | ---------------------------------------- |
| Управление     | Автоматическое                      | Ручное                                   |
| Освобождение   | Автоматически (при выходе из блока) | Обязательно в `finally`                  |
| Гибкость       | Низкая                              | Высокая (`tryLock`, `lockInterruptibly`) |
| Справедливость | Не гарантирует                      | Можно включить                           |

#### Основные методы
- `lock()` — захватить (ждёт, если занят)    
- `unlock()` — освободить (обязательно в `finally`)    
- `tryLock()` — попытаться захватить, не ждать    
- `tryLock(timeout, unit)` — ждать указанное время    
- `lockInterruptibly()` — можно прервать ожидание    
- `isHeldByCurrentThread()` — проверяет, удерживает ли текущий поток    

#### Правило использования
```java
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    // критическая секция
} finally {
    lock.unlock();  // обязательно!
}
```

#### Когда использовать вместо `synchronized`
- Нужен **таймаут** при ожидании блокировки (`tryLock`)    
- Нужна возможность **прервать** ожидание    
- Нужна **справедливая** очередь    
- Нужно проверить, удерживает ли текущий поток блокировку    

> **`ReentrantLock`** — гибкая альтернатива `synchronized` с ручным управлением, счётчиком повторного входа и расширенными возможностями (таймауты, прерывания, справедливость). Требует обязательного освобождения в `finally`.

## 36 Types of JOIN. Name and tell how they differ.

###### **Какие бывают типы JOIN в SQL? Назови и объясни, чем они отличаются.**

JOIN — это способ объединять данные из нескольких таблиц по общим столбцам. Представь две таблицы: «Пользователи» и «Заказы». JOIN решает, какие строки из них мы хотим увидеть.

- **INNER JOIN** (самый частый) Возвращает **только совпадающие** строки из обеих таблиц. Если у пользователя нет заказов — его не будет в результате. Аналогия: «только те, кто пришёл на свидание с обеих сторон».
```SQL
    SELECT * FROM users INNER JOIN orders ON users.id = orders.user_id;
```
- **LEFT JOIN** (LEFT OUTER JOIN) Берёт **все** строки из левой таблицы + совпадающие из правой. Если совпадений нет — в правых колонках будет NULL. Очень полезен, когда нужно показать всех пользователей, даже без заказов.
```SQL
    SELECT * FROM users LEFT JOIN orders ON users.id = orders.user_id;
```
- **RIGHT JOIN** (RIGHT OUTER JOIN) То же самое, но наоборот: все строки из правой таблицы + совпадающие из левой. На практике используют реже LEFT JOIN (просто меняют порядок таблиц).
```SQL
    SELECT * FROM users RIGHT JOIN orders ON users.id = orders.user_id;
```
- **FULL JOIN** (FULL OUTER JOIN) Все строки из обеих таблиц. Где нет совпадения — NULL в соответствующих колонках. Полезен для сравнения двух таблиц «что есть только слева / только справа / в обеих».
```SQL
    SELECT * FROM users FULL JOIN orders ON users.id = orders.user_id;
```
- **CROSS JOIN** Декартово произведение: каждая строка левой таблицы соединяется с каждой строкой правой. Без условия **ON**. Может взорвать объём данных — используй осторожно!
```SQL
    SELECT * FROM users CROSS JOIN orders;
```

**Ключевое отличие**: разница в том, как обрабатываются строки, не имеющие пар. INNER их отбрасывает, OUTER сохраняет.

> **Совет на собеседовании:** Всегда говори, что:  
> - `INNER` — по умолчанию самый быстрый и частый, а 
> - `OUTER JOIN`ы нужны, когда важно не потерять строки.

## 37. HAVING and WHERE distinctions.

###### **В чём разница между `HAVING` и `WHERE`?**

Это один из самых популярных вопросов!
- **WHERE** — фильтрует **отдельные строки** **до** группировки и агрегации. 
  Можно исп. без GROUP BY. Нельзя исп. агрегатные функции (`SUM`, `COUNT` и т.д.).
- **HAVING** — фильтрует **уже сгруппированные** данные **после** `GROUP BY`. 
  Здесь можно использовать агрегатные функции.

Простая аналогия: `WHERE` — это «отбор людей перед тем, как собрать их в группы». 
`HAVING` — «после того, как группы сформированы, отбираем только те группы, где средний возраст > 30».

	Пример:
```SQL
SELECT department, COUNT(*) as emp_count
FROM employees
WHERE salary > 50000                    -- фильтруем строки до группировки
GROUP BY department
HAVING COUNT(*) > 10 AND AVG(salary) > 70000;  -- фильтруем группы
```

**Правило:** 
`WHERE` — до `GROUP BY`, `HAVING` — после. 
Если нет `GROUP BY`, `HAVING` обычно не имеет смысла.

| Характеристика         | WHERE                                                                                         | HAVING                                                           |
| ---------------------- | --------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| **Этап выполнения**    | Фильтрует строки **до** группировки (GROUP BY) и вычисления агрегатных функций                | Фильтрует группы <br>**после** GROUP BY <br>и агрегации          |
| **Агрегатные функции** | Нельзя использовать (например — ошибка: `WHERE COUNT(*) > 5` )                                | Можно использовать (например, `HAVING COUNT(*) > 5`)             |
| **Псевдонимы**         | Нельзя использовать псевдонимы столбцов из SELECT в некоторых СУБД (из-за порядка выполнения) | Можно использовать агрегаты, но псевдонимы тоже часто недоступны |
**Порядок выполнения (логический)**: 
`FROM` → `WHERE` → `GROUP BY` → `HAVING` → `SELECT` → `ORDER BY`.

## 38. What is a primary key in SQL and why is it important?

###### **Что такое первичный ключ (Primary Key) в SQL и почему он важен?**

**Primary Key (PK)** — это один или несколько столбцов, которые **уникально идентифицируют** каждую строку в таблице.

#### Характеристики:
1. **Уникальность** — не может быть двух строк с одинаковым значением ключа.    
2. **NOT NULL** — значение обязательно должно присутствовать (не мож. б. `NULL`). 
3. **Единственность** — в таблице может быть только один первичный ключ.    

#### Почему это важно:
- **Идентичность** — гарантирует, что к каждой строке мож. обратиться однозначно  
- **Ссылочная целостность** — на него ссылаются внешние ключи (foreign keys) из других таблиц.    
- **Производительность** — по умолчанию создает кластеризованный индекс 
  (в большинстве СУБД), который физически упорядочивает данные на диске, ускоряя поиск.    
- **Семантика данных** — задает бизнес-правило: каждая запись в таблице уникальна.

> `id BIGINT PRIMARY KEY AUTO_INCREMENT` или `email VARCHAR(255) PRIMARY KEY`.

Без PK таблица считается «плохой» с точки зрения реляционной модели.

## 39. What is a foreign key and how is it used to establish relationships between tables?

###### **Что такое внешний ключ (Foreign Key) и как он используется для установления связей между таблицами?**

**Foreign Key (FK)** — это столбец (или набор столбцов) в одной таблице, который ссылается на **Primary Key** (или уникальный ключ) в другой таблице.

**Как устанавливает связи**:
- Определяет отношение "родитель-потомок" (master-detail) и обеспечивает **ссылочную целостность**.    
- Значение в столбце внешнего ключа **должно либо совпадать с существующим значением первичного ключа в родительской таблице**, либо быть `NULL` (если допускается).

**Пример**: Таблица `orders` имеет колонку `user_id` — это `Foreign Key`, который ссылается на `id` в таблице `users`.

**Зачем нужен / Что даёт FK:**
1. **Ссылочная целостность** — предотвращает "висячие" записи (сирот). Нельзя вставить заказ с `user_id = 999`, если пользователя с таким ID не существует.    
2. **Каскадные операции** — при удалении или обновлении родительской записи можно автоматически выполнить действия с дочерними (`ON DELETE CASCADE`, `ON DELETE SET NULL` и т.д.).    
3. **Документирование схемы** — явно показывает связи между сущностями на уровне базы данных, а не только в коде приложения. 
   Помогает понимать отношения: `one-to-one`, `one-to-many`, `many-to-many`

> Без `FK` связь существует только «на словах» — легко нарушить данные.

## 40. How to build a many-to-many connection?

###### **Как построить связь «многие ко многим» (Many-to-Many)?**

Связь "многие-ко-многим" (Many-to-Many) напрямую в реляционных базах данных не реализуется, так как это привело бы к дублированию данных. 
Она строится через **промежуточную (связующую) таблицу**.

**Схема**:  
Есть таблица `Students` (id, name) и таблица `Courses` (id, title).  
Создается третья таблица — `Enrollments` (или `Student_Course`), которая содержит:
- `student_id` — внешний ключ на `Students(id)`    
- `course_id` — внешний ключ на `Courses(id)`    
- Первичный ключ обычно составной из этих двух полей (или суррогатный `id`)    

**Результат**: один студент может быть записан на много курсов (много записей в `Enrollments` с его `student_id`), и один курс может иметь много студентов (много записей с одинаковым `course_id`).

**Бонус:** В промежуточной таблице можно хранить дополнительные атрибуты связи (оценка, дата записи и т.д.).

## 41. What is the difference between the DELETE and TRUNCATE statements?

###### **В чём разница между DELETE и TRUNCATE?**

Оба удаляют данные, но очень по-разному:

- **DELETE**
    - Удаляет **выбранные** строки (можно с WHERE)
    - Можно откатить (в транзакции)
    - Срабатывают триггеры
    - Логируется каждая удалённая строка → медленно на больших таблицах
    - Не сбрасывает автоинкремент
	
- **TRUNCATE**
    - Удаляет **все** строки таблицы сразу
    - Обычно быстрее (деаллокейт страниц, а не построчное удаление)
    - **Нельзя** использовать с WHERE
    - В большинстве СУБД нельзя откатить (DDL-подобная операция)
    - Сбрасывает автоинкремент (идентификаторы начинаются заново)
    - Триггеры обычно не срабатывают

**Когда что использовать:** DELETE — когда нужно удалить часть данных или важна транзакционность. TRUNCATE — когда нужно быстро очистить всю таблицу (тестовые данные, логи и т.д.).

### `DELETE` vs `TRUNCATE` в таблице:

| Характеристика             | DELETE                                                  | TRUNCATE                                                                                         |
| -------------------------- | ------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| **Тип операции**           | DML (Data Manipulation Language)                        | DDL (Data Definition Language)                                                                   |
| **Скорость**               | Медленная (удаляет построчно)                           | Очень быстрая (деаллокация страниц)                                                              |
| **Условие WHERE**          | Можно использовать (удаляет часть строк)                | Нельзя (удаляет ВСЕ строки таблицы)                                                              |
| **Транзакции**             | Можно откатить (ROLLBACK)                               | В разных СУБД по-разному (в PostgreSQL и MySQL с InnoDB — можно откатить; в SQL Server — нельзя) |
| **Триггеры**               | Срабатывают триггеры на удаление                        | Триггеры не срабатывают                                                                          |
| **Счетчик автоинкремента** | Не сбрасывается (следующая вставка продолжит нумерацию) | Сбрасывается (начинает с начального значения)                                                    |
| **Блокировки**             | Блокирует строки <br>(row-level locks)                  | Блокирует таблицу <br>(table-level lock)                                                         |

## 42. What is normalization and why is it needed? Name the number of forms of normalization.

###### **Что такое нормализация и зачем она нужна? Назови формы нормализации.**

**Нормализация** — это процесс проектирования базы данных, при котором мы избавляемся от **избыточности** (дублирования данных) и **аномалий** (проблем при вставке/обновлении/удалении).

**Зачем нужна:**
- Меньше места на диске
- Легче поддерживать *consistency* (изменения в одном месте)
- Избежание аномалий (например, обновил имя в одном месте, а в другом осталось старое)
- Чистая, понятная структура

#### Основные формы нормализации (снизу вверх — каждая следующая строже):
1. **1NF (Первая нормальная форма)** — атомарные значения (нет списков в ячейке), каждая строка уникальна.
2. **2NF** — в 1NF + нет частичных зависимостей (все не ключевые поля зависят от **всего** первичного ключа).
3. **3NF** — в 2NF + нет транзитивных зависимостей (не ключевые поля зависят только от ключа, а не друг от друга).
4. **BCNF (Бойса-Кодда)** — более строгая 3NF (каждый детерминант — кандидат в ключ) , устр. нек. редкие аномалии с перекрывающимися составными ключами.
5. **4NF** — избавление от многозначных зависимостей и зависимостями соединения; на практике используются реже.
6. **5NF (PJ/NF)** — проекционно-соединительная нормальная форма (самая строгая, редко используется на практике).

> - На практике чаще всего доводят до **3NF** или **BCNF** — это хороший баланс.
> - В реальных проектах часто применяют **денормализацию** — намеренное отступление от нормальных форм для ускорения чтения (SELECT) в ущерб избыточности, особенно в аналитических системах (OLAP) и высоконагруженных сервисах.
## 43. What is a transaction and why is it needed?

###### **Что такое транзакция и зачем она нужна?**

**Транзакция** — это логически неделимая последовательность операций с базой данных (одна или несколько SQL-команд), которая переводит базу из одного целостного состояния в другое.

**Зачем нужна**:
1. **Целостность данных** — транзакция гарантирует, что база данных не окажется в противоречивом состоянии (например, деньги списались со счета, но не зачислились на другой).    
2. **Надежность** — в случае сбоя (отключение питания, ошибка в коде) транзакция откатывается, и данные остаются в консистентном состоянии.    
3. **Изоляция** — позволяет множеству пользователей одновременно работать с базой, не мешая друг другу.    

**Границы транзакции**: начинается явно (`BEGIN` / `START TRANSACTION`) или неявно (при первом запросе), завершается `COMMIT` (фиксация) или `ROLLBACK` (откат).

## 44. ACID. Name the principles and levels of transaction isolation.

###### **ACID. Назови принципы и уровни изоляции транзакций.**

**ACID** — четыре свойства, которые должна гарантировать надёжная транзакционная система:

- **A — Atomicity (Атомарность)** — всё или ничего: транзакция неделима. 
  Если любая операция внутри транзакции завершается ошибкой, вся транзакция откатывается. Ничего не происходит "наполовину".
- **C — Consistency (Согласованность)** — транзакция переводит базу данных из одного корректного состояния в другое. Не нарушаются все ограничения (констрейнты): внешние ключи, уникальность, NOT NULL и т.д.
- **I — Isolation (Изоляция)** — параллельно выполняющиеся транзакции не должны влиять друг на друга. Степень изоляции регулируется уровнями.
- **D — Durability (Долговечность)** — после фиксации (COMMIT) изменения сохраняются даже при аварийном отключении/ сбое сервера (write-ahead log).

#### **Уровни изоляции** (от самого слабого к самому строгому):
1. **Read Uncommitted** — можно читать «грязные» (незафиксированные) данные других транзакций. *Самый быстрый, но небезопасный уровень.*
2. **Read Committed** — "неповторяемое чтение". Видит только зафиксированные данные. Но если между двумя чтениями внутри одной транзакции другая транзакция изменила и зафиксировала данные, результаты чтений могут различаться. Это уровень по умолчанию во многих СУБД (<u>PostgreSQL</u>, SQL Server, Oracle).
3. **Repeatable Read** — "фантомное чтение". Гарантирует, что если строка была прочитана, при повторном чтении она не изменится. Однако могут появиться новые строки ("фантомы"), удовлетворяющие условию запроса. Это уровень по умолчанию в <u>MySQL</u> (InnoDB).
4. **Serializable** — самый строгий. Транзакции выполняются так, как будто они идут последовательно одна за другой. Полностью защищает от любых аномалий (грязное чтение, неповторяемое, фантомы), но ценой производительности.

> - В **PostgreSQL** по умолчанию — `Read Committed`,  
> - в **MySQL InnoDB** — `Repeatable Read`.

## 45. Main types of indexes and how they work.

###### **Основные типы индексов и как они работают.**

Индекс — это «оглавление» таблицы, которое ускоряет поиск, сортировку и соединения.

#### **Основные типы**:
- **B-tree (B+ tree)** — самый распространенный. Работает как сбалансированное дерево. Позволяет выполнять поиск по равенству (`=`), диапазонам (`BETWEEN`, `>`, `<`), а также сортировку (`ORDER BY`) за O(log n). B+Tree хранит данные только в листьях, связанных в список, что оптимизирует последовательное чтение.
- **Hash Index** — использует хеш-таблицу. **Идеален только для поиска по точному равенству** (`=`), но не поддерживает диапазонные запросы и сортировку. 
  Часто используется в движке Memory (HEAP) в MySQL.
- **Bitmap Index** — эффективен для столбцов с малым количеством уникальных значений (низкая кардинальность: пол, статус, тип). Использует битовые карты для быстрого выполнения сложных логических условий (AND, OR, NOT). Часто применяется в аналитических базах (Data Warehouses), но реже в OLTP.
  
- **Full-text (GIN в PostgreSQL, FULLTEXT в MySQL)** — предназначен для полнотекстового поиска по текстовым полям. Позволяет искать слова и фразы с учетом морфологии, игнорируя стоп-слова.
- **GiST / SP-GiST (PostgreSQL)** — для работы с геоданными, массивами, полнотекстовым поиском (расширенные возможности).

**Правило:** Индексы ускоряют чтение, но замедляют запись (INSERT/UPDATE/DELETE) — нужно балансировать.

#### Как работает **B-Tree** индекс:
- Корень дерева содержит ссылки на промежуточные узлы.    
- Проходя от корня к листу, индекс сужает диапазон поиска на каждом уровне.    
- В листьях хранятся указатели на фактические строки в таблице (или сами данные, если индекс кластеризованный).

## 46. What is a query execution plan and how to view it.

###### **Что такое план выполнения запроса (Query Execution Plan) и как его посмотреть?**

**Query Execution Plan** — это «рецепт», который оптимизатор базы данных составляет перед выполнением твоего SQL-запроса. Он показывает:

- В каком порядке будут обрабатываться таблицы
- Какие индексы будут использоваться
- Типы соединений (Nested Loop, Hash Join, Merge Join)
- Сколько строк обрабатывается
- Стоимость операций, количество строк и т.д.

Это главный инструмент для понимания, **почему запрос медленный** и где можно оптимизировать.

**Как посмотреть:**
- **PostgreSQL:** `EXPLAIN ANALYZE SELECT ...;` 
	  (ANALYZE — реальное выполнение с таймингами)
- **MySQL:** `EXPLAIN SELECT ...;` или EXPLAIN FORMAT=JSON SELECT ...;
- **SQL Server:** `Include Actual Execution Plan` в SSMS (кнопка или Ctrl+M)
- **Oracle:** `EXPLAIN PLAN FOR ...;` затем `SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);`

В плане ищи:
- **Seq Scan** (полное сканирование таблицы) — плохо на больших таблицах
- **Index Scan** / **Index Seek** — хорошо
- Стрелки с большой толщиной — много данных передаётся между операциями

**Совет:** Всегда смотри Actual Execution Plan (а не Estimated), потому что оценки оптимизатора иногда ошибаются.

### PostgreSQL
```sql
EXPLAIN ANALYZE SELECT * FROM users u LEFT JOIN orders o ON u.id = o.user_id WHERE u.city = 'Минск';
```

### Пример вывода (реальный)
```text
Hash Left Join  (cost=12.34..67 rows=100 width=42) (actual time=0.123..234 rows=50 loops=1)
  Hash Cond: (u.id = o.user_id)
  ->  Seq Scan on users u  (cost=0.00..30 rows=150 width=20) (actual time=0.050..800 rows=200 loops=1)
        Filter: (city = 'Минск'::text)
        Rows Removed by Filter: 800
  ->  Hash  (cost=10.50..50 rows=150 width=22) (actual time=0.070..071 rows=150 loops=1)
        ->  Seq Scan on orders o  (cost=0.00..50 rows=150 width=22) (actual time=0.010..040 rows=150 loops=1)
Planning Time: 0.250 ms
Execution Time: 1.500 ms
```

### Разбор каждой части
#### 1. Заголовок узла (самый важный)
```text
Hash Left Join (cost=12.34..67 rows=100 width=42) (actual time=0.123..234 rows=50 loops=1)
```

|Параметр|Что означает|
|---|---|
|**Hash Left Join**|Тип JOIN (метод соединения)|
|**cost=12.34..67**|Оценка стоимости: `12.34` (начало) → `45.67` (финиш). Чем меньше, тем быстрее|
|**rows=100**|Оценка количества строк (до выполнения)|
|**width=42**|Средняя ширина строки в байтах (42 байта на строку)|
|**actual time=0.123..234**|Реальное время: `0.123` (первая строка) → `1.234` (последняя) в ms|
|**rows=50**|Реальное количество строк (после выполнения)|
|**loops=1**|Сколько раз выполнялся этот узел|

> **Важно:** `rows` (оценка) vs `rows` (actual) — если сильно отличаются, статистика устарела → нужен `ANALYZE`.

#### 2. Условие соединения
```text
Hash Cond: (u.id = o.user_id)
```
Как соединяются таблицы (условие из `ON` или `WHERE`).
#### 3. Дочерние узлы (выполняются первыми)

#### Левый узел (обычно сканирование таблицы)
```text
Seq Scan on users u (cost=0.00..30 rows=150 width=20) (actual time=0.050..800 rows=200 loops=1)
  Filter: (city = 'Минск'::text)
  Rows Removed by Filter: 800
```

|Параметр|Что означает|
|---|---|
|**Seq Scan**|Последовательное сканирование всей таблицы (медленно при больших таблицах)|
|**Filter**|Условие фильтрации (WHERE)|
|**Rows Removed by Filter**|Сколько строк отброшено (800 не из Минска)|

**Другие типы сканирования:**
- `Index Scan` — по индексу (быстро)    
- `Index Only Scan` — только из индекса (ещё быстрее)    
- `Bitmap Index Scan` — битовая карта (для нескольких условий)    

#### Правый узел (для Hash Join)
```text
Hash (cost=10.50..50 rows=150 width=22) (actual time=0.070..071 rows=150 loops=1)
  ->  Seq Scan on orders o (cost=0.00..50 rows=150 width=22) (actual time=0.010..040 rows=150 loops=1)
```

- **Hash** — построение хеш-таблицы из правой таблицы (для быстрого поиска)    
- Внутри — сканирование таблицы `orders`    

#### 4. Глобальные метрики (в конце)
```text
Planning Time: 0.250 ms
Execution Time: 1.500 ms
```

|Параметр|Что означает|
|---|---|
|**Planning Time**|Время на построение плана (разбор запроса, оптимизация)|
|**Execution Time**|Реальное время выполнения запроса (без планирования)|

### Типичные типы JOIN (в плане)

|В плане|Что значит|
|---|---|
|**Hash Join**|Строится хеш-таблица из правой таблицы. Хорош для больших данных|
|**Merge Join**|Обе таблицы отсортированы, сливаются как при слиянии массивов|
|**Nested Loop**|Для каждого из левой таблицы — поиск в правой. Хорош для маленьких таблиц|

#### Что важно отслеживать (красные флаги)

|Проблема|Что видно в плане|
|---|---|
|**Seq Scan на большой таблице**|`Seq Scan` + `rows=1000000` (нужен индекс)|
|**Неверная оценка строк**|`rows=100` (оценка) vs `actual rows=10000` (нужен `ANALYZE`)|
|**Много циклов**|`loops=100` — вложенный цикл выполняется 100 раз|
|**Медленный фильтр**|`Rows Removed by Filter: 90000` — отсекается 90% строк|
|**Большое Planning Time**|> 100ms — сложный запрос или много таблиц|

### Короткий чек-лист анализа плана
1. **Смотреть на тип сканирования** — искать `Index Scan` вместо `Seq Scan`    
2. **Сравнить оценку и actual rows** — если разница > 10x, статистика устарела    
3. **Найти самый дорогой узел** — у которого `cost` максимальный    
4. **Проверить `loops`** — если > 1, возможно, вложенный цикл    
5. **Убедиться, что Execution Time приемлем** — для запроса

## 47. Bean lifecycle. What it looks like, how the bean is created, etc.

###### **Жизненный цикл бина в Spring. Как он выглядит, как создаётся бин и т.д.**

Жизненный цикл бина в Spring IoC контейнере — это последовательность этапов от создания до уничтожения. Spring предоставляет множество точек вмешательства (*хуков*) для настройки поведения бина.

#### **Полный цикл**: (для Spring 6 / Spring Boot 3+ в 2026 году):
1. **Сканирование и чтение метаданных**    
    - Контейнер находит классы, аннотированные `@Component` (или объявленные 
      в XML/Java Config), и читает их метаданные, регистрирует описание бина.
    
2. **Создание экземпляра (Instantiation)**    
    - Через рефлексию (или фабричный метод) вызывается конструктор. 
      Если конструктор не указан явно, используется дефолтный. 
      Если есть несколько конструкторов, Spring выбирает тот, 
      который аннотирован `@Autowired` (*или тот, у которого* 
      *наибольшее количество зависимостей, начиная с `Spring 5`*). 
      Бин существует, но ещё «пустой».
    
3. **Заполнение свойств (*Population*)**    
    - Внедряются зависимости через сеттеры, поля или конструктор 
      (`@Autowired`, `@Inject`, `@Resource`).
    
4. **Установка имени бина (*Aware interfaces*)**    
    - Если бин реализует интерфейсы `BeanNameAware`, `BeanClassLoaderAware`, `BeanFactoryAware` — вызываются соответствующие методы, бин получает информацию о контейнере. 
      Бин «узнаёт» о своём имени, фабрике или контексте.
    
5. **Предварительная инициализация <br>(`BeanPostProcessor.postProcessBeforeInitialization`)**    
    - Здесь работают различные обработчики, например, `@PostConstruct` обрабатывается на этом этапе. Также сюда попадает обработка `@Autowired` из `AutowiredAnnotationBeanPostProcessor` (частично).
      Здесь работают прокси (например, для `@Transactional`).
    
6. **Инициализация (*Initialization*)**    
    - Вызывается метод, аннотированный `@PostConstruct`.        
    - Если бин реализует `InitializingBean`, вызывается `afterPropertiesSet()`.        
    - Если в конфигурации указан (в конфигурации) `init-method`, вызывается он.
    
7. **Постобработка после инициализации <br>`(BeanPostProcessor.postProcessAfterInitialization)`**    
    - На этом этапе создаются **прокси** (если бин аннотирован `@Transactional`, `@Async` и т.д.). Spring оборачивает бин в прокси-объект, который подменяет оригинальный бин в контейнере. 
      Финальные доработки.
    
8. **Использование (*Bean in use*)**    
    - Бин живет в контейнере, участвует в работе приложения, 
      его можно инжектить и вызывать..
    
9. **Уничтожение (*Destruction*)**    
    - При закрытии контекста (например, при остановке приложения) вызывается метод, аннотированный `@PreDestroy`.        
    - Если бин реализует `DisposableBean`, вызывается `destroy()`.        
    - Если указан `destroy-method`, вызывается он.

**Важный момент**: Если бин имеет **прототипную область видимости (*prototype*)**, Spring **не управляет его полным жизненным циклом** — он создает бин, внедряет зависимости, вызывает постпроцессоры, но не хранит его и не вызывает методы уничтожения. Уничтожение *прототипного* бина — ответственность клиентского кода.

На практике чаще всего используют **@PostConstruct** и **@PreDestroy** — 
они простые и современные.

## 48 . Methods for declaring beans. List and name the advantages and disadvantages.

###### **Способы объявления бинов. Перечисли и расскажи плюсы/минусы каждого.**

В *Spring* есть несколько способов сказать контейнеру: 
«Вот этот класс должен стать бином». Вот основные:

1. **Аннотации 
   (`@Component`, `@Service`, `@Repository`, `@Controller`, `@Configuration` + `@Bean`)** Самый популярный и современный способ. 
	   **Плюсы:** минимум кода, легко читать, поддержка component scanning, отличная интеграция с Spring Boot. 
	   **Минусы:** сканирование пакетов может быть чуть медленнее на очень больших проектах, сложнее контролировать порядок создания.
	   
2. **Java-based конфигурация 
   `(@Configuration` + `@Bean`)** Методы в конфигурац-м классе возвращают бины. 
	   **Плюсы:** полный контроль над созданием (*можно вызывать другие `@Bean` внутри*), легко читать логику инициализации, отлично для библиотек. 
	   **Минусы:** больше кода, чем просто `@Component`.
	   
3. **XML-конфигурация** 
   (`applicationContext.xml`) Старый классический способ. 
	   **Плюсы:** полностью внешняя конфигурация (*можно менять без перекомпиляции*), удобно для очень крупных *legacy*-проектов. 
	   **Минусы:** много *boilerplate*, сложно читать, нет поддержки современных фич типа *SpEL* в полной мере, почти не используется в новых проектах.
	   
4. **`@Import`, `@ImportResource`, `FactoryBean`** — вспомогательные способы.

**На собеседовании обычно говорят:** в 99% современных проектов используют аннотации + @Configuration. XML оставили только в старых корпоративных системах.

| Способ                                         | Пример                                                                                                                                             | Преимущества                                                                                                                                                                                           | Недостатки                                                                                                                                                 |
| ---------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **XML-конфигурация**                           | `<bean id="userService" class="com.example.UserService"/>`                                                                                         | — Полное отделение конфигурации от кода  <br>— Подходит для случаев, когда нужно подключать разные конфигурации без перекомпиляции  <br>— Исторически стабильный                                       | — Громоздко, много XML-кода  <br>— Не type-safe (ошибки только в runtime)  <br>— Сложно рефакторить  <br>— Устаревает, редко используется в новых проектах |
| **Java-конфигурация (@Configuration + @Bean)** | `@Configuration<br>public class AppConfig {<br> @Bean<br> public UserService userService() {<br> return new UserService(repository());<br> }<br>}` | — Type-safe, ошибки на этапе компиляции  <br>— Полный контроль над созданием объекта  <br>— Можно использовать любую Java-логику внутри метода  <br>— Удобно для создания бинов из сторонних библиотек | — Класс конфигурации может разрастаться  <br>— Требует явного указания зависимостей (ручной вызов методов)                                                 |
| **Аннотации (@Component + стереотипы)**        | `@Service<br>public class UserService {<br> @Autowired<br> private UserRepository repository;<br>}`                                                | — Минимальный код, высокая читаемость  <br>— Автоматическое обнаружение через component-scan  <br>— Удобно для собственных классов приложения                                                          | — Меньше контроля над процессом создания  <br>— Нельзя использовать для бинов из сторонних библиотек (их код нельзя изменить)                              |
| **@Import + FactoryBean**                      | `@Import(MyFactoryBean.class)`                                                                                                                     | — Позволяет реализовать сложную логику создания бина (например, с условной логикой)                                                                                                                    | — Сложнее в понимании и отладке  <br>— Используется редко, в специфических случаях                                                                         |

**Дополнительно**: с появлением *Spring Boot* основным способом стала комбинация: 
`@Component`-аннотаций для **своих** классов и 
`@Configuration` + `@Bean` для **сторонних** библиотек и тонкой настройки.

## 49. Bean scopes. Name and tell about their life cycle.

###### **Области видимости бинов (Bean Scopes). Назови и расскажи про их жизненный цикл.**

**Scope** определяет, сколько экземпляров бина создаётся и как долго они живут. 
Основные:
- **Singleton** (по умолчанию) Один экземпляр на весь *ApplicationContext*. 
  Создаётся при старте (или лениво с lazy-init) и живёт до закрытия контекста. 
	  **Когда использовать:** сервисы, репозитории, компоненты без состояния.
	  
- **Prototype** Новый экземпляр при каждом запросе (getBean() или инъекции). 
  Жизненный цикл короткий — уничтожается сразу после использования 
  (если нет сильных ссылок). 
	  **Когда использовать:** бины с состоянием (например, корзина покупок), тяжёлые объекты, которые нужно создавать заново.
	  
- **Request** (только в веб-приложениях) Новый бин на каждый HTTP-запрос. 
	  Живёт от начала до конца запроса.
- **Session** Один бин на всю HTTP-сессию пользователя.
- **Application** Один бин на всё ServletContext (похоже на singleton, но для веб).
- **WebSocket** (редко) — на время WebSocket-сессии.

**Важный нюанс:** Для `prototype`-бинов внутри `singleton` нужно использовать **ObjectFactory**, **Provider** или **@Lookup** — иначе `singleton` «захватит» первый созданный prototype.

| Скоуп                            | Описание                                                                         | Жизненный цикл                                                                | Когда использовать                                                                                                          |
| -------------------------------- | -------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **singleton** <br>(по умолчанию) | Один экземпляр на весь Spring IoC контейнер                                      | Создается при старте контейнера (если не lazy), уничтожается при его закрытии | **По умолчанию**. Для stateless-сервисов, репозиториев, утилитных бинов. Безопасен, если не имеет изменяемого состояния     |
| **prototype**                    | Новый экземпляр при каждом запросе (при каждом внедрении или вызове `getBean()`) | Создается при запросе; Spring НЕ управляет его уничтожением                   | Для stateful-компонентов, где каждый клиент требует свой экземпляр. Редко используется, т.к. усложняет управление ресурсами |
| **request** <br>(web)            | Один экземпляр на HTTP-запрос                                                    | Создается при обработке запроса, уничтожается после его завершения            | Для хранения данных, специфичных для одного запроса (например, корзина покупок)                                             |
| **session** <br>(web)            | Один экземпляр на HTTP-сессию                                                    | Создается при создании сессии, уничтожается при ее истечении или завершении   | Для данных пользователя на протяжении всей сессии                                                                           |
| **application** <br>(web)        | Один экземпляр на ServletContext (на всё веб-приложение)                         | Создается при старте веб-приложения, уничтожается при его остановке           | Для глобальных данных уровня приложения (шире, чем singleton в рамках одного контейнера, если контейнеров несколько)        |
| **websocket**<br> (web)          | Один экземпляр на WebSocket-соединение                                           | Привязан к жизненному циклу WebSocket-сессии                                  | Для real-time приложений с WebSocket                                                                                        |

**Важные особенности**:
- **Singleton-бины с prototype-зависимостями**: если в singleton-бин внедрить prototype-бин, prototype-бин создастся **один раз** при создании singleton-бина, а не при каждом вызове. Чтобы получить новый prototype при каждом вызове, нужно использовать `@Lookup` или `ObjectFactory`.    
- **Веб-скоупы доступны только в веб-приложениях** и требуют регистрации `RequestContextListener` или использования `DispatcherServlet`.
```java
@Lookup
    public abstract Notification getNotification();
```

## 50. Tools for creating proxies. Differences between Dynamic Proxy and CGLib.

**Инструменты для создания прокси. Разница между `Dynamic Proxy` и `CGLIB`.**

**Spring AOP** (и транзакционность, и кэширование, и асинхронность) 
реализуются через прокси. Spring использует **два** механизма:

### **JDK Dynamic Proxy** (*стандартный Java*):
- **Как работает**: создает прокси на лету, реализующий **все интерфейсы** целевого класса. Прокси является объектом, который реализует те же интерфейсы, что и целевой объект.    
- **Требования**: целевой класс **обязан реализовывать хотя бы один интерфейс**.    
- **Механизм**: использует `java.lang.reflect.Proxy` и `InvocationHandler`.    
- **Плюсы**: встроен в JDK, не требует дополнительных зависимостей, работает с интерфейсами.    
- **Минусы**: не может проксировать методы, которые не объявлены в интерфейсе. Если класс не реализует интерфейс — не подходит.    

### **CGLib** (*Code Generation Library*):
- **Как работает**: создает прокси через **наследование** — генерирует подкласс целевого класса и переопределяет его методы.    
- **Требования**: целевой класс не должен быть `final` (нельзя наследоваться от `final` класса), и методы не должны быть `final` (нельзя переопределить).    
- **Механизм**: использует байт-код манипуляции (ASM) для генерации подкласса.    
- **Плюсы**: может проксировать классы без интерфейсов.    
- **Минусы**: требует дополнительной зависимости (`cglib` или `spring-core` уже включает его); немного медленнее при создании прокси (хотя разница для большинства приложений незначительна); не может проксировать `final` методы и классы.    

#### **Как Spring выбирает**:
- Если целевой класс реализует хотя бы один интерфейс — по умолчанию используется **JDK Dynamic Proxy**.    
- Если интерфейсов нет — используется **CGLib**.    
- Можно принудительно включить **CGLib** для всех бинов 
  через настройку: `spring.aop.proxy-target-class=true` (в *Spring Boot*) 
  или `<aop:config proxy-target-class="true"/>` (в *XML*).    

**На практике**: в современных приложениях (Spring Boot) часто включают CGLib по умолчанию, так как это упрощает работу — не нужно создавать интерфейсы для каждого сервиса, если в них нет необходимости.

**Как Spring выбирает:** По умолчанию с Spring 5+ (и Boot 2.2+) используется **CGLIB** для всех случаев (*даже если есть интерфейс*). Можно переключить на JDK Dynamic Proxy через proxyTargetClass=false.

> **Совет:** Если видишь ошибку «Cannot subclass final class» — значит используется CGLIB и класс final. Решение — убрать final или явно задать JDK-прокси.

## 51. Spring MVC vs Spring Boot. Differences.

**Spring MVC vs Spring Boot. В чём различия?**

- **Spring MVC** — это **модуль** внутри Spring Framework, который реализует паттерн Model-View-Controller для построения веб-приложений. 
  Он отвеч. за обработку HTTP-запросов, контроллеры, вьюхи (JSP, Thymeleaf и т.д.).
- **Spring Boot** — это **целый фреймворк** поверх Spring, который сильно упрощает разработку. Его главная идея — 
  «*конвенция вместо конфигурации*» + *авто-конфигурация*.

#### **Ключевые различия:**
- **Конфигурация** — Spring MVC требует ручной настройки (DispatcherServlet, ViewResolver, XML/Java Config). Spring Boot делает почти всё автоматически (embedded Tomcat, Jackson, DataSource и т.д.).
- **Встраиваемый сервер** — в MVC нужен внешний сервер (Tomcat). В Boot сервер встроенный — запускаешь приложение одной main-методой.
- **Скорость разработки** — Boot в разы быстрее: стартеры, автоконфигурация, Actuator, DevTools.
- **Размер и управление зависимостями** — Boot использует starters (одна зависимость вместо десятка).
- **Production-ready фичи** — Boot даёт Actuator, metrics, health-checks из коробки.

**Простыми словами:** *Spring MVC* — это «двигатель» для веб-приложений. 
*Spring Boot* — это «автомобиль с автоматической коробкой передач», в котором уже всё настроено и готово к поездке.

Сегодня почти все новые проекты делают на *Spring Boot*, а чистый *Spring MVC* используют только в очень специфических *legacy*-системах.

| Аспект                     | Spring MVC (без Boot)                                                                                                                      | Spring Boot                                                                                   |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------- |
| **Конфигурация**           | Требует ручной настройки: `web.xml` или <br><br>`WebApplicationInitializer`, `@EnableWebMvc`, настройка ViewResolver, диспетчера сервлетов | Автоконфигурация через `spring-boot-starter-web`; все работает "из коробки"                   |
| **Сервер**                 | Требуется внешний <br>сервер приложений (Tomcat, Jetty, etc.)                                                                              | Встраиваемый сервер (Tomcat по умолчанию) — приложение запускается как `main()` метод         |
| **Зависимости**            | Нужно вручную подключать <br>все зависимости с правильными <br>версиями                                                                    | Стартеры управляют версиями (Bill of Materials)                                               |
| **Конфигурационные файлы** | XML или Java Config, часто много boilerplate кода                                                                                          | `application.properties` / <br>`application.yml`; автоконфигурация по умолчанию               |
| **Actuator**               | Нет из коробки                                                                                                                             | `spring-boot-starter-actuator` дает готовые endpoints для мониторинга (health, metrics, etc.) |

## 52. Distinguishes @Component from @Service and @Repository.

**Чем отличается @Component от @Service и @Repository?**

Все три аннотации делают одно и то же технически — помечают класс как бин и включают его в component scanning.

Но семантически они разные (это важно для читаемости и будущих инструментов):
- **@Component** — общая аннотация для любого компонента. Используй, когда не подходит ни Service, ни Repository.
- **@Service** — маркерная для бизнес-логики (сервисный слой). Показывает, что класс содержит правила бизнеса.
- **@Repository** — для слоя доступа к данным (DAO/репозиторий). Spring дополнительно переводит исключения БД (DataAccessException) в свои unchecked-исключения.

**На практике:** Используй правильную аннотацию — код становится самодокументированным. В будущем Spring может добавить разное поведение (например, дополнительные аспекты для @Repository).

Многие разработчики по привычке везде ставят @Component — это работает, но считается не лучшей практикой.

| Аннотация           | Назначение                                                                                 | Дополнительное поведение                                                                                                                                                                                                                                                     |
| ------------------- | ------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **@Component**      | Общая аннотация для любого Spring-бина, который не попадает под более конкретные категории | Базовая. Указывает, что класс должен быть обнаружен component-scan'ом                                                                                                                                                                                                        |
| **@Service**        | Обозначает слой бизнес-логики (сервисы)                                                    | Чисто семантическая. Указывает, что класс содержит бизнес-логику, валидацию, вызовы репозиториев. Может использоваться для специальной обработки (например, в некоторых AOP-сценариях)                                                                                       |
| **@Repository**     | Обозначает слой доступа к данным (DAO)                                                     | **Добавляет важное поведение**: Spring преобразует специфичные для БД исключения (например, `SQLException`) в непроверяемые исключения из иерархии `DataAccessException`. Это унифицирует обработку ошибок независимо от используемой технологии (JPA, JDBC, MongoDB и т.д.) |
| **@Controller**     | Обозначает веб-контроллер в Spring MVC                                                     | Обнаруживается `DispatcherServlet`; может использоваться с `@RequestMapping`. В сочетании с `@ResponseBody` превращается в `@RestController`                                                                                                                                 |
| **@RestController** | Специализация `@Controller` для REST API                                                   | Комбинация `@Controller` + `@ResponseBody` на уровне класса. Все методы автоматически возвращают JSON/XML, а не представления (views)                                                                                                                                        |

**Почему это важно**:
- **Семантика**: код становится самодокументируемым. 
  Глядя на `@Repository`, сразу понятно, что класс работает с базой данных.    
- **AOP-точки входа**: можно настроить аспекты, 
  которые применяются только к сервисам или только к репозиториям.    
- **Обработка исключений**: `@Repository` — единственный, 
  кто добавляет реальное техническое поведение (трансляцию исключений).    

**Технически**: если заменить `@Service` на `@Component`, приложение будет работать идентично. Но делать так не рекомендуется, так как теряется выразительность и потенциальные возможности фреймворка.

## 53. How to perform a native database query when using Spring Data JPA?

**Как выполнить нативный SQL-запрос при использовании Spring Data JPA?**

Spring Data JPA предоставляет несколько способов выполнения нативных SQL-запросов (native queries):

#### **1. Аннотация `@Query` с параметром `nativeQuery = true` в репозитории**
```java
public interface UserRepository extends JpaRepository<User, Long> {
    
    // Нативный запрос с позиционными параметрами
    @Query(value = "SELECT * FROM users WHERE email = ?1", nativeQuery = true)
    User findByEmailNative(String email);
    
    // Нативный запрос с именованными параметрами
    @Query(
	    value = "SELECT * FROM users WHERE age >= :minAge", nativeQuery = true)
    List<User> findByMinAge(@Param("minAge") int minAge);
    
    // Запрос с проекцией (возвращаем не сущность, а интерфейс/массив объектов)
    @Query(
	    value = "SELECT id, name, email FROM users WHERE active = true", 
	    nativeQuery = true)
    List<Object[]> findActiveUsersRaw();
}
```

#### **2. Использование `EntityManager` напрямую**
```java
@PersistenceContext
private EntityManager entityManager;

public List<User> findByCustomQuery(String name) {
    Query query = entityManager.createNativeQuery(
        "SELECT * FROM users WHERE name LIKE ?1", User.class);
    query.setParameter(1, "%" + name + "%");
    return query.getResultList();
}
```

#### **3. Использование `NamedNativeQuery` (через аннотации на сущности)**
```java
@Entity
@NamedNativeQuery(
    name = "User.findByEmailNative",
    query = "SELECT * FROM users WHERE email = :email",
    resultClass = User.class
)

public class User { ... }

// В репозитории
public interface UserRepository extends JpaRepository<User, Long> {
    List<User> findByEmailNative(@Param("email") String email);
}
```

**Важные моменты**:
- Нативные запросы **не переносимы** между СУБД — они привязаны к конкретному диалекту.    
- При использовании нативных запросов с `@Query` и `nativeQuery = true`, Spring Data JPA **не применяет** к результату механизм "first-level cache" так же, как для JPQL-запросов.    
- Для вставки/обновления/удаления нужно добавлять `@Modifying` и часто `@Transactional`:    

```java
@Modifying
@Transactional
@Query(value = "UPDATE users SET status = 'inactive' WHERE last_login < NOW() - INTERVAL '1 year'", 
       nativeQuery = true)
int deactivateInactiveUsers();
```

**JdbcTemplate** или **NamedParameterJdbcTemplate** — когда вообще не нужна JPA-маппинг.

**Важно:** При nativeQuery результат нужно маппить вручную или использовать @SqlResultSetMapping, если возвращаешь не сущность. Также native-запросы могут ломать переносимость между БД.

## 54. How does @Transactional work in Spring? What happens if one service calls a Transactional method from another Transactional method?

**Как работает @Transactional в Spring? Что происходит, если один сервис вызывает транзакционный метод из другого транзакционного метода?**

#### **Механизм работы `@Transactional`**:
1. **Создание прокси**: Spring создает прокси-объект для бина, у которого есть методы, аннотированные `@Transactional` (или для всего класса, если аннотация на уровне класса).    
   
2. **Перехват вызова**: когда вызывается метод, прокси перехватывает вызов.    
   
3. **Логика управления транзакцией**:    
    - Проверяет, активна ли уже транзакция 
	      (через `TransactionSynchronizationManager`).        
    - Определяет поведение распространения (**propagation**).        
    - При необходимости открывает **новую** транзакцию 
	      (через `PlatformTransactionManager`).        
    - Вызывает оригинальный метод целевого бина.        
    - Если метод завершился успешно — коммитит транзакцию.        
    - Если метод выбросил исключение — откатывает транзакцию 
      (по умолчанию откат происходит при `RuntimeException` и `Error`, 
      но не при checked-исключениях, если не указано иначе через `rollbackFor`). 

#### **Пропагации (Propagation)** — определяют, <br>как метод ведет себя относительно существующей транзакции:

- **`REQUIRED`** (по умолчанию) — если транзакция есть, присоединяется к ней; 
	  если нет — создает новую. Присоединяясь к уже открытой транзакции, Одна физическая транзакция. Если упадёт любой — откатится всё.   
- **`REQUIRES_NEW`** — всегда создает новую транзакцию, 
	  приостанавливая существующую. Коммитятся/откатываются независимо   
- **`MANDATORY`** — требует наличия транзакции, 
	  если нет — выбрасывает исключение.    
- **`SUPPORTS`** — если транзакция есть, присоединяется; 
	  если нет — выполняется без транзакции.    
- **`NOT_SUPPORTED`** — выполняется вне транзакции, 
	  существующая приостанавливается.    
- **`NEVER`** — выбрасывает исключение, 
	  если транзакция существует.    
- **`NESTED`** — создает вложенную транзакцию (*savepoint*/ точка сохранения), 
	  если текущая существует *(можно откатить только вложенную часть)*.    

#### **Что происходит при вызове одного transactional-метода из другого** (внутри одного класса):

Это классический "подводный камень" Spring AOP.
```java
@Service
public class UserService {
    
    @Transactional
    public void methodA() {
        // ...
        methodB();  // ВЫЗОВ ВНУТРИ ТОГО ЖЕ КЛАССА!
    }
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() {
        // ...
    }
}
```

**Реальность**: вызов `methodB()` из `methodA()` **не пойдет через прокси**. Он будет вызван напрямую, как обычный Java-метод. Аннотация `@Transactional` на `methodB()` будет **проигнорирована**, потому что транзакционная логика применяется только при вызове через прокси.

**Ключевой момент**: — прокси работает только при вызове через прокси 
(*извне бина*). Если метод вызывает другой метод внутри того же класса — 
прокси обходит, транзакция не применяется (self-invocation problem).

#### **Решения**:
1. **Вынести метод в отдельный бин** (самое чистое решение):
```java
@Service
public class UserService {
    @Autowired
    private TransactionalHelper helper;
    
    @Transactional
    public void methodA() {
        helper.methodB();  // Теперь вызов через прокси
    }
}
```
    
2. **Использовать `@Autowired private Self self` 
	   и вызывать через self** (менее элегантно).    
	   
3. **Использовать `AopContext.currentProxy()`** 
	   (считается анти-паттерном, но работает).    

**Что происходит при вызове из разных классов**:
- Если `ServiceA.methodA()` (транзакционный) вызывает `ServiceB.methodB()` (транзакционный), то каждый вызов проходит через свой прокси.    
- Если оба используют `REQUIRED`, они будут выполняться в **одной транзакции** (если `methodB` не помечен `REQUIRES_NEW`).    
- Spring передает транзакционный контекст через ThreadLocal, поэтому внутри одного потока транзакция "видна" всем бинам, вызванным из нее.

## 56. What is DI and IoC?

**Что такое DI и IoC?**

- **IoC (Inversion of Control — Инверсия управления)** — это принцип, при котором **контроль над созданием и управлением объектами** передаётся от разработчика фреймворку (Spring контейнеру). Вместо того чтобы самому создавать объекты через new, ты говоришь: «Вот мои классы, сам создавай и связывай их».
- **DI (Dependency Injection — Внедрение зависимостей)** — это конкретный способ реализации IoC. Фреймворк сам «внедряет» нужные зависимости в класс (через конструктор, сеттер или поле).

#### **Способы внедрения зависимостей в Spring**:
	
1. **Конструкторная инъекция (рекомендованный способ)**:
```java
@Service
public class UserService {
    private final UserRepository repository;
    
    public UserService(UserRepository repository) {
        this.repository = repository;
    }
}
```
_Плюсы_: поля можно сделать `final`, обеспечивается иммутабельность; 
удобно для тестирования; явно указывает обязательные зависимости.
    
2. **Сеттерная инъекция**:
```java
@Service
public class UserService {
    private UserRepository repository;
    
    @Autowired
    public void setRepository(UserRepository repository) {
        this.repository = repository;
    }
}
```
_Плюсы_: позволяет сделать зависимости опциональными; удобно для конфигураций, которые могут меняться после создания объекта.
    
3. **Полевая инъекция (field injection)**:
```java
@Service
public class UserService {

    @Autowired
    private UserRepository repository;
}
```
 _Минусы_: усложняет тестирование (нужна рефлексия для подмены), нарушает принцип единственной ответственности, скрывает зависимости. Не рекомендуется для production-кода, хотя часто встречается в учебных примерах.

#### **Почему это важно**:
- **Слабая связанность (loose coupling)**: классы зависят от абстракций (интерфейсов), а не от конкретных реализаций.    
- **Тестируемость**: легко подменить реальные зависимости на моки.    
- **Управление жизненным циклом**: контейнер берет на себя создание и уничтожение объектов.    
- **Единая точка конфигурации**: все зависимости настраиваются централизованно.    

**Spring как IoC-контейнер**: Spring управляет бинами (объектами), их зависимостями, временем жизни, скоупами, предоставляет механизмы для внедрения (Autowired), а также добавляет возможности вроде AOP, транзакционности, кэширования, которые работают поверх этой инфраструктуры.

## 57. What is the essence of Docker and what problem does it solve?

**В чём суть Docker и какую проблему он решает?**

**Docker** — это платформа для **контейнеризации** приложений. Он позволяет упаковать приложение вместе со всеми его зависимостями (библиотеки, runtime, конфиги, системные утилиты) в лёгкий, переносимый **контейнер**.

**Основная суть** (в одном предложении): Docker создаёт **стандартизированную среду выполнения**, которая работает одинаково на любом хосте — на ноутбуке разработчика, на тестовом сервере, в продакшене в облаке (AWS, Azure, Kubernetes) или даже на другой ОС.

**Какую проблему решает?** Классическую проблему «**у меня работает на моей машине**» (It works on my machine!).

До Docker типичные боли:
- На машине разработчика Java 17 + определённые версии библиотек, а на сервере — Java 11 и другие зависимости → приложение падает.
- Разные ОС (Windows/Mac/Linux) → разные пути, права доступа, поведение.
- Сложное развёртывание: нужно устанавливать Nginx, PostgreSQL, Redis вручную на каждом сервере.
- Конфликты портов и версий между несколькими приложениями на одном сервере.

**Docker решает это через**:
1. **Единообразие окружения**: контейнер включает в себя всё необходимое для запуска. Образ, протестированный на dev-машине, будет работать идентично на production.    
2. **Изоляция**: контейнеры изолированы друг от друга (файловая система, сеть, процессы), но используют общее ядро ОС, что делает их легче виртуальных машин.    
3. **Повторяемость**: образы версионируются, хранятся в реестрах (Docker Hub, приватные registry), что позволяет легко разворачивать и откатывать версии.    
4. **Стандартизация доставки**: "Build, Ship, Run" — сборка образа, доставка в registry, запуск на любой инфраструктуре.    

**Важное отличие от виртуальных машин**:
- VM: гостевая ОС (гигабайты), гипервизор, полная изоляция, медленный старт.    
- Docker: контейнер использует ядро хостовой ОС, изоляция на уровне процессов (namespaces, cgroups), старт за миллисекунды, размер образа — мегабайты.


## 58. Describe in general terms what should be in the .Dockerfile.

**Опиши в общих чертах, что должно быть в *Dockerfile*.**

**Dockerfile** — это текстовый файл-инструкция (рецепт), по которому Docker собирает образ (image). Каждая строка — это команда, которая создаёт новый **слой** в образе.

**Типичная структура Dockerfile для Java-приложения**:
```dockerfile
# 1. Базовый образ (от чего отталкиваемся)
#    Используем многоступенчатую сборку для уменьшения размера финального образа
FROM maven:3.8.4-openjdk-17-slim AS builder

# 2. Установка рабочей директории внутри контейнера
WORKDIR /app

# 3. Копирование файлов проекта (оптимизация: сначала pom.xml для кэширования слоев)
COPY pom.xml .
COPY src ./src

# 4. Сборка приложения (команда, специфичная для технологии)
RUN mvn clean package -DskipTests

# 5. Второй этап — финальный образ (без инструментов сборки)
FROM openjdk:17-slim

# 6. Копируем собранный JAR из предыдущего этапа
COPY --from=builder /app/target/*.jar /app/app.jar

# 7. Указываем порт, который будет слушать приложение (документирующий характер)
EXPOSE 8080

# 8. Команда запуска (exec-форма — предпочтительнее shell-формы)
ENTRYPOINT ["java", "-jar", "/app/app.jar"]
```

**Основные инструкции Dockerfile**:

| Инструкция   | Назначение                                                                  | Пример                                          |
| ------------ | --------------------------------------------------------------------------- | ----------------------------------------------- |
| `FROM`       | Базовый образ (обязательная первая инструкция)                              | `FROM openjdk:17-slim`                          |
| `WORKDIR`    | Установка рабочей директории <br>(создает, если нет)                        | `WORKDIR /app`                                  |
| `COPY`       | Копирование файлов <br>с хоста в контейнер                                  | `COPY ./target/app.jar /app/`                   |
| `ADD`        | Расширенная версия COPY (поддерживает URL и распаковку архивов)             | `ADD https://example.com/file.tar.gz /tmp/`     |
| `RUN`        | Выполнение команд во время сборки образа                                    | `RUN apt-get update && apt-get install -y curl` |
| `ENV`        | Установка переменных окружения                                              | `ENV JAVA_OPTS="-Xmx512m"`                      |
| `EXPOSE`     | Декларация порта (документация, не открывает порт автоматически)            | `EXPOSE 8080`                                   |
| `CMD`        | Команда по умолчанию (может быть переопределена при запуске)                | `CMD ["java", "-jar", "app.jar"]`               |
| `ENTRYPOINT` | Основная команда (не может быть переопределена полностью, только дополнена) | `ENTRYPOINT ["java", "-jar"]`                   |
| `USER`       | Смена пользователя для запуска (безопасность)                               | `USER appuser`                                  |
| `VOLUME`     | Создание точки монтирования для томов                                       | `VOLUME /data`                                  |
| `ARG`        | Переменные сборки (доступны только во время `docker build`)                 | `ARG VERSION=1.0`                               |

#### **Важные принципы**:
- **Многоступенчатая сборка (multi-stage build)** — позволяет разделить окружение сборки и окружение выполнения, уменьшая размер финального образа.    
- **Порядок инструкций важен** — Docker кэширует слои. Часто меняющиеся инструкции (например, `COPY src`) должны быть в конце, редко меняющиеся (`FROM`, `RUN apt-get`) — в начале.    
- **Минимизация слоев** — объединять команды `RUN` через `&&` и очищать кэш в том же слое: `RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*`.

**Дополнительно хорошие практики:**
- `.dockerignore` файл (исключай `target/`, `node_modules/`, `.git` и т.д.)
- Многоэтапная сборка (multi-stage) — финальный образ в 5–10 раз меньше.
- Кэширование слоёв: часто изменяемые файлы копируй в конец.
- Минимизация слоёв: объединяй RUN команды через &&.
- Безопасность: не используй latest, добавляй sha256-хэш образа, сканируй на уязвимости (Trivy).

**Почему важно:** Правильный Dockerfile ускоряет сборку в CI/CD (GitHub Actions, GitLab CI), уменьшает размер образов и повышает безопасность. На собеседовании часто просят написать простой Dockerfile для Spring Boot приложения.

#### **Минимальный Dockerfile** (рабочий)
```dockerfile
FROM alpine:latest
CMD ["echo", "Hello, World!"]
```

**Что здесь происходит:**

| Инструкция                      | Что делает                                             |
| ------------------------------- | ------------------------------------------------------ |
| `FROM alpine:latest`            | Берёт минимальный образ Linux (Alpine — ~5 МБ)         |
| `CMD ["echo", "Hello, World!"]` | При запуске контейнера выполнит `echo "Hello, World!"` |

**Как проверить:**
```bash
docker build -t my-minimal-image .
docker run my-minimal-image
# Вывод: Hello, World!
```

#### **Минимальный Dockerfile** для реального приложения (Java)
```dockerfile
FROM openjdk:17-slim
COPY target/my-app.jar /app.jar
CMD ["java", "-jar", "/app.jar"]
```

**Пояснение каждой строки:**

| Инструкция                         | Объяснение                                                                                       |
| ---------------------------------- | ------------------------------------------------------------------------------------------------ |
| `FROM openjdk:17-slim`             | Базовый образ с Java 17 (урез. версия, меньше размер)                                            |
| `COPY target/my-app.jar /app.jar`  | Копирует ваш собранный JAR-файл из локальной папки `target/` в корень образа с именем `/app.jar` |
| `CMD ["java", "-jar", "/app.jar"]` | При запуске контейнера выполняет `java -jar /app.jar`                                            |

**Что ещё обычно добавляют** (но не минимально)
```dockerfile
FROM openjdk:17-slim
WORKDIR /app                    # ← задаёт рабочую папку (куда перейти)
COPY target/my-app.jar app.jar  # ← копирует в рабочую папку
EXPOSE 8080                     # ← документирует порт (информация)
RUN apk add --no-cache curl     # ← устанавливает утилиты (диагностика)
ENTRYPOINT ["java", "-jar"]     # ← вместо CMD (разница в гибкости)
CMD ["app.jar"]                 # ← аргумент для ENTRYPOINT
```

**Главное правило**
> **Минимум — это `FROM` + `CMD`.**  
> Всё остальное — для вашего конкретного приложения.

## 59. What is docker-compose and why is it needed?

**Что такое docker-compose и зачем он нужен?**

**Docker Compose** — это инструмент для определения и запуска многоконтейнерных Docker-приложений. С помощью YAML-файла (`docker-compose.yml` или `compose.yaml`) описываются все сервисы приложения (например, база данных, кэш, бэкенд, фронтенд), их конфигурация, сети, тома и зависимости.

Вместо того чтобы вручную запускать 10 команд docker run, ты описываешь всё в одном файле: сервисы, сети, volumes, переменные окружения, зависимости.

#### **Зачем нужен**:
1. **Оркестрация нескольких контейнеров**: вместо запуска 5-10 контейнеров вручную с длинными командами `docker run`, всё описывается в одном файле.
    
2. **Единая конфигурация**: сети создаются автоматически, контейнеры могут обращаться друг к другу по именам сервисов.
    
3. **Воспроизводимость окружения**: разработчик запускает `docker-compose up` и получает полностью настроенное окружение для разработки.
    
4. **Управление жизненным циклом**: одной командой можно запустить, остановить, пересобрать, посмотреть логи всех сервисов.

**Пример `docker-compose.yml` для Spring Boot + PostgreSQL**:
```yaml
version: '3.8'

services:
  postgres:
    image: postgres:15
    container_name: myapp-postgres
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: user
      POSTGRES_PASSWORD: secret
    volumes:
      - postgres_data:/var/lib/postgresql/data
    ports:
      - "5432:5432"
    networks:
      - myapp-network
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user"]
      interval: 10s
      timeout: 5s
      retries: 5

  app:
    build: .
    container_name: myapp-spring
    depends_on:
      postgres:
        condition: service_healthy
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/mydb
      SPRING_DATASOURCE_USERNAME: user
      SPRING_DATASOURCE_PASSWORD: secret
    ports:
      - "8080:8080"
    networks:
      - myapp-network
    restart: unless-stopped

volumes:
  postgres_data:

networks:
  myapp-network:
    driver: bridge
```

**Ключевые возможности**:
- `depends_on` — управление порядком запуска (но не ожидание готовности сервиса, поэтому часто используют `healthcheck`).    
- `volumes` — сохранение данных между перезапусками.    
- `networks` — изоляция и связь между контейнерами.    
- `profiles` — запуск только определенных сервисов для разных сценариев (dev, test, prod).

## 60. What are layers in images and what problem do they solve?

**Что такое слои в образах Docker и какую проблему они решают?**

Docker-образ **не является одним большим файлом**. Это **стек (stack) неизменяемых *read-only* слоёв** (layers, *каждый из которых является результатом выполнения одной инструкции в Dockerfile*), которые накладываются друг на друга с помощью union filesystem (обычно overlay2).
При запуске контейнера поверх этих слоев добавляется тонкий *read-write* слой (container layer).

Каждый слой — это набор изменений файловой системы:

```text
+------------------+
|  Container Layer | (read-write, временные изменения)
+------------------+
|  Layer: COPY app | (read-only)
+------------------+
|  Layer: RUN mvn  | (read-only)
+------------------+
|  Layer: COPY pom | (read-only)
+------------------+
|  Layer: FROM     | (read-only, базовый образ)
+------------------+
```

**Каждый слой immutable** и имеет уникальный идентификатор (digest).

#### **Какую проблему решают слои**:
1. **Кэширование и ускорение сборки**:    
    - Docker кэширует каждый слой. Если слой не изменился (например, `RUN apt-get install` остался тем же), Docker исп. кэш и не пересобирает этот слой.      
    - Это критически важно: если вы меняете только исходный код (последние слои), все предыдущие слои (базовый образ, установка зависимостей) переиспользуются из кэша.
    
2. **Экономия места и сетевого трафика**:    
    - Слои переиспользуются между разными образами. Если у вас есть 10 образов на базе `openjdk:17-slim`, этот базовый слой хранится на диске **1 раз**.  
    - При `docker pull` загружаются только те слои, которых нет локально.
    
3. **Эффективное хранение**:    
    - Docker использует Union File Systems (overlay2, aufs), которые позволяют объединять несколько слоев в единую файловую систему.
      
4.  **Быстрый запуск контейнеров** 
	- Контейнер — это просто «верхний» writable слой поверх read-only слоёв.
	
5. **Версионирование и наследование**
	- Можно наследовать образы и добавлять только свои изменения.

#### **Практические следствия**:
- **Порядок инструкций в Dockerfile имеет значение**. Часто меняющиеся инструкции должны быть внизу, редко меняющиеся — вверху.
```dockerfile
# Хорошо (зависимости кэшируются)
COPY pom.xml .
RUN mvn dependency:go-offline   # Слой с зависимостями
COPY src ./src                   # Слой с кодом (меняется часто)
RUN mvn package

# Плохо (каждое изменение кода инвалидирует все последующие слои)
COPY . .
RUN mvn package
```

- **Многоступенчатая сборка** позволяет финальному образу содержать только последний слой с приложением, не включая слои с инструментами сборки (Maven, JDK для компиляции и т.д.).

> **На практике:** Плохая организация слоёв приводит к долгим сборкам и большим образам (иногда по 1–2 ГБ вместо 100–200 МБ). Всегда ставь часто меняющиеся инструкции в конец Dockerfile.

## 61. What types of networks does docker-compose allow you to install for services?

**Какие типы сетей позволяет устанавливать docker-compose для сервисов?**

Docker Compose позволяет явно задавать сети и их драйверы. Основные типы (network drivers):

1. **Bridge** (самый распространённый, используется по умолчанию)
    - Создаёт изолированную сеть на одном хосте.
    - Контейнеры общаются друг с другом по имени сервиса (например, app → db).
    - Хост может достучаться через *published ports*.
    - NAT + встроенный DNS. Используй для большинства приложений.
2. **Host**
    - Контейнер использует сетевой стек хоста напрямую (без изоляции).
    - Порты контейнера = порты хоста (не нужно -p).
    - Выше производительность, но меньше безопасности. Когда использовать: когда нужна максимальная скорость сети (например, высоконагруженные сервисы).
3. **None**
    - Полностью отключает сеть у контейнера.
    - Нет доступа никуда. Используется редко — для супер-изолированных задач.
4. **Overlay**
    - Для **многохостовых** окружений (Docker Swarm или Kubernetes-подобных).
    - Контейнеры на разных физических серверах могут общаться как в одной сети.
    - Использует VXLAN для туннелирования. В простом docker-compose почти не нужен, но важен при переходе на Swarm.

**Дополнительно:** Можно создавать пользовательские сети:
```yaml
networks:
  my_backend_net:
    driver: bridge
    driver_opts:
      com.docker.network.bridge.name: mynet
```

Или attachable сети для ручного подключения.

**На практике:** В 95% случаев достаточно дефолтной **bridge**-сети. 
Overlay нужен, когда приложение растёт до кластера.

#### Пример настройки нескольких сетей в docker-compose.yml:
```yaml
version: '3.8'

services:
  app:
    build: .
    networks:
      - frontend
      - backend
    ports:
      - "8080:8080"

  postgres:
    image: postgres:15
    networks:
      - backend      # Доступна только в backend-сети
    # Не имеет доступа к frontend сети

  nginx:
    image: nginx:alpine
    networks:
      - frontend
    ports:
      - "80:80"

networks:
  frontend:
    driver: bridge
    ipam:
      config:
        - subnet: 172.20.0.0/16
  backend:
    driver: bridge
    internal: true   # internal = true — запрещает доступ извне (только между сервисами)
```

#### **Важные особенности**:
- По умолчанию Compose создает одну сеть для проекта. Все сервисы в этой сети видят друг друга по имени сервиса (DNS-резолвинг).
    
- `internal: true` делает сеть полностью изолированной от внешнего мира — полезно для баз данных и бэкенд-сервисов, которые не должны быть доступны с хоста напрямую.
    
- Сервис может быть подключен к нескольким сетям одновременно (как `app` в примере выше) — это позволяет реализовать паттерн "сетевая сегментация".

## 62. Case: I launched PostgreSQL in docker, but it is not available on the system via standard port 5432. Where to look and how to diagnose?

**Кейс: Я запустил PostgreSQL в Docker, но он недоступен на системе по стандартному порту 5432. Где смотреть и как диагностировать?**

Это классическая проблема. Диагностика должна идти пошагово.
#### **Шаг 1: Проверить, запущен ли контейнер**
```bash
docker ps
# или
docker ps -a   # включая остановленные
```

**Ожидаем**: увидеть контейнер с PostgreSQL в статусе `Up`. 
Если его нет — контейнер не запустился или упал.

#### **Шаг 2: Проверить маппинг портов**
```bash
docker ps --format "table {{.Names}}\t{{.Ports}}"
# или
docker port <container_name>
```

**На что смотреть**:
- Хорошо: `0.0.0.0:5432->5432/tcp` — порт проброшен на все интерфейсы хоста.    
- Плохо: `5432/tcp` — порт проброшен только внутри сети Docker, не на хост (отсутствует `-p 5432:5432`).    
- Плохо: `127.0.0.1:5432->5432/tcp` — порт проброшен только на localhost (недоступен извне, но с хоста должен быть доступен).    

#### **Шаг 3: Проверить, слушает ли контейнер внутри**
Зайти в контейнер и проверить, что PostgreSQL действительно запущен и слушает порт:
```bash
docker exec -it <container_name> bash
# внутри контейнера:
netstat -tlnp | grep 5432
# или
ss -tlnp | grep 5432
```
Если внутри контейнера порт не слушается — проблема в PostgreSQL (не запустился, ошибка конфигурации).

#### **Шаг 4: Проверить логи контейнера**
```bash
docker logs <container_name>
# или с tail и follow
docker logs --tail 100 -f <container_name>
```
Типичные ошибки в логах:
- `port 5432 already in use` — порт занят другим процессом на хосте.    
- `FATAL: could not create lock file` — проблемы с правами на volume.    
- `database "postgres" does not exist` — проблемы с инициализацией.    

#### **Шаг 5: Проверить, что порт не занят на хосте**
```bash
# Linux/Mac
sudo lsof -i :5432
# или
netstat -tlnp | grep 5432

# Windows (PowerShell)
netstat -ano | findstr :5432
```
Если порт занят — либо другой экземпляр PostgreSQL, либо другой контейнер, либо какой-то процесс. 
Решение: остановить конфликтующий процесс или изменить порт проброса (`-p 5433:5432`).

#### **Шаг 6: Проверить сетевые настройки контейнера**
```bash
docker inspect <container_name> | grep -A 10 "NetworkSettings"
```
Убедиться, что контейнер подключен к сети и имеет IP-адрес. Если контейнер запущен с `--network host`, проброс портов `-p` не работает — порты становятся доступны напрямую на хосте.

#### **Шаг 7: Проверить файрволл (на Linux)**
```bash
sudo iptables -L -n | grep 5432
# или
sudo ufw status
```
Docker автоматически управляет iptables, но если файрволл настроен жестко, доступ может блокироваться.

#### **Шаг 8: Проверить конфигурацию PostgreSQL внутри контейнера**

В файле `postgresql.conf` параметр `listen_addresses` должен быть `'*'` (слушать все интерфейсы), а не `'localhost'`.
```bash
docker exec <container_name> cat /var/lib/postgresql/data/postgresql.conf | grep listen_addresses
```
Если там `localhost`, PostgreSQL не принимает соединения извне контейнера.

#### **Быстрая диагностическая команда (всё в одном)**:
```bash
# Проверка статуса, портов и последних логов для контейнера с именем postgres
CONTAINER_NAME="postgres"
echo "=== STATUS ===" && docker ps -a --filter "name=$CONTAINER_NAME" && \
echo "=== PORTS ===" && docker port $CONTAINER_NAME 2>/dev/null && \
echo "=== LOGS (last 20) ===" && docker logs --tail 20 $CONTAINER_NAME 2>&1 && \
echo "=== HOST PORT CHECK ===" && (lsof -i :5432 2>/dev/null || netstat -tlnp 2>/dev/null | grep 5432 || echo "Port 5432 not in use on host")

```

#### **Самые частые причины проблемы**:

|Причина|Симптом|Решение|
|---|---|---|
|Не указан `-p 5432:5432` или `ports:` в compose|`docker port` не показывает маппинг|Добавить проброс портов|
|Порт 5432 занят на хосте|Логи содержат `bind: address already in use`|Остановить конфликтующий процесс или сменить порт|
|Контейнер не запустился|`docker ps -a` показывает статус `Exited`|Смотреть логи, проверить volume permissions|
|PostgreSQL не слушает на всех интерфейсах|Внутри контейнера netstat показывает `127.0.0.1:5432`|Переопределить `listen_addresses` или использовать кастомный `postgresql.conf`|
|Соединение с localhost, но PostgreSQL в другом сетевом пространстве (Linux)|При запуске использовался `--network host` или нестандартные настройки|Проверить сетевой режим, использовать `127.0.0.1` с проброшенным портом|

#### **Полезная команда для полной диагностики запуска**:
```bash
# Запускаем PostgreSQL с явной диагностикой
docker run -d \
  --name postgres-test \
  -e POSTGRES_PASSWORD=secret \
  -p 5432:5432 \
  -v pg_data:/var/lib/postgresql/data \
  postgres:15
```
