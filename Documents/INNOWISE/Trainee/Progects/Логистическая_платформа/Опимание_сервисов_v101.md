Основные компоненты (микросервисы)

Логистическая платформа обычно включает следующие функциональные блоки:

### архитектура (6–7 сервисов)

| Сервис                   | База данных        | Задачи                                                                     | Взаимодействие                                                  |
| ------------------------ | ------------------ | -------------------------------------------------------------------------- | --------------------------------------------------------------- |
| **API Gateway**          | —                  | Маршрутизация, Keycloak                                                    | HTTP → все сервисы                                              |
| **User Service**         | PostgreSQL + Mongo | Роли (клиент/водитель/админ), профиль (+аватар, документы)                 | HTTP (синхронно)                                                |
| **Delivery Service**     | PostgreSQL +Redis  | **Грузы, доставки, статусы, расчёт стоимости, отмена**                     | HTTP → Transport для проверки ТС; Kafka → `delivery.events`     |
| **Transport Service**    | PostgreSQL         | **ТС, водители, статус AVAILABLE/IN_USE**                                  | HTTP (синхронно для проверки); слушает Kafka для сброса статуса |
| **Notification Service** | PostgreSQL         | Читает Kafka `delivery.events`, отправляет email/sms/telegram              | Kafka (асинхронно)                                              |
| **Paymant Service**      |                    |                                                                            |                                                                 |
| **Reporting Service**    | PostgreSQL         | Читает Kafka `delivery.events` + `user.actions`, формирует отчёты (п.5 ТЗ) | Kafka (асинхронно)                                              |
| **Monitoring Service**   | —                  | Healthchecks (Spring Actuator) + ELK для логов                             | HTTP / Actuator                                                 |


---
## **API Getway**   *(Точка входа)*
### Задачи:
- Маршрутизация запросов от мобильных приложений/веб-клиентов к нужным сервисам;
- Единая точка аутентификации (OAuth2/JWT).
	- для администраторов системы;
	- для пользователей/ клиентов;
	- для водителей;
- Роутинг пользователей;
### Стек: 
- Spring Security;
- KeyCloak;
- Kafka;

---
## **User Service**
### Задачи:
- Вся детальная информация о пользователях системы;
- Хранение аватаров и личных документов пользователей;
- Назначение и хранение ролей пользователей;
### Стек:
- DB: Postgres (информация о пользователе), Mongo (аватар, документы)

------
## **Reporting Service**

### Задачи:
- Собирает и хранит сообщения со всех сервисов (хранение информации);
- Протоколирует заходы и действия пользователей в системе;
- Генерация отчетов, актов выполненных работ, выгрузка данных для 1С.;
### Стек:
- DB: Postgres;
- Kafka;

------
## **Cargo Service**

### Задачи:
- Вся доступная информация о грузе (хранение информации);
- Управление операциями по приемке и отгрузке;
- Управление складом (WMS): Контроль запасов, размещение товаров;
### Особенности:
- Низкая нагрузка (тут скорее - книжный шкаф);
- Тут информация об упаковке груза, таре, партии, методе складирования...;
- Если происходит ЧП с грузом (порча и т.д...) - то уведомляет сервис доставки;
- Все протоколирует в Reporting Service;
### Стек:
- DB: Postgres;
- Kafka;

---
## **Delivery Service**
### Задачи:
- Прием, валидация всей приходящей от заказчика информации;
- Отслеживание статуса заказов;
### Особенности:
- Центральный микросервис системы, управляет почти всеми остальными;
- Высокая нагрузка (собирает почти со всех сервисов);
- Получает уведомления о ЧП с грузом или с ТС, принимает решения о судьбе груза;
- Предоставляет заказчику информацию (счет) за доставку груза на основании финансовой информации с `Paymant Service`;
- Принимает решения о том, когда отправлять груз на основании информации об оплате с `Paymant Service` ;
- Все протоколирует в `Reporting Service`;
### Стек:
- DB: Postgres;
- Kafka;

---
## **Transport Service**
### Задачи:
- реестр водителей;
- назначение водителей на транспортные средства (ТС); 
- реестр ТС;
- графики ТО TC;
- назначение грузов на ТС;
- Мониторинг и телеметрия: Сбор данных с GPS-трекеров для оценки времени прибытия (ETA) в реальном времени;
### Особенности:
- Высокая нагрузка (работа с множеством заказчиков);
- Масштабируемость / Лекговесность;
- Фиксирует усредненное местоположение (т.е. только город, только для заказчиков)
- Если происходит ЧП с транспортом - то уведомляет сервис доставки;
- Все протоколирует в Reporting Service;
### Стек:
- DB: Postgres;
- Kafka;

- Актуализирует из внешних источников инфу для расчета стоимости доставки;
- Контролирует оплату пользователя за доставку;
### Стек:

### Задачи:
- Рассчитывает и формирует стоимость доставки груза;
- Актуализирует из внешних источников инфу для расчета стоимости доставки;

### Стек:

## **Route Service**
Самый алгоритмически сложный сервис.
### Задачи:
- Построение оптимального маршрута для курьера (с учетом пробок, временных окон доставки).;
- Интеграция с внешними картографическими API (Google Maps, 2GIS, OSRM);
- Группировка заказов для одного курьера в один рейс;
### Особенности:
- Низкая нагрузка (нужен только при расчетах маршрутов);
- Высокая сложность (нужно учитывать много факторов);
- Когда маршрут построен, он отдает его в `Delivery Service` доставки, который назначает заказ на конкретное ТС и водителя;
### Стек:
- DB: Redis;
- Интеграция с Yandex - картами;

---
## **Tracking Service**

### Задачи:
- Отслеживает и хранит инфу о местоположении ТС в реальном времени;
### Особенности:
- Высокая нагрузка (постоянный мониторинг каждой единицы ТС);
- Масштабируемость / Лекговесность;
- Фиксирует <u>томное</u> местоположение (только для компании - владельца)
### Стек:
- DB: Postgres;
