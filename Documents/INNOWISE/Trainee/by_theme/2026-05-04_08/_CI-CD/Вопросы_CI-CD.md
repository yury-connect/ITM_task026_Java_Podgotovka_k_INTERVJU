Ниже представлен иерархический перечень, ранжированный от самых «горячих» и базовых тем до узкоспециализированных вопросов для уровня Middle/Middle+.

---
## 1. Фундаментальные основы и концепции (*must-know*)

1. **Различия CI, Continuous Delivery и Continuous Deployment:** определение понятий, в какой момент заканчивается работа разработчика и начинается автоматизация, роль ручного подтверждения (manual approval).
    
2. **Типичный жизненный цикл пайплайна для Spring Boot:** подробное описание всех стадий от коммита (checkout) до продакшена (build, test, quality gate, package, dockerize, push, deploy).
    
3. **Управление конфигурациями и профилями:** как разделять `application.yml` для разных окружений (dev, test, prod) и передавать нужные Spring Profiles через переменные окружения в CI/CD.
    
4. **Управление секретами:** безопасное хранение паролей БД и API-ключей (Vault, Kubernetes Secrets, GitHub/GitLab Secrets) и их отличие от обычных конфигов.

### 2. Сборка, артефакты и Docker

5. **Оптимизация Docker-образов для Java:** использование Multi-stage builds, выбор базовых образов (Alpine, Distroless), запуск от не-root пользователя и использование JRE вместо JDK.
    
6. **Ускорение сборки и кэширование:** работа с кэшем зависимостей Maven/Gradle и использование слоев Docker (Docker Layer Caching) для сокращения времени билда.
    
7. **Управление артефактами и версионирование:** разница между SNAPSHOT и Release, стратегии тегирования (SemVer vs Git Hash), использование Nexus или Artifactory.
    
8. **Неизменяемость артефактов (Immutability):** почему один и тот же бинарный файл (JAR/Image) должен проходить через все среды без пересборки.

## 3. Тестирование и качество кода в пайплайне

9. **Автоматизация тестирования:** иерархия запусков (Unit -> Integration -> E2E), условия «падения» сборки при провале тестов, использование Testcontainers в CI (проблема Docker-in-Docker).
    
10. **Статический анализ и Quality Gates:** интеграция SonarQube, Checkstyle, Snyk или OWASP Dependency Check. Что делать, если сборка не проходит порог качества.
    
11. **Работа с внешними зависимостями:** как пайплайн поднимает временные БД или моки (WireMock) для прохождения интеграционных тестов.

## 4. Деплой и эксплуатация

12. **Стратегии развертывания:** подробное сравнение Blue-Green, Canary и Rolling Update. Механизмы реализации и применимость для микросервисов.
    
13. **Проверка здоровья приложения (Health Checks):** использование Spring Boot Actuator (`/liveness`, `/readiness`) для контроля готовности приложения принимать трафик в Kubernetes/Cloud.
    
14. **Миграции баз данных:** интеграция Liquibase или Flyway в пайплайн. Проблема обратной совместимости схем и выполнение миграций без простоя (Zero Downtime).
    
15. **Стратегии отката (Rollback):** как автоматизировать возврат на предыдущую версию при росте ошибок и как при этом обрабатывать уже примененные миграции БД.
    
16. **Graceful Shutdown:** настройка корректного завершения работы Spring Boot, чтобы не прерывать текущие запросы при обновлении сервиса.

## 5. Инструменты и продвинутые подходы

17. **Сравнение CI-систем:** опыт работы с Jenkins (Declarative vs Scripted Pipeline), GitLab CI (.gitlab-ci.yml) или GitHub Actions. Плюсы и минусы каждой для Java-стека.
    
18. **Infrastructure as Code (IaC):** зачем разработчику понимать Terraform, Helm или Ansible и как они вплетаются в процесс доставки кода.
    
19. **GitOps (ArgoCD / Flux):** понимание разницы между Push и Pull моделями деплоя и как Git становится единственным источником правды для состояния кластера.
    
20. **Монорепозитории vs Микросервисы:** особенности настройки CI/CD, когда в одном репозитории живет несколько независимых Spring Boot сервисов.

## 6. Узкие и редкие вопросы (Strong Middle / Senior)

21. **Оптимизация параллелизма:** распараллеливание джобов, тест-сплиттинг (разделение тестов на группы) для экстремально быстрых пайплайнов.
    
22. **Feature Flags:** как использование флагов в коде влияет на процесс CI/CD и позволяет деплоить «выключенную» фичу.
    
23. **Review Environments:** создание временных динамических окружений для каждой ветки (Pull Request) для ручного тестирования или демонстрации.
    
24. **Jenkins Shared Libraries:** создание переиспользуемого кода для пайплайнов в больших компаниях.
    
25. **Contract Testing:** внедрение Pact-тестов в пайплайн для проверки совместимости микросервисов.

---
**Совет:** Для уровня Middle важно не просто перечислить инструменты, а объяснить **зачем** нужен тот или иной этап. Например, не просто «мы прогоняем тесты», а «мы используем Quality Gates, чтобы код с покрытием ниже 80% не попал в мастер-ветку».
