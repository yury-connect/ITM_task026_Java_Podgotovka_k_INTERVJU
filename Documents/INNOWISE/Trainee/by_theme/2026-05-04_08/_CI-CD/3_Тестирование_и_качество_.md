# 3. **Тестирование** и **качество кода**

---
## 9. Автоматизация тестирования и Testcontainers

- **Fail-fast:** Пайплайн должен прерываться на самом раннем этапе провала теста.
    
- **Testcontainers:** Использование реальных Docker-контейнеров (Postgres, Kafka) в интеграционных тестах. Это снимает проблему "в H2 работает, в Postgres — нет".
    
- **CI Setup:** Для работы Testcontainers в CI нужен доступ к Docker Daemon (Dind или Remote Docker Host).
    
- **E2E:** Запускаются только после успешного деплоя на временное или тестовое окружение.

## 10. Static Analysis и Quality Gates

- **Инструменты:** SonarQube (качество кода), Trivy (уязвимости в образе), Checkstyle (стиль).
    
- **Quality Gate:** Набор условий, без которых сборка считается проваленной.
    
    - _Примеры:_ Покрытие тестами (Coverage) > 80%, отсутствие критических уязвимостей, отсутствие дублирования кода > 5%.

## 11. Работа с внешними зависимостями (Mocks)

- Если приложение зависит от внешнего API, которое недоступно в CI, используется **WireMock** или **MockServer**.
    
- **Pact (Contract Testing):** Современный подход для проверки совместимости микросервисов. Пайплайн проверяет, не сломал ли "поставщик" API ожидания "потребителя" еще до деплоя.

---
**Совет:** На Middle-уровне важно подчеркивать, что CI/CD — это не просто скрипты, а способ **снижения рисков** и **сокращения Time-to-Market**.
