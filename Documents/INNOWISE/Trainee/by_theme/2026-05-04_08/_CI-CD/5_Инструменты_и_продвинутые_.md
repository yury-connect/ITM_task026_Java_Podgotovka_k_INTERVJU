## 17. Сравнение CI-систем для Java-стека

- **Jenkins:**    
    - _Подход:_ **Declarative Pipeline** (рекомендуемый YAML-подобный синтаксис) или **Scripted Pipeline** (Groovy-код для сверхсложной логики).
        
    - _Плюсы:_ Огромная экосистема (1800+ плагинов), полная свобода в настройке, отличная работа с монорепозиториями.
        
    - _Минусы:_ Требует отдельного администрирования (Self-hosted), риск "плагинного ада", сложный UI.
    
- **GitHub Actions:**    
    - _Подход:_ YAML-воркфлоу напрямую в репозитории.
        
    - _Плюсы:_ Идеальная интеграция с GitHub (PR, Issues), огромный Marketplace готовых шагов (Actions), нативная поддержка матричных сборок (тестирование на разных JDK одной джобой).
        
    - _Минусы:_ Зависимость от облака вендора, ограниченная кастомизация self-hosted раннеров.
    
- **GitLab CI:**    
    - _Подход:_ `.gitlab-ci.yml` с мощной системой шаблонов.
        
    - _Плюсы:_ "Всё в одном" (Registry, CI, Security), отличный встроенный механизм кэширования и управления артефактами.

## 18. Infrastructure as Code (IaC) в жизни Java-разработчика

Разработчик Middle+ уровня использует IaC для управления окружением без ручных манипуляций.

- **Terraform:** Создание и конфигурация облачной инфраструктуры (БД, S3 бакеты, K8s кластеры).
    
- **Helm:** Стандарт де-факто для деплоя в Kubernetes. Позволяет описывать Spring Boot приложение как набор шаблонов (Deployment, Service, Ingress), куда в рантайме подставляются нужные переменные.
    
- **Роль в CD:** Инструменты IaC запускаются на этапе деплоя. Это гарантирует, что стейджинг и прод настроены идентично.

## 19. GitOps (ArgoCD / Flux)

- **Push-модель:** Пайплайн (например, Jenkins) сам выполняет команду деплоя в кластер. Минус: у CI должны быть полные права на кластер.
    
- **Pull-модель (GitOps):** Внутри кластера живет агент (ArgoCD), который "тянет" изменения из Git.
    
- #### **Преимущества:**
    
    - **Single Source of Truth:** Git — единственный источник правды.
        
    - **Drift Detection:** Если кто-то вручную изменит настройки в K8s, ArgoCD автоматически вернет их к состоянию, описанному в Git.
        
    - **Безопасность:** Кластер сам забирает изменения, внешним системам не нужны права на доступ к API Kubernetes.

### 20. CI/CD в Монорепозиториях

Когда в одном репозитории несколько Spring Boot сервисов:

- **Selective Builds:** Пайплайн анализирует `git diff` и запускает сборку только тех модулей, где были изменения кода или их зависимостей.
    
- **Shared Libraries:** Общие модули (Common Libs) собираются и тестируются первыми.
    
- **Оптимизация:** Важно настроить изолированное кэширование Maven/Gradle для каждого сервиса, чтобы изменения в одном API не сбрасывали кэш всего проекта.
