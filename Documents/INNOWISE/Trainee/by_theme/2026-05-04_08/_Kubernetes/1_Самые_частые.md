# 1. Самые частые (Core концепции и Health Checks)

Ниже представлен переработанный список вопросов и ответов. Ответы скорректированы для обеспечения технической точности (с учетом специфики JVM и актуальных версий Kubernetes 2025–2026), структурированы и избавлены от избыточной информации.

---
## 1. Что такое Kubernetes и какие проблемы он решает по сравнению с Docker?

**Kubernetes (K8s)** — это платформа для оркестрации контейнеров. В отличие от Docker (который управляет жизненным циклом отдельного контейнера на одной хостовой машине), K8s управляет кластером машин.

#### **Решаемые проблемы:**

- **Self-healing:** Автоматический перезапуск упавших контейнеров и их перенос на здоровые узлы (Node).
    
- **Масштабирование:** Горизонтальное увеличение количества экземпляров приложения (HPA) в зависимости от нагрузки.
    
- **Service Discovery:** Встроенные механизмы связи между микросервисами без жесткой привязки к IP-адресам.
    
- **Zero-downtime deployment:** Обновление приложений без прерывания обслуживания пользователей (Rolling Updates).
    
- **Управление ресурсами:** Оптимальное распределение контейнеров по серверам на основе их потребностей в CPU/RAM.

## 2. Что такое Pod? Почему нельзя запускать контейнер напрямую?

**Pod** — это минимальный объект развертывания в K8s. Это логическая оболочка для одного или нескольких контейнеров.

- **Особенности:** Контейнеры внутри одного Pod всегда запускаются на одном узле, делят общую сеть (один IP, localhost для связи друг с другом) и могут делить общие диски (Volumes).
    
- **Почему не контейнер напрямую:** K8s управляет не процессами, а абстракциями. Pod позволяет добавлять вспомогательные контейнеры (**Sidecars**) к основному приложению (например, для сбора логов или проксирования трафика), не меняя само приложение.

## 3. Что такое Deployment и как он обеспечивает отказоустойчивость?

**Deployment** — это контроллер, описывающий желаемое состояние приложения (образ, количество реплик, конфигурация).

- **Механизм:** Deployment создает под собой объект **ReplicaSet**, который следит, чтобы в кластере всегда было запущено ровно столько Pod, сколько указано.
    
- **Отказоустойчивость:** Если Pod падает или узел, на котором он работал, выходит из строя, Deployment (через ReplicaSet) автоматически создает новый экземпляр.
    
- **Обновления:** Позволяет декларативно менять версию приложения, постепенно заменяя старые Pod на новые.

## 4. В чем разница между Liveness, Readiness и Startup пробами для Spring Boot?

В контексте Spring Boot Actuator эти пробы критичны для стабильности:

|**Проба**|**Назначение**|**Реакция K8s на провал**|
|---|---|---|
|**Startup**|Проверка, завершилась ли инициализация (актуально для Java).|Пока проба не пройдена, остальные пробы отключены. Если не проходит — Pod рестартует.|
|**Readiness**|Готово ли приложение принимать входящий трафик (БД доступна, кэш прогрет).|Pod исключается из балансировки Service (трафик не идет), но не перезапускается.|
|**Liveness**|Не зависло ли приложение (deadlock, внутренний сбой).|Pod принудительно перезапускается.|

> **Настройка:** В Spring Boot 2.3+ достаточно включить `management.endpoint.health.probes.enabled: true`. Используйте пути `/actuator/health/liveness` и `/actuator/health/readiness`.

## 5. ConfigMap vs Secret: как Spring Boot получает настройки?

- **ConfigMap:** Используется для открытых данных (URL баз данных, профили, логи). Хранит данные в открытом виде.
    
- **Secret:** Для конфиденциальной информации (пароли, токены). Данные хранятся в Base64 (в базовой версии) или могут быть зашифрованы на уровне хранилища (Etcd).

#### **Интеграция со Spring Boot:**

1. **Environment Variables:** Значения из ConfigMap/Secret пробрасываются в переменные окружения контейнера. Spring Boot автоматически мапит их на свойства (например, `DB_PASSWORD` -> `spring.datasource.password`).
    
2. **Mounting Files:** Конфиг монтируется как файл (например, `application.yaml`) в файловую систему Pod.

### 6. Типы Service в Kubernetes и механизм Service Discovery

**Service** — это стабильный виртуальный IP-адрес и DNS-имя для группы Pod.

- **ClusterIP (default):** Сервис доступен только внутри кластера.
    
- **NodePort:** Открывает порт на каждом узле кластера (для внешнего доступа без балансировщика).
    
- **LoadBalancer:** Создает облачный балансировщик (AWS ELB, GCP LB).
    
- **Headless Service:** (IP: None) — возвращает IP-адреса всех Pod напрямую. Используется для Stateful-приложений (например, Kafka).

**Discovery:** Микросервис А обращается к микросервису Б просто по DNS-имени: `http://auth-service:8080`. Внутренний DNS-сервер K8s (CoreDNS) разрешит это имя в ClusterIP сервиса.

### 7. Requests vs Limits: что будет с JVM при OOMKilled?

- **Requests:** Ресурсы, которые K8s гарантирует выделить контейнеру. Используются для планирования (Scheduling).
    
- **Limits:** Жесткий потолок потребления.

#### **Важно для Java:**

Если JVM потребляет больше памяти, чем указано в **Memory Limit**, срабатывает механизм **OOM Killer** ОС Linux. Процесс убивается мгновенно, Pod получает статус `OOMKilled` (Exit Code 137). При этом внутри JVM вы не увидите `java.lang.OutOfMemoryError`, так как память закончилась на уровне контейнера, а не кучи (Heap).

> **Рекомендация:** Указывайте `-Xmx` чуть меньше, чем Memory Limit (оставляя 20-30% для non-heap памяти: стеки потоков, метаспейс, Direct Buffer).

### 8. Deployment vs StatefulSet: когда выбирать второй?

- **Deployment:** Для **Stateless** сервисов. Pod-ы взаимозаменяемы, имеют случайные имена (например, `web-5f89c`). При перезапуске не сохраняют состояние диска.
    
- **StatefulSet:** Для приложений, требующих идентичности.
    
    - **Стабильное имя:** `db-0`, `db-1`, которое сохраняется после рестарта.
        
    - **Стабильное хранилище:** При перезапуске Pod `db-0` всегда получит тот же самый диск (Persistent Volume), что был у него до падения.

**Для Spring Boot:** Используйте StatefulSet только если вы реализуете сложную распределенную систему с локальным состоянием (например, In-memory БД или специфический кластерный кэш).

### 9. Как реализовать Graceful Shutdown в связке со Spring Boot?

Чтобы не оборвать запросы пользователей при деплое:

1. **В Spring Boot:** Включите `server.shutdown: graceful`.
    
2. **В Kubernetes:** Настройте `terminationGracePeriodSeconds` (по умолчанию 30с). Это время, которое K8s дает Pod на завершение после отправки сигнала `SIGTERM`.
    
3. **Нюанс:** K8s исключает Pod из списка живых эндпоинтов Service одновременно с отправкой `SIGTERM`. Чтобы избежать ошибок 502, часто добавляют `preStop` hook (команда `sleep`), чтобы приложение поработало лишние несколько секунд, пока обновляются таблицы маршрутизации в кластере.

### 10. В чем разница между Ingress и Service LoadBalancer?

- **Service LoadBalancer:** На каждый сервис создается отдельный внешний IP и физический балансировщик в облаке. Это дорого и сложно в управлении при наличии 50+ микросервисов.
    
- **Ingress:** Это единая точка входа (L7-балансировщик).
    
    - Один внешний IP на весь кластер.
        
    - Позволяет делать маршрутизацию на основе путей (`/api/orders` -> `order-service`) или доменов (`auth.myapp.com` -> `auth-service`).
        
    - Управляет SSL-сертификатами в одном месте.

Для работы Ingress в кластере должен быть запущен **Ingress Controller** (например, NGINX Ingress Controller).
