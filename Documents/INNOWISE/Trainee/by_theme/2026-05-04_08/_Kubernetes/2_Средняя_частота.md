## 11. Что такое Horizontal Pod Autoscaler (HPA)? Метрики для Spring Boot

**HPA** — это контроллер, который автоматически изменяет количество реплик в Deployment на основе текущей нагрузки.

#### **Как это работает:**

K8s опрашивает метрики через Metrics API (обычно раз в 15 секунд). Если текущее значение метрики выше целевого, HPA рассчитывает необходимое количество реплик по формуле:

$\text{DesiredReplicas} = \lceil \text{CurrentReplicas} \times (\text{CurrentMetricValue} / \text{TargetMetricValue}) \rceil$.

#### **Метрики для Spring Boot:**

- **CPU Utilization:** Основная метрика. Java — ресурсоемкая среда, рост CPU часто коррелирует с ростом бизнес-нагрузки.
    
- **Custom Metrics (Prometheus/Micrometer):** Более точный подход. Позволяет масштабироваться по:
    
    - Количеству HTTP-запросов в секунду (RPS).
        
    - Размеру очереди в Kafka/RabbitMQ (если воркер не справляется с обработкой).
        
    - Количеству активных сессий или потоков (Thread Count).

> **Важно:** Масштабирование по **Memory** для Java не всегда эффективно, так как JVM неохотно отдает память ОС, и HPA может не «увидеть» реального снижения нагрузки.

## 12. Хранение данных: PV, PVC и временные файлы

В Kubernetes разделяют временное и постоянное хранилище:

- **emptyDir (Временное):** Создается в момент запуска Pod и удаляется вместе с ним. Идеально для кэша, временных файлов Spring Boot (каталог `tmp`) или обмена данными между Sidecar и основным контейнером.
    
- **PersistentVolume (PV):** Реальный сетевой диск (например, AWS EBS или NFS), выделенный админом кластера.
    
- **PersistentVolumeClaim (PVC):** Запрос разработчика на получение диска (аналог интерфейса в Java). Приложение «говорит»: «Мне нужно 10ГБ памяти», и K8s подбирает подходящий PV.

**Рекомендация для Middle:** Помните, что микросервисы должны быть **Stateless**. Если приложению нужно сохранить загруженный пользователем файл, лучше использовать объектное хранилище (S3), а не локальные диски.

### 13. Init-контейнеры и Sidecar паттерн в Java

- **Init Containers:** Запускаются **до** основного контейнера и должны завершиться успешно.
    - _Пример:_ Запуск миграций БД (Liquibase/Flyway). Это гарантирует, что приложение не начнет работу с устаревшей схемой.
        
    - _Пример:_ Ожидание доступности внешнего API или БД (скрипт проверки коннекта).
    
- **Sidecar Containers:** Работают **параллельно** с основным контейнером.
    - _Пример:_ **Log Shipper** (Fluent Bit), который читает лог-файл Java-приложения и отправляет его в ELK.
        
    - _Пример:_ **Service Mesh прокси** (Istio/Envoy), который берет на себя шифрование трафика (mTLS) и ретраи запросов.

### 14. Namespace: Изоляция и окружения

**Namespace** — это логический раздел внутри кластера.

- **Изоляция окружений:** Вместо создания разных кластеров для `dev`, `stage` и `prod`, их можно разделить нэймспейсами (хотя для `prod` чаще используют отдельный кластер).
    
- **Квоты (Resource Quota):** Можно ограничить потребление ресурсов командой или проектом (например, не более 50 ГБ RAM на весь `dev` нэймспейс).
    
- **DNS:** Внутри одного нэймспейса сервисы доступны по коротким именам (`db-service`). Между нэймспейсами нужно полное имя: `db-service.database-ns.svc.cluster.local`.

### 15. Сеть: Kube-proxy, CoreDNS и CNI

Сетевая модель K8s подразумевает, что любой Pod может достучаться до любого другого Pod по IP без NAT.

- **CNI (Container Network Interface):** Плагин (Calico, Flannel), который «прокладывает кабели» — выдает IP-адреса и настраивает маршруты.
    
- **CoreDNS:** Внутренняя телефонная книга. Когда Java-код обращается к `http://order-service`, CoreDNS возвращает IP-адрес этого сервиса.
    
- **Kube-proxy:** Настройщик на каждой ноде. Он создает правила (iptables/IPVS), благодаря которым трафик, идущий на виртуальный IP Сервиса, перенаправляется на реальные IP-адреса живых Pod-ов.

### 16. Обновление без простоя (Rolling Update)

**Rolling Update** — это стратегия по умолчанию. K8s постепенно убивает старые Pod и создает новые.

- **Как это работает:** Если у вас 4 реплики, K8s поднимет 1 новую, дождется прохождения ее `Readiness Probe` и только потом удалит 1 старую. Так до полной замены.
    
- **Критические параметры:**
    
    - `maxSurge`: Сколько «лишних» Pod можно создать в процессе обновления (например, +25%).
        
    - `maxUnavailable`: Сколько Pod могут быть недоступны в процессе (например, 0).
    
- **Важно для Java:** Так как Spring Boot стартует долго, наличие корректной **Readiness Probe** обязательно. Без неё K8s начнет переключать трафик на еще не загрузившуюся JVM.

### 17. Отказоустойчивость и Self-healing

Kubernetes — это система, стремящаяся к «желаемому состоянию»:

1. **Node Failure:** Если нода «умирает», Control Plane видит это и пересоздает все Pod, которые на ней были, на других свободных нодах.
    
2. **Process Failure:** Если Java-процесс падает с ошибкой, Kubelet на ноде немедленно перезапускает контейнер.
    
3. **Application Freeze:** Если приложение зависло (deadlock) и не отвечает на Liveness пробу, K8s сам пристрелит контейнер и поднимет новый.
    
4. **PDB (Pod Disruption Budget):** Гарантия того, что при плановых работах (например, обновлении самого K8s) всегда будет доступно минимум N реплик вашего критичного сервиса.

Благодаря этому приложение продолжает работать даже при падении отдельных контейнеров, Pod’ов или целых нод.
