## 25. Control Plane и Worker Node: архитектура K8s

Кластер делится на две ключевые зоны:

#### **Control Plane («Мозг» кластера):**

- **kube-apiserver:** Единственная точка входа для всех запросов (от `kubectl` до внутренних компонентов).
    
- **etcd:** Распределенное хранилище «ключ-значение». Здесь хранится всё состояние кластера. Если данных нет в etcd, их нет в кластере.
    
- **kube-scheduler:** Выбирает оптимальную ноду для запуска новых Pod, учитывая ресурсы (requests/limits) и ограничения (affinity).
    
- **kube-controller-manager:** Следит за состоянием объектов (например, если Pod упал, он дает команду на создание нового).    

#### **Worker Node (Рабочие лошадки):**

- **kubelet:** Агент на каждой ноде. Он получает инструкции от API-сервера и следит, чтобы контейнеры в Pod были запущены и здоровы.
    
- **kube-proxy:** Сетевой регулировщик. Настраивает правила трафика, чтобы запросы попадали в нужные Pod-ы через Service.
    
- **Container Runtime:** Среда запуска (чаще всего **containerd**).

## 26. DaemonSet: когда использовать?

**DaemonSet** гарантирует, что копия конкретного Pod будет запущена на **каждой** ноде кластера.

- **Применение в Java-инфраструктуре:**
    
    - Сбор логов (например, Fluent Bit), который должен забирать логи со всех приложений на ноде.
        
    - Мониторинг ресурсов ноды (Node Exporter).
        
    - Сетевые плагины (CNI).
    
- **Отличие от Deployment:** Если вы добавите в кластер 5 новых нод, DaemonSet автоматически поднимет на них свои Pod-ы, в то время как Deployment сохранит фиксированное число реплик.

## 27. Job и CronJob: разовые задачи в Java

В отличие от Deployment, эти объекты предназначены для задач, которые должны **завершиться**, а не работать вечно.

- **Job:** Однократный запуск. _Пример:_ Тяжелая миграция данных в БД при релизе новой версии.
    
- **CronJob:** Запуск по расписанию (аналог linux cron). _Пример:_ Очистка временных папок в хранилище каждое воскресенье в 3:00 или генерация ночного отчета.

## 28. Kubernetes Operator: зачем он Java-разработчику?

**Operator** — это продвинутый контроллер, который расширяет возможности K8s, добавляя в него знания о конкретном приложении.

- **Когда нужен:** Для сложных Stateful-систем (PostgreSQL, Kafka, Redis). Оператор умеет делать бэкапы, обновлять версию БД без потери данных и восстанавливать кластер после сбоев.
    
- **Для Spring Boot:** Обычно избыточен. Но если вы создаете облачную платформу (PaaS) внутри компании, оператор на Java (с помощью **Java Operator SDK**) может автоматизировать создание всей инфраструктуры под каждый новый микросервис.

## 29. kubectl apply vs kubectl create

- **kubectl create:** Императивный подход. «Создай объект сейчас». Если объект уже есть, команда выдаст ошибку.
    
- **kubectl apply:** Декларативный подход (рекомендуемый). «Сделай так, чтобы состояние в кластере соответствовало этому файлу». Если объекта нет — создаст, если есть — обновит только измененные поля. Именно `apply` используется в CI/CD пайплайнах.

## 30. Логирование Spring Boot: EFK/ELK стек

В Kubernetes меняется подход к логам: приложение **не должно** писать в файлы.

1. **Stdout/Stderr:** Spring Boot пишет логи в консоль.
    
2. **Collector (Fluent Bit):** Работает как DaemonSet на каждой ноде, подхватывает консольные выводы всех контейнеров.
    
3. **Storage (Elasticsearch/Loki):** Коллектор пересылает логи в централизованное хранилище.
    
4. **UI (Kibana/Grafana):** Вы ищете логи через интерфейс.

> **Совет для Middle:** Упомяните, что для удобного поиска в ELK лучше перевести логи Spring Boot в формат **JSON**. Это позволит фильтровать логи по конкретным полям (например, `level`, `traceId`, `serviceName`) без сложных регулярных выражений.
