## 7. Конфигурация в *Spring Boot*

#### **Как настроить Kafka через application.yml vs Java Config? В чем разница между Factory и Template?**

- **application.yml (Рекомендуемый):** Spring Boot автоматически настраивает `KafkaTemplate`, `ConsumerFactory` и `ProducerFactory` на основе префикса `spring.kafka`. Этого достаточно для 90% задач.
    
- **Java Config:** Необходим, если в приложении нужно работать с **несколькими кластерами Kafka** одновременно или требуются специфические настройки бинов, которые нельзя передать через `.yml`.

#### **Ключевые сущности:**
- **ProducerFactory / ConsumerFactory:** Стратегии создания клиентов Kafka. Они содержат настройки (адреса брокеров, сериализаторы).
    
- **KafkaTemplate:** Высокоуровневая обертка над Producer API. Инкапсулирует логику отправки сообщений (синхронно/асинхронно) и позволяет не работать с низкоуровневым кодом.

## 8. Аннотация @KafkaListener

#### **Как она работает «под капотом» и как управлять метаданными?**

**Механизм работы:** При старте приложения Spring сканирует бины на наличие `@KafkaListener` и создает для каждого метода `MessageListenerContainer`. Он запускает цикл `poll()` в отдельном потоке (или нескольких, если настроен параметр `concurrency`).

#### **Продвинутые возможности:**
- **Метаданные:** Через аннотацию `@Header` можно получить любую информацию о сообщении:
```Java
@KafkaListener(topics = "orders")
    public void listen(String data, 
                       @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
                       @Header(KafkaHeaders.OFFSET) long offset) { ... }
```
    
- **Динамический Group ID:** Можно использовать SpEL (Spring Expression Language): `groupId = "${spring.application.name}"`.
    
- **Несколько топиков:** Можно передать массив строк или использовать `topicPattern` для подписки по регулярному выражению.

## 9. Обработка ошибок в Spring

#### **Как настроить DefaultErrorHandler и стратегии BackOff?**

`DefaultErrorHandler` пришел на смену устаревшим обработчикам. Он позволяет настроить политику повторов перед тем, как признать обработку неудачной.

#### **Стратегии BackOff:**
- **FixedBackOff:** Фиксированный интервал между попытками (например, 2 секунды).
    
- **ExponentialBackOff:** Интервал растет с каждой попыткой (например, 1с, 2с, 4с, 8с). Это снижает нагрузку на систему при массовых сбоях.

**Важно:** Можно указать исключения, при которых ретраи **не нужны** (например, `ValidationException`), используя метод `addNotRetryableExceptions()`.

## 10. Retry и DLT (*Dead Letter Topics*)

#### **Чем отличаются блокирующие повторы от неблокирующих?**

В современной Spring Kafka (2.7+) рекомендуется использовать `@RetryableTopic`.

- **Неблокирующий повтор (@RetryableTopic):** Если обработка упала, сообщение отправляется в промежуточный топик-ретрай (например, `topic-retry-1000`). Основной топик продолжает читаться другими консьюмерами. После исчерпания попыток сообщение уходит в `DLT`.
    - _Плюс:_ Высокая пропускная способность.
        
    - _Минус:_ Нарушается строгий порядок сообщений (сообщения за «проблемным» могут быть обработаны раньше).
    
- **Блокирующий повтор (DefaultErrorHandler):** Потребитель «замирает» и пытается переобработать то же самое сообщение.
    - _Плюс:_ Сохраняется строгий порядок.
        
    - _Минус:_ Останавливается вся партиция (Consumer Lag растет).

> **Рекомендация:** Если порядок критичен (финансовые транзакции) — используйте блокирующие повторы. В остальных случаях — `@RetryableTopic`.

## 11. Проблема «*Poison Pill*» и ErrorHandlingDeserializer

#### **Что делать, если сообщение невозможно десериализовать?**

**Poison Pill** — это сообщение с битой структурой. Стандартный десериализатор выкинет исключение **до** того, как сообщение попадет в `@KafkaListener`. Консьюмер попадет в бесконечный цикл: `poll` -> ошибка десериализации -> `poll` того же офсета.

**Решение: ErrorHandlingDeserializer** Он оборачивает основной десериализатор (например, JSON). Если десериализация провалилась, он не бросает исключение, а:
1. Возвращает `null` вместо данных.
    
2. Добавляет в заголовки (`Headers`) информацию об ошибке.
    
3. Передает управление в `DefaultErrorHandler`, который видит ошибку десериализации и может сразу отправить битое сообщение в DLT, не блокируя поток.

**Настройка в yml:**
```YAML
spring.kafka.consumer:
  value-deserializer: org.springframework.kafka.support.serializer.ErrorHandlingDeserializer
  
  properties:
  
    spring.deserializer.value.delegate.class: org.springframework.kafka.support.serializer.JsonDeserializer
```
