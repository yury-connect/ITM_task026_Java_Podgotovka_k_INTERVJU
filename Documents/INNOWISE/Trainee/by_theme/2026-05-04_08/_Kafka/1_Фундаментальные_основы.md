# 1. Фундаментальные основы <br>и архитектура (*Must-have*)

---
## 1. Архитектура `Kafka` и отличия от `RabbitMQ`

#### **Основные компоненты:**
- **Broker:** Сервер, отвечающий за хранение и передачу сообщений.
    
- **Topic:** Логический поток данных.
    
- **Partition:** Физический сегмент топика. 
	  Обеспечивает параллелизм и упорядоченность внутри себя.
    
- **Offset:** Порядковый номер сообщения в партиции. 
	  Используется для отслеживания прогресса чтения.
    
- **Producer:** Источник данных, отправляющий сообщения.
    
- **Consumer & Consumer Group:** Потребители, объединенные в группу 
	  для параллельного чтения топика.
    
- **Replica (Leader/Follower):** Копии партиций для отказоустойчивости. 
	  Запись/чтение всегда идут через **Leader**.    

#### **Отличия от `RabbitMQ`:**
- **Модель данных:** 
	  **Kafka** — это распределенный лог (хранит историю). 
	  **RabbitMQ** — классическая очередь 
		  (*сообщения обычно удаляются после подтверждения*).
    
- **Pull vs Push:** 
	  В **Kafka** потребитель сам забирает данные (***Pull***), контролируя темп. 
	  В **RabbitMQ** брокер заталкивает данные (***Push***) потребителю.
    
- **Хранение:** 
	  **Kafka** хранит данные согласно Retention Policy (по времени или размеру). 
	  **RabbitMQ** хранит сообщения, пока они не будут обработаны.

## 2. Устройство `Topic` и `Partition`

#### **Зачем нужны партиции:**
- **Масштабируемость:** Позволяют распределять один топик 
	  между несколькими брокерами.
    
- **Параллелизм:** Одна партиция — единица параллелизма. В группе 
	  количество активных потребителей ограничено количеством партиций.
    
- **Производительность:** Позволяет вести запись на диск в несколько потоков.

#### **Изменение количества партиций:**
- **Увеличение:** Возможно «на лету». Однако сообщения с одним ключом могут попасть в разные партиции (нарушится порядок), так как изменится результат хэш-функции `hash(key) % partitions`.
    
- **Уменьшение:** **Невозможно.** Для этого требуется создание нового топика и полная миграция данных.

### 3. Группы потребителей и *Rebalancing*

#### **Механизм распределения:**
Внутри группы одна партиция может принадлежать только одному потребителю. Потребитель может обрабатывать несколько партиций.

#### **Ребалансировка (*Rebalancing*):**
- **Триггеры:** Изменение состава группы (ушел/пришел консьюмер), изменение количества партиций, падение брокера.
    
- **Риски:** Главный минус — эффект **Stop-the-world**, когда на время перераспределения чтение данных полностью останавливается.

#### **Стратегии:**
- **Range:** Потопиковое распределение (по умолчанию).
    
- **Round Robin:** Равномерное циклическое распределение партиций.
    
- **Sticky:** Пытается сохранить старые назначения при перераспределении.
    
- **Cooperative Sticky:** Современный стандарт (с Kafka 2.4). Проводит ребаланс инкрементально, не останавливая работу всей группы.

## 4. Гарантии доставки (*Delivery Semantics*)

| **Семантика**     | **Описание**                        | **Producer Config**                            | **Consumer Config**                                              |
| ----------------- | ----------------------------------- | ---------------------------------------------- | ---------------------------------------------------------------- |
| **At-most-once**  | Без дублей, <br>но возможны потери. | `acks=0`                                       | `enable.auto.commit =true`                                       |
| **At-least-once** | Без потерь, <br>но возможны дубли.  | `acks=all`, `retries>0`                        | `enable.auto.commit =false` <br>+ ручной коммит <br>после логики |
| **Exactly-once**  | Ровно один раз (*идеально*).        | `enable.idempotence =true`, `transactional.id` | `isolation.level =read_committed`                                |

**Примечание**: _Exactly-once (EOS) требует использования транзакционного API Kafka для атомарности записи и коммита офсета._

---
## 5. Гарантия порядка сообщений

#### **Как Kafka хранит порядок:**
1. **Только внутри партиции:** Гарантируется строгая последовательность по офсету.
    
2. **Роль ключа:** Сообщения с одинаковым ключом всегда попадают в одну партицию (гарантирует порядок для конкретного ID, например `order_id`).

#### **Порядок во всем топике:**
Достигается только при наличии **одной партиции**. 
Это убивает масштабируемость и превращает Kafka в бутылочное горлышко.

## 6. Надежность и Acks

#### **Параметр `acks` (Producer):**
- `acks=0`: Подтверждение не требуется. Максимальная скорость, минимальная надежность.
    
- `acks=1`: Только Leader подтвердил запись. Риск потери при падении лидера до репликации.
    
- `acks=all (-1)`: Подтверждение от лидера и всех **ISR** (In-Sync Replicas).

#### **Связь с `min.insync.replicas`:**
Это настройка брокера/топика. Если `acks=all`, запись будет успешной только если число синхронных реплик (ISR) не меньше `min.insync.replicas`.

- _Пример:_ Если `replication.factor=3`, а `min.insync.replicas=2`, система выдержит падение одного брокера без потери возможности записи. Если упадут два — запись будет заблокирована для надежности.
