## 8. Оптимизация размера образа

Для Middle-разработчика важно уметь создавать компактные и безопасные образы:

- Использование **JRE** вместо полного JDK (экономия ~200МБ).
    
- **Alpine Linux** (очень маленький дистрибутив) или **Distroless** (образы без shell и лишних утилит, что повышает безопасность).
    
- Использование `.dockerignore`, чтобы не тащить в образ папку `.git`, тесты и локальные файлы сборки.

## 9. Multi-stage builds

Техника, позволяющая использовать один образ для сборки (с Maven/Gradle), а другой — для запуска.
```Dockerfile
# Stage 1: Сборка
FROM maven:3.9-eclipse-temurin-21 AS build
COPY src /home/app/src
COPY pom.xml /home/app
RUN mvn -f /home/app/pom.xml clean package

# Stage 2: Runtime
FROM eclipse-temurin:21-jre
COPY --from=build /home/app/target/*.jar app.jar
ENTRYPOINT ["java", "-jar", "/app.jar"]
```

**Результат:** В финальный образ не попадает исходный код и инструменты сборки, только JRE и JAR.

## 10. Spring Boot Layered JAR

Начиная с версии 2.3, Spring Boot позволяет разбивать сам JAR-файл на слои (Dependencies, Loader, Snapshot-dependencies, Application).

- Это позволяет Docker кэшировать тяжелый слой с зависимостями отдельно от слоя с вашим кодом. При изменении кода вы пересобираете слой размером в несколько КБ, а не весь "толстый" JAR в сотни МБ.

## 11. Volumes vs Bind Mounts

- **Bind Mounts:** Привязка конкретной папки хоста к контейнеру. Используется чаще в разработке (например, для hot-reload).
    
- **Volumes:** Управляются самим Docker. Это стандарт для продакшена. Они независимы от жизненного цикла контейнера и используются для баз данных или хранения логов, так как данные в них сохраняются даже после удаления контейнера.

## 12. Resource Limits и JVM

- **Проблема:** Старые версии Java (до 8u191/10) не понимали ограничений Docker и видели всю память сервера, что приводило к `OOM Killer` (ОС убивала контейнер).
    
- **Современное решение:** Начиная с Java 11+, JVM корректно распознает лимиты `cgroups`.
    
- **Best Practice:** Использовать флаг `-XX:MaxRAMPercentage` (например, 75.0), чтобы оставить место для Native memory и стеков потоков, не ограничивая Heap жестким значением в мегабайтах.

## 13. Healthchecks

- **Docker Healthcheck:** Проверка на уровне Docker-демона (команда в Dockerfile).
    
- **Spring Boot Actuator:** Предоставляет эндпоинт `/actuator/health`.
    
- **Интеграция:** В Dockerfile или Compose стоит прописать вызов `curl` к актуатору. Это позволяет оркестратору (или Compose) понять, что приложение не просто "запущено", а готово обрабатывать трафик (Readiness).

## 14. Логирование

Золотое правило контейнеризации: **логи должны писаться в STDOUT и STDERR**.

- Не нужно писать логи в файлы внутри контейнера (они пропадут при его удалении).
    
- Docker перехватывает поток вывода и сохраняет его. Вы можете прочитать их через `docker logs`.
    
- В продакшене Docker-драйверы (например, `fluentd` или `gelf`) автоматически пересылают эти логи в централизованные системы (ELK, Graylog).
