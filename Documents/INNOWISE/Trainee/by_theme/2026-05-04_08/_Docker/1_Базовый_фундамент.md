# 1. Базовый фундамент

---
## 1. Разница между Docker Image и Docker Container

- **Docker Image** — это инертный, неизменяемый (immutable) файл-шаблон, состоящий из набора слоев. Он содержит код, среду выполнения (JRE), библиотеки и конфигурацию. Образ хранится в Registry.
    
- **Docker Container** — это развернутый и запущенный экземпляр образа. С технической точки зрения это изолированный процесс в ОС хоста, использующий `namespaces` (для изоляции ресурсов) и `cgroups` (для ограничения ресурсов). У контейнера есть свой тонкий записываемый слой поверх слоев образа.

## 2. Dockerfile: разбор основных инструкций

- **FROM** — задает базовый образ (например, JRE).
    
- **WORKDIR** — устанавливает рабочую директорию (аналог `cd`).
    
- **COPY** — копирует локальные файлы в образ. **Предпочтительнее ADD**, так как `ADD` имеет избыточную логику (скачивание по URL и автоматическую распаковку архивов), что может снизить безопасность и предсказуемость сборки.
    
- **RUN** — выполняет команду при сборке (создает новый слой). Обычно это `apt-get install` или сборка проекта.
    
- **CMD** — параметры по умолчанию для запуска контейнера. Легко переопределяются при старте.
    
- **ENTRYPOINT** — основная команда, которая будет запущена. Параметры из `CMD` или CLI добавляются к ней.
    
- **EXPOSE** — носит информативный характер (документация порта), реально порт на хосте не открывает.    

## 3. Разница между ENTRYPOINT и CMD в контексте Java

Для Java-приложений обычно используют комбинацию в **exec-форме** (в виде JSON-массива):
```Dockerfile
ENTRYPOINT ["java", "-jar"]
CMD ["app.jar"]
```

Это позволяет гибко управлять запуском. Если вы вызовете `docker run my-app other-app.jar`, то `other-app.jar` заменит значение в `CMD`, но `java -jar` из `ENTRYPOINT` останется. **Важно:** использование exec-формы позволяет Java-процессу получить `PID 1` и корректно обрабатывать сигналы завершения (SIGTERM) от Docker.

## 4. Docker Compose: запуск связки Spring Boot + БД

Docker Compose позволяет описать инфраструктуру в одном YAML-файле.

- **Проблема:** Инструкция `depends_on` в базовом виде гарантирует только порядок запуска контейнеров, но не готовность приложения внутри (БД может еще инициализироваться).
    
- **Решение:** Использовать `healthcheck` в описании сервиса БД. В сервисе приложения указать:
```YAML
depends_on:
  db:
    condition: service_healthy
```

Это заставит Spring Boot ждать момента, когда база данных будет готова принимать соединения.

## 5. Слои (Layers) и кэширование

Каждая команда `RUN`, `COPY`, `ADD` создает новый слой. Docker кэширует их. Если содержимое слоя не изменилось, Docker берет его из кэша.

- **Оптимизация:** Нужно располагать команды от "редко меняющихся" к "часто меняющимся".
    
- **Для Java:** Сначала копируем файлы сборки (`pom.xml` или `build.gradle`) и скачиваем зависимости, а только потом копируем исходный код. Так при изменении одной строки кода Docker не будет заново скачивать все библиотеки из Maven Central.

## 6. Управление переменными окружения

Spring Boot отлично интегрирован с Docker. Любая переменная окружения, переданная через `-e` или в `compose` (например, `SPRING_PROFILES_ACTIVE`), автоматически переопределяет значение в `application.properties`.

- **JAVA_OPTS:** Часто используется для передачи параметров JVM (например, `-Xmx`, логирование). Важно правильно прописать их в `ENTRYPOINT`, чтобы они подхватились при старте.

## 7. Проброс портов (Port mapping)

- **-p 8080:9090** — связывает порт `8080` на хост-машине с портом `9090` внутри контейнера.
    
- Если контейнеры находятся в одной **Docker Network** (создается автоматически в Compose), они могут общаться напрямую по именам сервисов и их внутренним портам (например, `http://backend:9090`), не выставляя их на хост-машину.
