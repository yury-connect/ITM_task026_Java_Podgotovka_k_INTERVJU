# Группа 4: Управление транзакциями 
(Очень высокая частота)

Транзакции — это одна из самых частых тем на собеседовании, особенно на Middle позиции. Отвечаю максимально точно и с аргументацией.

---
### 35. Что такое `@Transactional`? Как он работает под капотом? (Прокси + AOP)

**Ответ:** `@Transactional` — это декларативный способ управления транзакциями в Spring. Вместо ручного открытия/коммита/отката через `transactionManager` вы просто ставите аннотацию на метод или класс.

**Как работает под капотом (обязательно рассказать этот механизм):**

1. **Spring создает прокси** (JDK Dynamic Proxy или CGLIB) для каждого бина, содержащего `@Transactional`.
    
2. **Прокси перехватывает вызов метода** и перед вызовом целевого метода:
    
    - Проверяет, есть ли уже открытая транзакция (с учетом propagation).
        
    - При необходимости создает новую транзакцию через `PlatformTransactionManager`.
        
    - Устанавливает уровень изоляции, таймаут, readOnly.
        
3. **Выполняет целевой метод**.
    
4. **После выполнения:**
    
    - Если метод завершился **нормально** (без исключения) → прокси делает `commit`.
        
    - Если метод выбросил **исключение**, подпадающее под `rollbackFor` (по умолчанию `RuntimeException` и `Error`) → прокси делает `rollback`.
        
    - Если исключение не должно вызывать откат → `commit`.
        
5. **Прокси не создается**, если метод `private` или вызван из того же класса (self-invocation) — это частая ошибка.    

**Ключевая суть:** `@Transactional` работает только через прокси, поэтому вызов `this.method()` внутри того же класса не открывает транзакцию.

### 36. Какие `propagation` (*распространение транзакций*) бывают?

**Ответ:** Propagation определяет, **как** метод ведет себя, 
когда вызывается внутри уже существующей транзакции. *Всего их **7***:

| Propagation                       | Поведение                                                                                                                                           | Типичное использование                                                                   |
| --------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| **`REQUIRED`** <br>(по умолчанию) | Если транзакция есть — **присоединяется** к ней. <br>Нет — создает **новую**.                                                                       | 95% методов                                                                              |
| **`REQUIRES_NEW`**                | **Всегда** приостанавливает <br>текущую транзакцию (*если есть*) <br>и создает **новую**. <br>После завершения новой — <br>**возобновляет** старую. | Логирование, аудит, отправка email (*должны закоммититься <br>даже при откате основной*) |
| **`SUPPORTS`**                    | Если транзакция есть — присоединяется. <br>Если нет — работает без <br>транзакции.                                                                  | Методы только для чтения, которым не нужна транзакция, но могут участвовать              |
| **`MANDATORY`**                   | **Требует** наличия транзакции. <br>Если нет — бросает исключение.                                                                                  | Внутренние методы, которые должны вызываться только в контексте транзакции               |
| **`NOT_SUPPORTED`**               | **Приостанавливает** <br>текущую транзакцию <br>и выполняет метод <br>**без транзакции**. <br>После завершения — <br>возобновляет.                  | Операции, которые не должны быть в транзакции (например, массовые отчеты)                |
| **`NEVER`**                       | Запрещает транзакцию. Если транзакция есть — бросает исключение.                                                                                    | Операции, которые не могут участвовать в транзакции                                      |
| **`NESTED`**                      | Если транзакция есть — создает **вложенную** (savepoint). <br>Если нет — создает новую. <br>Откат nested не откатывает родительскую.                | Частичный откат внутри большой транзакции                                                |

**Важное замечание:** `NESTED` работает только с `DataSourceTransactionManager` (JDBC). JTA не поддерживает savepoint'ы.

### 37. Что делает `REQUIRES_NEW`? <br>Будет ли транзакция приостановлена?

**Ответ:** `@Transactional(propagation = Propagation.REQUIRES_NEW)` 
гарантирует, что метод всегда выполняется в **новой, независимой транзакции**.

#### **Алгоритм работы:**
1. Проверяет, есть ли уже открытая транзакция в текущем потоке.
    
2. **Если есть** — **приостанавливает её** (suspend). Текущая транзакция временно "замораживается" — её изменения не видны другим, она не коммитится.
    
3. Создает **совершенно новую** транзакцию (с новым соединением с БД, если это физическая транзакция, или с новым транзакционным контекстом).
    
4. Выполняет метод.
    
5. **Коммитит или откатывает** новую транзакцию **независимо** от родительской.
    
6. **Возобновляет** приостановленную родительскую транзакцию.

#### **Ключевые последствия:**
- Откат новой транзакции **не влияет** на родительскую.
    
- Откат родительской транзакции **не откатывает** уже закоммиченную новую.
    
- Каждая `REQUIRES_NEW` транзакция использует <u>отдельное</u> соединение с БД 
  (*или транзакционный ресурс*), что может привести к **deadlock'ам** 
  и пулу соединений.    

**Пример использования:** Логирование действий пользователя. Даже если основная бизнес-операция откатится, запись в лог "пользователь пытался сделать X" должна сохраниться. Именно поэтому `REQUIRES_NEW` идеален для аудита.

**Ответ на вопрос "будет ли транзакция приостановлена?":** **Да, обязательно**. Приостановка происходит на уровне транзакционного менеджера: текущая транзакция отвязывается от текущего потока и сохраняется в стеке, а новая привязывается.

### 38. Какие `isolation levels` (*уровни изоляции*) <br>поддерживает Spring?

**Ответ:** Spring поддерживает **5** уровней изоляции, соответствующих стандарту SQL-92:

| Уровень                | Грязное чтение (Dirty Read) | Неповторяемое чтение (Non-repeatable Read) | Фантомное чтение (Phantom Read) | Описание                                                                                                                                               |
| ---------------------- | --------------------------- | ------------------------------------------ | ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **`DEFAULT`**          | Зависит от БД               | Зависит от БД                              | Зависит от БД                   | Использует настройку БД по умолчанию (обычно `READ_COMMITTED` в PostgreSQL, `READ_COMMITTED` в Oracle, `REPEATABLE_READ` в MySQL)                      |
| **`READ_UNCOMMITTED`** | ✅ **Может быть**            | ✅                                          | ✅                               | Самая низкая изоляция. Транзакция видит незакоммиченные изменения других транзакций. Риск грязного чтения.                                             |
| **`READ_COMMITTED`**   | ❌ Нет                       | ✅                                          | ✅                               | **Стандарт для многих БД**. Видит только закоммиченные данные. Но при повторном чтении той же строки данные могут измениться.                          |
| **`REPEATABLE_READ`**  | ❌ Нет                       | ❌ Нет                                      | ✅                               | Гарантирует, что при повторном чтении той же строки в рамках транзакции данные не изменятся. Фантомные строки (INSERT/ DELETE) возможны.               |
| **`SERIALIZABLE`**     | ❌ Нет                       | ❌ Нет                                      | ❌ Нет                           | **Максимальная изоляция**. Транзакции выполняются последовательно (или как будто последовательно). Самые высокие накладные расходы и риск deadlock'ов. |

**Как задать:**
```java
@Transactional(isolation = Isolation.REPEATABLE_READ)
public void myMethod() { }
```

**Важное замечание:** Уровень изоляции применяется **только** при создании новой транзакции (propagation `REQUIRED` или `REQUIRES_NEW`, когда транзакция реально создается). Если метод присоединяется к существующей транзакции, указанный isolation игнорируется.

### 39. Как работает `@Transactional(readOnly = true)`? <br>Оптимизации (*Hibernate, flush mode*)

**Ответ:** `readOnly = true` — это подсказка Spring и базе данных о том, 
что транзакция будет <u>только читать</u> данные, не изменяя их.

#### **Что происходит под капотом:**
1. **На уровне JDBC:** Для некоторых БД (*например, `PostgreSQL`, `H2`*) устанавливается флаг `readOnly` на соединении. БД может оптимизировать блокировки, отключить механизмы журналирования, убрать накладные расходы на ожидание блокировок.
    
2. **На уровне Hibernate (самое важное):**    
    - **Flush mode** принудительно устанавливается в `FlushMode.MANUAL` или `FlushMode.NEVER` (*зависит от версии*).
        
    - Это означает, что Hibernate **не будет автоматически синхронизировать** (*flush*) `persistence context` с БД перед коммитом.
        
    - **Никакие `INSERT`, `UPDATE`, `DELETE` не будут отправлены в БД** даже если вы вызвали `save()` или `merge()` (*изменения накопятся, но не сбросятся*).
        
    - Отключается dirty checking — Hibernate не сравнивает состояние сущностей 
	      со *snapshot*-ом, так как обновлений всё равно не будет.
    
3. **На уровне Spring:** Может быть задействована оптимизация 
	   на уровне ресурсов (например, connection pool).

#### **Практический эффект:**
- **Производительность выше** — меньше блокировок, 
	  меньше накладных расходов на `flush`.
    
- **Безопасность** — защита от случайной модификации данных.
    
- **Ограничение:** Если вы всё же вызовете `save()` внутри `readOnly = true`, Hibernate может выбросить исключение (*зависит от настройки*) или просто проигнорировать изменения до выхода из транзакции.

**Важный нюанс:** `readOnly = true` **не запрещает** изменение данных на уровне БД через прямые SQL-запросы или нативные запросы. 
Это лишь подсказка, а не строгий контроль доступа.

### 40. Что такое `rollbackFor`? <br>По умолчанию откат только по `RuntimeException` и `Error`

**Ответ:** `rollbackFor` — это параметр `@Transactional`, который указывает, 
	**какие исключения должны вызывать откат транзакции**.

#### **Правило по умолчанию (очень важно):**
- **Откат происходит** — при `RuntimeException` (и его наследниках, например `NullPointerException`, `IllegalArgumentException`) и `Error`.
    
- **Откат НЕ происходит** — при **проверяемых исключениях** (`Exception` и его наследниках, кроме `RuntimeException`, например `IOException`, `SQLException`).

**Почему так:** Проверяемые исключения считаются "ожидаемыми" бизнес-сценариями (например, "товар не найден", "недостаточно средств"), и они не должны откатывать транзакцию по умолчанию.

**Как изменить:**
```java
// Откатывать при любом Exception (включая проверяемые)
@Transactional(rollbackFor = Exception.class)

// Откатывать при конкретных исключениях
@Transactional(rollbackFor = {SQLException.class, IOException.class})

// НЕ откатывать при конкретном RuntimeException
@Transactional(noRollbackFor = NullPointerException.class)
```

**Пример:** Если у вас метод бросает `BusinessException` (которое extends `Exception`), транзакция **закоммитится** по умолчанию. Чтобы это изменить, нужно явно указать `rollbackFor = BusinessException.class`.

### 41. Почему проверяемые исключения (`Exception`) <br>не откатывают транзакцию по умолчанию?

**Ответ:** Это **архитектурное решение** Spring, основанное на философии, что проверяемые исключения (checked exceptions) представляют **ожидаемые, восстанавливаемые бизнес-ситуации**, а не системные ошибки.

#### **Аргументация:**
1. **Бизнес-сценарий vs Системная ошибка:**    
    - `SQLException` (*при нарушении уникальности*) — может означать "пользователь уже существует". Это бизнес-логика, которая не требует отката всей операции.
        
    - `IOException` (файл не найден) — может быть ожидаемым сценарием.
        
    - А вот `NullPointerException`, `IllegalArgumentException` — это баги, при которых состояние системы неопределено, и транзакцию нужно откатить.
    
2. **Принцип наименьшего удивления:** Разработчики ожидают, что если метод бросает `IOException`, то можно перехватить его и попробовать альтернативный путь (*например, создать файл заново*). Автоматический откат в этом случае был бы слишком агрессивным.
    
3. **Совместимость с EJB:** Поведение `@Transactional` унаследовано от спецификации EJB, где проверяемые исключения тоже не вызывали откат.
    
4. **Явное лучше неявного:** Если разработчик хочет откат при проверяемом исключении, он должен явно указать `rollbackFor`. 
   Это делает код самодокументированным.

**Что делать, если нужно откатывать по проверяемому исключению:**
```java
@Transactional(rollbackFor = BusinessException.class)
public void doSomething() throws BusinessException {
    // ...
}
```

### 42. Как задать таймаут транзакции? (`timeout`)

**Ответ:** Таймаут транзакции задается через параметр `timeout` (в секундах) 
	в аннотации `@Transactional`.
```java
@Transactional(timeout = 30) // 30 секунд
public void longRunningOperation() { }
```

#### **Что происходит:**
- Таймаут отсчитывается с **момента создания транзакции** 
  (*начала выполнения метода, если создается новая транзакция*).
    
- Если метод не завершился за отведенное время, Spring автоматически 
  **помечает транзакцию как rollback-only** и при попытке коммита 
  бросает `TransactionTimedOutException`.
    
- На уровне БД может быть установлен соответствующий таймаут 
  на запрос (*но зависит от драйвера и БД*).

#### **Важные нюансы:**
1. **`timeout` применяется только при создании новой транзакции** (propagation = `REQUIRED` или `REQUIRES_NEW`, когда реально создается новая). Если метод присоединяется к существующей транзакции, его таймаут **игнорируется**.
    
2. **Единица измерения:** секунды. Можно использовать `timeoutString` для более гибкого формата (например, `"5s"`, `"1m"`).
    
3. **Значение по умолчанию:** `-1` (без таймаута, используется таймаут по умолчанию транзакционного менеджера или БД).
    
4. **Транзакционный менеджер должен поддерживать таймауты.** `DataSourceTransactionManager` поддерживает, JTA — тоже.
    
5. **Таймаут не гарантирует прерывание выполнения метода** — он лишь гарантирует, что при попытке commit'а после таймаута транзакция откатится. Долго выполняющийся SQL-запрос может не прерваться, пока не завершится.

### 43. Проблема: `@Transactional` <br>в `private` методе — работает?

**Ответ:** **Нет, не работает.** И это прямое следствие того, 
	как Spring реализует транзакции через прокси.

#### **Почему:**
1. **JDK Dynamic Proxy** — проксирует только **публичные методы интерфейсов**. `private` методы даже не видны через интерфейс.
    
2. **CGLIB** — создает подкласс. `private` методы не наследуются и не могут быть переопределены. Прокси не имеет доступа к `private` методу родительского класса.
    
3. **Даже если метод вызван публично** из другого класса, но сам метод `private`, Spring <u>физически не может его перехватить</u> через прокси.

**Пример НЕРАБОТАЮЩЕГО кода:**
```java
@Service
public class UserService {
    
    @Transactional  // НЕ БУДЕТ РАБОТАТЬ!
    private void saveUser(User user) {
        // ...
    }
    
    public void register(User user) {
        saveUser(user); // self-invocation + private
    }
}
```

**Что происходит:** Никакой прокси не создается для `private` методов. 
Транзакция не открывается, метод выполняется без транзакционного контекста.

**Решение:** Вынести логику в **публичный метод**
	(или `protected`, если используется CGLIB). Либо выделить в отдельный сервис.

**Исключение (*теоретическое*):** При использовании **compile-time weaving** 
или **load-time weaving** с полноценным AspectJ (не Spring AOP) можно зааспектить `private` методы, но это нестандартный подход и обычно не используется.

**Вывод для собеседования:** `@Transactional` должен ставиться только 
на **публичные методы**. Это архитектурное ограничение прокси-механизма.

### 44. Как **мониторить** транзакции и видеть, **открыта ли** транзакция на самом деле?

**Ответ:** Есть несколько способов мониторинга транзакций в Spring:

#### **Способ 1: Логирование через `application.properties`**
```properties
logging.level.org.springframework.transaction=DEBUG
logging.level.org.springframework.orm.jpa.JpaTransactionManager=DEBUG
logging.level.org.hibernate.engine.transaction=DEBUG
```
В логах появятся записи: "Creating new transaction", "Committing JPA transaction", "Rolling back".

#### **Способ 2: Использовать `TransactionSynchronizationManager` <br>(см. вопрос 45)**
```java
boolean isActive = 
	TransactionSynchronizationManager.isActualTransactionActive();
	
boolean isReadOnly = 
	TransactionSynchronizationManager.isCurrentTransactionReadOnly();
	
String name = 
	TransactionSynchronizationManager.getCurrentTransactionName();
```

#### **Способ 3: Включить `DEBUG` для `DataSource`** <br>(увидеть реальные `BEGIN`, `COMMIT`, `ROLLBACK`)
```properties
logging.level.java.sql.Connection=DEBUG
logging.level.java.sql.Statement=DEBUG
```

#### **Способ 4: Использовать `@Transactional` <br>с `TransactionAspectSupport`**
```java
@Transactional
public void myMethod() {
    TransactionStatus status = 
					TransactionAspectSupport.currentTransactionStatus();
    boolean isRollbackOnly = status.isRollbackOnly();
    // ...
}
```

#### **Способ 5: Внедрить `PlatformTransactionManager` напрямую**
```java
@Autowired
private PlatformTransactionManager txManager;
public void check() {
    TransactionStatus status = txManager.getTransaction(new DefaultTransactionDefinition());
    boolean isNew = status.isNewTransaction();
    txManager.rollback(status); // откатить, чтобы не оставлять открытую транзакцию
}
```

#### **Способ 6: `Actuator` и метрики (*для production*)**
```properties
management.endpoints.web.exposure.include=metrics

```

Эндпоинт `/actuator/metrics/spring.transaction.*` покажет статистику 
	по транзакциям (*коммиты, откаты*).

**Практический совет:** Самый простой способ на разработке — включить `DEBUG` логгирование для `org.springframework.transaction`. Это покажет, создается ли транзакция, приостанавливается ли, коммитится.

### 45. Что такое `TransactionSynchronizationManager`?

**Ответ:** `TransactionSynchronizationManager` — это центральный **утилитный класс** в Spring, который управляет **ресурсами, привязанными к текущему потоку** для каждой транзакции.

#### **Основные возможности:**
1. **Проверка состояния транзакции в текущем потоке:**
```java
boolean isActive = TransactionSynchronizationManager.isActualTransactionActive();
boolean isReadOnly = TransactionSynchronizationManager.isCurrentTransactionReadOnly();
String txName = TransactionSynchronizationManager.getCurrentTransactionName();
Integer isolationLevel = TransactionSynchronizationManager.getCurrentTransactionIsolationLevel();
```
    
2. **Регистрация синхронизаций (`TransactionSynchronization`)** — объектов, которые получают уведомления о событиях транзакции (beforeCommit, afterCommit, afterCompletion). Полезно для действий, которые должны выполниться только после успешного коммита (например, отправка сообщения в очередь).
    
3. **Привязка ресурсов к транзакции** (низкоуровневая операция). Spring сам привязывает соединение с БД (`Connection`) к текущему потоку, чтобы несколько репозиториев использовали одно и то же соединение в рамках транзакции.
    
4. **Управление приостановленными транзакциями** (для `REQUIRES_NEW`, `NOT_SUPPORTED`).

**Пример регистрации синхронизации:**
```java
@Transactional
public void doSomething() {
    TransactionSynchronizationManager.registerSynchronization(
        new TransactionSynchronization() {
            @Override
            public void afterCommit() {
                // Выполнится только после успешного коммита
                sendEmailNotification();
            }
        }
    );
}
```

**Внутреннее устройство:** Хранит все данные в **`ThreadLocal`** переменных, поэтому каждая транзакция в своем потоке изолирована.

**Для Middle важно:** Понимать, что `TransactionSynchronizationManager` — это внутренний механизм Spring, но знание его существования помогает отлаживать проблемы с транзакциями и писать корректный код, зависящий от фазы транзакции.

### 46. Как работает менеджер транзакций <br>(`PlatformTransactionManager`)? <br>Реализации

**Ответ:** `PlatformTransactionManager` — это **центральный интерфейс** Spring Transaction API, который абстрагирует управление транзакциями от конкретной технологии (JDBC, JPA, JTA и т.д.).

#### **Основные методы:**
```java
public interface PlatformTransactionManager {
    TransactionStatus getTransaction(TransactionDefinition definition) throws TransactionException;
    void commit(TransactionStatus status) throws TransactionException;
    void rollback(TransactionStatus status) throws TransactionException;
}
```

#### **Как работает (стандартный алгоритм):**
1. **`getTransaction()`** — проверяет, есть ли уже активная транзакция в текущем потоке (через `TransactionSynchronizationManager`).
    
    - Если есть и `propagation` позволяет присоединиться → возвращает существующий статус.
        
    - Если нужно создать новую → открывает соединение/сессию, начинает транзакцию БД, привязывает ресурс к потоку.
        
    - Если нужно приостановить текущую → сохраняет её, отвязывает от потока.
        
2. **`commit()`** — если статус помечен `rollbackOnly` → вызывает `rollback()`. Иначе вызывает `doCommit()` (реальная операция на БД).
    
3. **`rollback()`** — откатывает транзакцию и освобождает ресурсы.

#### **Основные реализации:**

| Реализация                                         | Когда используется                                     | Особенности                                                                                                                               |
| -------------------------------------------------- | ------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **`DataSourceTransactionManager`**                 | Обычный JDBC, MyBatis, прямой `DataSource`             | Работает с `java.sql.Connection`. Поддерживает `REQUIRES_NEW`, <br>`NESTED` (savepoint'ы).                                                |
| **`JpaTransactionManager`**                        | **Самая частая** — для JPA (Hibernate, EclipseLink)    | Работает с `EntityManager`. Делегирует `DataSourceTransactionManager` для JDBC-операций. Умеет синхронизировать JPA-сессию с транзакцией. |
| **`JtaTransactionManager`**                        | Распределенные транзакции (XA), несколько БД, JMS + БД | Делегирует JTA-провайдеру (Atomikos, Bitronix, Narayana, WebLogic/WebSphere). Тяжелый, сложный в настройке.                               |
| **`HibernateTransactionManager`**<br> (устаревший) | Старые версии Hibernate без JPA                        | В новых проектах не используется — вместо него `JpaTransactionManager`.                                                                   |

#### **Как *Spring* выбирает менеджер:**
- Если есть `@Bean` типа `PlatformTransactionManager` — использует его.
    
- Spring Boot автоматически создает `JpaTransactionManager` при наличии `spring-boot-starter-data-jpa`.
    
- Можно явно указать `@Transactional("transactionManagerName")`.

**Важный нюанс:** `JpaTransactionManager` требует, чтобы `EntityManager` был привязан к текущему потоку. Spring делает это автоматически через `@PersistenceContext` и `SharedEntityManagerCreator`.

**Для собеседования:** Уверенно рассказать, что менеджер транзакций — это абстракция над БД-специфичными операциями, и Spring Boot автоматически настраивает подходящий менеджер в зависимости от зависимостей в classpath.
