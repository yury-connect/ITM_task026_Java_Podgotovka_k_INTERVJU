### 17. Что такое `application.properties` / `application.yml`?

**Ответ:** Это главные конфигурационные файлы Spring Boot приложения.

- **`application.properties`** — классический формат `key=value`. 
	  Простой, но при большом количестве настроек становится многословным.
    
- **`application.yml`** — формат на основе YAML (Yet Another Markup Language). 
  Поддерживает иерархическую структуру с отступами, что делает конфигурацию более читаемой для вложенных свойств (например, `spring.datasource.url` 
  вместо плоского списка).    

**Важный нюанс:**  Spring Boot автоматически загружает эти файлы из нескольких мест 
	**по убыванию приоритета** (выше = важнее):
1. `file:./config/` (подкаталог config в текущей директории)
2. `file:./` (текущий каталог)
3. `classpath:/config/`
4. `classpath:/`

Это позволяет легко переопределять настройки на разных окружениях (например, через монтирование volume в Docker) без изменения JAR-файла.

### 18. Как связать свойства из файла с полем в классе? <br>(`@Value`, `@ConfigurationProperties`)

**Ответ:** Spring Boot предоставляет два основных способа:

**1. `@Value` — для точечного внедрения отдельных свойств:**
```java
@Value("${app.name:defaultName}")
private String appName;
```

Поддерживает SpEL выражения и значения по умолчанию через `:`.

**2. `@ConfigurationProperties` — для группового маппинга:**
```java
@Component
@ConfigurationProperties(prefix = "app")
public class AppProperties {
    private String name;
    private int timeout;
    private List<String> servers;
    // геттеры и сеттеры
}
```

Spring автоматически маппит `app.name`, `app.timeout`, `app.servers` в поля объекта.

**Ключевое различие:** 
- `@Value` — для единичных значений, 
- `@ConfigurationProperties` — для связанных групп свойств.

---
### 19. В чем преимущество `@ConfigurationProperties` <br>перед `@Value`?

**Ответ:** `@ConfigurationProperties` значительно мощнее 
	для структурированной конфигурации:

1. **Type safety** — *Spring Boot* выполняет конвертацию и валидацию типов на этапе <u>старта приложения</u>. При ошибке (*например, строка вместо числа*) приложение **не запустится** с понятным сообщением. С `@Value` ошибка проявится только при первом обращении к полю (*часто уже в продакшене*).
    
2. **Валидация** — поддерживает JSR-303 аннотации (`@NotNull`, `@Min`, `@Max`, `@Email`) прямо на полях конфигурационного класса. `@Value` такой возможности не имеет.
    
3. **Вложенные объекты** — можно маппить сложные структуры:
```java
@ConfigurationProperties(prefix = "db")
public class DbProperties {
    private ConnectionSettings connection = new ConnectionSettings();
    private PoolSettings pool = new PoolSettings();
    // вложенные объекты маппятся автоматически
}
```
    
4. **Метаданные** — при использовании `spring-boot-configuration-processor` генерируются метаданные для автодополнения в IDE.
    
5. **Удобство** — один объект вместо десятка аннотаций `@Value`.

**Вывод:** 
- `@Value` хорош для одного-двух свойств, 
- `@ConfigurationProperties` — для целых групп и сложной конфигурации.

### 20. Как задать **профили** (*profiles*)? <br>(`@Profile`, `spring.profiles.active`)

**Ответ:** Профили в Spring — это механизм изоляции конфигурации для разных окружений (**`dev`, `test`, `prod`, `staging`**).

#### **Способы задания активных профилей:**
1. **Через `application.properties`:**
```properties
spring.profiles.active=dev,prod
```
    
2. **Через системное свойство JVM:**
```bash
java -Dspring.profiles.active=prod -jar app.jar
```
    
3. **Через переменную окружения:**
```bash
export SPRING_PROFILES_ACTIVE=prod
```
    
4. **В коде при запуске:**
```java
SpringApplication app = new SpringApplication(MyApp.class);
app.setAdditionalProfiles("prod");
app.run(args);
```
    
5. **Через `@ActiveProfiles` в тестах.**

**Использование:** Аннотация `@Profile` ограничивает регистрацию бина 
	или `@Configuration` класса только при активном указанном профиле.

**Приоритет задания профилей** (*от высшего к низшему*):
- `spring.profiles.active` в командной строке / 
	  переменной окружения / JVM property 
- `spring.profiles.active` в `application.properties` / `application.yml` 
- Значение по умолчанию (`spring.profiles.default`)

### 21. Как сделать конфигурацию, зависящую от профиля?

**Ответ:** Spring Boot поддерживает **профильные файлы конфигурации** 
по соглашению об именовании:
- `application-dev.yml` — для **dev**-профиля    
- `application-prod.yml` — для **prod**-профиля    
- `application-test.properties` — для **test**-профиля    

**Как это работает:**
1. Базовый блок (до первого `---`) загружается **всегда**.
2. Каждый последующий документ активируется только если выполнены условия 
	   в `spring.config.activate.on-profile` (*или других `on-` условиях*).
3. Свойства из активированных профильных документов **переопределяют** 
	   свойства из предыдущих блоков.

**Пример:**
- `application.yml` содержит `server.port=8080`    
- `application-prod.yml` содержит `server.port=443`    
- При профиле `prod` приложение стартует на порту 443.    

**Альтернативный подход:** 
Один YAML-файл с мульти-документной структурой (через `---`), 
где каждый документ имеет `spring.config.activate.on-profile`.

### 22. Что такое `@PropertySource`?

**Ответ:** `@PropertySource` — аннотация для загрузки дополнительных `.properties` файлов в `Environment` Spring, **кроме** стандартного `application.properties`.

**Пример:**
```java
@Configuration
@PropertySource("classpath:db.properties")
@PropertySource(value = "file:/etc/app/custom.properties", ignoreResourceNotFound = true)
public class AppConfig {
    // db.properties и custom.properties теперь доступны через @Value или Environment
}
```

**Важные возможности:**
- `ignoreResourceNotFound = true` — не падать, если файл отсутствует.
- `encoding` — указать кодировку.
- `factory` — кастомный `PropertySourceFactory`.

**Ограничение:** Не работает с YAML-файлами (только `.properties`). 
Для YAML используется `@PropertySource` с `YamlPropertySourceFactory` 
(*требует дополнительной настройки*).

**Приоритет:** `@PropertySource` загружается **после** стандартных `application.properties`/`application.yml`, поэтому свойства из `@PropertySource` **могут переопределять** свойства из application-файлов (если ключ совпадает).

### 23. Что такое `SpringEnvironment`? <br>Как получить доступ к пропертям из кода?

**Ответ:** `Environment` — это интерфейс Spring, представляющий абстракцию 
<u>над всеми источниками конфигурации</u>: системные переменные, переменные окружения, JVM-аргументы, `application.properties`, `@PropertySource` файлы.

#### **Способы получения доступа:**

1. **Внедрение через `@Autowired`:**
```java
@Autowired
private Environment env;

public void doSomething() {
    String dbUrl = env.getProperty("spring.datasource.url");
    int timeout = env.getRequiredProperty("app.timeout", Integer.class);
    String[] activeProfiles = env.getActiveProfiles();
}
```
    
2. **Через `ApplicationContext`:**
```java
Environment env = context.getEnvironment();
```
    
3. **В `@Configuration` классах:**
```java
@Configuration
public class MyConfig {
    @Autowired
    private Environment env;
}
```

#### **Методы `Environment`:**
- `getProperty(String key)` — получить значение или *null*.    
- `getRequiredProperty(String key)` — бросит `IllegalStateException` при отсутствии.
- `containsProperty(String key)` — проверить наличие.    
- `getActiveProfiles()` / `getDefaultProfiles()` — работа с профилями.    
- `acceptsProfiles(Profiles)` — проверка, активен ли профиль.

### 24. Как задать случайный порт для тестов <br>(`server.port=0`)?

**Ответ:** В Spring Boot можно указать `server.port=0` в `application-test.properties` или через `@TestPropertySource`. Это заставит встроенный сервер (Tomcat/Jetty/Undertow) выбрать **случайный свободный порт** при старте.

#### **Как получить номер случайного порта в тесте:**
```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class MyIntegrationTest {
    
    @LocalServerPort
    private int randomPort; // порт автоматически внедрится
    
    @Test
    void test() {
        System.out.println("Server is running on port: " + randomPort);
        // Используем порт для создания URL, RestTemplate и т.д.
    }
}
```

#### **Варианты:**
- `@SpringBootTest(webEnvironment = RANDOM_PORT)` — поднимает полный контекст со случайным портом.
    
- `@WebMvcTest` также поддерживает `@LocalServerPort`.
    
- Можно также внедрить `int port` в любой бин через `@Value("${local.server.port}")`, но только после того, как сервер стартовал.    

> `@Value("${local.server.port}")` работает только после старта встроенного сервера (поэтому лучше использовать в тестах с `@SpringBootTest(webEnvironment = RANDOM_PORT)` или через `WebEnvironment.DEFINED_PORT` + `@LocalServerPort`).

**Зачем это нужно:** При параллельном запуске тестов фиксированный порт 
(например, 8080) может быть занят. <u>Случайный порт исключает конфликты</u>.

### 25. Поддержка мульти-документ файлов YAML <br>в Spring Boot?

**Ответ:** Да, Spring Boot полностью поддерживает мульти-документные YAML-файлы, где документы разделяются тремя дефисами `---`.

**Пример `application.yml`:**
```yaml
spring:
  application:
    name: myapp
  profiles:
    active: dev
---
spring:
  config:
    activate:
      on-profile: dev
server:
  port: 8080
---
spring:
  config:
    activate:
      on-profile: prod
server:
  port: 443
  ssl:
    enabled: true
```

### **Как работает:**
- Первый блок загружается всегда.
    
- Блоки с `spring.config.activate.on-profile` 
	  загружаются только при активном соответствующем профиле.
    
- Свойства из профильных блоков переопределяют свойства из первого блока.

### **Важные ограничения <br>по сравнению с отдельными файлами:**

1. **Списки** (lists) в разных документах **не мержатся автоматически** — 
	   последний активированный документ полностью заменяет предыдущий список (в отличие от отдельных `application-{profile}.yml` файлов, где Spring Boot умеет мержить списки при определённых условиях).
    
2. **Нельзя использовать** `@PropertySource` для YAML напрямую 
	   (требуется `YamlPropertySourceFactory`).
    
3. **Переопределение свойств** между документами 
	   может запутать при сложной структуре.

**Рекомендация:** Для простых приложений мульти-документный YAML удобен. Для сложных проектов с большим количеством профилей лучше использовать отдельные файлы `application-{profile}.yml` — они понятнее и их проще поддерживать.
