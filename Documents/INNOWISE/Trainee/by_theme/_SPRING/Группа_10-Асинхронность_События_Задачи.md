### 99. Что такое `@Async`? Как включить (`@EnableAsync`)

**Ответ:** `@Async` — аннотация Spring, которая позволяет выполнять методы **асинхронно** в отдельном потоке, не блокируя вызывающий поток.

#### **Как работает:**
1. Spring создает прокси для бина (через AOP).
    
2. При вызове метода, помеченного `@Async`, вызов **не выполняется сразу**.
    
3. Он **упаковывается в задачу** и передается в `TaskExecutor` (пул потоков).
    
4. Вызывающий поток продолжает выполнение, не дожидаясь результата.    

**Как включить:**
```java
@Configuration
@EnableAsync  // включает обработку @Async
public class AsyncConfig {
    
    // Опционально: кастомизация пула потоков
    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("Async-");
        executor.initialize();
        return executor;
    }
}
```

**Пример использования:**
```java
@Service
public class NotificationService {
    
    @Async  // метод выполнится в отдельном потоке
    public void sendEmail(String to, String message) {
        // долгая операция (отправка email)
	        System.out.println("Sending email to " + to + " in thread: " 
		        + Thread.currentThread().getName());
    }
    
    @Async("customExecutor")  // указание конкретного пула
    public CompletableFuture<String> sendAsyncWithResult(String to) {
        // асинхронный метод с возвратом результата
        return CompletableFuture.completedFuture("OK");
    }
}
@RestController
public class UserController {
    @Autowired
    private NotificationService notificationService;
    
    @PostMapping("/register")
    public ResponseEntity<String> register(@RequestBody User user) {
        // Сохраняем пользователя (синхронно)
        userService.save(user);
        
        // Отправляем email асинхронно (не ждем)
        notificationService.sendEmail(user.getEmail(), "Welcome!");
        
        // Ответ возвращается сразу, не дожидаясь отправки email
        return ResponseEntity.ok("User registered");
    }
}
```

**Важные нюансы:**

|Проблема|Решение|
|---|---|
|`@Async` на `private` методе|Не работает (как и любой AOP через прокси)|
|`@Async` внутри того же класса (self-invocation)|Не работает — через `this.method()` прокси не вызывается|
|Исключения в `@Async` void методе|Не видны вызывающему коду (логируются)|
|Возвращаемый тип|`void` или `CompletableFuture<T>` (или `Future<T>`)|

**Возврат результата из `@Async`:**
```java
@Async
public CompletableFuture<User> findUserAsync(Long id) {
    User user = userRepository.findById(id).orElse(null);
    return CompletableFuture.completedFuture(user);
}
// Вызов
CompletableFuture<User> future = userService.findUserAsync(1L);
User user = future.get(5, TimeUnit.SECONDS);  // ждем результат с таймаутом
```

### 100. Как работает `TaskExecutor` в Spring? Пулы потоков по умолчанию

**Ответ:** `TaskExecutor` — это интерфейс Spring, абстрагирующий выполнение задач в пуле потоков. Он используется `@Async` для асинхронного выполнения.

**Иерархия:**
```text
TaskExecutor (Spring)
    ↓
AsyncTaskExecutor
    ↓
ThreadPoolTaskExecutor (самый распространенный)
```

**Реализации `TaskExecutor`:**

|Реализация|Описание|Когда использовать|
|---|---|---|
|`SyncTaskExecutor`|Выполняет задачу **синхронно** (в том же потоке)|Для тестов|
|`SimpleAsyncTaskExecutor`|Создает **новый поток** для каждой задачи (не пул)|Не рекомендуется для production|
|`ThreadPoolTaskExecutor`|Настоящий пул потоков с настройками|**Стандарт для production**|
|`ConcurrentTaskExecutor`|Адаптер для `java.util.concurrent.Executor`|Для совместимости|

**Пул потоков по умолчанию (если не задан свой):**
```java
// Spring Boot создает默认的 ThreadPoolTaskExecutor с настройками:
// - corePoolSize: 8 (количество ядер процессора)
// - maxPoolSize: Integer.MAX_VALUE (неограничен)
// - queueCapacity: Integer.MAX_VALUE (неограниченная очередь)
// - keepAliveSeconds: 60
// - threadNamePrefix: "task-"
```

**Настройка своего `TaskExecutor`:**
```java
@Configuration
@EnableAsync
public class AsyncConfig {
    
    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);    // базовое количество потоков
        executor.setMaxPoolSize(20);    // максимальное количество потоков
        executor.setQueueCapacity(100); //размер очереди перед созд.новых потоков
        executor.setKeepAliveSeconds(60);// время жизни лишних потоков
        executor.setThreadNamePrefix("MyPool-");
        executor.setRejectedExecutionHandler(
	        new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }
}
```

**Правила работы `ThreadPoolTaskExecutor` (важно!):**

|Ситуация|Действие|
|---|---|
|Потоков меньше `corePoolSize`|Создает **новый поток** для каждой задачи|
|Потоков больше `corePoolSize`, но очередь не заполнена|Кладет задачу в **очередь**|
|Очередь заполнена, потоков меньше `maxPoolSize`|Создает **новый поток**|
|Очередь заполнена, потоков == `maxPoolSize`|Срабатывает **RejectedExecutionHandler**|

**RejectedExecutionHandler (стратегии отклонения):**

|Стратегия|Поведение|
|---|---|
|`AbortPolicy` (по умолчанию)|Бросает `RejectedExecutionException`|
|`CallerRunsPolicy`|Выполняет задачу в **вызывающем потоке** (блокирует)|
|`DiscardPolicy`|Молча отбрасывает задачу|
|`DiscardOldestPolicy`|Удаляет самую старую задачу из очереди и пробует добавить новую|

**Мониторинг пула:**
```java
@Component
public class PoolMonitor {
    @Autowired
    private ThreadPoolTaskExecutor taskExecutor;
    
    @Scheduled(fixedDelay = 60000)
    public void monitor() {
        log.info("Active threads: {}, Pool size: {}, Queue size: {}",
            taskExecutor.getActiveCount(),
            taskExecutor.getPoolSize(),
            taskExecutor.getThreadPoolExecutor().getQueue().size());
    }
}
```

### 101. Что такое `@Scheduled`? Как включить (`@EnableScheduling`)

**Ответ:** `@Scheduled` — аннотация Spring для выполнения методов **по расписанию** (cron, фиксированная задержка, фиксированная частота).

**Как включить:**
```java
@Configuration
@EnableScheduling  // включает обработку @Scheduled
public class SchedulingConfig {
    
    // Опционально: настройка пула для шедулера
    @Bean
    public TaskScheduler taskScheduler() {
        ThreadPoolTaskScheduler scheduler = new ThreadPoolTaskScheduler();
        scheduler.setPoolSize(10);
        scheduler.setThreadNamePrefix("Scheduler-");
        return scheduler;
    }
}
```

**Пример использования:**
```java
@Component
public class ScheduledTasks {
    
    private static final Logger log = LoggerFactory.getLogger(ScheduledTasks.class);
    
    // Фиксированная задержка между окончанием предыдущего и началом следующего
    @Scheduled(fixedDelay = 5000)
    public void taskWithFixedDelay() {
        log.info("Fixed delay task - last execution finished 5 seconds ago");
    }
    
    // Фиксированная частота (не зависит от времени выполнения)
    @Scheduled(fixedRate = 5000)
    public void taskWithFixedRate() {
        log.info("Fixed rate task - started every 5 seconds");
    }
    
    // С initialDelay (первый запуск через 10 секунд)
    @Scheduled(initialDelay = 10000, fixedRate = 5000)
    public void taskWithInitialDelay() {
        log.info("Starts after 10 seconds, then every 5 seconds");
    }
    
    // Cron выражение
    @Scheduled(cron = "0 0 9 * * MON-FRI")  // каждый будний день в 9:00
    public void taskWithCron() {
        log.info("Weekday 9 AM task");
    }
    
    // Cron с часовым поясом
    @Scheduled(cron = "0 0 12 * * *", zone = "America/New_York")
    public void taskWithTimezone() {
        log.info("Noon New York time");
    }
}
```

**Важные нюансы:**
- Методы с `@Scheduled` должны быть **void** и не принимать параметров.
    
- Методы должны быть **public** (как и любой AOP через прокси).
    
- Шедулер работает в **одном потоке по умолчанию** (если не настроен `TaskScheduler` с пулом).
    
- Если задача выбросила исключение, следующая по расписанию **все равно выполнится**.

### 102. Поддерживаемые cron-выражения в Spring? А также `fixedDelay` vs `fixedRate`

**Ответ:** Spring поддерживает **6-польные cron-выражения** (секунды, минуты, часы, день месяца, месяц, день недели).

**Формат cron в Spring:**
```text
секунды минуты часы день-месяца месяц день-недели
|       |     |    |           |     |
|       |     |    |           |     +----- день недели (0-7, 0 и 7 = воскресенье)
|       |     |    |           +------- месяц (1-12 или JAN-DEC)
|       |     |    +------------------- день месяца (1-31)
|       |     +------------------------ час (0-23)
|       +------------------------------ минута (0-59)
+------------------------------------- секунда (0-59)
```

**Специальные символы:**

|Символ|Описание|Пример|
|---|---|---|
|`*`|Любое значение|`* * * * * *` — каждую секунду|
|`?`|Не указано (только для дня месяца и дня недели)|`0 0 12 ? * MON` — каждый понедельник в 12:00|
|`-`|Диапазон|`0 0 9-17 * * *` — каждый час с 9 до 17|
|`,`|Несколько значений|`0 0 9,13,17 * * *` — в 9, 13 и 17 часов|
|`/`|Шаг|`*/5 * * * * *` — каждые 5 секунд|
|`L`|Последний (день месяца или недели)|`0 0 12 L * *` — последний день месяца в 12:00|
|`#`|N-й день недели|`0 0 12 ? * MON#3` — третий понедельник месяца|

**Примеры cron-выражений:**

|Cron|Описание|
|---|---|
|`0 0 9 * * *`|Каждый день в 9:00|
|`0 */15 * * * *`|Каждые 15 минут|
|`0 0 12 * * MON`|Каждый понедельник в 12:00|
|`0 0 0 1 * *`|Первое число каждого месяца в полночь|
|`0 0 0 * * SUN`|Каждое воскресенье в полночь|
|`0 0 9-17 * * MON-FRI`|Каждый час с 9 до 17 по будням|

**`fixedDelay` vs `fixedRate` (важное различие):**

|Параметр|Значение|Поведение при долгой задаче|
|---|---|---|
|**`fixedDelay`**|Время **между окончанием** предыдущего выполнения и **началом** следующего|Если задача выполняется 10 секунд, а `fixedDelay=5000`, интервал между стартами = 15 секунд|
|**`fixedRate`**|Время **между началами** выполнения|Если задача выполняется 10 секунд, а `fixedRate=5000`, задачи будут **накладываться** (если пул позволяет) или ждать в очереди|

**Визуальное объяснение:**
```text
fixedDelay=5000 (задача выполняется 3 секунды):
[====TASK====].....[====TASK====].....[====TASK====]
0s            8s            16s           24s
(5s задержка после окончания)
fixedRate=5000 (задача выполняется 3 секунды):
[====TASK====][====TASK====][====TASK====]
0s            5s            10s           15s
(каждые 5 секунд, независимо от длительности)
```

**Проблема с `fixedRate`:** Если задача выполняется дольше, чем `fixedRate`, и пул只有一个 поток, задачи будут накапливаться в очереди. Решение — увеличить размер пула шедулера.

### 103. Что такое `ApplicationEventPublisher` и `ApplicationListener`? (Синхронные события по умолчанию)

**Ответ:** `ApplicationEventPublisher` и `ApplicationListener` — механизм **событий** в Spring, реализующий паттерн Observer (Наблюдатель) внутри приложения.

#### **Как работает (по умолчанию — синхронно):**
1. Компонент **публикует** событие через `ApplicationEventPublisher`.
    
2. Spring синхронно (в том же потоке) уведомляет все зарегистрированные **слушатели**.
    
3. Если слушатель бросает исключение, оно пробрасывается в публикующий код.    

#### **Определение события:**
```java
// Событие — это POJO, расширяющий ApplicationEvent (или просто Object с версии 4.2+)
public class UserRegisteredEvent extends ApplicationEvent {
    private final String email;
    private final LocalDateTime registeredAt;
    
    public UserRegisteredEvent(Object source, String email) {
        super(source);
        this.email = email;
        this.registeredAt = LocalDateTime.now();
    }
    
    // геттеры
}
// Альтернатива (без extends ApplicationEvent, работает с 4.2)
public class OrderCreatedEvent {
    private final Long orderId;
    // конструктор, геттеры
}
```

#### **Публикация события:**
```java
@Service
public class UserService {
    
    @Autowired
    private ApplicationEventPublisher eventPublisher;  // для публикации событий
    
    public void registerUser(String email) {
        // бизнес-логика
        System.out.println("User registered: " + email);
        
        // Публикуем событие (синхронно)
        UserRegisteredEvent event = new UserRegisteredEvent(this, email);
        eventPublisher.publishEvent(event);
        
        // Код продолжит выполнение ТОЛЬКО после того, как все слушатели обработают событие
    }
}
```

#### **Обработка события (слушатели):**
```java
// Способ 1: Реализация ApplicationListener
@Component
public class WelcomeEmailListener implements ApplicationListener<UserRegisteredEvent> {
    @Override
    public void onApplicationEvent(UserRegisteredEvent event) {
        System.out.println("Sending welcome email to: " + event.getEmail());
        // долгая операция (отправка email)
    }
}
// Способ 2: Аннотация @EventListener (рекомендуемый)
@Component
public class NotificationListener {
    
    @EventListener
    public void handleUserRegistered(UserRegisteredEvent event) {
        System.out.println("Sending welcome email to: " + event.getEmail());
    }
    
    @EventListener
    @Order(1)  // порядок выполнения, если несколько слушателей
    public void sendWelcomeSms(UserRegisteredEvent event) {
        System.out.println("Sending SMS to: " + event.getEmail());
    }
    
    // Условный слушатель (только для определенных значений)
    @EventListener(condition = "#event.email.endsWith('@company.com')")
    public void handleInternalUser(UserRegisteredEvent event) {
        System.out.println("Internal user special handling");
    }
}
```

#### **Преимущества синхронных событий:**
- Простота (нет проблем с многопоточностью)    
- Гарантия, что слушатели выполнятся до завершения транзакции (если событие опубликовано внутри `@Transactional`)    

#### **Недостатки:**
- Долгие слушатели блокируют основной поток    
- Исключение в слушателе может откатить транзакцию основного метода    

**Решение для долгих операций:** Асинхронные события (следующий вопрос).

### 104. Как сделать асинхронные события? <br>(`@Async` + `ApplicationEventPublisher`)

**Ответ:** Чтобы события обрабатывались **асинхронно** (в отдельном потоке), нужно добавить `@Async` к слушателю или настроить `ApplicationEventMulticaster` с пулом потоков.

#### **Способ 1: `@Async` на методе-слушателе** (простой):
```java
@Configuration
@EnableAsync  // обязательно включить
public class AsyncEventConfig { }
@Component
public class AsyncNotificationListener {
    
    @Async  // этот слушатель выполнится в отдельном потоке
    @EventListener
    public void handleUserRegistered(UserRegisteredEvent event) {
        // долгая операция — не блокирует основной поток
        sendWelcomeEmail(event.getEmail());
    }
    
    @Async("customExecutor")  // с указанием конкретного пула
    @EventListener
    public void sendSms(UserRegisteredEvent event) {
        // другой асинхронный обработчик
    }
}
```

#### **Способ 2: Кастомный `ApplicationEventMulticaster`** <br>(глобальная настройка):
```java
@Configuration
public class AsyncEventMulticasterConfig {
    
    @Bean(name = "applicationEventMulticaster")  // специальное имя!
    public ApplicationEventMulticaster applicationEventMulticaster() {
        SimpleApplicationEventMulticaster multicaster = 
	        new SimpleApplicationEventMulticaster();
        multicaster.setTaskExecutor(taskExecutor());  // задаем пул потоков
        return multicaster;
    }
    
    @Bean
    public ThreadPoolTaskExecutor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("Event-");
        executor.initialize();
        return executor;
    }
}
```

**Сравнение синхронных и асинхронных событий:**

|Характеристика|Синхронные события|Асинхронные события|
|---|---|---|
|**Блокировка основного потока**|Да (ждет всех слушателей)|Нет (основной поток продолжает)|
|**Исключения в слушателе**|Пробрасываются в публикующий код|Логируются, не влияют на публикацию|
|**Порядок выполнения**|Гарантирован|Не гарантирован|
|**Транзакционный контекст**|Наследуется|**Не наследуется** (важно!)|
|**Производительность**|Ниже при долгих операциях|Выше|

**Важный нюанс — транзакции и асинхронные события:**
```java
@Service
public class UserService {
    
    @Transactional
    public void registerUser(String email) {
        userRepository.save(new User(email));
        eventPublisher.publishEvent(new UserRegisteredEvent(this, email));
        // При синхронном событии: слушатель видит незакоммиченные данные 
	        // (flush еще не было)
        // При асинхронном событии: слушатель выполняется в другом потоке, 
        // может не увидеть данные, если транзакция еще не закоммичена!
    }
}
```

**Решение проблемы с транзакциями и асинхронными событиями:**
```java
@Component
public class TransactionAwareEventListener {
    
    @Async
    @EventListener
    @TransactionalEventListener(
	    phase = TransactionPhase.AFTER_COMMIT)  // после коммита!
    public void handleAfterCommit(UserRegisteredEvent event) {
        // Выполнится ТОЛЬКО после успешного коммита транзакции
        // Данные уже точно в БД
        sendEmail(event.getEmail());
    }
}
```

**Фазы `@TransactionalEventListener`:**

|Фаза|Описание|
|---|---|
|`BEFORE_COMMIT`|До коммита (по умолчанию)|
|`AFTER_COMMIT`|После успешного коммита (рекомендуется для асинхронной обработки)|
|`AFTER_ROLLBACK`|После отката транзакции|
|`AFTER_COMPLETION`|После завершения (commit или rollback)|

**Рекомендация:** Для долгих операций (email, SMS, генерация отчетов) 
используйте **асинхронные события** 
с `@Async` + `@TransactionalEventListener(phase = AFTER_COMMIT)`.  
Для быстрых операций (кэширование, логирование) подходят синхронные.
