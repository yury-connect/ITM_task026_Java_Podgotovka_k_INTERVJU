# Группа 9: Тестирование 
(Важно для Middle)

Тестирование — это то, что отличает профессионального Middle от Junior, который только пишет код. Отвечаю максимально точно и с аргументацией.

---
### 93. Какие типы тестов поддерживает Spring Boot? (`@SpringBootTest`, `@WebMvcTest`, `@DataJpaTest`)

**Ответ:** Spring Boot предоставляет несколько **срезов (slices)** тестирования, которые поднимают только определенную часть контекста для быстрого и целенаправленного тестирования.

**Основные аннотации для тестирования:**

|Аннотация|Что поднимает|Типичное использование|Скорость|
|---|---|---|---|
|**`@SpringBootTest`**|Полный контекст приложения|Интеграционные тесты, end-to-end|Медленный|
|**`@WebMvcTest`**|Только MVC слой (контроллеры, фильтры, интерсепторы)|Тестирование REST контроллеров, валидации, сериализации|Быстрый|
|**`@DataJpaTest`**|Только JPA слой (репозитории, EntityManager)|Тестирование репозиториев, запросов, маппинга|Быстрый|
|**`@JsonTest`**|Jackson сериализация/десериализация|Тестирование JSON маппинга|Очень быстрый|
|**`@WebFluxTest`**|Reactive WebFlux контроллеры|Тестирование WebFlux эндпоинтов|Быстрый|
|**`@RestClientTest`**|REST клиенты (RestTemplate, WebClient)|Тестирование клиентов к внешним API|Быстрый|
|**`@JdbcTest`**|Только JDBC (без JPA)|Тестирование нативных JDBC запросов|Быстрый|

**Примеры использования:**
```java
// Полный контекст (медленно, но полноценно)
@SpringBootTest
@AutoConfigureMockMvc
class FullIntegrationTest {
    @Autowired
    private MockMvc mockMvc;
    // тесты...
}

// Только MVC слой (быстро, контроллеры изолированы)
@WebMvcTest(UserController.class)
class UserControllerTest {
    @Autowired
    private MockMvc mockMvc;
    
    @MockBean
    private UserService userService; // мокаем зависимости контроллера
}

// Только JPA слой (быстро, репозитории с реальной БД)
@DataJpaTest
class UserRepositoryTest {
    @Autowired
    private UserRepository userRepository;
    
    @Autowired
    private TestEntityManager entityManager;
}
```

**Важный принцип:** Используйте минимально необходимый срез. `@SpringBootTest` только для самых важных сквозных тестов. Остальное — через специализированные аннотации.

### 94. Чем отличается `@MockBean` от `@Mock`? <br>(*Spring context* vs *plain Mockito*)

**Ответ:** Это принципиальное различие между **Mockito** (обычный мок) 
и **Spring Boot Test** (мок в контексте Spring).

| Характеристика              | `@Mock` (Mockito)                | `@MockBean` (Spring Boot Test)                          |
| --------------------------- | -------------------------------- | ------------------------------------------------------- |
| **Принадлежность**          | Только Mockito                   | Mockito + Spring Context                                |
| **Замена бина в контексте** | Нет                              | **Да** — заменяет реальный бин в ApplicationContext     |
| **Использование**           | В unit-тестах без Spring         | В интеграционных тестах с Spring контекстом             |
| **Внедрение в бины**        | Вручную через конструктор/сеттер | Автоматически, Spring внедрит мок вместо реального бина |
| **Сброс между тестами**     | Нужно управлять вручную          | Spring Boot автоматически сбрасывает                    |

**Пример с `@Mock` (unit-тест, нет Spring):**
```java
@ExtendWith(MockitoExtension.class)  // только Mockito, без Spring
class UserServiceUnitTest {
    
    @Mock
    private UserRepository userRepository;  // просто мок
    
    @InjectMocks
    private UserService userService;  // Mockito сам внедрит мок
    
    @Test
    void test() {
        when(userRepository.findById(1L)).thenReturn(Optional.of(new User()));
        // тестируем userService
    }
}
```

**Пример с `@MockBean` (интеграционный тест с Spring):**
```java
@WebMvcTest(UserController.class)  // Spring поднимает контекст
class UserControllerTest {
    
    @Autowired
    private MockMvc mockMvc;
    
    @MockBean  // заменяет реальный UserService в Spring контексте на мок
    private UserService userService;
    
    @Test
    void testController() throws Exception {
        when(userService.findById(1L)).thenReturn(new UserDto(1L, "John"));
        
        mockMvc.perform(get("/users/1"))
               .andExpect(status().isOk())
               .andExpect(jsonPath("$.name").value("John"));
    }
}
```

**Когда что использовать:**

|Сценарий|Рекомендация|
|---|---|
|Чистый unit-тест сервиса (без Spring)|`@Mock` + `@InjectMocks`|
|Тест контроллера с `@WebMvcTest`|`@MockBean` для зависимостей контроллера|
|Тест репозитория с `@DataJpaTest`|`@MockBean` не нужен (используем реальную БД)|
|Интеграционный тест с `@SpringBootTest`|`@MockBean` для подмены внешних сервисов|

**Важный нюанс:** `@MockBean` создает мок **внутри Spring контекста**. Если у вас есть два теста с разными моками одного и того же бина, Spring может переиспользовать контекст, и моки "загрязнятся". Используйте `@DirtiesContext` или разные конфигурации.

### 95. Как протестировать слой Controller <br>(`@WebMvcTest` + `MockMvc`)?

**Ответ:** `@WebMvcTest` + `MockMvc` — стандартный способ тестирования REST контроллеров в изоляции от сервисного слоя.

**Полный пример:**
```java
// поднимаем только UserController и MVC инфраструктуру
@WebMvcTest(UserController.class)  
class UserControllerTest {
    
    @Autowired
    private MockMvc mockMvc;  // основной инструмент для отправки запросов
    
    @MockBean
    private UserService userService;  // мокаем зависимость
    
    @Test
    void getUserById_ShouldReturnUser() throws Exception {
    
        // given
        UserDto userDto = new UserDto(1L, "John Doe", "john@example.com");
        when(userService.findById(1L)).thenReturn(userDto);
        
        // when & then
        mockMvc.perform(get("/api/users/1")
                .contentType(MediaType.APPLICATION_JSON))
                .andExpect(status().isOk())
                .andExpect(jsonPath("$.id").value(1))
                .andExpect(jsonPath("$.name").value("John Doe"))
                .andExpect(jsonPath("$.email").value("john@example.com"));
        
        verify(userService).findById(1L);
    }
    
    @Test
    void createUser_ShouldReturnCreated() throws Exception {
        // given
        UserCreateRequest request = 
			        new UserCreateRequest("Jane", "jane@example.com");
        UserDto created = new UserDto(2L, "Jane", "jane@example.com");
        
        when(userService.create(any())).thenReturn(created);
        
        // when & then
        mockMvc.perform(post("/api/users")
                .contentType(MediaType.APPLICATION_JSON)
                .content(asJsonString(request)))
                .andExpect(status().isCreated())
                .andExpect(header()
	                .string("Location", containsString("/api/users/2")))
                .andExpect(jsonPath("$.id").value(2));
    }
    
    @Test
    void getUserById_NotFound_ShouldReturn404() throws Exception {
        when(userService.findById(999L))
	        .thenThrow(new UserNotFoundException("User not found"));
        
        mockMvc.perform(get("/api/users/999"))
                .andExpect(status().isNotFound())
                .andExpect(jsonPath("$.message").value("User not found"));
    }
    
    @Test
    void createUser_InvalidInput_ShouldReturnBadRequest() throws Exception {
        UserCreateRequest invalid = new UserCreateRequest("", ""); //пустые поля
        
        mockMvc.perform(post("/api/users")
                .contentType(MediaType.APPLICATION_JSON)
                .content(asJsonString(invalid)))
                .andExpect(status().isBadRequest())
                .andExpect(jsonPath("$.errors").isArray());
    }
    
    private String asJsonString(Object obj) {
        try {
            return new ObjectMapper().writeValueAsString(obj);
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
}
```

**Основные методы `MockMvc`:**

|Метод|Описание|
|---|---|
|`perform()`|Выполнить запрос|
|`get()`, `post()`, `put()`, `delete()`, `patch()`|HTTP методы|
|`contentType()`|Установить Content-Type|
|`content()`|Установить тело запроса|
|`param()`|Добавить query параметр|
|`header()`|Добавить заголовок|
|`cookie()`|Добавить cookie|

**Основные `ResultMatcher` (проверки ответа):**

|Метод|Описание|
|---|---|
|`status().isOk()`|Проверка HTTP статуса|
|`jsonPath("$.field").value(expected)`|Проверка JSON поля|
|`jsonPath("$.field").exists()`|Проверка существования поля|
|`content().json(expectedJson)`|Сравнение всего JSON|
|`header().string("Location", expected)`|Проверка заголовка|

**Важный нюанс:** `@WebMvcTest` не поднимает сервисный слой, поэтому все зависимости контроллера должны быть замоканы через `@MockBean`. Реальная бизнес-логика не выполняется.

### 96. Как протестировать слой `Repository` <br>(`@DataJpaTest`, `TestEntityManager`, транзакционный откат)?

**Ответ:** `@DataJpaTest` — аннотация для тестирования JPA репозиториев 
с **реальной (*in-memory или тестовой*) базой данных**.

**Пример полного теста репозитория:**
```java
@DataJpaTest  // поднимает JPA слой, использует in-memory БД (H2 по умолчанию)
@AutoConfigureTestDatabase(replace = Replace.ANY)  // заменяет реальную БД на тестовую
class UserRepositoryTest {
    
    @Autowired
    private UserRepository userRepository;
    
    @Autowired
    // альтернатива EntityManager с удобными методами
    private TestEntityManager entityManager;  
    
    @Test
    void save_ShouldPersistUser() {
        // given
        User user = User.builder()
                .name("John Doe")
                .email("john@example.com")
                .build();
        
        // when
        User saved = userRepository.save(user);
        
        // then
        assertThat(saved.getId()).isNotNull();
        assertThat(saved.getName()).isEqualTo("John Doe");
        
        // проверяем, что реально сохранилось в БД
        User found = entityManager.find(User.class, saved.getId());
        assertThat(found).isNotNull();
        assertThat(found.getEmail()).isEqualTo("john@example.com");
    }
    
    @Test
    void findByName_ShouldReturnUser() {
        // given
        User user = User.builder().name("Jane").email("jane@example.com").build();
        entityManager.persistAndFlush(user);  // сохраняем и сбрасываем в БД
        
        // when
        Optional<User> found = userRepository.findByName("Jane");
        
        // then
        assertThat(found).isPresent();
        assertThat(found.get().getEmail()).isEqualTo("jane@example.com");
    }
    
    @Test
    void customQuery_ShouldReturnActiveUsers() {
        // given
        User active1 = User.builder().name("Active1").active(true).build();
        User active2 = User.builder().name("Active2").active(true).build();
        User inactive = User.builder().name("Inactive").active(false).build();
        
        entityManager.persist(active1);
        entityManager.persist(active2);
        entityManager.persist(inactive);
        entityManager.flush();
        
        // when
        List<User> activeUsers = userRepository.findAllActive();
        
        // then
        assertThat(activeUsers).hasSize(2);
        assertThat(activeUsers).extracting(User::getName)
                .containsExactlyInAnyOrder("Active1", "Active2");
    }
    
    @Test
    @Transactional
    @Rollback  // по умолчанию @DataJpaTest уже откатывает, но можно явно
    void transaction_ShouldBeRolledBack() {
        User user = User.builder().name("Temp").build();
        userRepository.save(user);
        
        assertThat(userRepository.count()).isEqualTo(1);
        // после теста БД откатится, запись не останется
    }
}
```

**Особенности `@DataJpaTest`:**

|Характеристика|Описание|
|---|---|
|**БД по умолчанию**|H2 (in-memory) — быстрая, изолированная|
|**Транзакции**|Каждый тест обернут в транзакцию, которая **откатывается** после теста|
|**Поднимаемые компоненты**|`DataSource`, `EntityManager`, `JpaRepository`, `TransactionManager`|
|**НЕ поднимаются**|`@Service`, `@Controller`, `@Component`|
|**Кэш**|Hibernate кэш очищается между тестами|

**TestEntityManager vs EntityManager:**

|Метод `TestEntityManager`|Аналог в `EntityManager`|Назначение|
|---|---|---|
|`persist(entity)`|`persist()`|Сохранить сущность|
|`persistAndFlush(entity)`|`persist()` + `flush()`|Сохранить и сбросить в БД|
|`find(class, id)`|`find()`|Найти по ID|
|`flush()`|`flush()`|Сбросить изменения в БД|
|`clear()`|`clear()`|Очистить persistence context|

**Использование реальной БД для тестов (не H2):**
```java
@DataJpaTest
@AutoConfigureTestDatabase(replace = Replace.NONE)  // не заменять реальную БД
@Testcontainers  // использование Testcontainers
class PostgresRepositoryTest {
    // тесты с реальным PostgreSQL в Docker
}
```

**Важный нюанс:** `@DataJpaTest` не поднимает `@Transactional` на сервисном слое. Если в репозитории есть методы, требующие открытой транзакции, они будут работать внутри тестовой транзакции.

### 97. Что такое `@TestConfiguration`?

**Ответ:** `@TestConfiguration` — аннотация для определения **специальных бинов только для тестов**, которые не должны попадать в основной контекст приложения.

**Когда использовать:**

- Нужен мок или заглушка для внешнего сервиса.
    
- Нужен альтернативный бин для тестов (например, другой `RestTemplate`).
    
- Нужно переопределить часть конфигурации для конкретного теста.    

**Пример 1: Дополнительные бины для теста:**
```java
@DataJpaTest
class UserRepositoryTest {
    
    @TestConfiguration
    static class TestConfig {
        @Bean
        public AuditListener auditListener() {
            return new AuditListener();  // дополнительный бин для теста
        }
        
        @Bean
        public DataSource testDataSource() {
            return new HikariDataSource();  // кастомный DataSource для тестов
        }
    }
    
    @Autowired
    private UserRepository userRepository;
    // тесты...
}
```

**Пример 2: Переопределение бина в `@SpringBootTest`:**
```java
@SpringBootTest
class UserServiceIntegrationTest {
    
    @TestConfiguration
    static class MockConfig {
        @Bean
        @Primary  // этот бин будет использоваться вместо реального
        public NotificationService mockNotificationService() {
            return mock(NotificationService.class);
        }
    }
    
    @Autowired
    private UserService userService;
    
    @Autowired
    private NotificationService notificationService;  // это мок!
    
    @Test
    void test() {
        // notificationService уже замокан
    }
}
```

**Пример 3: Внутри тестового класса (без отдельного файла):**
```java
@WebMvcTest(UserController.class)
class UserControllerTest {
    
    @TestConfiguration
    static class TestConfig {
        @Bean
        public UserService userService() {
            return mock(UserService.class);  // создаем мок вручную
        }
    }
    
    // можно не использовать @MockBean, так как UserService уже определен в TestConfiguration
    @Autowired
    private UserService userService;  // это мок из TestConfiguration
}
```

**Отличие `@TestConfiguration` от обычной `@Configuration`:**

|Характеристика|`@TestConfiguration`|`@Configuration`|
|---|---|---|
|**Применение**|Только в тестах|В основном коде|
|**Переопределение бинов**|Да (с `@Primary` или через `@MockBean`)|Нет (приведет к конфликту)|
|**Автоматическое сканирование**|Нет (должен быть явно использован)|Да (при component scan)|

**Важный нюанс:** Бины из `@TestConfiguration` **не переопределяют** бины из основной конфигурации, если не использовать `@Primary`. Они добавляются **дополнительно**. Для замены бина используйте `@MockBean` или `@Primary`.

### 98. Как протестировать асинхронные методы или таймеры?

**Ответ:** Тестирование асинхронного кода требует особого подхода, так как выполнение происходит в другом потоке. Spring Boot предоставляет несколько инструментов.

**Проблема:** Асинхронный метод возвращает управление до завершения работы, тест может завершиться раньше, чем выполнится асинхронная задача.

**Способы тестирования `@Async` методов:**

**Способ 1: `CountDownLatch` (ручное ожидание):**
```java
@Service
public class AsyncService {
    @Async
    public CompletableFuture<String> asyncMethod() {
        // ... асинхронная работа
        return CompletableFuture.completedFuture("result");
    }
}
@SpringBootTest
class AsyncServiceTest {
    
    @Autowired
    private AsyncService asyncService;
    
    @Test
    void testAsyncMethod() throws Exception {
        CompletableFuture<String> future = asyncService.asyncMethod();
        String result = future.get(5, TimeUnit.SECONDS);  // ждем с таймаутом
        assertThat(result).isEqualTo("result");
    }
}
```

**Способ 2: `Awaitility` (рекомендуемый для сложных сценариев):**
```xml
<dependency>
    <groupId>org.awaitility</groupId>
    <artifactId>awaitility</artifactId>
    <scope>test</scope>
</dependency>
```

```java
@Test
void testAsyncWithAwaitility() {
    asyncService.processAsync();
    
    await().atMost(5, SECONDS)
           .until(() -> asyncService.isProcessComplete());
}
```

**Способ 3: Синхронный прокси для тестов (замена Executor):**
```java
@TestConfiguration
public class TestAsyncConfig {
    
    @Bean
    @Primary
    public TaskExecutor syncTaskExecutor() {
        return new SyncTaskExecutor();  // выполняет задачи синхронно в том же потоке
    }
}
@SpringBootTest
@Import(TestAsyncConfig.class)
class AsyncServiceTestWithSyncExecutor {
    // @Async методы будут выполняться синхронно
}
```

**Тестирование `@Scheduled` (таймеров):**

**Способ 1: Ожидание срабатывания:**
```java
@SpringBootTest
class ScheduledServiceTest {
    
    @Autowired
    private ScheduledService scheduledService;
    
    @Test
    void testScheduledTask() throws Exception {
        // Ждем, пока задача выполнится (например, каждые 5 секунд)
        Thread.sleep(10_000);  // плохо, но просто
        
        assertThat(scheduledService.getCounter()).isGreaterThan(0);
    }
}
```

**Способ 2: Отключение шедулера и ручной вызов (рекомендуемый):**
```java
@SpringBootTest
@EnableScheduling  // обычно включен, но для тестов можно выключить
class ScheduledServiceTest {
    
    @Autowired
    private ScheduledService scheduledService;
    
    @Test
    void testTaskLogic() {
        // Вызываем метод напрямую, без ожидания шедулера
        scheduledService.performCleanup();
        
        assertThat(scheduledService.getCleanedCount()).isEqualTo(5);
    }
}
```

**Способ 3: Использование `Awaitility` с шедулером:**
```java
@Test
void testScheduledTaskWithAwaitility() {
    await().atMost(10, SECONDS)
           .pollInterval(1, SECONDS)
           .until(() -> scheduledService.getLastExecutionTime() != null);
}
```

**Общие рекомендации:**

|Сценарий|Рекомендация|
|---|---|
|`@Async` с возвращаемым значением|Используйте `CompletableFuture.get(timeout)`|
|`@Async` void метод|Используйте `CountDownLatch` или `Awaitility`|
|`@Scheduled`|**Не тестируйте сам шедулер** — тестируйте логику метода, вызывая его напрямую|
|Асинхронные события|Используйте `ApplicationEvent` + `@EventListener` + `Awaitility`|

**Важный нюанс:** В тестах с `@Async` нужно добавлять `@EnableAsync` на тестовый конфигурационный класс или использовать `@SpringBootTest`, который подхватывает основную конфигурацию. Также не забывайте про `ThreadPoolTaskExecutor` — в тестах он может быть не инициализирован.
