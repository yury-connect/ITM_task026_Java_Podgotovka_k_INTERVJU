# Как мы можем пофиксить dead lock

---
## Краткий ответ (*для собеседования*)

**Deadlock** возникает, когда выполняются **4 условия Коффмана**: 
- взаимное исключение, 
- удержание и ожидание, 
- отсутствие вытеснения и 
- циклическое ожидание.

#### Чтобы его "пофиксить", нужно **нарушить хотя бы 1 из этих условий**:
	
1. **Нарушить удержание и ожидание** — захватывать все нужные ресурсы атомарно (*одним блоком `synchronized` или через `tryLock` с таймаутом*).
    
2. **Нарушить циклическое ожидание** — ввести глобальный порядок захвата блокировок (*всегда захватываем мониторы в строго определенной последовательности, например, по `hashCode` или по имени*).
    
3. **Нарушить отсутствие вытеснения** — использовать `ReentrantLock` с методом `tryLock(timeout)`, чтобы поток мог отказаться от блокировки и освободить уже захваченные ресурсы.
    
4. **Нарушить взаимное исключение** — где возможно, использовать конкурентные структуры данных без блокировок (`ConcurrentHashMap`, `Atomic*`).

**Главный инструмент в продакшене** — это детекция через дампы потоков (`jstack`, `VisualVM`) и исправление порядка захвата блокировок.

## Глубокий разбор (*чтобы было максимально понятно*)

### 1. Что такое Deadlock простыми словами

Представь двух людей за столом. У одного нож, у другого вилка. Каждый хочет есть, но для этого нужны и нож, и вилка. Первый не отдает нож, пока не получит вилку. Второй не отдает вилку, пока не получит нож. В итоге — оба голодные и зависли навечно.

В Java это выглядит так:
```java
class Friend {
    private final String name;
    public Friend(String name) { this.name = name; }
    public String getName() { return name; }
	
    public synchronized void bow(Friend bower) {
        System.out.format(
	        "%s: %s поклонился мне!%n", this.name, bower.getName());
        bower.bowBack(this);
    }
	
    public synchronized void bowBack(Friend bower) {
        System.out.format(
	        "%s: %s поклонился в ответ!%n", this.name, bower.getName());
    }
}

// В двух потоках:
final Friend alphonse = new Friend("Alphonse");
final Friend gaston = new Friend("Gaston");

new Thread(() -> alphonse.bow(gaston)).start();
new Thread(() -> gaston.bow(alphonse)).start();
```

Поток 1 захватил монитор `alphonse` и ждет монитор `gaston`.  
Поток 2 захватил монитор `gaston` и ждет монитор `alphonse`.  
Циклическое ожидание налицо.

### 2. Четыре условия Коффмана (классика)

Для deadlock'а нужны все 4 одновременно:
	
1. **Взаимное исключение (Mutual Exclusion)** — ресурс может удерживаться только одним потоком.
    
2. **Удержание и ожидание (Hold and Wait)** — поток удерживает один ресурс и ждет другой.
    
3. **Отсутствие вытеснения (No Preemption)** — ресурс освобождается только самим потоком, никто не может его отобрать.
    
4. **Циклическое ожидание (Circular Wait)** — цепочка потоков, где каждый ждет ресурс, занятый следующим.

Чтобы пофиксить — **ломаем любое из них**.

---
### 3. Практические способы исправления

#### `Способ 1`. Глобальный порядок захвата (ломаем циклическое ожидание) — **САМЫЙ ЧАСТЫЙ В ПРОДАКШЕНЕ**

Если все потоки захватывают блокировки в строго одинаковом порядке — цикла не будет.

**Пример:**
```java
// Вместо случайного порядка — сортируем по hashCode
public void transfer(Account from, Account to, int amount) {
    Account first = from.getId() < to.getId() ? from : to;
    Account second = from.getId() < to.getId() ? to : from;
    
    synchronized (first) {
        synchronized (second) {
            // безопасный перевод
        }
    }
}
```

Если id не уникальны, можно использовать `System.identityHashCode()` или внешний порядковый номер.

---
#### `Способ 2`. Атомарный захват всех ресурсов (ломаем "удержание и ожидание")

Захватываем все нужные блокировки **одним транзакционным действием**.

**`Вариант А`: Вложенные `synchronized`** (но осторожно, если их много):
```java
synchronized (lock1) {
    synchronized (lock2) {
        // работа
    }
}
```
Это не гарантирует атомарность захвата, но если порядок фиксирован — работает.

**`Вариант Б` (*надёжнее*):** Используем `ReentrantLock` и захватываем через `tryLock` все сразу:
```java
ReentrantLock lock1 = new ReentrantLock();
ReentrantLock lock2 = new ReentrantLock();

public void safeMethod() {
    while (true) {
        if (lock1.tryLock()) {
            try {
                if (lock2.tryLock()) {
                    try {
                        // безопасная работа
                        return;
                    } finally {
                        lock2.unlock();
                    }
                }
            } finally {
                lock1.unlock();
            }
        }
        // небольшая пауза перед повторной попыткой
        Thread.sleep(10);
    }
}
```
Здесь мы либо захватываем оба, либо освобождаем всё и пробуем снова.

---
#### `Способ 3`. Использование таймаутов (*ломаем "отсутствие вытеснения"*)

`ReentrantLock.tryLock(long timeout, TimeUnit unit)` позволяет подождать ограниченное время. Если не дождались — освобождаем всё и откатываем операцию.
```java
if (lock1.tryLock(1, TimeUnit.SECONDS)) {
    try {
        if (lock2.tryLock(1, TimeUnit.SECONDS)) {
            try {
                // работа
            } finally { lock2.unlock(); }
        }
    } finally { lock1.unlock(); }
} else {
    // откат, логирование, повторная попытка
}
```
Это не предотвращает deadlock, но позволяет из него **выбраться** — система не зависает навечно.

---
#### `Способ 4`. Отказ от блокировок (*ломаем "взаимное исключение"*)

Используем неблокирующие структуры данных:	
- `ConcurrentHashMap` вместо `HashMap` + `synchronized`    
- `AtomicInteger`, `AtomicReference` для счётчиков и ссылок    
- `ConcurrentLinkedQueue` для очередей    
- `StampedLock` с оптимистичным чтением

Или используем **акторы** (например, Akka) — каждый объект владеет своим состоянием и общается через сообщения, а не через общие блокировки.

---
#### `Способ 5`. Уменьшение области видимости блокировки

Чем меньше кода внутри `synchronized` — тем меньше шансов, что один поток удержит блокировку слишком долго, пока ждет другой ресурс.

**Плохо:**
```java
synchronized (this) {
    // долгий расчет
    // вызов внешнего сервиса
    // запись в БД
}
```

**Хорошо:**
```java
// захватываем только для чтения/записи критической переменной
synchronized (this) {
    localCopy = this.sharedData;
}
// долгие операции делаем вне блокировки
result = process(localCopy);
synchronized (this) {
    this.sharedData = result;
}
```

### 4. Диагностика (*как найти deadlock в работающей системе*)

1. **jstack** — получить дамп потоков:
```bash
jps -l   # найти PID
jstack <PID>
```
  В выводе будет четко написано: `Found one Java-level deadlock:` и показан цикл.
    
2. **VisualVM / JConsole** — графические тулзы, показывают заблокир-е мониторы.
    
3. **ThreadMXBean** программно:
```java
ThreadMXBean bean = ManagementFactory.getThreadMXBean();
long[] threadIds = bean.findDeadlockedThreads();
if (threadIds != null) {
    // логируем информацию о deadlock'е
}
```
 Можно настроить мониторинг и алертинг.

---
### 5. Продвинутые техники (архитектурные)

- **Иерархические блокировки** — назначаем каждому ресурсу уровень (level). Поток может захватывать ресурсы только в порядке возрастания уровня.
    
- **Блокировка с ранжированием** — как в БД: сначала таблица A, потом B, никогда наоборот.
    
- **Deadlock detection + восстановление** — разрешаем deadlock'ам случаться, но детектируем их и прерываем один из потоков (через `Thread.interrupt()` при использовании `ReentrantLock.lockInterruptibly()`).
    
- **Использование `Semaphore`** вместо `synchronized` — можно настроить количество разрешений.

---
### 6. Что НЕЛЬЗЯ делать (антипаттерны)

- ❌ `Thread.stop()` — устарел и опасен (ломает состояние объекта).
    
- ❌ Просто увеличивать таймауты — это маскировка, а не решение.
    
- ❌ Использовать `synchronized(this)` в методах класса с публичными методами — рискуете, что внешний код тоже захватит ваш монитор.
    
- ❌ Игнорировать дампы потоков — если deadlock происходит раз в месяц, надо разбираться, а не перезапускать сервер.

---
### 7. Итоговый чек-лист при подозрении на deadlock

1. Сделать дамп потоков → найти циклическое ожидание.
    
2. Посмотреть, какие блокировки захвачены и в каком порядке.
    
3. Ввести глобальный порядок захвата (самый простой и надёжный фикс).
    
4. Если порядок невозможно определить — использовать `tryLock` с таймаутом и откатом.
    
5. Переписать код на неблокирующие структуры, где это уместно.
    
6. Добавить мониторинг (алерт при `findDeadlockedThreads()`).
