Сначала **краткий ответ для интервью**, потом **глубокое погружение**, а в конце — **таблица принятия решений** (когда что использовать).

---
## Краткий ответ (*для собеседования*)

**`synchronized`** — это встроенная в JVM языковая конструкция.  
Простая, надежная, с оптимизациями на уровне JVM (*биасинг, легковесные блокировки*), но жесткая: нельзя прервать ожидание, нельзя проверить, занят ли монитор, нет таймаутов, все блокировки неявные.

**`ReentrantLock`** — это `API`-шная реализация из `java.util.concurrent.locks`.  
Гибкая: есть `tryLock()` с таймаутом, `lockInterruptibly()`, честный режим (fairness), возможность создать несколько условий (`Condition`) на один лок.  
Цена — немного больший overhead и необходимость явно вызывать `unlock()` в `finally`.

#### **Главные различия:**
	
1. **Гибкость:** `ReentrantLock` богаче по функционалу.
    
2. **Производительность:** раньше `ReentrantLock` был быстрее при высокой конкуренции, но с Java 6+ и улучшениями `synchronized` разница практически стерлась (часто `synchronized` даже быстрее за счет биасинга).
    
3. **Безопасность:** `synchronized` безопаснее, потому что блокировка автоматически освобождается при выходе из блока. В `ReentrantLock` можно забыть вызвать `unlock()` и получить дедлок или трудноуловимый баг.
    
4. **Условия:** `ReentrantLock` позволяет несколько `Condition` на один лок (аналогично `wait/notify`, но более гибко).

**Мое правило:** если хватает `synchronized` — использую его. `ReentrantLock` беру только когда нужен таймаут, прерывание или несколько условий ожидания.

# Глубокое погружение

### 1. Как они устроены внутри (важно для понимания)

#### `synchronized` — мониторы внутри объекта

Каждый объект в Java имеет **монитор** (в заголовке объекта есть поле под это). Когда поток входит в `synchronized`-блок, он пытается захватить этот монитор. JVM использует три уровня оптимизации:
	
1. **Biased Locking (*смещенная блокировка*)** — если лок захватывается всегда одним потоком, JVM просто "*запоминает*" этот поток и проверка стоит почти бесплатно (*один сравнение*).
    
2. **Lightweight Locking (*легковесная блокировка*)** — если несколько потоков конкурируют, но без реального ожидания (спин-локинг), JVM использует CAS-операции в заголовке объекта.
    
3. **Heavyweight Locking (*тяжелая блокировка*)** — если конкуренция высокая, JVM переключается на системный мьютекс (функции ядра) — это дорого.    

Всё это происходит **автоматически**, без твоего участия. JVM сама эскалирует уровень блокировки.

#### `ReentrantLock` — построен на `AbstractQueuedSynchronizer` (AQS)
	
`ReentrantLock` внутри использует **AQS** — фреймворк для создания блокировок и синхронизаторов. Там хранится:
	
- `volatile int state` — счетчик захватов (для реентерабельности, т.е. повторного входа)
    
- `Queue` ожидающих потоков (CLH-очередь)
    
- Все операции строятся на **CAS** (Compare-And-Swap) без использования `synchronized` внутри себя (кроме низкоуровневых `Unsafe`).

`ReentrantLock` также может работать в режиме **fair** (честный) — тогда потоки обслуживаются в порядке очереди, или **unfair** (нечестный, по умолчанию) — может быть "перехват" блокировки новым потоком.

### 2. Детальное сравнение по параметрам

| Параметр                         | `synchronized`                              | `ReentrantLock`                                              |
| -------------------------------- | ------------------------------------------- | ------------------------------------------------------------ |
| **Освобождение блокировки**      | Автоматическое (при выходе из блока/метода) | Явное: `lock.unlock()` <br>в `finally`                       |
| **Таймаут при захвате**          | ❌ Нет                                       | ✅ `tryLock(1, TimeUnit.SECONDS)`                             |
| **Прерывание ожидания**          | ❌ Нельзя прервать поток, ждущий монитор     | ✅ `lock.lockInterruptibly()` <br> — можно прервать           |
| **Проверка занятости**           | ❌ Нет                                       | ✅ `isLocked()`, `tryLock()`                                  |
| **Условия (Condition)**          | Только один (через `wait/notify`)           | Несколько `Condition` на один лок                            |
| **Честность (Fairness)**         | ❌ Нет, полностью нечестная                  | ✅ Можно включить через конструктор `new ReentrantLock(true)` |
| **Производительность (Java 8+)** | Часто быстрее благодаря биасингу            | Чуть выше overhead из-за CAS и AQS                           |
| **Глубокая диагностика**         | Ограничена (`Thread.holdsLock()`)           | Есть `getHoldCount()`,  <br>`hasQueuedThreads()` и др.       |
| **ReadWriteLock**                | ❌ Нет                                       | ✅ Есть отдельный  <br>`ReentrantReadWriteLock`               |
| **Риск ошибки**                  | Низкий (авто-освобождение)                  | Высокий (забыли `unlock()` → дедлок)                         |

### 3. Ключевые фичи `ReentrantLock` с примерами

#### А) `tryLock()` с таймаутом — спасение от дедлоков
```java
ReentrantLock lock = new ReentrantLock();
if (lock.tryLock(100, TimeUnit.MILLISECONDS)) {
    try {
        // делаем работу
    } finally {
        lock.unlock();
    }
} else {
    // не получили лок — делаем что-то другое
    System.out.println("Не удалось захватить блокировку");
}
```
Это позволяет не зависать навсегда.

#### Б) Прерываемое ожидание — `lockInterruptibly()`
```java
ReentrantLock lock = new ReentrantLock();
try {
    lock.lockInterruptibly(); // если прервут — выбросит InterruptedException
    try {
        // работа
    } finally {
        lock.unlock();
    }
} catch (InterruptedException e) {
    // поток прервали во время ожидания
    Thread.currentThread().interrupt(); // восстанавливаем статус
}
```
Полезно в долгих операциях, которые надо отменять.

#### В) Несколько `Condition` — вместо `wait/notify`

Это суперфича! Один лок, но несколько очередей ожидания.
```java
class BoundedBuffer {
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();
    private final Object[] items = new Object[100];
    private int putptr, takeptr, count;
    public void put(Object x) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) 
                notFull.await(); // ждем, пока не освободится место
            items[putptr] = x;
            if (++putptr == items.length) putptr = 0;
            count++;
            notEmpty.signal(); // сигналим, что элемент появился
        } finally {
            lock.unlock();
        }
    }
    public Object take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) 
                notEmpty.await();
            Object x = items[takeptr];
            if (++takeptr == items.length) takeptr = 0;
            count--;
            notFull.signal();
            return x;
        } finally {
            lock.unlock();
        }
    }
}
```

В `synchronized` для этого пришлось бы использовать один монитор и `wait/notify` без разделения на "полное" и "пустое" — были бы лишние пробуждения (spurious wakeups), и код был бы менее эффективным.

#### Г) Честный режим (fairness)
```java
ReentrantLock fairLock = new ReentrantLock(true);
```
В честном режиме лок достанется самому долгоожидающему потоку. Это предотвращает "голодание" (starvation), но снижает пропускную способность примерно на порядок. Используй, только если нужна строгая справедливость.

---
### 4. Чего нет у `ReentrantLock`, но есть у `synchronized`

1. **Автоматическая очистка при исключении** — в `synchronized` если внутри блока вылетело исключение, блокировка гарантированно освободится. В `ReentrantLock` если ты забыл `finally` — блокировка останется запертой навсегда.
    
2. **Интеграция с JVM-оптимизациями** — `synchronized` может быть "урезан" компилятором (lock elision), если JVM докажет, что объект не используется несколькими потоками. `ReentrantLock` так не умеет.
    
3. **Простота чтения** — код с `synchronized` чище:
```java
synchronized(lock) {
    // код
}
```
против
```java
lock.lock();
try {
    // код
} finally {
    lock.unlock();
}
```
    
4. **Монитор одного объекта** — `synchronized` можно повесить на любой объект. `ReentrantLock` — только на экземпляре самого лока.

### 5. Производительность (спорный момент)

**Исторически:**
	
- **Java 1.4 — 5:** `ReentrantLock` был сильно быстрее.
    
- **Java 6 — 8:** JVM значительно оптимизировала `synchronized` (biased locking, lock coarsening). Разница почти стерлась.
    
- **Java 9+:** В некоторых сценариях `synchronized` даже быстрее, потому что он полностью внутри JVM, без вызовов через `Unsafe` (хотя и там CAS).

**Мой совет:** не выбирай по производительности, пока не измерил в своём конкретном сценарии (JMH-тесты). В 95% случаев разница не имеет значения.

### 6. Где что применять — матрица решений

|Сценарий|Выбор|
|---|---|
|Простая синхронизация метода или короткого блока|✅ `synchronized`|
|Нужен таймаут при захвате|✅ `ReentrantLock`|
|Нужно прервать поток, ожидающий блокировку|✅ `ReentrantLock`|
|Несколько очередей ожидания (producer-consumer)|✅ `ReentrantLock` + `Condition`|
|Нужна читаемость и минимальный риск ошибок|✅ `synchronized`|
|Строгая справедливость (FIFO)|✅ `ReentrantLock(true)`|
|Чтение и запись разделены (много читателей, мало писателей)|✅ `ReentrantReadWriteLock` (не сравнивали, но упомянуть)|
|Внутренняя синхронизация в коллекциях|`synchronized` (как у `Vector`, `Hashtable`) или лучше `Concurrent*`|
|Высоконагруженный трейдинговый движок с микросекундными требованиями|Пиши lock-free структуры, но если выбирать — тестируй оба|

### 7. Частые ошибки с `ReentrantLock`
```java
// ❌ ОШИБКА 1: забыли unlock
lock.lock();
if (someCondition) {
    return; // выход без unlock!
}
lock.unlock();

// ❌ ОШИБКА 2: unlock не в finally
try {
    lock.lock(); // lock до try — правильно
    // работа
} catch (Exception e) {
    // ...
} finally {
    lock.unlock();
}

// ❌ ОШИБКА 3: повторный lock без счетчика
lock.lock();
lock.lock(); // второй раз — счетчик становится 2
lock.unlock(); // один раз — блокировка осталась!
// Нужно столько же unlock, сколько lock
```

**Правильный паттерн:**
```java
Lock lock = new ReentrantLock();
lock.lock(); // перед try
try {
    // critical section
} finally {
    lock.unlock(); // обязательно
}
```

### 8. Чего нет в обоих, но есть в `StampedLock` (это бонусный балл на собеседовании)

`StampedLock` — это улучшенная версия для чтения/записи, появившаяся в Java 8. У неё есть **оптимистичное чтение**:
```java
StampedLock sl = new StampedLock();
long stamp = sl.tryOptimisticRead();
// читаем данные
if (!sl.validate(stamp)) { // проверяем, не изменились ли
    stamp = sl.readLock(); // если изменились — переходим к пессимистичному чтению
    try {
        // читаем заново
    } finally {
        sl.unlockRead(stamp);
    }
}
```
Это дает максимальную производительность, когда данных много, а записей мало.

### 9. Итоговый ответ для собеседования (расширенный)

> "В повседневной работе я начинаю с `synchronized`. Он проще, безопаснее и хорошо оптимизируется JVM. Перехожу на `ReentrantLock` только тогда, когда появляется конкретная потребность:
> 	
> - захват с таймаутом (чтобы не зависать),
>     
> - прерывание ожидания (через `lockInterruptibly()`),
>     
> - несколько условий ожидания (через `Condition`),
>     
> - или честный режим.
> 
> При этом я всегда помню, что `ReentrantLock` требует ручного освобождения в `finally` — это частая причина багов. Также важно знать, что `ReentrantLock` реентерабельный (поток может захватывать его несколько раз, но счетчик захватов обязан быть сброшен таким же числом разблокировок).
> 
> По производительности в современных JVM разница незначительна, поэтому я никогда не выбираю один вместо другого без предварительного бенчмаркинга. В сложных сценариях чтения-записи я рассматриваю `ReentrantReadWriteLock` или даже `StampedLock` для оптимистичного чтения."

### 10. Вопрос-ловушка, который часто задают

**Вопрос:** _"Может ли `synchronized` быть прерван, если поток ждет блокировку?"_

**Ответ:** Нет. Поток, который пытается войти в `synchronized`-блок и не может захватить монитор, находится в состоянии `BLOCKED`. Его **нельзя прервать** через `Thread.interrupt()` — прерывание будет проигнорировано, пока он не получит блокировку.

В отличие от `ReentrantLock.lockInterruptibly()`, где поток переходит в состояние `WAITING` и реагирует на прерывание. Это важное архитектурное отличие.
