**Кратко (для памяти):**
> [!IMPORTANT]
#### **4 стандартных способа фикса:**
1. **Перезапуск** — временно, но не решение.
2. **jstack** — найти виновника (*какие потоки и какие locks*).
3. **Изменить порядок захвата мониторов** — главное правило: <u>всегда в одном и том же порядке</u>.
4. **Использовать `tryLock()` с таймаутом** — отказ от бесконечного ожидания.

---
### **Структура** (*что делать по шагам*):

|Шаг|Действие|Инструмент|
|---|---|---|
|1|Обнаружить|`jps` → `jstack <pid>` или `ThreadMXBean`|
|2|Локализовать|Найти потоки в состоянии `BLOCKED` с циклическими зависимостями|
|3|Исправить код|Выбрать один из вариантов ниже|
|4|Проверить|Нагрузочное тестирование + мониторинг|

---
### **Варианты исправления** (*от простого к сложному*):

- **Вариант А:** Всегда захв-ть ресурсы в **одинаковом порядке** (*глобальный порядок*).
    
- **Вариант Б:** Использовать `ReentrantLock.tryLock(long timeout, TimeUnit)` вместо `synchronized`.
    
- **Вариант В:** Уменьшить границы синхронизации (*критические секции короче*).
    
- **Вариант Г:** Использовать высокоуровневые `ConcurrentHashMap`, `BlockingQueue` — они уже внутренне оптимизированы.

---
### КОНСПЕКТ: Фикс deadlock’а

**1. Обнаружение (в продакшене)**
```

```
bash

jps                     # список Java-процессов
jstack <PID>            # смотрим потоки BLOCKED

- Ищем циклические зависимости: `Thread-1` ждёт lock B, `Thread-2` ждёт lock A.
    

**2. Исправление в коде (4 варианта)**

|Вариант|Суть|Когда использовать|
|---|---|---|
|**Порядок**|Всегда захватывать мониторы в одном порядке|Если можно договориться в команде|
|**tryLock()**|`if (lock.tryLock(100, ms)) { ... } else { откат }`|Когда критично не висеть|
|**Уменьшить синхронизацию**|Выносить из `synchronized` всё, что не нужно|Всегда полезно|
|**Concurrent-классы**|`ConcurrentHashMap`, `BlockingQueue` вместо ручных локов|Для типовых структур данных|

**3. Пример (порядок — самый частый фикс)**

Было (deadlock):

java

// Поток 1: lockA → lockB
// Поток 2: lockB → lockA

// Оба потока: lockA → lockB (единый порядок)

**4. tryLock() — шаблон**

java

ReentrantLock lock1 = new ReentrantLock();
if (lock1.tryLock(500, TimeUnit.MILLISECONDS)) {
    try {
        // работа
    } finally { lock1.unlock(); }
} else {
    // делаем что-то другое или логируем
}

**5. Мониторинг после фикса**

- Прогоняем нагрузку с несколькими потоками.
    
- Смотрим через `jstack` — нет ли `BLOCKED` цепочек.
    

---

**Итоговая памятка (запомнить):**

> Deadlock ≠ баг JVM. Это всегда **ошибка проектирования**.  
> Главное оружие — **порядок захвата** и **таймауты**.  
> В 80% случаев помогает просто переставить `synchronized` в одном направлении.
