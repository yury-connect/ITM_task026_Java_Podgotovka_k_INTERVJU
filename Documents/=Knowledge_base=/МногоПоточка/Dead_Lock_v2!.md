# Как мы можем пофиксить dead lock

---
**дедлок (deadlock)** — это ситуация «взаимного тупика».

> **Пример:** Поток №1 заблокировал _Товар А_ и хочет обновить _Товар Б_. В эту же секунду Поток №2 заблокировал _Товар Б_ и пытается обновить _Товар А_. Они заблокировали друг друга и зависли навсегда.

Вот **4 основных способа** его исправить и предотвратить:

### 1. Одинаковый **порядок** (*Сортировка `ID`*)
Самый частый и надежный способ. Если твоим транзакциям нужно обновить несколько товаров за раз, **всегда сортируй их ID по возрастанию** перед отправкой в базу данных.
	
- **Как это лечит проблему:** Оба потока попытаются сначала заблокировать товар с меньшим ID (например, №5), а уже потом №12. Поток №2 просто подождет на первом шаге, вместо того чтобы устроить тупик.

### 2. Делай транзакции **максимально короткими**
Чем дольше открыта транзакция (аннотация `@Transactional`), тем дольше база держит блокировки на строках.
	
- Убери из транзакций любые запросы во внешние API, тяжелые расчеты в памяти и долгие конвертации.
    
- Схема должна быть быстрой: зашел в базу -> мгновенно обновил -> вышел (транзакция закрылась, блокировки снялись).

### 3. Настройка **таймаутов** (*Lock Timeout*)
По умолчанию база данных может ждать освобождения строки очень долго.
	
- Установи ограничение по времени (например, 2-3 секунды). Если за это время блокировка не снялась, транзакция просто прервется по таймауту и откатится (Rollback), освободив ресурсы для других.

### 4. Оптимистичная блокировка (`@Version`)
Вместо жесткой блокировки строк в базе данных (`SELECT FOR UPDATE`, которая и вызывает дедлоки), используй оптимист. блокировку через аннотацию `@Version` в JPA.
	
- База не будет блокировать строки при чтении. При записи Hibernate просто проверит версию строки. Если её кто-то изменил параллельно, транзакция упадет с понятной ошибкой `OptimisticLockException`, которую можно просто попробовать запустить повторно (retry).
