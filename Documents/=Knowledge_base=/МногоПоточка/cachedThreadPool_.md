# Чем может быть опасен **`Cached Thread Pool`**?

---
Java Concurrency. Пул потоков `Executors.newCachedThreadPool()` на первый взгляд кажется идеальным: он автоматически масштабируется, создаёт новые потоки, когда они нужны, и утилизирует их, если они простаивают (по умолчанию 60 секунд).

Однако в enterprise-системах (особенно под высокой нагрузкой) использовать его «из коробки» **смертельно опасно**. Вот почему:

### 1. Риск **OutOfMemoryError** (*OOM*)
Это главная и самая страшная угроза.
	
- **Как он устроен внутри:** `cachedThreadPool` использует внутри себя `SynchronousQueue` в качестве очереди задач. Эта очередь имеет нулевую емкость — она не может хранить задачи, а сразу передает их потокам. Если все потоки заняты, пул **создает новый поток**.
    
- **В чём опасность:** Максимальное количество потоков (`maximumPoolSize`) у него по умолчанию равно **`Integer.MAX_VALUE`** (более 2 миллиардов!).
    
- **Сценарий катастрофы:** Если на твой сервис пойдет резкий лавинообразный поток запросов (DDOS-атака или просто пиковая нагрузка), пул начнет бесконтрольно плодить операционные потоки Java. Каждый поток требует выделения памяти под свой стек (по умолчанию около 1 МБ в 64-битной JVM). Память на сервере мгновенно закончится, и приложение упадет с критической ошибкой `java.lang.OutOfMemoryError: unable to create new native thread`.

### 2. Падение производительности из-за *CPU Thrashing*
Когда потоков становится слишком много (больше, чем ядер процессора на сервере), операционная система начинает тратить больше ресурсов на переключение контекста (`context switching`), чем на выполнение полезной работы.
	
- Процессор начинает «захлёбываться», пытаясь уделить время каждому из тысяч созданных потоков. В результате производительность всей системы падает практически до нуля.

### 3. Отсутствие контроля над нагрузкой (*Backpressure*)
Правильный дизайн микросервисов требует предсказуемости. Если нагрузка превышает возможности системы, система должна либо временно складывать задачи в ограниченную очередь, либо мягко отказывать клиенту (клиент получит ошибку `429 Too Many Requests` или `503 Service Unavailable`, но сам сервис выживет).
	
- С `cachedThreadPool` у тебя нет механизма сдерживания (backpressure). Он будет пытаться переварить абсолютно всё, пока не упадет вместе с сервером.

### 🛠 Как это пофиксить (*Безопасная альтернатива*)
В реальном продакшене вместо готовых фабричных методов `Executors` всегда создают `ThreadPoolExecutor` вручную, явно контролируя все лимиты:
```Java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
	10, // corePoolSize: сколько потоков живет всегда
	50, // maximumPoolSize: лимит сверху (>50 потоков не создастся никогда!)
	60L, TimeUnit.SECONDS,    // время жизни лишних простаивающих потоков
	new LinkedBlockingQueue<>(1000), // Ограниченная очередь! Если прилетит 1001-я задача, сработает защита
	new ThreadPoolExecutor.AbortPolicy() // Стратегия отказа: выбросит RejectedExecutionException при переполнении
);
```
С такой конфигурацией твой микросервис будет вести себя предсказуемо при любых пиках нагрузки!
