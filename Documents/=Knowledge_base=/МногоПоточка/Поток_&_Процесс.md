Когда тебя спросят: _«Что такое процесс и поток? Чем они отличаются?»_, ты можешь выдать такую структуру:

> **Ответ:**
> 
> _«**Процесс** — это экземпляр программы, который выполняется в изолированном адресном пространстве. У него есть своя память, файловые дескрипторы, и он не может напрямую трогать память другого процесса — для общения нужно использовать IPC (сокеты, очереди, файлы)._
> 
> _**Поток** — это легковесный исполнитель внутри процесса. Все потоки одного процесса разделяют общую память (Heap) и метаданные классов, но у каждого есть свой стек (Stack) для локальных переменных и цепочки вызовов._
> 
> _В Java используется нативная потоковая модель — каждый Java-поток маппится на поток ОС. Но с Java 21 появились виртуальные потоки, которые управляются JVM и не привязаны 1:1 к ОС._
> 
> _Из-за того, что потоки разделяют память, возникают классические проблемы многопоточки: гонки данных, дедлоки и проблемы видимости. Их решают через синхронизацию.»_

| Характеристика            | Процесс                                     | Поток                                                     |
| ------------------------- | ------------------------------------------- | --------------------------------------------------------- |
| **Адресное пространство** | ✅ Изолированное                             | ❌ Общее (разделяют Heap)                                  |
| **Свой стек**             | ✅ Есть                                      | ✅ Есть                                                    |
| **Разделяют Heap**        | ❌ Нет                                       | ✅ Да                                                      |
| **Создание**              | Дорогое (копирование ресурсов)              | Дешёвое (только стек + регистры)                          |
| **IPC / синхронизация**   | Сложная (сокеты, файлы, очереди)            | Простая (через общие объекты, `synchronized`)             |
| **Сбой**                  | Падение процесса не убивает другие процессы | Падение потока может убить весь процесс (если не поймать) |

- **Куча (Heap)** — **одна на весь процесс**. Все потоки **внутри одного процесса** разделяют одну и ту же кучу.
    
- **У разных процессов** — **свои собственные кучи** (они изолированы друг от друга).

> - **Одна JVM = один дом (процесс).**   
> - **Потоки = жильцы (у каждого своя комната-стек, но общая кухня-куча).**   
**Другой процесс = соседний дом, куда мы можем сходить в гости (ProcessBuilder).**

### 1. **Общая куча (Heap) — одна на все потоки**

- Все потоки внутри JVM работают с **одной и той же кучей**.
    
- Именно поэтому нужна синхронизация (`synchronized`, `Lock`) — чтобы потоки не поломали друг другу данные.    

### 2. **JVM никогда не запускает второй процесс "сама по себе"**

- Даже если у тебя в приложении 1000 потоков — это всё еще **один процесс** с одним PID (идентификатором процесса).
    
- Если приложение упадет с `OutOfMemoryError` — упадет **весь процесс**, и все потоки умрут вместе с ним.

---
## 🎤 Как ответить на собеседовании (твоими словами)

> _«Когда мы запускаем Java-приложение, стартует **один процесс** JVM. Он имеет **один PID** и **одну общую кучу**. Внутри этого процесса мы можем создавать множество потоков — они разделяют кучу, но имеют свои стеки. Сама JVM никогда не создает новые процессы без нашего явного указания — только потоки. Если мы хотим запустить что-то отдельное (другую программу или скрипт), мы используем ProcessBuilder, но это уже будет внешний процесс, живущий отдельно от нашей JVM.»_
