**REST (*Representational State Transfer*)** — архитектурный стиль для построения распределенных систем, чаще всего поверх HTTP.

### 🔴 6 обязательных принципов (Constraints)

| №   | Принцип                          | Что значит                                     | Пример нарушения                         |
| --- | -------------------------------- | ---------------------------------------------- | ---------------------------------------- |
| 1   | **Client-Server**                | Разделение клиента и сервера                   | Бизнес-логика на клиенте                 |
| 2   | **Stateless**                    | Каждый запрос содержит всю информацию          | Сессия на сервере (`HttpSession`)        |
| 3   | **Cacheable**                    | Ответы должны быть кешируемыми                 | Всегда `Cache-Control: no-cache`         |
| 4   | **Uniform Interface**            | Единообразие интерфейса (HTTP методы, ресурсы) | `POST /getUser` вместо `GET /users/{id}` |
| 5   | **Layered System**               | Клиент не знает о промежуточных слоях          | Клиент знает про балансировщик           |
| 6   | **Code on Demand** (опционально) | Сервер может передать клиенту код (JS, applet) | —                                        |

### 🎯 Uniform Interface — *4 подпринципа*

| Подпринцип                      | Что требует                             | Правильно                               | Неправильно                                     |
| ------------------------------- | --------------------------------------- | --------------------------------------- | ----------------------------------------------- |
| **Identification of resources** | Ресурс идентифицируется **URI**         | `GET /users/123`                        | `GET /getUserById?id=123`                       |
| **Representation**              | Ресурс отделён от его **представления** | `Accept: application/json`              | Всегда возвращает JSON                          |
| **Self-descriptive messages**   | Сообщение **описывает себя**            | `POST /users` создаёт                   | `POST /users` удаляет                           |
| **HATEOAS**                     | Гиперссылки для **переходов**           | `{"id":1, "links":{"self":"/users/1"}}` | Нет ссылок, клиент должен знать все URL заранее |
1. **Identification of resources** — URI = имя ресурса, а не действие (`/users/123`, а не `/getUserById`).
    
2. **Representation** — клиент выбирает формат через `Accept` (JSON, XML, и т.д.).
    
3. **Self-descriptive messages** — метод HTTP + URI = действие. `POST` = создание, `DELETE` = удаление.
    
4. **HATEOAS** — ответ содержит ссылки на связанные действия, клиент "гуляет" по API без жёстко зашитых URL.

### 🟢 Уровни зрелости REST (Richardson Maturity Model)
```text
Уровень 0 (POX/SOAP) → Одна точка входа, один метод
Уровень 1 (Resources) → Разные URI для разных ресурсов
Уровень 2 (HTTP Verbs) → GET, POST, PUT, DELETE
Уровень 3 (HATEOAS)   → Гиперссылки в ответах
```
