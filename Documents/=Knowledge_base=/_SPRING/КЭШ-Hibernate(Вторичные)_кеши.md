## Многоуровневое кеширование Hibernate (Вторичные кеши)

---
## Что такое кеширование в Hibernate и зачем оно нужно?

**Hibernate** — это ORM (*Object-Relational Mapping*) фреймворк, который преобразует Java-объекты в записи базы данных и обратно. Каждый раз, когда вы вызываете `entityManager.find(User.class, 1L)`, Hibernate выполняет SQL-запрос к базе данных. Это дорогостоящая операция (сетевое взаимодействие, парсинг SQL, чтение с диска).

**Цель кеширования** — максимально сократить количество обращений к базе данных, сохраняя уже загруженные объекты в памяти. Hibernate предоставляет **три уровня кеша**, которые работают вместе как единая система.

---
## Уровень 1. Кеш первого уровня (***L1** Cache*) <BR>— «Кеш сессии»

### Что это?
Это **обязательный, всегда включенный** кеш, который живет в рамках одной **сессии** (или `EntityManager`).

### Как работает?
Когда вы загружаете объект через `session.get(User.class, 1L)`:
	
1. Hibernate сначала проверяет **L1**-кеш.
    
2. Если объект там есть — он возвращается **без запроса к БД**.
    
3. Если объекта нет — выполняется SQL-запрос, объект сохраняется в **L1**-кеш и возвращается.

### Ключевые особенности:
- **Включен всегда** — его нельзя отключить.
    
- **Живет только в рамках одной сессии**. Как только сессия закрывается — кеш уничтожается.
    
- **Гарантирует**, что в рамках одной сессии для одного и того же ID базы данных будет существовать **только один объект** (это называется **persistence context**).
    
- **Изменения объектов** автоматически отслеживаются, и при коммите транзакции Hibernate синхронизирует их с БД (без лишних запросов на чтение).

### Простой пример:
```java
// Открываем сессию
Session session = sessionFactory.openSession();

// Первый запрос — идет в БД
User user1 = session.get(User.class, 1L);

// Второй запрос — БД НЕ трогается, объект берется из L1-кеша
User user2 = session.get(User.class, 1L);

// Проверяем, что это один и тот же объект в памяти
System.out.println(user1 == user2); // true
session.close(); // L1-кеш уничтожен
```

### Когда L1-кеш сбрасывается?
- При вызове `session.evict(entity)` — удаляет конкретный объект.
    
- При вызове `session.clear()` — очищает весь кеш.
    
- При закрытии сессии (`session.close()`).
    
- При вызове `session.refresh(entity)` — перезагружает объект из БД, обновляя кеш.

### Важный нюанс
Если в рамках одной сессии вы загружаете один и тот же объект через **разные запросы** (JPQL, Criteria), Hibernate **все равно** использует L1-кеш и возвращает тот же самый объект:
```java
// Загружаем по ID
User user1 = session.get(User.class, 1L);

// Загружаем через JPQL запрос
User user2 = session.createQuery("FROM User WHERE id = 1", User.class)
                    .getSingleResult();
                    
// Это один и тот же объект
System.out.println(user1 == user2); // true
```

## Уровень 2. Кеш второго уровня (L2 Cache) — «Кеш фабрики сессий»

### Что это?
Это **опциональный, глобальный** кеш, который живет на уровне **фабрики сессий** (`SessionFactory` или `EntityManagerFactory`). Он разделяется между **всеми сессиями**.

### Зачем он нужен?
Представьте, что у вас есть 100 пользователей одновременно. Каждый из них запрашивает данные о стране (справочник). Без **L2**-кеша каждый запрос будет идти в БД. С **L2**-кешем — данные загружаются **один раз** и используются всеми.

### Как работает?
1. Вы загружаете объект через `session.get(User.class, 1L)`.
    
2. Hibernate **сначала** проверяет **L1**-кеш.
    
3. Если в **L1** нет — проверяет **L2-кеш** (*если он включен для этой сущности*).
    
4. Если объект есть в **L2** — он помещается в **L1** и возвращается.
    
5. Если объекта нет ни в **L1**, ни в **L2** — выполняется SQL-запрос, объект сохраняется **сначала в L2**, потом в **L1** и возвращается.

### Ключевые особенности:
- **Выключен по умолчанию** — нужно **явно** включать.
    
- **Глобальный** — живет все время работы приложения (пока жива `SessionFactory`).
    
- **Работает только с сущностями** (по ID). Для сложных запросов используется отдельный кеш — *Query Cache*.
    
- **Экономит ресурсы** БД, но потребляет больше памяти JVM.

### Как включить L2-кеш?

#### Шаг 1. Добавить зависимость (*провайдер*)
Например, для *Ehcache* (популярный выбор):
```xml
<dependency>
    <groupId>org.hibernate.orm</groupId>
    <artifactId>hibernate-ehcache</artifactId>
</dependency>
```

#### Шаг 2. Настроить в `application.properties` или `hibernate.cfg.xml`
```properties
# Включаем L2-кеш
spring.jpa.properties.hibernate.cache.use_second_level_cache = true

# Указываем провайдера (Ehcache)
spring.jpa.properties.hibernate.cache.region.factory_class = org.hibernate.cache.jcache.JCacheRegionFactory

# Включаем статистику (для отладки)
spring.jpa.properties.hibernate.generate_statistics = true
```

#### Шаг 3. Пометить сущность как **кешируемую**
Используйте аннотацию `@Cacheable` (из `jakarta.persistence`) и `@Cache` (из Hibernate):
```java
@Entity
@Cacheable
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE) // стратегия доступа
public class User {
    @Id
    private Long id;
    private String name;
    // ...
}
```

### Стратегии доступа к L2-кешу (`CacheConcurrencyStrategy`)

|Стратегия|Описание|Когда использовать|
|---|---|---|
|**READ_ONLY**|Данные никогда не изменяются. Лучшая производительность.|Справочники (список стран, валют).|
|**READ_WRITE**|Данные могут изменяться. Использует "мягкие" блокировки.|Обычные сущности, которые редко обновляются.|
|**NONSTRICT_READ_WRITE**|"Мягкая" синхронизация. Допускает небольшую рассинхронизацию.|Когда согласованность не критична.|
|**TRANSACTIONAL**|Используется в управляемых транзакциях (JTA).|Когда нужна строгая атомарность.|

### Пример работы L2-кеша:
```java
// Сессия 1
Session session1 = sessionFactory.openSession();
User user1 = session1.get(User.class, 1L); // Идет в БД, сохраняет в L2
session1.close();

// Сессия 2 (другой пользователь, другая сессия)
Session session2 = sessionFactory.openSession();
User user2 = session2.get(User.class, 1L); // НЕ идет в БД, берется из L2!
System.out.println(user1 == user2); // false (разные объекты в памяти)
session2.close();
```

### Когда L2-кеш сбрасывается?
- **Автоматически** — при обновлении, удалении или вставке сущности (если используется `READ_WRITE`).
    
- **Вручную** — через вызов `sessionFactory.getCache().evict(User.class, 1L)` (удалить конкретную запись) или `sessionFactory.getCache().evictEntityRegion(User.class)` (очистить весь кеш для сущности).

### Важно!
L2-кеш работает **только** для загрузки по **первичному ключу** (`get()`, `find()`). Для сложных запросов (WHERE, JOIN) используется отдельный механизм — **Query Cache**.

## Уровень 3. Кеш запросов (*Query Cache*) <br>— «Кеш результатов запросов»

### Что это?
Это отдельный кеш, который хранит **не сами сущности**, а **идентификаторы (ID)** сущностей, которые вернул запрос. Сами сущности при этом загружаются из **L2**-кеша (или из БД, если **L2** отключен).

### Зачем он нужен?
Выполнение сложного JPQL/HQL-запроса с JOIN, WHERE и GROUP BY — это дорого. Query Cache позволяет сохранить **результат запроса** (список ID), чтобы при повторном выполнении того же запроса не парсить JPQL, не строить план выполнения и не обращаться к БД.

### Как работает?
1. Вы выполняете запрос: `FROM User WHERE age > 18 ORDER BY name`.
    
2. Hibernate проверяет Query Cache по **хешу запроса** + **значениям параметров**.
    
3. Если в кеше есть список ID — Hibernate загружает **только эти ID**, а затем подгружает каждую сущность **по отдельности** из L2-кеша (или БД, если L2 выключен).
    
4. Если в кеше нет — выполняется SQL-запрос, извлекаются ID, сохраняются в Query Cache, а сами сущности — в L2-кеш.

### Как включить Query Cache?
```properties
# Включаем Query Cache
spring.jpa.properties.hibernate.cache.use_query_cache = true

# Опционально: кеширование запросов с параметрами
spring.jpa.properties.hibernate.cache.use_query_cache = true
```

И пометить конкретный запрос как кешируемый:
```java
List<User> users = session.createQuery("FROM User WHERE age > :age", User.class)
				   .setParameter("age", 18)
				   .setCacheable(true) // Включаем кеширование для этого запроса
				   .setCacheRegion("userAgeQuery") // Опционально: регион кеша
				   .getResultList();
```

### Ключевые особенности:
- **Работает только в связке с L2-кешем**. Если L2 выключен, то после нахождения ID в Query Cache каждый объект будет загружаться из БД отдельным запросом — это **МЕДЛЕННЕЕ**, чем просто выполнить один большой запрос.
    
- **Инвалидируется** при любом изменении данных, которые могут повлиять на результат запроса (например, INSERT/UPDATE/DELETE в таблице User).
    
- **Хранит только ID**, а не сами данные.

### Аналогия для понимания
Представьте, что у вас есть **книга** (сущность) и **каталог** (Query Cache), где указаны номера страниц с нужными темами (ID сущностей):
	
1. Сначала вы ищете тему в каталоге (Query Cache) — там написаны номера страниц.
    
2. Затем вы открываете книгу на нужной странице (L2-кеш) и читаете текст (данные).
    
3. Если в каталоге нет нужной темы — вы читаете всю книгу заново (SQL-запрос), заполняете каталог и запоминаете страницы.

## Схема работы всех трех уровней (пошагово)

Вот как Hibernate обрабатывает запрос `session.get(User.class, 1L)`:
```text
┌─────────────────────────────────────────────────────────────┐
│ 1. Вызов session.get(User.class, 1L)                        │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. Проверка L1-кеша (Session)                               │
│    Есть ли объект с ID = 1 в текущей сессии?                │
└─────────────────────────────────────────────────────────────┘
                            │
          ┌─────────────────┴─────────────────┐
          │ ЕСТЬ                              │ НЕТ
          ▼                                   ▼
┌────────────────────────────┐  ┌─────────────────────────────────────────────┐
│ Возврат объекта из L1      │  │ 3. Проверка L2-кеша (SessionFactory)        │
│ (БД не трогаем)            │  │    Включен ли L2 для User? Есть ли ID = 1?  │
└────────────────────────────┘  └─────────────────────────────────────────────┘
                                             │
                           ┌─────────────────┴─────────────────┐
                           │ ЕСТЬ                              │ НЕТ
                           ▼                                   ▼
         ┌────────────────────────────┐  ┌────────────────────────────────────┐
         │ Загружаем объект из L2     │  │ 4. Выполняется SQL-запрос к БД     │
         │ Копируем в L1 и возвращаем │  │    SELECT * FROM users WHERE id = 1│
         └────────────────────────────┘  └────────────────────────────────────┘
                                                     │
                                                     ▼
                                  ┌─────────────────────────────────────────────┐
                                  │ 5. Создается объект User из ResultSet       │
                                  │    Сохраняется в L2-кеш                     │
                                  │    Сохраняется в L1-кеш                     │
                                  │    Возвращается пользователю                │
                                  └─────────────────────────────────────────────┘
```

А вот так выглядит обработка запроса `FROM User WHERE age > 18`:
```text
┌─────────────────────────────────────────────────────────────┐
│ 1. Вызов session.createQuery("FROM User WHERE age > 18")    │
│    .setCacheable(true).getResultList()                      │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌───────────────────────────────────────────────────────────────┐
│ 2. Проверка Query Cache                                       │
│    Есть ли список ID для этого запроса (по хешу + параметрам)?│
└───────────────────────────────────────────────────────────────┘
                            │
          ┌─────────────────┴─────────────────┐
          │ ЕСТЬ                              │ НЕТ
          ▼                                   ▼
┌────────────────────────────┐  ┌─────────────────────────────────────────────┐
│ Получаем список ID из      │  │ 3. Выполняется SQL-запрос к БД              │
│ Query Cache                │  │    SELECT id FROM users WHERE age > 18      │
│ Для каждого ID:            │  └─────────────────────────────────────────────┘
│  1. Проверяем L1           │                     │
│  2. Если нет — проверяем L2│                     ▼
│  3. Если нет — грузим из БД│   ┌────────────────────────────────────────────┐
│  4. Сохраняем в L1 и L2    │   │ 4. Полученный список ID сохраняется в      │
└────────────────────────────┘   │    Query Cache (с привязкой к хешу запроса)│
                                │    Далее для каждого ID:                   │
                                │     1. Проверяем L1                        │
                                │     2. Если нет — проверяем L2             │
                                │     3. Если нет — грузим из БД             │
                                └────────────────────────────────────────────┘
```

## Практические рекомендации

### 1. Когда **использовать** L2-кеш?
- **Справочные данные** (список стран, валют, статусов заказов) — они редко меняются.
    
- **Данные, которые часто читаются** (например, профиль пользователя, если он просматривается тысячами пользователей).
    
- **Объекты, которые требуют дорогостоящего построения** (например, сложные агрегаты).

### 2. Когда **НЕ использовать** L2-кеш?
- **Часто изменяемые данные** (например, счет пользователя).
    
- **Огромные таблицы** (кеш займет слишком много памяти).
    
- **Данные, которые должны быть всегда актуальными** (финансовые транзакции).

### 3. Оптимизация Query Cache
- Используйте его ТОЛЬКО для запросов, которые:
	    
    - Часто выполняются.
        
    - Возвращают небольшое количество записей (до 100–200).
        
    - Имеют большой "попаданий" (hit ratio).
    
- Всегда включайте L2-кеш, если используете Query Cache.

### 4. Мониторинг эффективности
Включите статистику Hibernate и следите за метриками:
```properties
spring.jpa.properties.hibernate.generate_statistics = true
```

В логах вы увидите:
```text
Second level cache hits: 150
Second level cache misses: 30
Query cache hits: 45
Query cache misses: 12
```

### 5. Управление размерами кеша
Для Ehcache/Caffeine настройте:
	
- `maxEntriesLocalHeap` — максимальное количество объектов.
    
- `timeToLiveSeconds` — время жизни записи.
    
- `timeToIdleSeconds` — время бездействия перед удалением.

## Визуальная схема (запоминающийся образ)
	
Представьте, что Hibernate — это **библиотека**:
	
- **L1-кеш** = **рабочий стол библиотекаря**. На столе лежат книги, которые он сейчас читает (активная сессия). Как только он заканчивает работу — стол очищается.
    
- **L2-кеш** = **основной склад библиотеки**. Здесь хранятся самые популярные книги. Все библиотекари имеют к ним доступ.
    
- **Query Cache** = **каталог**. В нем указано, на каких полках (в L2) лежат книги по определенной теме. Если темы нет в каталоге — библиотекарь идет в подвал (БД) и ищет книги, попутно заполняя каталог.

---
## Заключение

**Многоуровневое кеширование Hibernate** — это мощная система, которая значительно ускоряет работу приложений, но требует осознанного подхода. Запомните:
	
1. **L1** — всегда включен, работает в рамках сессии.
    
2. **L2** — глобальный, включается настройкой и аннотациями на сущностях.
    
3. **Query Cache** — кеширует ID результатов запросов, должен использоваться вместе с L2.
    

**Главный секрет успеха**: правильно выбрать стратегию доступа (`READ_ONLY` для справочников, `READ_WRITE` для остального), настроить размеры кеша и всегда помнить про инвалидацию при изменении данных. И тогда ваше приложение будет работать молниеносно! 🚀
