### 26. Что такое **AOP**? Зачем нужен?

**Ответ:** **AOP (Aspect-Oriented Programming)** — это парадигма программирования, которая позволяет выносить **сквозные (cross-cutting) concerns** — логику, повторяющуюся в разных частях приложения — в отдельные модули, называемые аспектами.

**Зачем нужен:** Без AOP сквозная логика (логирование, транзакции, безопасность, мониторинг, кэширование) "размазывается" по всем методам, нарушая **DRY** и **Single Responsibility**.

#### **Типичные примеры сквозных concerns:**
- **Логирование** — логировать вход/выход из каждого метода.    
- **Транзакции** — открыть/коммит/откатить транзакцию вокруг бизнес-метода.    
- **Security** — проверить права доступа перед вызовом.    
- **Performance monitoring** — замерить время выполнения.    
- **Caching** — проверить кэш до вызова, сохранить результат после.    

**Ключевая ценность AOP:** Вы пишете аспект **один раз**, и он автоматически применяется ко всем методам, соответствующим условию (pointcut). Бизнес-код остается чистым, занимается только своей прямой обязанностью.

В *Spring AOP* реализован поверх **прокси** (JDK Dynamic Proxy или CGLIB) и является **proxy-based AOP**. Он не меняет байт-код класса (в отличие от полноценного AspectJ с compile-time или load-time weaving).

### 27. Основные **термины AOP**: <br>Aspect, JoinPoint, Advice, Pointcut, Weaving

**Ответ:** Это ключевые понятия AOP. Привожу с примерами.

| Термин        | Определение                                                                                                                                | Пример                                                  |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------- |
| **Aspect**    | Модуль, содержащий сквозную логику. Обычно класс с `@Aspect`.                                                                              | `@Aspect class LoggingAspect`                           |
| **JoinPoint** | Точка выполнения программы, <br>куда можно "вклиниться". <br>В Spring AOP — <br>**только вызовы методов**.                                 | Вызов `saveUser()` в сервисе                            |
| **Advice**    | Действие, которое выполняется<br> в JoinPoint. <br>Тип совета определяет **когда**.                                                        | `@Before`, `@AfterReturning`, `@Around`                 |
| **Pointcut**  | Выражение, <br>которое отбирает JoinPoint-ы, <br>где нужно применить Advice.                                                               | `@Pointcut("execution(* com.example.service.*.*(..))")` |
| **Weaving**   | Процесс связывания аспектов <br>с целевыми объектами <br>(создание прокси). <br>В Spring AOP происходит **во время выполнения (runtime)**. | Spring создаёт прокси и перехватывает вызовы            |

**Как это связано:** Вы определяете **Aspect**, внутри него — **Pointcut** (условие) и **Advice** (что делать). При выполнении программы **Weaving** встраивает Advice во все JoinPoint-ы, отобранные Pointcut-ом.

### 28. Какие типы Advice бывают?

**Ответ:** В Spring AOP есть 5 типов Advice, различающихся моментом выполнения относительно JoinPoint:

1. **`@Before`** — выполняется **до** выполнения метода.    
    - Используется для: проверка прав, логирование входа, инициализация контекста.
        
2. **`@After`** — выполняется **после** выполнения метода **в любом случае** (и нормальное завершение, и исключение). Аналог `finally`.    
    - Используется для: освобождение ресурсов, логирование выхода.
        
3. **`@AfterReturning`** — выполняется **только после успешного** выполнения метода (без исключения). Может получить доступ к возвращаемому значению через `returning`.    
    - Используется для: постобработка результата, аудит успешных операций.
        
4. **`@AfterThrowing`** — выполняется **только если метод выбросил исключение**. Может получить доступ к исключению через `throwing`.    
    - Используется для: обработка специфических исключений, логирование ошибок, отправка алертов.
        
5. **`@Around`** — **самый мощный**. Оборачивает метод полностью. Вы сами решаете, вызывать ли целевой метод (`proceed()`) и когда. Может:    
    - Вызвать метод несколько раз.        
    - Не вызывать вообще (например, вернуть из кэша).        
    - Изменить аргументы до вызова.        
    - Изменить возвращаемое значение.        
    - Обработать исключение.        

**Порядок выполнения при нескольких Advice на одном JoinPoint определяется порядком объявления аспектов или через `@Order`.**

---
### 29. Чем отличается `@Around` от остальных?

**Ответ:** `@Around` — единственный Advice, который **полностью контролирует выполнение целевого метода**.

**Ключевые отличия:**

1. **Вызов `proceed()` обязателен** (если вы хотите, чтобы целевой метод выполнился). Без него метод не будет вызван.
```java
@Around("@annotation(Cacheable)")
public Object around(ProceedingJoinPoint pjp) throws Throwable {
    Object cached = cache.get(pjp.getArgs());
    if (cached != null) return cached; // метод НЕ вызывается
    Object result = pjp.proceed(); // вызываем
    cache.put(pjp.getArgs(), result);
    return result;
}
```
    
2. **Может полностью заменить выполнение** — вернуть свой результат, не вызывая целевой метод (кэширование, security bypass).
    
3. **Может выполнить метод несколько раз** (retry-логика).
    
4. **Может изменить аргументы** перед передачей в целевой метод.
    
5. **Может перехватывать и обрабатывать исключения** по-своему, а не просто логировать.
    
6. **Может изменить возвращаемое значение** после выполнения.    

**Ограничения:** `@Around` требует больше кода и сложнее для понимания. Для простых случаев (простое логирование до или после) лучше использовать `@Before` / `@After` — они чище и производительнее.

**Правило:** Используйте `@Around` когда нужно **контролировать**, выполнится ли метод, или когда нужен доступ и к аргументам, и к результату, и к исключению одновременно. В остальных случаях — более специфичные Advice.

### 30. Как работают прокси в Spring AOP? (JDK Dynamic Proxy vs CGLIB)

**Ответ:** Spring AOP основан на **прокси-паттерне**. Когда вы применяете аспект к бину, Spring не изменяет ваш класс, а создает **прокси-объект**, который оборачивает оригинальный бин и перехватывает вызовы методов.

**Способ создания прокси зависит от того, реализует ли целевой класс интерфейс:**

| Характеристика         | JDK Dynamic Proxy                                          | CGLIB (Code Generation Library)                                                              |
| ---------------------- | ---------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| **Условие**            | Целевой класс <br>**реализует хотя бы <br>один интерфейс** | Целевой класс <br>**не реализует интерфейсы** <br>или принудительно задано                   |
| **Тип прокси**         | Прокси реализует **те же интерфейсы**, что и целевой класс | Прокси является **подклассом** <br>целевого класса                                           |
| **Создание**           | Встроенный в JDK (с версии 1.3)                            | Отдельная библиотека (включена в Spring, <br>в `spring-core`)                                |
| **Требования**         | Класс должен реализовывать интерфейс                       | Класс не должен быть `final` (нельзя наследовать `final`), <br>методы не должны быть `final` |
| **Производительность** | Немного быстрее при создании                               | Немного быстрее при вызовах <br>(современные версии <br>разница незначительна)               |

- В **чистом Spring Framework** (по умолчанию): если класс реализует хотя бы один интерфейс — **JDK Dynamic Proxy**, иначе — **CGLIB**.
- В **Spring Boot**: автоконфигурация по умолчанию устанавливает `spring.aop.proxy-target-class=true` → **CGLIB используется всегда** (даже если есть интерфейсы). Это упрощает работу и решает многие проблемы с self-invocation.

**Принудительно переключить:**
- Через свойство: `spring.aop.proxy-target-class=false` (JDK)
- Через аннотацию: `@EnableAspectJAutoProxy(proxyTargetClass = false)`

**Принудительное использование CGLIB:** Установить `spring.aop.proxy-target-class=true` (в `application.properties`) или `@EnableAspectJAutoProxy(proxyTargetClass = true)`. Это заставит Spring всегда использовать CGLIB, даже если интерфейсы есть.

**Важный нюанс:** Прокси — это **другой объект**, не оригинальный бин. Поэтому `this` внутри бина указывает на оригинал, и self-invocation (вызов метода из того же класса) **не проходит через прокси**, а значит, AOP не срабатывает. Это ключ к пониманию вопроса №31.

### 31. Почему метод `@Transactional`, вызванный из того же класса, не работает? (Проблема self-invocation)

**Ответ:** Это классическая проблема self-invocation, и она вытекает из того, как работают прокси в Spring AOP.

**Кратко:** Spring оборачивает ваш бин в прокси. Прокси перехватывает вызовы **снаружи**. Когда вы вызываете метод `this.method()` внутри того же класса, вы **обходите прокси** и вызываете оригинальный метод напрямую.

**Пример, который НЕ работает:**
```java
@Service
public class UserService {
    
    @Transactional
    public void saveUser(User user) {
        // этот метод обернут прокси
    }
    
    public void registerUser(User user) {
        this.saveUser(user); // self-invocation — @Transactional ИГНОРИРУЕТСЯ!
    }
}
```

**Что происходит под капотом:**

1. Spring создает прокси-объект (например, `UserServiceProxy`), который содержит ссылку на оригинальный `UserService`.
    
2. Внешний код работает с прокси.
    
3. Когда прокси перехватывает вызов `saveUser`, он открывает транзакцию, вызывает оригинальный `saveUser`, затем коммитит.
    
4. Но внутри `registerUser`, который уже выполняется внутри оригинального объекта, вызов `this.saveUser()` идет напрямую, без участия прокси. Транзакция не открывается.

**Аналогия:** Прокси — это как ассистент, который стоит перед вами. Когда кто-то говорит с ассистентом, ассистент может что-то сделать до и после вас. Но когда вы говорите сами с собой, ассистент не участвует.

**Решения:** В следующем вопросе (№32).

### 32. Как заставить работать AOP при self-invocation? (AopContext.currentProxy())

**Ответ:** Есть несколько способов обойти self-invocation и заставить AOP сработать даже при вызове из того же класса.

**Способ 1 (рекомендуемый):** **Реструктурировать код** — вынести вызываемый метод в отдельный бин.
```java
@Service
public class UserService {
    @Autowired
    private TransactionalHelper helper;
    
    public void registerUser(User user) {
        helper.saveUser(user); // теперь вызов идет через отдельный бин -> прокси
    }
}
```

Это самый чистый способ без привязки к Spring API.

**Способ 2:** **Использовать `AopContext.currentProxy()`** (прокси самого себя).
```java
@Service
@EnableAspectJAutoProxy(exposeProxy = true) // обязательно включить
public class UserService {
    
    @Transactional
    public void saveUser(User user) { }
    
    public void registerUser(User user) {
        ((UserService) AopContext.currentProxy()).saveUser(user);
    }
}
```

**Недостатки:** Нужно включать `exposeProxy=true`, код становится зависимым от Spring, нарушается чистота.

**Способ 3:** **Внедрить сам бин через самовнедрение (self-injection).**
```java
@Service
public class UserService {
    @Autowired
    private UserService self; // Spring внедрит прокси
    
    public void registerUser(User user) {
        self.saveUser(user); // вызов через прокси
    }
}
```

**Недостатки:** Неочевидно новичкам, может привести к циклическим ссылкам при неаккуратной настройке.

**Способ 4:** **Использовать `ApplicationContext.getBean(UserService.class)`** — самый тяжелый и нерекомендуемый.

**Вывод для собеседования:** Самое правильное решение — **выделить транзакционный метод в отдельный сервис**. Это соблюдает Single Responsibility и не привязывает код к деталям реализации Spring AOP.

### 33. Что такое `@EnableAspectJAutoProxy(proxyTargetClass=true)`?

**Ответ:** `@EnableAspectJAutoProxy` — аннотация, которая **включает поддержку AOP** в Spring-приложении (в Spring Boot она включена по умолчанию через автоконфигурацию).

**Параметр `proxyTargetClass` управляет типом создаваемых прокси:**

| `proxyTargetClass`                     | Тип прокси                              | Условия                     |
| -------------------------------------- | --------------------------------------- | --------------------------- |
| `false` (по умолчанию в чистом Spring) | JDK Dynamic Proxy (если есть интерфейс) | Стандартное поведение       |
| `true`                                 | **Всегда CGLIB**                        | Рекомендуется в Spring Boot |


**Когда нужно `proxyTargetClass = true`:**

1. **У вас есть бины, реализующие интерфейсы, но вам нужно проксировать не только методы интерфейса, но и другие публичные методы** (JDK прокси проксирует только методы интерфейса).
    
2. **Вы используете `@Configuration` классы, где нужно проксировать вызовы `@Bean` методов** (CGLIB может проксировать конкретные классы).
    
3. **Вы работаете с классами, которые не реализуют интерфейсы** (тут CGLIB используется автоматически, параметр не меняет поведение).
    
4. **Вам нужно, чтобы `final` классы?** Нет, CGLIB не может проксировать `final` классы — это ограничение.

**Когда ставить `proxyTargetClass = true`:** - Нужно проксировать методы, не объявленные в интерфейсе. - Используются `@Configuration` классы с `@Bean`-методами (CGLIB лучше работает с self-invocation внутри конфигурации). - Вы хотите единообразное поведение (CGLIB для всех бинов).

В Spring Boot по умолчанию уже `true`.

**Как включить:**
- Через аннотацию: `@EnableAspectJAutoProxy(proxyTargetClass = true)`    
- Через `application.properties`: `spring.aop.proxy-target-class=true`    

**Производительность:** Раньше CGLIB был медленнее, сейчас разница незначительна для 99% приложений.

**Внимание:** CGLIB создает подкласс, поэтому не может проксировать `final` методы и `final` классы.

### 34. Можно ли зааспектить `private` метод?

**Ответ:** **Нет, нельзя.** Это прямое следствие того, как работает прокси-механизм в Spring AOP.

**Почему нельзя:**

1. **JDK Dynamic Proxy** — проксирует только **публичные методы интерфейса**. `private` методы вообще не видны через интерфейс.
    
2. **CGLIB** — создает подкласс и переопределяет **публичные и защищенные** (`protected`) методы. `private` методы не наследуются и не могут быть переопределены.
    

**Детали:** Даже если использовать `proxyTargetClass=true` и AspectJ weaving, `private` методы остаются вне досягаемости для Spring AOP, потому что AOP в Spring основан на **перехвате вызовов методов** через прокси. `private` методы вызываются только внутри класса, не проходят через прокси-объект снаружи.

**Исключение (теоретическое):** Если использовать **load-time weaving (LTW)** с полноценным AspectJ (не Spring AOP), то можно зааспектить `private` методы, внося изменения непосредственно в байткод класса. Но **Spring AOP этого не делает**. На собеседовании Middle важно знать это различие.

**Что можно аспектить:**
- `public` методы — **да** (через любой прокси).    
- `protected` методы — **да** (через CGLIB).    
- package-private (без модификатора) — **да** (через CGLIB, если класс в том же пакете).
- `private` — **нет**.

**Рекомендация:** Если вам нужно применить аспект к логике, которую вы хотели сделать `private`, вынесите эту логику в отдельный `public` или `protected` метод и вызывайте его. Или пересмотрите дизайн: возможно, эту логику стоит вынести в отдельный бин.

> Поэтому private методы лучше не использовать для сквозной логики — 
> выносите её в public/protected методы или отдельные компоненты.
