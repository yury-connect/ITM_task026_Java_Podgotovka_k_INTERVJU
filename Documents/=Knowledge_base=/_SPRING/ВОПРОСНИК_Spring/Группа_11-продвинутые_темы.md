### 105. Как работает внедрение зависимостей в `@Bean` методах одной конфигурации? (CGLIB прокси перехватывает вызовы)

**Ответ:** Когда класс помечен `@Configuration`, Spring создает **CGLIB-прокси** этого класса. При вызове одного `@Bean` метода внутри другого, прокси **перехватывает вызов** и возвращает бин из контекста, а не создает новый объект.

**Пример:**
```java
@Configuration
public class AppConfig {
    
    @Bean
    public UserService userService() {
        // Этот вызов НЕ создает новый DataSource!
        return new UserService(dataSource());  // перехватывается прокси
    }
    
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource();
    }
}
```

**Что происходит под капотом (упрощенно):**
```java
// Spring создает примерно такой прокси (CGLIB)
public class AppConfig$$EnhancerByCGLIB extends AppConfig {
    
    private DataSource cachedDataSource;
    
    @Override
    public DataSource dataSource() {
        if (cachedDataSource == null) {
            cachedDataSource = super.dataSource();  // реальное создание один раз
        }
        return cachedDataSource;
    }
    
    @Override
    public UserService userService() {
        // Перехватываем вызов dataSource() и возвращаем бин из контекста
        DataSource ds = context.getBean(DataSource.class);
        return new UserService(ds);
    }
}
```

**Сравнение `@Configuration` vs `@Component`:**
```java
// Вариант 1: @Configuration (правильно)
@Configuration
public class CorrectConfig {
    @Bean
    public A a() { return new A(); }
    
    @Bean
    public B b() { return new B(a()); }  // a() возвращает синглтон из контекста
}
// Вариант 2: @Component (НЕПРАВИЛЬНО для синглтонов)
@Component
public class WrongConfig {
    @Bean
    public A a() { return new A(); }
    
    @Bean
    public B b() { return new B(a()); }  // a() создает НОВЫЙ объект каждый раз!
}
```

**Вывод:** В `@Configuration` межбеновые вызовы **проксируются** и возвращают управляемые Spring бины. В `@Component` — обычные Java вызовы, создающие новые объекты.

### 106. Что такое `BeanPostProcessor` и `BeanFactoryPostProcessor`? Разница и порядок выполнения

**Ответ:** Это интерфейсы-хуки для вмешательства в жизненный цикл бинов и конфигурации Spring.

|Характеристика|`BeanFactoryPostProcessor`|`BeanPostProcessor`|
|---|---|---|
|**Что обрабатывает**|`BeanDefinition` (метаданные о бинах)|Экземпляры бинов|
|**Когда выполняется**|**До** создания экземпляров бинов|**В процессе** создания бинов|
|**Типичное использование**|Модификация свойств бина до инициализации, подмена `BeanDefinition`|Проксирование, логирование, внедрение зависимостей|

**Порядок выполнения:**
```text
1. Сканирование и парсинг конфигурации
2. Выполняются все BeanFactoryPostProcessor (изменяют BeanDefinition)
   ├── PriorityOrdered
   ├── Ordered
   └── остальные
3. Создаются экземпляры бинов (singleton)
   Для каждого бина:
   ├── 3.1. Конструктор
   ├── 3.2. Внедрение зависимостей (setters, @Autowired)
   ├── 3.3. BeanPostProcessor.postProcessBeforeInitialization
   ├── 3.4. @PostConstruct / afterPropertiesSet()
   ├── 3.5. BeanPostProcessor.postProcessAfterInitialization
   └── 3.6. Бин готов
```

**Пример `BeanFactoryPostProcessor`:**
```java
@Component
public class CustomBeanFactoryPostProcessor implements BeanFactoryPostProcessor {
    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) {
        BeanDefinition bd = beanFactory.getBeanDefinition("dataSource");
        bd.getPropertyValues().add("url", "jdbc:mysql://localhost:3306/mydb");
    }
}
```

**Пример `BeanPostProcessor`:**
```java
@Component
public class LoggingBeanPostProcessor implements BeanPostProcessor {
    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) {
        System.out.println("Before init: " + beanName);
        return bean;
    }
    
    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        System.out.println("After init: " + beanName);
        // Можно вернуть прокси вместо исходного бина
        if (bean instanceof MyService) {
            return Proxy.newProxyInstance(...);
        }
        return bean;
    }
}
```

**Встроенные реализации:**
- `AutowiredAnnotationBeanPostProcessor` — обрабатывает `@Autowired`, `@Value`
- `PersistenceExceptionTranslationPostProcessor` — для JPA исключений
- `CommonAnnotationBeanPostProcessor` — для `@PostConstruct`, `@PreDestroy

### 107. Что такое `ImportSelector` и `ImportBeanDefinitionRegistrar`? (Для написания стартеров)

**Ответ:** Это интерфейсы для **условной и программной регистрации бинов**, используемые в автоконфигурации Spring Boot стартеров.

**`ImportSelector`** — выбирает имена классов для импорта на основе аннотаций:
```java
public interface ImportSelector {
    String[] selectImports(AnnotationMetadata importingClassMetadata);
}

// Пример: логика импорта на основе свойства
public class CacheImportSelector implements ImportSelector {
    @Override
    public String[] selectImports(AnnotationMetadata metadata) {
        boolean enableRedis = "redis".equals(System.getProperty("cache.type"));
        return enableRedis 
            ? new String[]{RedisCacheConfig.class.getName()}
            : new String[]{LocalCacheConfig.class.getName()};
    }
}

@Configuration
@Import(CacheImportSelector.class)
public class CacheAutoConfiguration { }
```

**`ImportBeanDefinitionRegistrar`** — программная регистрация бинов с полным контролем:
```java
public interface ImportBeanDefinitionRegistrar {
    void registerBeanDefinitions(AnnotationMetadata metadata, BeanDefinitionRegistry registry);
}
// Пример: регистрация бинов на основе аннотации @EnableFeature
public class FeatureRegistrar implements ImportBeanDefinitionRegistrar {

    @Override
    public void registerBeanDefinitions(
    
	    AnnotationMetadata metadata, BeanDefinitionRegistry registry) {
	    
        // Читаем атрибуты аннотации
        Map<String, Object> attributes = 
	        metadata.getAnnotationAttributes(EnableFeature.class.getName());
	        
        String mode = (String) attributes.get("mode");
        
        if ("async".equals(mode)) {
            BeanDefinitionBuilder builder = 
	            BeanDefinitionBuilder.rootBeanDefinition(AsyncFeature.class);
	            
            registry.registerBeanDefinition(
	            "feature", builder.getBeanDefinition());
	            
        } else {
            BeanDefinitionBuilder builder = 
	            BeanDefinitionBuilder.rootBeanDefinition(SyncFeature.class);
	            
            registry.registerBeanDefinition(
	            "feature", builder.getBeanDefinition());
        }
    }
}
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Import(FeatureRegistrar.class)
public @interface EnableFeature {
    String mode() default "sync";
}

// Использование
@SpringBootApplication
@EnableFeature(mode = "async")
public class Application { }
```

**Разница:**

|Характеристика|`ImportSelector`|`ImportBeanDefinitionRegistrar`|
|---|---|---|
|**Что возвращает**|Имена классов (строки)|Регистрирует `BeanDefinition` напрямую|
|**Гибкость**|Средняя|Максимальная|
|**Доступ к `BeanDefinitionRegistry`**|Нет|Да|
|**Типичное использование**|Выбор готовых конфигураций|Создание бинов с кастомными параметрами|

### 108. Что такое `Conditional` аннотации? <br>(`@ConditionalOnMissingBean`, `@ConditionalOnProperty`, `@ConditionalOnClass`)

**Ответ:** `@Conditional` — механизм условной регистрации бинов. Spring Boot предоставляет множество удобных conditional аннотаций для автоконфигурации.

**Основные conditional аннотации Spring Boot:**

|Аннотация|Условие|
|---|---|
|`@ConditionalOnClass`|Указанный класс присутствует в classpath|
|`@ConditionalOnMissingClass`|Указанный класс отсутствует|
|`@ConditionalOnBean`|Указанный бин уже существует в контексте|
|`@ConditionalOnMissingBean`|Указанный бин **отсутствует** (самое частое)|
|`@ConditionalOnProperty`|Свойство имеет определенное значение|
|`@ConditionalOnWebApplication`|Приложение является веб-приложением|
|`@ConditionalOnNotWebApplication`|Приложение НЕ веб-приложение|
|`@ConditionalOnExpression`|SpEL выражение истинно|
|`@ConditionalOnJava`|Версия Java соответствует|
|`@ConditionalOnSingleCandidate`|Только один кандидат бина|

**Примеры использования:**
```java
@Configuration
public class DataSourceAutoConfiguration {
    
    // Если DataSource уже есть в контексте — не создаем
    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    public DataSource defaultDataSource() {
        return new HikariDataSource();
    }
    
    // Только если есть драйвер PostgreSQL в classpath
    @Bean
    @ConditionalOnClass(name = "org.postgresql.Driver")
    public PostgreSQLDataSource postgresDataSource() {
        return new PostgreSQLDataSource();
    }
    
    // Включаем кэш только если свойство cache.enabled=true
    @Bean
    @ConditionalOnProperty(
	    name = "cache.enabled", 
	    havingValue = "true", 
	    matchIfMissing = false)
    public CacheManager cacheManager() {
        return new CaffeineCacheManager();
    }
    
    // Только для веб-приложений
    @Bean
    @ConditionalOnWebApplication
    public WebMvcConfigurer webMvcConfigurer() {
        return new MyWebMvcConfigurer();
    }
    
    // Сложное условие через SpEL
    @Bean
    @ConditionalOnExpression(
	    "'${app.mode}'.equals('prod') and ${app.feature.enabled:true}")
    public FeatureService featureService() {
        return new ProdFeatureService();
    }
}
```

**Создание своей `conditional` аннотации:**
```java
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Conditional(OnCustomPropertyCondition.class)
public @interface ConditionalOnCustomProperty {
    String value();
}
public class OnCustomPropertyCondition implements Condition {
    @Override
    public boolean matches(
			    ConditionContext context, 
			    AnnotatedTypeMetadata metadata) {
        String property = (String) 
        metadata.getAnnotationAttributes().get("value");
        return context.getEnvironment().getProperty(property) != null;
    }
}
```

### 109. Как написать свой собственный Spring Boot Starter?

**Ответ:** Spring Boot Starter — это JAR с автоконфигурацией, который автоматически настраивает бины при наличии определенных зависимостей.

**Шаги создания стартера:**

**1. Создайте Maven/Gradle проект с двумя модулями:**
```text
my-spring-boot-starter/
├── my-spring-boot-autoconfigure/   # автоконфигурация
└── my-spring-boot-starter/         # пустой стартер (только зависимости)
```

**2. Создайте класс конфигурации:**
```java
@Configuration
@ConditionalOnClass(MyService.class)  // если есть класс
@EnableConfigurationProperties(MyProperties.class)
public class MyAutoConfiguration {
    
    @Autowired
    private MyProperties properties;
    
    @Bean
    @ConditionalOnMissingBean
    public MyService myService() {
        return new MyService(properties.getTimeout(), properties.getUrl());
    }
    
    @Bean
    @ConditionalOnMissingBean
    public MyHealthIndicator myHealthIndicator(MyService myService) {
        return new MyHealthIndicator(myService);
    }
}
```

**3. Создайте класс свойств:**
```java
@ConfigurationProperties(prefix = "my")
public class MyProperties {

    private int timeout = 5000;
    private String url = "http://default.com";
    
    // геттеры и сеттеры
}
```

**4. Создайте `spring.factories` (*для Spring Boot 2.x*) в `META-INF/`:**
```properties
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.MyAutoConfiguration
```

**Для *Spring Boot 3.x* используйте <br>`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`:**
```text
com.example.MyAutoConfiguration
```

**5. Создайте модуль-стартер (`pom.xml`):**
```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>my-spring-boot-autoconfigure</artifactId>
</dependency>

<!-- зависимости, которые нужны для работы -->

<dependency>
    <groupId>com.example</groupId>
    <artifactId>my-core-lib</artifactId>
</dependency>
```

**6. Документация и метаданные (опционально):**  
Добавьте зависимость для генерации метаданных в autoconfigure модуль:
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-configuration-processor</artifactId>
    <optional>true</optional>
</dependency>
```

**Итоговая структура:**
```java
@SpringBootApplication
@EnableConfigurationProperties
public class Application {
    // Теперь пользователь может настроить:
    // my.timeout=10000
    // my.url=http://myservice.com
}
```

### 110. Что такое `ApplicationRunner` и `CommandLineRunner`? Когда выполняются?

**Ответ:** Это интерфейсы для выполнения кода **после полного старта Spring контекста**, но **до того, как приложение начнет принимать запросы** (для веб-приложений).

**Разница:**

|Характеристика|`CommandLineRunner`|`ApplicationRunner`|
|---|---|---|
|**Метод**|`run(String... args)`|`run(ApplicationArguments args)`|
|**Доступ к опциям**|Только массив строк|Парсинг опций (`--key=value`)|
|**Доступ к аргументам**|`args[0]`, `args[1]`|`args.getOptionNames()`, `args.getNonOptionArgs()`|

**Примеры:**
```java
@Component
@Order(1)  // порядок выполнения
public class MyCommandLineRunner implements CommandLineRunner {
    @Override
    public void run(String... args) throws Exception {
        System.out.println("CommandLineRunner: " + Arrays.toString(args));
        // Загрузка начальных данных, проверка конфигурации
    }
}

@Component
@Order(2)
public class MyApplicationRunner implements ApplicationRunner {
    @Override
    public void run(ApplicationArguments args) throws Exception {
        System.out.println(
	        "ApplicationRunner non-option args: " + args.getNonOptionArgs());
        System.out.println(
	        "ApplicationRunner option args: " + args.getOptionNames());
        
        // args --profile=prod --port=8080
        String profile = args.getOptionValues("profile").get(0);
    }
}
```

**Когда выполняются:**
- После инициализации `ApplicationContext`    
- После вызова всех `BeanPostProcessor`    
- После всех `@PostConstruct`    
- **До** того, как встроенный сервер (Tomcat) начнет принимать запросы    

**Типичное использование:**
- Загрузка начальных данных в БД    
- Проверка наличия обязательных ресурсов    
- Запуск batch-процесса    
- Разогрев кэша    

**Важный нюанс:** Если в runner возникает исключение, приложение **не стартует**. Для graceful обработки используйте `try-catch`.

### 111. Как Spring Boot обрабатывает `@Value` с выражениями (`${...}`) и SpEL (`#{...}`)?

**Ответ:** `@Value` поддерживает два типа выражений:

|Тип|Синтаксис|Описание|
|---|---|---|
|**Property placeholder**|`${property.name}`|Извлекает значение из `PropertySources` (properties, YAML, env vars)|
|**SpEL (Spring Expression Language)**|`#{expression}`|Выполняет динамическое выражение на бинах и их методах|

**Property placeholder (`${}`):**
```java
@Component
public class MyComponent {
    
    @Value("${app.name}")              // из application.properties
    private String appName;
    
    @Value("${app.timeout:1000}")      // со значением по умолчанию
    private int timeout;
    
    @Value("${random.int(1,100)}")     // случайное число
    private int randomValue;
    
    @Value("${JAVA_HOME}")             // из переменной окружения
    private String javaHome;
    
    @Value("${my.list}")               // строка "a,b,c" → List
    private List<String> list;          // требует бина-конвертера
}
```

**SpEL (`#{}`):**
```java
@Component
public class SpelComponent {
    
    @Value("#{systemProperties['user.home']}")      // системное свойство
    private String userHome;
    
    @Value("#{T(java.lang.Math).random() * 100}")   // статический метод
    private double random;
    
    @Value("#{@dataSource.url}")                    // поле другого бина
    private String dbUrl;
    
    @Value("#{userService.findDefaultEmail()}")     // вызов метода бина
    private String defaultEmail;
    
    @Value("#{user.age > 18 ? 'ADULT' : 'MINOR'}")  // тернарный оператор
    private String ageGroup;
}
```

**Комбинация `${}` и `#{}`:**
```java
// Сначала резолвится ${}, потом результат передается в #{}
@Value("#{'${app.names}'.split(',')}")
private List<String> names;
@Value("${app.base-url}#{'/api/v1/users'}")
private String usersUrl;
```

**Как работает под капотом:**
1. `AutowiredAnnotationBeanPostProcessor` обрабатывает `@Value`.    
2. `StringValueResolver` резолвит значения.    
3. Для `${}` используется `PropertySourcesPlaceholderConfigurer` (или `EmbeddedValueResolver` в Boot).    
4. Для `#{}` используется `BeanExpressionResolver`.

### 112. Что такое `DeferredResult` или `Callable` в Spring MVC? (Асинхронные запросы, освобождение tomcat-потока)

**Ответ:** `Callable` и `DeferredResult` — механизмы **асинхронной обработки запросов** в Spring MVC, которые освобождают Tomcat-поток на время долгой операции.

**Проблема:** Обычный запрос занимает Tomcat-поток на все время выполнения. При долгих операциях (вызов внешнего API, долгий расчет) потоки заканчиваются.

**`Callable` — простой асинхронный результат:**
```java
@RestController
public class AsyncController {
    
    @GetMapping("/async-callable")
    public Callable<String> asyncCallable() {
        // Tomcat-поток освобождается сразу
        return () -> {
            Thread.sleep(5000);  // долгая операция в отдельном потоке
            return "Result after 5 seconds";
        };
    }
    
    @GetMapping("/async-callable-with-timeout")
    public Callable<String> asyncWithTimeout() {
        return () -> {
            // долгая операция
            return "result";
        };
    }
}
```

**`DeferredResult` — ручное управление (более гибкий):**
```java
@RestController
public class DeferredResultController {
    
    private final Map<String, DeferredResult<String>> pendingRequests = new ConcurrentHashMap<>();
    
    @GetMapping("/deferred")
    public DeferredResult<String> deferred() {
		    
        DeferredResult<String> result = 
	        new DeferredResult<>(10000L);  // таймаут 10 сек
	        
        String requestId = UUID.randomUUID().toString();
        pendingRequests.put(requestId, result);
        
        result.onTimeout(() -> {
            pendingRequests.remove(requestId);
            result.setErrorResult("Timeout");
        });
        
        result.onCompletion(() -> pendingRequests.remove(requestId));
        
        return result;
    }
    
    @PostMapping("/callback/{requestId}")
    public void callback(
	    @PathVariable String requestId, @RequestBody String data) {
        DeferredResult<String> result = pendingRequests.remove(requestId);
        if (result != null) {
            result.setResult(data);  // завершаем ожидающий запрос
        }
    }
}
```

**Сравнение:**

|Характеристика|Обычный синхронный|`Callable`|`DeferredResult`|
|---|---|---|---|
|**Освобождает Tomcat-поток**|Нет|Да|Да|
|**Простота**|Просто|Средне|Сложно|
|**Управление из другого потока**|Нет|Нет|Да (callback, MQ, внешний вызов)|
|**Таймаут**|По соединению|Поддерживается|Поддерживается|
|**Обработка ошибок**|Стандартная|Через исключения|`setErrorResult()`|

**Когда что использовать:**
- **`Callable`** — для простых асинхронных операций внутри приложения    
- **`DeferredResult`** — когда ответ приходит из другого потока, очереди (JMS, Kafka), от внешнего callback-вызова    

**Важный нюанс:** Нужно настроить `TaskExecutor` для асинхронных запросов:
```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {
    
    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(50);
        executor.setQueueCapacity(200);
        executor.initialize();
        return executor;
    }
}
```

### 113. Как работает управление кешами (`@Cacheable`, `@CacheEvict`, `@EnableCaching`)?

**Ответ:** Spring Cache — это абстракция над кэш-провайдерами (Caffeine, Redis, EhCache, ConcurrentHashMap). Она работает через **AOP-прокси**, перехватывая вызовы методов и проверяя кэш.

**Как включить:**
```java
@Configuration
@EnableCaching  // включает Spring Cache
public class CacheConfig {
    
    @Bean
    public CacheManager cacheManager() {
        // Caffeine (in-memory, высокопроизводительный)
        CaffeineCacheManager cacheManager = 
	        new CaffeineCacheManager("users", "products");
        cacheManager.setCaffeine(Caffeine.newBuilder()
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .maximumSize(1000));
        return cacheManager;
    }
}
```

**Основные аннотации:**

|Аннотация|Назначение|
|---|---|
|`@Cacheable`|Кэширует результат метода. При повторном вызове с теми же аргументами возвращает значение из кэша|
|`@CacheEvict`|Удаляет записи из кэша (при изменении/удалении данных)|
|`@CachePut`|Выполняет метод и обновляет кэш (не проверяет кэш перед вызовом)|
|`@Caching`|Группирует несколько кэш-аннотаций|
|`@CacheConfig`|Общие настройки кэша на уровне класса|

**Примеры:**
```java
@Service
@CacheConfig(cacheNames = "users")  // общее имя кэша для всех методов
public class UserService {
    
    // Результат кэшируется по ключу = id
    @Cacheable(key = "#id")
    public User findById(Long id) {
        System.out.println("DB call for id: " + id);
        return userRepository.findById(id).orElse(null);
    }
    
    // Ключ из нескольких параметров
    @Cacheable(key = "#name + ':' + #age")
    public User findByNameAndAge(String name, int age) {
        return userRepository.findByNameAndAge(name, age);
    }
    
    // Кэширование всех пользователей
    @Cacheable(value = "allUsers", key = "'all'")
    public List<User> findAllUsers() {
        return userRepository.findAll();
    }
    
    // Удаление из кэша при обновлении
    @CacheEvict(key = "#user.id")
    public User update(User user) {
        return userRepository.save(user);
    }
    
    // Удаление всех записей из кэша
    @CacheEvict(value = "users", allEntries = true)
    public void clearCache() {
        // очистка кэша
    }
    
    // Conditional кэширование
    @Cacheable(key = "#id", condition = "#id > 10", unless = "#result == null")
    public User findByIdWithCondition(Long id) {
        return userRepository.findById(id).orElse(null);
    }
    
    // @CachePut — всегда вызывает метод и обновляет кэш
    @CachePut(key = "#user.id")
    public User refresh(User user) {
        return userRepository.save(user);
    }
}
```

**Генерация ключей (по умолчанию — `SimpleKeyGenerator`):**
- Без параметров → `SimpleKey.EMPTY`    
- Один параметр → сам параметр    
- Несколько параметров → `SimpleKey` из них    

**Кастомный ключ через SpEL:**
```java
@Cacheable(key = "#root.methodName + ':' + #id")           // метод + id
@Cacheable(key = "#root.targetClass.simpleName + ':' + #id")
@Cacheable(key = "#result.id")                              // из результата (только в @CachePut)
@Cacheable(key = "T(com.example.MyUtils).generateKey(#id)") // вызов статического метода
```

**`@CacheEvict` с параметрами:**
```java
@CacheEvict(key = "#id")                    // удалить конкретную запись
@CacheEvict(allEntries = true)              // очистить весь кэш
@CacheEvict(key = "#id", beforeInvocation = true) // удалить до вызова метода
```

**Поддерживаемые кэш-провайдеры:**

|Провайдер|Описание|
|---|---|
|`ConcurrentMapCache`|По умолчанию, in-memory, для разработки|
|`CaffeineCache`|Высокопроизводительный, для production|
|`RedisCache`|Распределенный кэш|
|`EhCacheCache`|Зрелый, с дисковым хранилищем|
|`HazelcastCache`|Распределенный|

**Аннотация `@Caching` для группировки:**
```java
@Caching(
    evict = {
        @CacheEvict(key = "#id"),
        @CacheEvict(value = "allUsers", key = "'all'")
    }
)
public void delete(Long id) {
    userRepository.deleteById(id);
}
```
