# Модификаторы доступа в Java и их применимость к классам

В Java модификаторы доступа определяют **границы видимости** (область инкапсуляции) для классов, интерфейсов, полей и методов. Они делятся на четыре уровня.

## 🧩 1. Четыре уровня доступа в Java (Сводная матрица)
Для понимания иерархии уровней важно помнить, как они соотносятся друг с другом по степени «открытости» (от самого строгого к самому свободному):

|**Модификатор**|**Внутри того же класса**|**Внутри того же пакета**|**В наследниках (даже в других пакетах)**|**Во внешнем мире (везде)**|
|---|---|---|---|---|
|`private`|✅ Да|❌ Нет|❌ Нет|❌ Нет|
|`package-private` (default)|✅ Да|✅ Да|❌ Нет|❌ Нет|
|`protected`|✅ Да|✅ Да|✅ Да|❌ Нет|
|`public`|✅ Да|✅ Да|✅ Да|✅ Да|

_Примечание: Ключевого слова `default` как модификатора доступа в коде не существует (оно используется только для default-методов интерфейсов или в `switch`). Если перед классом или методом модификатор не указан вовсе — этот уровень называется **package-private**._

## 🏗️ 2. Применимость к верхнеуровневым (Top-Level) классам
К **верхнеуровневым классам** (которые лежат непосредственно в файле `.java` внутри пакета) можно применить **только два** модификатора доступа из четырех:

### ── ✅ Что МОЖНО использовать:

- **`public class MyClass`**: Класс виден абсолютно любому другому классу в проекте (и во внешних библиотеках, если модуль экспортирует этот пакет).
    
- **`class MyClass` (package-private)**: Класс доступен для использования и наследования **строго внутри своего пакета**. Из других пакетов обратиться к нему невозможно.

### ── ❌ Что ИСПОЛЬЗОВАТЬ НЕЛЬЗЯ (Ошибка компиляции):
```Java
private class PrivateClass { }   // ❌ Ошибка компиляции!
protected class ProtectedClass { } // ❌ Ошибка компиляции!
```

## 🧠 3. Глубокое техническое обоснование: ПОЧЕМУ введен такой запрет?

Интервьюеры уровня Middle/Senior часто спрашивают не просто «что можно, а что нельзя», а именно логику создателей языка: _«Почему `private` и `protected` запрещены на верхнем уровне?»_

### ── 1. Почему `private` для top-level класса не имеет смысла?

Смысл модификатора `private` — запереть видимость элемента **внутри фигурных скобок `{}` родительской структуры**.

Если мы сделаем верхнеуровневый класс приватным, то его областью видимости станет сам этот файл. В итоге этот класс **не сможет использовать вообще никто во всей JVM**, даже классы из его собственного пакета. Такой класс превратился бы в абсолютно мертвый код. Компилятор Java пресекает это на корню.

### ── 2. Почему `protected` для top-level класса избыточен и неприменим?

Модификатор `protected` расширяет package-private доступ, добавляя возможность видеть элемент **классам-наследникам из чужих пакетов**.

Но чтобы чужой класс в принципе смог унаследоваться от нашего класса (`class Child extends Parent`), он должен сначала **увидеть** сам этот `Parent`.

- Если бы `Parent` был package-private, чужой класс из другого пакета его бы не увидел и не смог бы написать `extends`.
    
- Если бы мы сделали его `public`, то `protected` уже не нужен, ведь класс и так виден всем.

Таким образом, на уровне пакетов `protected` полностью теряет свой смысл и логически дублирует package-private.

## ⚡ 4. Исключение из правил: Вложенные (Nested) классы

Как только класс перестает быть верхнеуровневым и объявляется **внутри другого класса**, он приобретает статус полноправного члена (Member) этого класса — наравне с обычными полями и методами.

Для **вложенных (nested) классов** становятся доступны **все четыре модификатора доступа**:
```Java
public class OuterClass {

    // Доступен только внутри скобок OuterClass
    private class InnerPrivate { } 

    // Доступен в OuterClass, внутри его пакета и всем его наследникам
    protected class InnerProtected { } 

    // Доступен внутри пакета
    class InnerPackagePrivate { } 

    // Доступен всему миру через OuterClass.InnerPublic
    public class InnerPublic { } 
}
```
