# Архитектура JVM, Классы-загрузчики (ClassLoaders) и Динамическая загрузка

---
## 🌐 1. Концепция JVM и исполнение кода
**Виртуальная машина Java (JVM)** — это изолированная программная среда (*виртуальное представление физического компьютера*), выступающая ядром экосистемы Java.
	
- **Гибридный подход:** В отличие от компилируемых (C, C++) или интерпретируемых (Python, JS) языков, Java совмещает оба метода. Компилятор (`javac`) переводит исходный код `.java` в независимый от платформы **байт-код** (`.class` файлы). Затем JVM интерпретирует этот байт-код в машинные инструкции конкретной ОС, реализуя принцип _"Write Once, Run Anywhere"_ (WORA).
    
- **Языки JVM:** Платформа поддерживает не только Java, но и другие языки (Kotlin, Scala, Groovy), компилируемые в аналогичный байт-код.

---
## 🏗️ 2. Три главных компонента архитектуры JVM
Архитектурно виртуальная машина Java состоит из трех ключевых подсистем:
1. **Подсистема загрузчика классов (*Class Loader Subsystem*)**.    
2. **Область данных среды выполнения (*Runtime Data Areas / Память*)**.    
3. **Механизм выполнения (*Execution Engine*)**.    

---
## 🔌 3. Подсистема загрузчика классов (Class Loader Subsystem)
Когда приложение запускается или запрашивает новый класс, подсистема `Class Loader` находит байт-код и загружает его в память. Процесс состоит из трех последовательных этапов: **Загрузка ──► Связывание ──► Инициализация**.

### 🛠️ Этап I. Загрузка (*Loading*)
JVM считывает бинарный байт-код файла `.class` по его полному имени (FQN) и создает его репрезентацию в памяти. Загрузка построена на **трех встроенных загрузчиках**, организованных в строгую иерархию:
```text
[1. Bootstrap ClassLoader]  <── (Корневой / Самый доверенный)
			▲
			│
[2. Extension ClassLoader]  <── (Загрузчик расширений)
			▲
			│
[3. Application ClassLoader] <── (Системный / Локальный classpath)
			▲
			│
[User-Defined ClassLoader]  <── (Пользовательские кастомные загрузчики)
```

1. **Bootstrap ClassLoader (Базовый / Корневой):**    
    - Самый доверенный источник, написан на нативном коде (C/C++), 
	      не является Java-классом.        
    - **Что загружает:** Базовые пакеты Java Core API 
	      (`java.lang.*`, `java.util.*`, `java.net.*`, `java.io.*`).        
    - **Откуда:** Исторически из архива `rt.jar` и основных библиотек 
	      в каталоге `$JAVA_HOME/jre/lib`.
    
2. **Extension ClassLoader (Загрузчик расширений):**
    - Является подклассом _Bootstrap_ и родителем для _Application_.
    - **Что загружает:** Расширения стандартных библиотек Java.
    - **Откуда:** Каталог `$JAVA_HOME/jre/lib/ext/` или системный путь, 
	      указанный в `java.ext.dirs`.
        
3. **Application ClassLoader (Системный загрузчик приложений):**
    - Конечный встроенный загрузчик, подкласс _Extension_.        
    - **Что загружает:** Классы самого приложения 
	      и подключенные внешние зависимости (библиотеки).
    - **Откуда:** Путь, прописанный в переменной окружения **`CLASSPATH`** или переданный через параметры командной строки **`-classpath`** / **`-cp`**.

#### 🔄 Принцип делегирования (Delegation Principle)

Загрузка классов строго подчиняется иерархическому принципу делегирования снизу вверх:
	
- Когда `Application ClassLoader` получает запрос на загрузку класса, он не ищет его сам, а **делегирует** запрос наверх к `Extension ClassLoader`.
    
- `Extension` делегирует запрос еще выше — к `Bootstrap`.
    
- `Bootstrap` проверяет свои базовые пути. Если класс найден — он загружается. Если нет — управление возвращается вниз к `Extension`.
    
- Если ни один родитель не нашел класс, текущий дочерний загрузчик пытается прочитать файл сам. Если класс так и не обнаружен на самом нижнем уровне, выбрасывается исключение **`ClassNotFoundException`** или **`NoClassDefFoundError`**.

### 🔗 Этап II. Связывание (Linking)

Связывание подготавливает загруженный класс к использованию и разделяется на три шага:

1. **Проверка (Verification):** Проверяется структурная и синтаксическая корректность байт-кода на соответствие спецификациям JVM. Защищает систему от вредоносного изменения байт-кода. Если класс скомпилирован на новой версии Java (например, Java 11), а запускается на старой (Java 8), проверка упадет с ошибкой **`VerifyException`**.
    
2. **Подготовка (Preparation):** Выделяется память под статические поля (`static`) класса. На этом шаге поля инициализируются **значениями по умолчанию** (для числовых типов — `0`, для объектов — `null`, для boolean — `false`).
	    
    - _Пример:_ Для строки `private static final boolean enabled = true;` на этапе подготовки в памяти выделится место, но значение будет установ. в `false`.
    
3. **Разрешение (Resolution):** Необязательный (может быть ленивым) этап. Символические ссылки, находящиеся в пуле констант времени выполнения, заменяются на прямые физические ссылки в памяти.

### 🏁 Этап III. Инициализация (Initialization)

Заключительный этап. Выполняется специальный скрытый метод инициализации класса — **`<clinit>`**.

- Происходит вызов статических блоков инициализации (`static { ... }`) и присвоение статическим переменным их **реальных (фактических) значений**, заданных в коде.
    
    - _Пример:_ Переменная `enabled` из шага подготовки окончательно получает свое истинное значение `true`.
        
- **Потокобезопасность:** JVM гарантирует, что инициализация класса является потокобезопасной. Если несколько потоков одновременно запрашивают инициализацию одного класса, процесс будет заблокирован и выполнен строго один раз.

## 💾 4. Область данных среды выполнения (Runtime Data Areas / Память)

### 🌐 Общие области (Доступны всем потокам, не потокобезопасны):

- **Область метода (Method Area / Metaspace):** Хранит данные уровня класса: пул констант времени выполнения, метаданные полей и методов, код конструкторов и методов. Создается при старте JVM. Нехватка памяти вызывает **`OutOfMemoryError`**.
    
- **Область кучи (Heap Area):** Главное хранилище объектов. Здесь выделяется память под все экземпляры классов (`new MyObject()`) и массивы. Объект создается в куче, а ссылка на него живет в стеке потока. Нехватка места приводит к **`OutOfMemoryError`**.

### 🔒 Изолированные области (Выделяются на каждый поток, потокобезопасны):

- **Область стека (Stack Area):** Каждому потоку выделяется свой стек. При каждом вызове метода в стек помещается структура — **Фрейм стека (Stack Frame)**, которая уничтожается при выходе из метода. Превышение лимита глубины стека (например, бесконечная рекурсия) вызывает **`StackOverflowError`**.
    
    - _Анатомия фрейма стека:_
        1. _Локальные переменные:_ Массив, хранящий примитивы и ссылки на объекты, используемые внутри метода.
            
        2. _Стек операндов:_ LIFO-структура, рабочая зона для выполнения промежуточных математических и логических операций.
            
        3. _Данные фрейма:_ Символьные ссылки метода и информация для обработки исключений (блоки `catch`).
        
- **Регистры счетчика программ (PC Registers):** Хранят адрес текущей исполняемой инструкции потока. После выполнения инструкции счетчик автоматически обновляется адресом следующей.
    
- **Стеки нативных методов (Native Method Stacks):** Выделенный стек для поддержки выполнения кода, написанного на других языках (C/C++), вызываемого через механизмы Java.

## ⚙️ 5. Механизм выполнения (Execution Engine)
Преобразует байт-код, лежащий в областях данных, в машинные инструкции процессора. Работает через три подсистемы:

### ── Интерпретатор против JIT-компилятора

- **Интерпретатор:** Читает байт-код строка за строкой и сразу выполняет его. Метод работает быстро при старте, но медленно на длинной дистанции, так как при повторном вызове метода строка кода будет интерпретироваться заново.
    
- **JIT-компилятор (Just-In-Time):** Повышает общую производительность. Специальный компонент (**Профилировщик**) ищет "горячие точки" (Hot Spots) — участки кода и методы, которые выполняются чаще всего. JIT компилирует этот байт-код целиком в нативный машинный код. Повторные вызовы используют нативный код напрямую.
	    
    - _Пример оптимизации:_ В цикле `for` обычный интерпретатор на каждой итерации будет перезаписывать значение переменной в память. JIT-компилятор соптимизирует код: сохранит значение в регистре процессора на время работы цикла и запишет в память один раз по его завершении.

### ── Сборщик мусора (Garbage Collector, GC)

Автоматически сканирует область Кучи (Heap), находит объекты, на которые больше никто не ссылается, и уничтожает их. Процесс включает два этапа: **Пометка (Mark)** — идентификация живых и неиспользуемых объектов, и **Очистка (Sweep)** — освобождение памяти.

#### Три основных типа GC в JVM:
1. **Последовательный (Serial GC) [`-XX:+UseSerialGC`]:** Работает в один поток. При запуске полностью останавливает работу всего приложения (эффект _Stop-The-World_ — STW). Подходит для простых однопоточных консольных утилит.
    
2. **Параллельный (Parallel GC) [`-XX:+UseParallelGC`]:** GC по умолчанию для старых версий Java. Использует несколько потоков для очистки памяти, сокращая время паузы, но эффект STW все еще присутствует.
    
3. **Garbage First (G1 GC) [`-XX:+UseG1GC`]:** Современный стандарт для многопоточных приложений с большими объемами кучи (более 4 ГБ). Делит кучу на множество мелких областей одинакового размера, находит регионы с наибольшим количеством мусора и очищает их в первую очередь, минимизируя время пауз STW.

## 🔌 6. Нативный интерфейс Java (JNI) и Нативные библиотеки
- **JNI (Java Native Interface):** Мост-интерфейс, позволяющий Java-коду вызывать функции, написанные на языках C, C++ или Ассемблере (например, для прямого взаимодействия с низкоуровневым оборудованием или ОС).
    
- Методы, реализованные нативно, помечаются в коде ключевым словом **`native`**, а сама библиотека подгружается через метод `System.loadLibrary()`.
    
- **Нативные библиотеки методов:** Скомпилированные файлы платформозависимого кода в формате `.dll` (для Windows) или `.so` (для Linux).

## ⚡ 7. Динамическая загрузка классов
Динамическая загрузка — это механизм, позволяющий JVM загружать новые классы в память прямо во время выполнения программы, когда они физически не были известны на этапе компиляции. Это основа архитектур плагинов, рефлексии (Reflection API) и работы фреймворков (Spring, Hibernate).

В Java динамически загрузить класс можно двумя основными инструментами, имеющими принципиальное отличие:

### 🛠️ Инструмент 1: <br>`Class.forName("ПолноеИмяКласса")`

- **Что делает:** Загружает байт-код класса, проводит связывание и **обязательно запускает инициализацию** (выполняет статические блоки кода и инициализирует статические поля).
    
- **Сценарий:** Классический пример — старая загрузка драйверов СУБД JDBC (`Class.forName("org.postgresql.Driver")`), когда при вызове метода драйвер должен был сам статически зарегистрировать себя в системе.

### 🛠️ Инструмент 2: <br>`ClassLoader.loadClass("ПолноеИмяКласса")`

- **Что делает:** Напрямую задействует подсистему загрузчиков. Он только **загружает** класс в память, но **НЕ инициализирует его**. Статические блоки выполнены не будут, пока к классу не обратятся напрямую (например, через создание экземпляра).

## 🚨 8. Главный чек-лист распространенных ошибок JVM

| **Ошибка / Исключение**      | **Тип**           | **Почему возникает на самом деле**                                                                                                                                                                                                                                |
| ---------------------------- | ----------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`ClassNotFoundException`** | Checked Exception | Запрос на динамическую загрузку класса (`Class.forName()`, `loadClass()`) выполнен по строковому имени, но класс с таким именем **физически отсутствует** в доступных путях поиска загрузчиков.                                                                   |
| **`NoClassDefFoundError`**   | Error             | Компиляция прошла успешно, файл `.class` существовал. Однако во время выполнения JVM попыталась обратиться к этому классу, но загрузчик классов **не смог найти его файл в текущем classpath** (например, jar-архив удалили или не донесли на сервер при деплое). |
| **`OutOfMemoryError`**       | Error             | Заполнена физическая память Области Кучи (Heap) или Области метода (Metaspace). Новые объекты не могут быть созданы, так как свободного места нет, а Сборщик мусора (GC) не может высвободить память (все объекты удерживаются сильными ссылками).                |
| **`StackOverflowError`**     | Error             | В области стека конкретного потока закончилось свободное место для создания нового фрейма. Самая частая причина — некорректная, бесконечная рекурсия или чрезмерно глубокий вызов методов.                                                                        |
