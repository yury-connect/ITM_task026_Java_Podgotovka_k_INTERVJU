Эти страшные слова из теории типов на самом деле описывают очень простую вещь: **«Как переносятся отношения наследования между базовыми типами на производные от них типы (например, коллекции или массивы)»**.

Простой пример для старта: мы все знаем, что `Integer` — это наследник `Number`. А вот является ли `List<Integer>` наследником `List<Number>`? Ответ на этот вопрос как раз и зависит от того, является ли коллекция _инвариантной_, _ковариантной_ или _контрвариантной_.

Давай разберем эти три понятия раз и навсегда на простых примерах из Java.

---
## 1. **Инвариантность** (*Invariance*)

> **Суть:** Родственные связи исходных типов **НЕ переносятся** на производные типы. Никакой связи между коллекциями нет.

Если `A` является родителем `B`, то `Коллекция<A>` и `Коллекция<B>` — это абсолютно чужие друг другу типы. Ты не можешь присвоить один другому.

По умолчанию **дженерики в Java инвариантны**.
```Java
// Integer — наследник Number, но...
List<Number> list = new ArrayList<Integer>(); // ОШИБКА КОМПИЛЯЦИИ!
```

- **Зачем это нужно:** Для стопроцентной безопасности типов (`Type Safety`). Если бы Java разрешила такую строчку, то через ссылку `list` (которая думает, что работает с `Number`) мы могли бы легально положить туда `Double (3.14)`. Но в памяти-то лежит список `Integer`! Программа бы упала в рантайме. Инвариантность защищает нас от этого еще на этапе компиляции.    

---
## 2. **Ковариантность** (*Covariance*)

> **Суть:** Родственные связи исходных типов **сохраняются в том же виде** и для производных типов.

Если `A` — родитель `B`, то `Коллекция<B>` считается наследником `Коллекция<A>`. Направление стрелки наследования сохраняется.

В Java ковариантность можно включить двумя способами:

1. **Через массивы (они ковариантны по умолчанию):**
```Java
Number[] array = new Integer[5]; // Компилируется! Направление сохранилось.
```
    
2. **Через Wildcard `? extends T` (Upper Bound Wildcard):**
```Java
List<? extends Number> list = new ArrayList<Integer>(); // Компилируется!
```

- **Ограничение ковариантности (Только чтение):** Ковариантная коллекция превращается в **Producer** (вспоминаем правило PECS). Из списка `List<? extends Number>` можно безопасно **читать** элементы (они гарантированно будут `Number`), но в него **нельзя ничего записать**, так как компилятор не знает, список _каких именно_ наследников (`Integer` или `Double`) там скрывается.

---
## 3. **Контрвариантность** (*Contravariance*)

> **Суть:** Родственные связи исходных типов **разворачиваются наизнанку** для производных типов.

Если `A` — родитель `B`, то `Коллекция<A>` внезапно становится «наследником» (подтипом) для `Коллекция<B>`. Стрелка наследования переворачивается в противоположную сторону.

В Java контрвариантность включается через Wildcard `? super T` (Lower Bound Wildcard):
```Java
// Мы можем присвоить список РОДИТЕЛЕЙ (Number) в переменную для списка НАСЛЕДНИКОВ
List<? super Integer> list = new ArrayList<Number>(); // Компилируется!
```

- **Ограничение контрвариантности (Только запись):** Контрвариантная коллекция становится **Consumer**. В этот список можно безопасно **записывать** элементы типа `Integer` (и его подтипы), потому что они гарантированно влезут и в `List<Number>`, и в `List<Object>`. А вот **читать** из него можно только в переменную типа `Object`, так как истинный верхний предел структуры в памяти компилятору неизвестен.

---
## Шпаргалка для собеседования

|**Понятие**|**Как пишется в Java**|**Родство коллекций**|**Для чего подходит (PECS)**|
|---|---|---|---|
|**Инвариантность**|`List<T>`|Полностью независимы|И чтение, и запись (но тип строго фиксирован)|
|**Ковариантность**|`List<? extends T>`|Направление то же самое|**Producer** (Только чтение)|
|**Контрвариантность**|`List<? super T>`|Направление инвертируется|**Consumer** (Только запись)|
