Методы `equals()` и `hashCode()` определены в классе `Object` и являются фундаментальными для всей экосистемы Java. Они отвечают за идентификацию объектов и их корректное поведение в коллекциях (особенно в `HashMap`, `HashSet` и `Hashtable`).

Ниже представлен исчерпывающий разбор контракта, деталей реализации, подводных камней и частных случаев.

---
## 📜 1. Контракт между `equals()` и `hashCode()`
Контракт — это свод железных правил, определяющих математическое взаимодействие двух методов. На собеседованиях его требуют отскакивающим от зубов.

- **Правило 1 (Главное):** Если два объекта равны по результатам вызова `equals()`, то их вызовы `hashCode()` **обязаны возвращать одинаковые целые числа**.
    $$\text{if } a.\text{equals}(b) \implies a.\text{hashCode}() == b.\text{hashCode}()$$
    
- **Правило 2 (Следствие):** Если два объекта имеют одинаковый `hashCode()`, это **НЕ означает**, что они равны по `equals()`. Данная ситуация называется **коллизией**.
    
- **Правило 3:** Если два объекта возвращают разные `hashCode()`, они **гарантированно не равны** по `equals()`.
    $$\text{if } a.\text{hashCode}() \neq b.\text{hashCode}() \implies !a.\text{equals}(b)$$
    
- **Правило 4 (Консистентность):** Вызов `hashCode()` должен возвращать один и тот же результат в течение всей жизни объекта, при условии, что поля, участвующие в проверке `equals()`, не изменялись.

### ── Свойства контракта `equals()` (Математические требования)

Реализация метода `equals()` обязана удовлетворять пяти свойствам **отношения эквивалентности**, определенным в JLS:

- **Рефлексивность:** Объект должен быть равен самому себе.    
    $$x.\text{equals}(x) == \text{true}$$
    
- **Симметричность:** Если объект $x$ равен $y$, то и $y$ должен быть равен $x$.    
    $$x.\text{equals}(y) \implies y.\text{equals}(x)$$
    
- **Транзитивность:** Если $x$ равен $y$, а $y$ равен $z$, то $x$ должен быть равен $z$.    
    $$\text{if } x.\text{equals}(y) \text{ and } y.\text{equals}(z) \implies x.\text{equals}(z)$$
    
- **Консистентность (Согласованность):** Повторные вызовы `equals()` должны возвращать один и тот же результат, пока поля объекта не изменены.
    
- **Проверка на null:** Сравнение любого объекта с `null` всегда должно возвращать `false`.
    $$x.\text{equals}(\text{null}) == \text{false}$$

## ⚙️ 2. Реализация по умолчанию в классе `Object`
Если ты не переопределяешь эти методы в своем классе, Java использует базовую логику:

- **По умолчанию в `equals()`:** Происходит банальное сравнение ссылок в памяти через оператор `==`. Обьект равен только самому себе.    
```Java
public boolean equals(Object obj) {
	return (this == obj);
}
```
    
- **По умолчанию в `hashCode()`:** Возвращается так называемый **Identity Hash Code**. Вопреки мифам, это _не прямой адрес объекта в памяти_. JVM вычисляет его на основе внутреннего уникального идентификатора объекта (часто через генератор псевдослучайных чисел, зашитый в нативный код виртуалки, который затем кэшируется в заголовке объекта — _Object Header_).

## 🛠️ 3. Пошаговый алгоритм правильного переопределения
При ручном написании `equals()` нужно строго соблюдать **5 шагов** для обеспечения безопасности (именно так генерируют методы современные IDE):
```Java
@Override
public boolean equals(Object o) {
    // 1. Проверка на идентичность ссылок (оптимизация скорости)
    if (this == o) return true;
    
    // 2. Проверка на null
    if (o == null) return false;
    
    // 3. Проверка строгого совпадения классов (почему не instanceof — ниже)
    if (this.getClass() != o.getClass()) return false;
    
    // 4. Приведение типа
    User user = (User) o;
    
    // 5. Покомпонентное сравнение значимых полей
    return this.age == user.age && 
           Objects.equals(this.name, user.name); // Безопасно сравнивает null
}

@Override
public int hashCode() {
    // Использует алгоритм на базе простых чисел (обычно 31) 
    // для минимизации коллизий
    return Objects.hash(name, age); 
}
```

## 🚨 4. Что будет, если переопределить один метод, а другой оставить?

Это <u>самый катастрофический сценарий</u> для коллекций на базе хэш-таблиц (`HashMap`, `HashSet`).

### ── Сценарий А: Переопределен `equals()`, но НЕ переопределен `hashCode()`

Ты создал класс `User` с логическим равенством по паспорту. У тебя есть `user1` и `user2` с одинаковыми именами. `user1.equals(user2)` возвращает `true`.
	
1. Ты кладешь `user1` в `HashMap` в качестве ключа: `map.put(user1, "Admin");`
    
2. `HashMap` берет дефолтный `hashCode()` от `user1` (например, `100500`) и кладет его в соответствующую корзину (bucket).
    
3. Пытаешься достать значение по `user2`: `map.get(user2);`
    
4. `HashMap` берет дефолтный `hashCode()` от `user2` (например, `200600`).
    
5. Хэши разные! `HashMap` идет искать в совершенно другую корзину, даже не вызывая `equals()`. **Результат: объект потерян, возвращается `null`.**

### ── Сценарий Б: Переопределен `hashCode()`, но НЕ переопределен `equals()`
	
1. Ты заставил все объекты возвращать одинаковый `hashCode() = 42`.
    
2. Ты кладешь 1000 разных объектов в `HashMap`.
    
3. Все 1000 объектов попадают в **одну и ту же корзину**, образуя огромную коллизию.
    
4. При попытке достать объект, `HashMap` находит корзину №42 и начинает перебирать элементы по цепочке, вызывая `equals()`.
    
5. Но `equals()` не переопределен! Он сравнивает ссылки (`==`). Ты никогда не найдешь объект по эквивалентному ключу, а производительность карты падает с $O(1)$ до катастрофических $O(N)$.

## 🧠 5. Детали, рекомендации и частные случаи

### ── Почему `getClass()` предпочтительнее, чем `instanceof`?

Если использовать `instanceof`, нарушается правило **симметричности** контракта в условиях наследования.

Представь, что у тебя есть `class User` и `class PremiumUser extends User`.

- Если использовать `instanceof`, то `user.equals(premiumUser)` вернет `true` (ведь PremiumUser является экземпляром User).
    
- Но при обратном вызове `premiumUser.equals(user)` метод класса-наследника проверит специфичные поля или strict-тип и вернет `false`.
    
- Использование `this.getClass() != o.getClass()` жестко гарантирует: объекты равны только если они принадлежат **абсолютно одному классу**.
    

### ── Проблема изменяемых (Mutable) полей в качестве ключей

> **Золотое правило:** Поля, участвующие в расчете `equals` и `hashCode`, должны быть неизменяемыми (`final`).

Если ты положил объект в `HashMap`, а затем изменил значение поля, которое влияло на хэш:
```Java
User user = new User("Иван");
map.put(user, "Data");
user.setName("Пётр"); // Хэш-код объекта внутри карты изменился!
map.get(user); // Вернет null, потому что карта ищет по новому хэшу в другой корзине
```

Ты заживо «замуровал» объект в памяти, создав классическую утечку памяти (Memory Leak), так как GC не сможет его удалить.

### ── Как сравнивать типы с плавающей точкой (`float`/`double`)?

Из-за особенностей округления бинарных дробей в компьютере сравнивать `float` и `double` напрямую через `==` нельзя. Используй обертки:

- Для `float`: `Float.floatToIntBits(f1) == Float.floatToIntBits(f2)`
    
- Для `double`: `Double.doubleToLongBits(d1) == Double.doubleToLongBits(d2)`
    
    _(Или просто делегируй это `Float.compare()` / `Double.compare()`)_.
    

### ── Специфика работы с массивами

Если поле твоего класса — это массив (`int[] data`), обычный метод `Objects.equals(this.data, other.data)` проверит только равенство ссылок на массивы, но не их содержимое.

- **Решение:** Для массивов в `equals` нужно использовать `Arrays.equals(this.data, other.data)`, а в `hashCode` — `Arrays.hashCode(this.data)`.
