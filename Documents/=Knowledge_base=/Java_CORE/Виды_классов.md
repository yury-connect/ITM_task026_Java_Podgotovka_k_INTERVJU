# Какие **виды классов** есть в *Java*?

В Java классификацию классов принято разделять по трем фундаментальным признакам: **по месту объявления (*топологии*)**, **по модификаторам/особому синтаксису** и **по роли в архитектуре приложения**.

---
## 1. Классификация по **месту объявления** (*Топология*)
Все классы в Java глобально делятся на **Top-level** (обычные классы, объявленные в собственном файле) и **Nested** (Вложенные — объявленные внутри другого класса).

### ── Вложенные классы (*Nested Classes*)
	
* **Статические вложенные классы (*Static nested classes*)**:
  Объявляются с модификатором `static` внутри внешнего класса. Физически не привязаны к экземпляру внешнего класса. Имеют доступ только к статическим полям и методам внешнего класса (включая `private static`).
	  
* **Внутренние классы (*Inner classes*)**:
  Нестатические классы внутри внешнего класса. Каждый экземпляр такого класса намертво привязан к объекту внешнего класса и имеет прямой доступ ко всей его структуре (включая `private` поля и методы, а так-же `static` членам внеш. класса).

### ── Разновидности внутренних (*Inner*) классов:
	
* **Локальные классы (*Local classes*)**:
  Классы, объявленные внутри какого-либо блока кода (обычно внутри метода). Доступны только в рамках этого блока. Могут использовать переменные метода, только если они являются `final` или *effectively final* (не изменяются после инициализации).
  Локальные классы в Java привязаны не просто к абстрактному «методу», а именно к **блоку кода**, ограниченному фигурными скобками `{}` (то есть к конкретной области или зоне видимости — _Block Scope_).
	  
* **Анонимные классы (*Anonymous classes*)**:
  Классы «на ходу» без имени. Одновременно объявляют класс-наследник (или реализацию интерфейса) и создают его единственный экземпляр в месте вызова. Часто заменяются лямбда-выражениями.

---
## 2. Классификация по **спецификаторам** и **синтаксису** <br>(*Спец-классы*)
	
* **Abstract классы**:
  Классы, помеченные ключевым словом `abstract`. Служат для создания базовых абстракций. Напрямую создать объект такого класса через `new` нельзя. 
  **Важно:** может вообще <u>не содержать</u> абстрактных методов.
	  
* **Final классы**:
  Классы с модификатором `final`. Запрещают наследование (*их нельзя расширить*). Яркий пример из JDK — класс `String`.
	  
* **[Enum](class-Enum_(Перечисления)) (*Перечисления*)**:
  Специальный вид классов, представляющий фиксированный набор констант. Напрямую наследуется от `java.lang.Enum`. Не может иметь публичных конструкторов.
	  
* **[Record](class-Record_(Записи).md) (*Записи*)** — *Добавлено в Java 16*:
  Специальный неизменяемый (immutable) вид классов, предназначенный только для переноса данных (DTO). Компилятор автоматически генерирует для него конструктор, методы `equals()`, `hashCode()`, `toString()` и геттеры.
	  
* **[Sealed](Sealed_(Запечатанные классы)) (*Запечатанные классы*)** — *Добавлено в Java 17*:
  Классы, которые строго ограничивают круг своих потенциальных наследников с помощью ключевого слова `permits`.

---
## 3. Классификация по **логической роли** <br>(*JDK и Архитектура*)
Эти категории представляют собой обычные классы с точки зрения синтаксиса, но выполняют специфические задачи:

### ── По роли в JDK:
	
* **Классы-обёртки (*Wrapper classes*)**:
  Объекты-аналоги для примитивных типов (`Integer`, `Long`, `Boolean`, `Double` и т.д.). Нужны для работы с коллекциями и дженериками, которые не поддерживают примитивы.

### ── По ООП-иерархии:
	
* **Суперклассы (*Родители*)** — базовые классы, отдающие свои свойства через `extends`.
	  
* **Подклассы (*Наследники*)** — расширяют функциональность суперклассов.
	  
* **Реализации** — классы, реализующие контракты одного или нескольких `interface`.

### ── По роли в архитектуре и паттернах (*Стереотипы*):
	
* **Singleton** — класс, структура которого гарантирует существование строго одного экземпляра на JVM.
	  
* **Factory** — класс, инкапсулирующий логику создания других объектов (порождающий паттерн).
	  
* **Proxy** — класс-заглушка, перехватывающий вызовы к реальному объекту для внедрения побочной логики (например, транзакций).
	  
* **Сервисы (Services)** — классы, содержащие чистую бизнес-логику приложения (в Spring помечаются аннотацией `@Service`).

## Анонимные классы и виды ссылок:
Когда ты создаешь анонимный класс в Java (например, для настройки соединения с БД или описания колбэка), под капотом всегда работает **Strong Reference (Сильная ссылка)**. Никакой автоматической магии с другими типами ссылок (вроде `WeakReference` или `SoftReference`) Java сама по себе здесь не применяет.

Однако в этой теме кроется одна из самых коварных ловушек Java, из-за которой анонимные классы часто становятся причиной **утечек памяти (Memory Leaks)**.

Давай разберем, как именно анонимный класс удерживает объекты в памяти и как это связано с типами ссылок.

### 🚨 Ловушка скрытой сильной ссылки (Implicit Strong Reference)
Главный секрет любого анонимного класса, созданного в нестатическом контексте (например, внутри обычного метода сервиса): **он всегда хранит скрытую сильную ссылку на объект внешнего класса**, в котором он был создан.

Представь такой сценарий (сервис управления соединениями):
```Java
public class DatabaseService {
    private byte[] bigDataBuffer = new byte[100 * 1024 * 1024]; // 100 МБ данных
	
    public ConnectionFactory createConnection() {
        // Создаем анонимный класс
        return new ConnectionFactory() {
            @Override
            public void connect() {
                System.out.println("Соединение установлено.");
            }
        };
    }
}
```

#### Что происходит в памяти?
1. Ты вызываешь метод `createConnection()`, получаешь объект анонимного класса и отдаешь его куда-то далеко (например, регистрируешь в глобальном пуле или фоновом потоке на долгое время).
    
2. Сам объект `DatabaseService` тебе больше не нужен, и ты зануляешь ссылку на него.
    
3. Ты ожидаешь, что Garbage Collector (GC) очистит `DatabaseService` и освободит 100 МБ памяти.
    
4. **Этого не произойдет!** Компилятор Java автоматически сгенерировал внутри анонимного класса скрытое поле вида `final DatabaseService this$0;` — это **сильная ссылка** на внешний сервис. Пока жив объект анонимного класса (твое соединение), он железно удерживает в памяти весь огромный `DatabaseService`.

### 🛠️ Применяются ли другие виды ссылок в анонимных классах?

**Автоматически — нет.** Но если тебе критически важно, чтобы анонимный класс не удерживал внешние объекты сильной ссылкой (например, при подписке на события или долгоживущих соединениях), ты **должен применить WeakReference вручную**.

В случае с анонимным классом сделать это сложно, потому что ссылку `this$0` контролирует компилятор. Единственный надежный способ разорвать эту связь — **перейти на лямбда-выражение (если это функциональный интерфейс) или статический вложенный класс**, и внутри них использовать `WeakReference`.

#### Пример безопасного решения через `WeakReference`:
```Java
public class DatabaseService {
    
    public ConnectionFactory createConnection() {
        // Создаем слабую ссылку на текущий сервис
        WeakReference<DatabaseService> serviceRef = new WeakReference<>(this);
		
        // Лямбда-выражение БЕЗ обращения к полям через this 
        // не захватывает внешний класс сильной ссылкой!
        return () -> {
            DatabaseService service = serviceRef.get();
            if (service != null) {
                System.out.println("Соединение установлено для живого сервиса.");
            } else {
                System.out.println(
	                "Сервис уже удален из памяти, соединение не актуально.");
            }
        };
    }
}
```

### 💡 Итог для собеседования
	
- По умолчанию в анонимных классах (и вокруг них) используются **только Strong References**.
    
- Анонимный класс держит сильную ссылку на то, что в него передали, а также **скрытую сильную ссылку** на своего «родителя» (внешний класс).
    
- Другие виды ссылок (`Weak`/`Soft`) применяются в таких сценариях исключительно **вручную разработчиком**, чтобы предотвратить утечки памяти, когда время жизни соединения или колбэка превышает время жизни основного бизнес-сервиса.
