**Record (Записи)** — это специальный вид классов в Java (появившийся как стабильная фича в Java 16), созданный исключительно для решения одной конкретной задачи: **быть удобным, лаконичным и неизменяемым контейнером для переноса данных (DTO / Data Carrier)**.

До появления записей создание простого класса данных превращалось в генерацию огромного количества шаблонного кода (Boilerplate code). Нам приходилось писать (или генерировать через IDE/Lombok) конструкторы, геттеры, `equals()`, `hashCode()` и `toString()`.

`Record` избавляет от этого раз и навсегда на уровне самого компилятора.

### 🛠️ **Было** vs **Стало**: Сила лаконичности
Допустим, нам нужен простой объект для передачи данных пользователя с тремя полями.

#### Как мы писали раньше (или с Lombok):
```Java
// Огромная простыня кода ради трех полей
public final class UserDTO {
    private final Long id;
    private final String username;
    private final String email;

    public UserDTO(Long id, String username, String email) {
        this.id = id;
        this.username = username;
        this.email = email;
    }

    public Long getId() { return id; }
    public String getUsername() { return username; }
    public String getEmail() { return email; }

    @Override
    public boolean equals(Object o) { /* ... */ }
    
    @Override
    public int hashCode() { /* ... */ }
    
    @Override
    public String toString() { /* ... */ }
}
```

#### Как это пишется через **Record**:
```Java
public record UserDTO(Long id, String username, String email) { }
```

**И всё!** Эта одна строчка кода компилируется в полноценный, защищенный класс.

### ⚙️ Что компилятор генерирует «под капотом» автоматически?
Когда ты пишешь `public record UserDTO(...)`, Java автоматически создает стандартный Java-класс со следующими свойствами:
	
1. **Неизменяемость (Immutable):** Все заголовки (параметры) превращаются в приватные финальные поля (`private final`). После создания объекта изменить его данные нельзя.
    
2. **Финальный класс:** Сам класс автоматически помечается как `final`. От него нельзя унаследоваться.
    
3. **Геттеры нового образца:** Компилятор создает методы для чтения полей, но **без префикса `get`**. Вместо `user.getId()` ты вызываешь просто `user.id()`.
    
4. **Конструктор по умолчанию:** Создается канонический конструктор, принимающий все параметры в том же порядке.
    
5. **Методы `equals()` и `hashCode()`:** Они генерируются автоматически на основе всех полей записи. Два разных объекта-рекорда с одинаковыми данными будут считаться равными (`equals() -> true`).
    
6. **Красивый `toString()`:** Автоматически выводит данные в понятном виде, например: `UserDTO[id=1, username=john, email=john@example.com]`.

### 🧠 Кастомная валидация: Компактный конструктор

Что если при создании записи нам нужно проверить данные (например, чтобы `id` не был отрицательным)? Для этого у рекордов есть уникальная фича — **Компактный конструктор (Compact Constructor)**. В нем не нужно заново переназначать переменные через `this.x = x` — компилятор сделает это сам в конце блока.
```Java
public record UserDTO(Long id, String username, String email) {
    
    // Компактный конструктор: без круглых скобок и параметров!
    public UserDTO {
        if (id == null || id < 0) {
            throw new IllegalArgumentException(
	            "ID не может быть отрицательным или null!");
        }
        if (username == null || username.isBlank()) {
            throw new IllegalArgumentException(
	            "Имя пользователя не может быть пустым!");
        }
        // В конце этого блока Java сама сделает: this.id = id; и т.д.
    }
}
```

### 🚫 Важные ограничения Record (Что делать НЕЛЬЗЯ)

Плата за лаконичность и безопасность — строгие правила спецификации:

- Рекорд **нельзя унаследовать** от другого класса (так как он скрыто уже наследуется от базового класса `java.lang.Record`).
    
- Рекорд **не может быть родителем** для другого класса (он всегда `final`).
    
- Внутрь рекорда **нельзя добавлять дополнительные instance-поля** (обычные переменные класса). Все данные объекта обязаны быть объявлены строго в заголовке `record Имя(...)`.
    
- **Но можно:** объявлять внутри статические поля (`static`), статические методы, обычные методы бизнес-логики, а также реализовывать любые интерфейсы (`implements MyInterface`).
