# 🧠 ClassLoader в Java — Полный конспект для собеседования
## 1. Что такое ClassLoader?
**ClassLoader** — это компонент JVM, который **динамически** загружает скомпилированный байт-код Java-классов (`.class`) в память.
> 📌 **Главная идея:** JVM не загружает все классы при старте. Класс загружается **только тогда**, когда он впервые понадобится (ленивая загрузка).
---
## 2. Иерархия загрузчиков (иерархия доверия)
```text
Bootstrap ClassLoader    ← Уровень 0 (Самый доверенный)
			↑
Platform ClassLoader     ← Уровень 1 (Загрузчик платформы)
	(до Java 9 — Extension)
			↑
 Application / System        ← Уровень 2 (Загружает ваш код)
			↑
User-Defined ClassLoader   ← Уровень 3 (Пользовательский)
```

---
### 🔹 **Bootstrap ClassLoader** (*Корневой / Базовый*)

|Свойство|Описание|
|---|---|
|**Что загружает**|Базовые классы JDK: `java.lang.*`, `java.util.*`, `java.io.*` и др.|
|**Откуда**|**Java 8 и ниже:** `$JAVA_HOME/jre/lib/rt.jar`  <br>**Java 9+:** Системные модули (например, `java.base`)|
|**Реализация**|Написана на **нативном коде** (C/C++). Не является Java-объектом.|
|**Уровень доверия**|**Максимальный.** Именно поэтому он не поддается подмене.|
|**Как определить**|`String.class.getClassLoader()` вернёт `null`|

---
### 🔹 **Platform ClassLoader** (*Платформенный*)

|Свойство|Описание|
|---|---|
|**До Java 9**|Назывался **Extension ClassLoader**. Загружал классы из `$JAVA_HOME/jre/lib/ext/*.jar`.|
|**После Java 9**|Переименован. Механизм `lib/ext` удалён, загрузка идет из модулей платформы.|
|**Роль**|Загружает API платформы, которые не входят в `java.base`, но являются частью JDK.|

---
### 🔹 **Application** / **System ClassLoader** <br>(*Системный / Прикладной*)

|Свойство|Описание|
|---|---|
|**Что загружает**|Классы вашего приложения и сторонних библиотек.|
|**Откуда**|Из **Classpath** (переменная `CLASSPATH`, параметры `-cp` или `-classpath`).|
|**Кто использует**|Именно этот загрузчик загружает ваш `main()` и всё, что вы пишете.|

---
### 🔹 **User-Defined ClassLoader** (*Пользовательский*)

Разработчик может создать свой загрузчик, унаследовавшись от `java.lang.ClassLoader`.

**Где применяется:**
	
- Плагинные системы (*поддержка динамической подгрузки плагинов*).
    
- Загрузка классов из **нестандартных источников**: база данных, сеть, зашифрованные файлы.
    
- **Горячая перезагрузка** (*HotSwap*) в веб-контейнерах (*Tomcat, Jetty*).
    
- **Изоляция модулей** (*например, OSGi, где каждое приложение живет в своем ClassLoader-е*).

---
## 3. **Модель делегирования родителю** *<br>(Parent Delegation Model*) 🪜

Это **ключевой механизм** работы ClassLoader-ов.

### Как это работает по шагам:
	
1. **Проверка кэша:** Загрузчик смотрит, не загружен ли этот класс уже.
    
2. **Делегирование родителю:** Запрос передаётся **родительскому** загрузчику.
    
3. **Рекурсия:** Родитель передаёт своему родителю и так до **Bootstrap**.
    
4. **Попытка загрузки:** Если Bootstrap не может найти класс, запрос "спускается" вниз.
    
5. **Самостоятельная загрузка:** Загрузчик, получивший запрос обратно, пытается загрузить класс сам.

### 🛡 Зачем это нужно?
Для **безопасности** и **целостности** системы.  
Например, вы не сможете подменить `java.lang.String` своим классом — запрос сначала уйдёт к Bootstrap ClassLoader, который загрузит настоящий системный класс.

> 💡 **Правило, которое легко запомнить:**  
> **Доверие — вверх (к родителям), загрузка — вниз (к дочернему).**

## 4. Жизненный цикл класса: <br>3 этапа работы ClassLoader ⚙️
```text
  ╔═══════════════╗
  ║   LOADING     ║   ← Поиск и импорт байт-кода
  ╚═══════════════╝
         ↓
  ╔═══════════════╗
  ║   LINKING     ║   ← Проверка, подготовка, разрешение
  ╚═══════════════╝
         ↓
  ╔═══════════════╗
  ║ INITIALIZATION║   ← Выполнение static-блоков и инициализация полей
  ╚═══════════════╝
```

---
### ① **Loading** (*Загрузка*)

ClassLoader ищет байт-код класса (файл `.class` или JAR) и создаёт в памяти объект `Class<?>`.

**Источник может быть любым:**
	
- Файловая система
    
- Сеть (URL)
    
- База данных
    
- Зашифрованный ресурс (через кастомный загрузчик)

---
### ② **Linking** (*Связывание*)

Этот этап делится на **три подэтапа**:

|Этап|Что происходит|
|---|---|
|**Verification** (Проверка)|Проверка корректности байт-кода: структура, соответствие JVM, безопасность.|
|**Preparation** (Подготовка)|Выделение памяти для **статических** полей и установка **значений по умолчанию** (`0`, `false`, `null`).|
|**Resolution** (Разрешение)|Замена **символических ссылок** (например, `"java/lang/String"`) на **реальные** ссылки в памяти JVM.|

> ⚠️ **Важно:** Resolution может выполняться **лениво** (lazy) — не сразу, а при первом реальном использовании класса.

---
### ③ **Initialization** (*Инициализация*)

Выполняется **Java-код инициализации** класса:

- Статические блоки `static { ... }`
    
- Присвоение реальных значений статическим полям (не default, а именно ваши значения).

---

## 5. Важные факты для собеседования 🎯

|Факт|Пояснение|
|---|---|
|**ClassLoader ≠ JVM Memory**|ClassLoader загружает классы в память, а управляет ими уже сама JVM (через Metaspace / PermGen).|
|**Класс идентифицируется не только именем**|Полный идентификатор класса = **полное имя + ClassLoader**. Один и тот же класс, загруженный разными ClassLoader-ами, считается разным типом для JVM.|
|**Кэширование классов**|Если класс уже загружен, JVM не будет загружать его повторно (до тех пор, пока ClassLoader жив).|
|**Динамическая загрузка**|Класс может быть загружен не только при старте, но и в рантайме: через `Class.forName()`, рефлексию, или кастомный загрузчик.|
|**Связь с исключениями**|`ClassNotFoundException` — класс не найден при попытке загрузить его динамически.  <br>`NoClassDefFoundError` — класс был доступен во время компиляции, но отсутствует в рантайме (проблема с ClassPath).|

## 6. Схема для запоминания 🧩
```text
       CLASS LOADER
             ↓
    ┌────────────────┐
    │    LOADING     │  ← Найти .class
    └────────────────┘
             ↓
    ┌────────────────┐
    │    LINKING     │  ← Проверить + Подготовить + Разрешить
    └────────────────┘
             ↓
    ┌────────────────┐
    │ INITIALIZATION │  ← Выполнить static { }
    └────────────────┘
```

**Мнемоника для запоминания порядка:**  
🟢 **З**агрузил → **С**вязал → **И**нициализировал — **ЗСИ**

**А для делегирования:**  
⬆️ Сначала вверх (к родителям), потом ⬇️ вниз (к себе).

## 7. Короткий ответ на собеседовании (шпаргалка)

> _"ClassLoader — это механизм JVM для динамической загрузки байт-кода классов. В Java используется трехуровневая иерархия: **Bootstrap** (загружает ядро JDK), **Platform** (загружает платформенные API) и **Application** (загружает классы приложения из Classpath)._  
> _Работа строится на модели делегирования родителю: сначала запрос уходит вверх до Bootstrap, и только если там класса нет, текущий загрузчик пытается загрузить его сам. Это обеспечивает безопасность и защиту системных классов._  
> _Жизненный цикл загрузки включает три этапа: **Loading** (поиск байт-кода), **Linking** (проверка, подготовка, разрешение ссылок) и **Initialization** (выполнение static-инициализаторов)."_

## 8. Бонус: как это связано с памятью JVM

Раньше (до Java 8) загруженные классы хранились в **PermGen** (Permanent Generation).  
С Java 8 PermGen убрали, и теперь метаданные классов хранятся в **Metaspace** — области памяти вне кучи, которая может динамически расширяться.

Это важно, потому что:

- `OutOfMemoryError: Metaspace` — если загружено слишком много классов (например, при горячем деплое без выгрузки старых ClassLoader-ов).
    
- Утечки памяти часто связаны с тем, что ClassLoader не может быть собран GC, потому что на него держат ссылки.
