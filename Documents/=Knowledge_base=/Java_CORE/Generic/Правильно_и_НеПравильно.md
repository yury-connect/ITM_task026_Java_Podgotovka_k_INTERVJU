# Всё о параметризации (Generics) в Java

Разберём тему максимально подробно, с примерами правильного и неправильного использования, а также со скрытыми нюансами.

---
## 1. Что можно параметризировать в Java

|Что параметризируется|Синтаксис|Пример|
|---|---|---|
|**Класс**|`class Name<T>`|`class Box<T> { private T item; }`|
|**Интерфейс**|`interface Name<T>`|`interface Repository<T, ID> {}`|
|**Метод**|`<T> T method(T t)`|`public <T> List<T> toList(T[] array)`|
|**Конструктор**|`<T> Name(T t)`|`public <T> Box(T initialValue)` (редко)|
|**Аннотация** (ограниченно)|через `@Repeatable` — не generics|Не параметризируется напрямую|

> **Важно:** Статические поля и статические методы (в части использования типа класса) **НЕ** могут использовать тип-параметр класса. Только свой собственный `<T>` в сигнатуре метода.

---

## 2. Параметризация класса (правильно и неправильно)

### ✅ Правильно: класс параметризован
```java
public class Box<T, U> {
    private T content;
    
    public void put(T item) { this.content = item; }
    public T get() { return content; }
}

// Использование
Box<String, Integer> stringBox = new Box<>();
stringBox.put("Hello");
String s = stringBox.get();  // Без приведения
```

### ❌ Неправильно: использование raw type (сырого типа)
```java
Box box = new Box();  // Warning: Raw use of parameterized class
box.put("Hello");     // Warning: Unchecked call
String s = (String) box.get();  // 👈 Приведение обязательно, потеря типобезопасности
```

> На собеседовании спросят: _"Что такое **raw type**?"_ — это **класс с generics**, используемый **без указания типа**. Он существует д<u>ля обратной совместимости с Java 4</u>, но использовать его **нельзя** в новом коде.

---
## 3. Параметризация метода

### ✅ Правильно: метод параметризован **своим** типом
```java
public class Util {
    // Метод параметризован своим <T>
    public static <T> T getFirst(List<T> list) {
        return list.isEmpty() ? null : list.get(0);
    }
}

// Использование
String first = Util.getFirst(List.of("a", "b"));  // T выводится автоматически
```

### ❌ Ошибка: метод использует **`T`** класса, но класса нет
```java
public class Util {
    // Ошибка! T не объявлен
    public static T getFirst(List<T> list) {  // ❌ Cannot resolve symbol 'T'
        return list.get(0);
    }
}
```

### ❌ Ошибка: метод пытается использовать **`T`** класса в статическом контексте
```java
public class Box<T> {
    // Ошибка! static метод не может использовать T класса
    // ❌ 'com.innowise.logistics.Box.this' cannot be ref_ from a static context
    public static T getDefault() {  
        return null;
    }
}
```

> **Правило:** `static`-метод не знает, что такое `T` класса, потому что `T` привязан к **экземпляру** класса. Единственный способ — сделать метод обобщённым отдельно:
```java
public static <T> T getDefault(Class<T> clazz) { ... }
```

## 4. Ограничения (bounded type parameters)

### ✅ Правильно: ограничение сверху (extends)
```java
// Только числа
public class NumberBox<T extends Number> {
    private T value;
    
    public double doubleValue() {
        return value.doubleValue();  // ✅ знаем, что есть метод doubleValue()
    }
}

// Использование
NumberBox<Integer> intBox = new NumberBox<>();
NumberBox<Double> doubleBox = new NumberBox<>();
// NumberBox<String> stringBox = new NumberBox<>(); // ❌ Ошибка компиляции
```

### ✅ Правильно: несколько ограничений
```java
// T должен быть и Number, и Comparable
public class MaxFinder<T extends Number & Comparable<T>> {
    public T max(T a, T b) {
        return a.compareTo(b) > 0 ? a : b;
    }
}
```

### ❌ Ошибка: ограничение снизу (super) в объявлении класса — запрещено!
```java
// ❌ Ошибка! super нельзя использовать в объявлении класса
public class Box<T super Number> {  // ❌ Unexpected bound
}
```
> **`super`** используется **только** в wildcard-ах (см. ниже), но не в объявлении класса/метода.

---
## 5. Wildcard (?) — неизвестный тип

### ✅ Правильно: `? extends T` (Producer Extends)
```java
// Метод, который читает из списка
public void printNumbers(List<? extends Number> list) {
    for (Number n : list) {
        System.out.println(n);
    }
}

// Можно передать List<Integer>, List<Double>, List<Float>
```

### ✅ Правильно: `? super T` (Consumer Super)
```java
// Метод, который добавляет в список
public void addNumbers(List<? super Integer> list) {
    list.add(1);
    list.add(2);
    // list.add(3.14); // ❌ Ошибка: нельзя добавить Double
}

// Можно передать List<Number>, List<Object>, List<Integer>
```

### ❌ Ошибка: использование wildcard в объявлении класса
```java
// ❌ Ошибка! Wildcard нельзя использовать в объявлении класса
public class Box<?> {  // ❌ Unexpected token
}
```

### ❌ Ошибка: добавление в `? extends`
```java
List<? extends Number> list = new ArrayList<Integer>();
list.add(5);  // ❌ Ошибка! Нельзя добавить элемент
```

### ❌ Ошибка: чтение из `? super`
```java
List<? super Integer> list = new ArrayList<Number>();
Integer i = list.get(0);  // ❌ Ошибка! Возвращается Object, не Integer
```

### **Запомни правило (PECS):**
	
- **Producer** = читаешь → `? extends T`
    
- **Consumer** = пишешь → `? super T`

## 6. Типичные ошибки с generics

### ❌ Ошибка: создание экземпляра T
```java
public class Factory<T> {
    public T create() {
        return new T();  // ❌ Ошибка! Нельзя создать T
    }
}
```

**Решение:** передавать `Class<T>` и использовать рефлексию:
```java
public class Factory<T> {
    private final Class<T> clazz;
    
    public Factory(Class<T> clazz) { this.clazz = clazz; }
    
    public T create() throws Exception {
        return clazz.getDeclaredConstructor().newInstance();
    }
}
```

### ❌ Ошибка: массив из T
```java
public class Box<T> {
	// ❌ Ошибка! Нельзя создать массив обобщённого типа
    private T[] array = new T[10];  
}
```

**Решение:** использовать `List<T>` или `Object[]` с приведением:
```java
public class Box<T> {
    private List<T> list = new ArrayList<>();  // ✅
    // или
    private T[] array = (T[]) new Object[10];  // ⚠️ с предупреждением
}
```

### ❌ Ошибка: `instanceof` с T
```java
if (obj instanceof T) {  // ❌ Ошибка! Illegal generic type for instanceof
}
```

**Решение:** использовать `Class<T>`:
```java
if (clazz.isInstance(obj)) {  // ✅
}
```

### ❌ Ошибка: статическое поле типа T
```java
public class Box<T> {
	// ❌ Ошибка! static поле не может быть типа T
    private static T defaultValue;  
}
```

### ❌ Ошибка: исключение типа T
```java
// ❌ Хотя компилируется, но бессмысленно
public class MyException<T> extends Exception {  
}
```

## 7. Type Erasure (стирание типов)

**Что происходит под капотом:** в Java generics существуют **только на этапе компиляции**. В байт-коде все `T` заменяются на `Object` (или на границу ограничения).
```java
// Исходный код
public class Box<T> {
    private T value;
}

// После компиляции (байт-код)
public class Box {
    private Object value;  // T стёрт в Object
}
```

**Следствия:**
	
1. Нельзя перегружать методы, отличающиеся только типом:
```java
public void process(List<String> list) { }  // ❌
public void process(List<Integer> list) { } // ❌ Ошибка: метод уже определён
```
	
2. Рефлексия не видит типов *(только при аннотировании или через `TypeToken`)*.

## 8. Рекомендации по именованию (best practices)

|Буква|Значение|
|---|---|
|`T`|Type (основной тип)|
|`E`|Element (коллекции)|
|`K`|Key (Map)|
|`V`|Value (Map)|
|`N`|Number|
|`S`, `U`, `V`|для нескольких типов (например, `Map<K, V>`)|
|`R`|Return type (в функциональных интерфейсах)|

## 9. Сложный пример (всё вместе)

```java
// Параметризованный интерфейс
public interface Repository<T, ID> {
    T findById(ID id);
    List<T> findAll();
    <S extends T> S save(S entity);  // Метод со своим типом
}

// Реализация
public class UserRepository implements Repository<User, Long> {
    @Override
    public User findById(Long id) { ... }
    
    @Override
    public List<User> findAll() { ... }
    
    @Override
    public <S extends User> S save(S entity) { ... }  // S extends User
}
```

## 10. Шпаргалка для собеседования (что говорить)

> **Как работают generics в Java?**  
> _«Generics реализованы через стирание типов (Type Erasure): во время компиляции все обобщённые типы заменяются на Object или на их границу. Это сделано для обратной совместимости с Java 4, но создаёт ограничения: нельзя создавать экземпляры T, массивы T, использовать instanceof. При этом компилятор добавляет неявные приведения, поэтому код остаётся типобезопасным.»_

> **В чём разница между `<T>` и `<?>`?**  
> _«`<T>` — это объявление типа-параметра, который связывается с именем и используется в нескольких местах. `<?>` — это wildcard, неизвестный тип, который используется, когда конкретный тип не важен. Wildcard нельзя использовать как тип возврата или в объявлении класса.»_
