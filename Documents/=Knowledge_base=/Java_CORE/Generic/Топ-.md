### Вопрос 1. Почему этот код не компилируется и как его починить?

Интервьюер просит тебя написать метод, который копирует элементы из одного списка в другой. Ты пишешь:
```Java
public <T> void copy(List<T> dest, List<T> src) {
    for (T item : src) {
        dest.add(item);
    }
}
```

Код компилируется. Но затем тебя просят вызвать его вот так:
```Java
List<Number> numList = new ArrayList<>();
List<Integer> intList = List.of(1, 2, 3);

copy(numList, intList); // Ошибка компиляции здесь!
```

**В чем проблема и как исправить метод `copy` с помощью Wildcards?**

#### Почему не работает:
Из-за инвариантности дженериков компилятор ожидает, что оба списка будут иметь **абсолютно одинаковый** тип `T`. Когда ты передаешь `List<Number>` и `List<Integer>`, компилятор не может выбрать один общий `T` для обоих аргументов в таком объявлении.

#### Как починить (Применяем PECS):
Список `src` отдает данные — он **Producer** (значит, `? extends T`). Список `dest` принимает данные — он **Consumer** (значит, `? super T`).

Переписываем сигнатуру метода:
```Java
public <T> void copy(List<? super T> dest, List<? extends T> src) {
    for (T item : src) {
        dest.add(item);
    }
}
```

Теперь код `copy(numList, intList);` скомпилируется идеально, так как `Integer` расширяет `T` (в данном случае `Number`), а `Number` является `super` типом для самого себя. К слову, именно так подписан метод `Collections.copy()` в самой Java JDK.

### Вопрос 2. Что такое «Захват маски» (Wildcard Capture) и почему падает этот код?

Тебе дали задачу: написать метод, который меняет местами (делает реверс) первый и последний элементы в `List<?>`.
```Java
public void swapFirstAndLast(List<?> list) {
    if (!list.isEmpty()) {
        Object first = list.get(0);
        // Ошибка компиляции на следующей строке: 
        // capture of ? extends Object inside list.set(...)
        list.set(0, list.get(list.size() - 1)); 
        list.set(list.size() - 1, first);
    }
}
```

**Почему компилятор ругается на `list.set()`, ведь мы берем элемент из этого же списка и кладем его обратно? Как это обойти?**

#### Почему не работает:
Компилятор строго следует правилу: в `List<?>` ничего записывать нельзя, потому что тип `?` неизвестен. Компилятор «захватывает» маску (Wildcard Capture) и ставит ей в соответствие внутреннее имя (например, `capture#1 of ?`). Когда ты пытаешься сделать `list.set()`, он не может доказать, что тип вставляемого элемента совпадает с этим тайным типом `capture#1`.

#### Как починить (Паттерн Helper-метод):
Нужно создать приватный вспомогательный метод (Helper), который преобразует неизвестный `?` в конкретный дженерик `T`. Для этого метода «захваченный» тип станет легитимным `T`:
```Java
public void swapFirstAndLast(List<?> list) {
    swapHelper(list); // Вызываем хелпер
}

// Хелпер "фиксирует" тип под именем T
private <T> void swapHelper(List<T> list) {
    if (!list.isEmpty()) {
        T first = list.get(0);
        list.set(0, list.get(list.size() - 1));
        list.set(list.size() - 1, first); // Теперь компилятор счастлив!
    }
}
```

### Вопрос 3. Массив против Списка. Почему массивы разрешают то, что дженерики запрещают?

Посмотри на два примера кода. Один из них упадет во время компиляции, а второй успешно скомпилируется, но громко грохнется прямо в рантайме. Кто есть кто?
```Java
// Пример А
List<Number> list = new ArrayList<Integer>(); 
list.add(3.14);

// Пример Б
Number[] array = new Integer[5];
array[0] = 3.14; 
```

- **Пример А — Умрет при компиляции.** Как мы уже знаем, списки **инвариантны**. Компилятор сразу бьет по рукам на первой строчке, защищая нас от рантайм-ошибок.
    
- **Пример Б — Скомпилируется, но упадет в рантайме** с ошибкой `ArrayStoreException`.

#### Почему так произошло?
Массивы в Java являются **ковариантными** (`Integer[]` является подтипом `Number[]`). Более того, массивы «помнят» свой реальный тип в рантайме (они reified). На второй строчке компилятор видит, что ты пытаешься положить `Double (3.14)` в массив, который по документам числится как `Number[]`. Для компилятора всё ок. Но сама JVM в рантайме знает, что массив в памяти на самом деле `Integer[]`, и выбрасывает исключение.

> **Вывод для интервью:** Дженерики были созданы как раз для того, чтобы перенести подобные ошибки из рантайма (как у массивов) на этап компиляции.
