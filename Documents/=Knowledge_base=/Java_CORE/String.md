# Какие нюансы у строк в Java?

---
## `String`, `StringBuilder`, `StringBuffer`

## 🔹 Класс `String`
`String` — это ссылочный тип данных, представляющий неизменяемую (Immutable) последовательность символов.

### Финализированность и неизменяемость
- Класс `String` объявлен как `final`. Это значит, что от него **нельзя унаследоваться**.
    
- Под капотом (*начиная с Java 9*) строки хранятся в виде **массива байт**: `private final byte[] value;` (*до Java 9 использовался массив `char[]`*). Ключевое слово `final` гарантирует, что после инициализации массив нельзя заменить.
    
- **Зачем сделана неизменяемость (`Immutable`)?**    
    1. **Безопасность (Security):** Строки часто используются для передачи конфиденциальных данных (пароли, URL БД, сетевые адреса). Если бы строки были изменяемыми, злоумышленник мог бы изменить значение ссылки прямо во время проверки прав доступа.
        
    2. **Многопоточность (Thread-Safety):** Строки абсолютно потокобезопасны. Их можно одновременно читать из разных потоков без риска получить поврежденные данные или состояния гонки.
        
    3. **Кэширование хэш-кода:** Хэш-код строки вычисляется один раз при создании и кэшируется в поле `private int hash;`. Это делает строки идеальными ключами для `HashMap`.
        
    4. **Экономия памяти (String Pool):** Позволяет переиспользовать одинаковые строковые литералы.

### **Пул строк** (*String Pool*)
Это специальная область в куче (**Heap**), которая хранит уникальные объекты строк.

- _Исправление популярного мифа:_ Пул строк находился в PermGen только до Java 6 включительно. Начиная с Java 7 и во всех современных версиях (Java 8, 11, 17, 21, 25) Пул строк живет в **обычной куче (Heap)**. Это позволяет сборщику мусора (GC) эффективно удалять неиспользуемые строки из пула, когда на них нет ссылок.

#### Способы создания строк:
1. **Литерал:** `String s = "cat";` — JVM ищет строку `"cat"` в пуле. Если нашла — возвращает ссылку на неё. Если нет — создает её в пуле и возвращает ссылку.
    
2. **Через `new`:** `String s = new String("cat");` — JVM гарантированно создаст **новый объект** строки в куче (Heap), даже если `"cat"` уже есть в пуле строк.

#### Метод **`intern()`**
Принудительно переносит строку в Пул строк. Если строка с таким содержимым уже есть в пуле, метод просто возвращает ссылку на существующую строку. Если её там нет — она добавляется в пул, и возвращается ссылка на неё.

**Пример (Подвох на собеседованиях):**
```Java
String s1 = "Home"; 
String s2 = new String("Home");

System.out.println(s1 == s2);     // false (разные объекты: один в пуле, другой в куче)
System.out.println(s1.equals(s2)); // true  (значения одинаковые)

String s3 = s2.intern(); 
System.out.println(s1 == s3);      // true  (s3 теперь ссылается на объект из пула)
```

# 🔹 Где хранится пул строк в разных версиях Java?

| **Версия Java**   | **Где хранится пул строк?**            | **Особенности**                                                                                                                                                  |
| ----------------- | -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Java 6 и ниже** | В **PermGen** _(Permanent Generation)_ | Область имеет строго фиксированный размер, из-за чего часто возникали ошибки `OutOfMemoryError: PermGen space`.                                                  |
| **Java 7**        | Перенесен в **Heap** _(Кучу)_          | Сам пул строк уже переехал в общую кучу, но область PermGen всё ещё существовала в структуре памяти.                                                             |
| **Java 8+**       | Полностью в **Heap** _(Кучу)_          | Пул строк окончательно закреплен в куче. Область PermGen полностью удалена, а метаданные классов вынесены в **Metaspace** (вне кучи, в системную native-память). |
### 📌 Почему пул строк перенесли в Heap?

- **✔ Защита от утечек памяти:** Размер области PermGen был жестко ограничен сверху. Если приложение активно генерировало или интернировало много динамических строк, оно неизбежно падало с критической ошибкой `OutOfMemoryError`.
    
- **✔ Гибкость и масштабируемость:** Куча (Heap) гораздо объемнее и может динамически расширяться (в пределах флага `-Xmx`), подстраиваясь под нужды приложения в рантайме.
    
- **✔ Эффективность очистки памяти:** Сборщик мусора (*Garbage Collector*) гораздо лучше, чаще и глубже оптимизирован под работу с кучей. Перенос пула строк в *Heap* позволил *GC* своевременно удалять из пула те строки, на которые в программе больше никто не ссылается, эффективно освобождая ресурсы.

# 🔴 Как очищался String Pool до Java 8?
До Java 8 пул строк (`String Pool`) находился в области **PermGen**, где *Garbage Collector* (GC) практически не работал.

### 1. Очистка происходила только при выгрузке классов
- Строки из пула удалялись **исключительно в момент выгрузки соответствующего класса** (*это было возможно, только если класс загружался через `Custom ClassLoader`*).
    
- Классы, загруженные с помощью стандартного `Bootstrap ClassLoader`, **никогда не выгружались** из памяти. Соответственно, связанные с ними строки оставались в пуле мертвым грузом до полного завершения работы JVM.

### 2. Хроническая проблема утечек памяти
- Если приложение в процессе работы динамически генерировало или считывало **большое количество новых уникальных строк** (*например, при парсинге тяжелых XML или JSON файлов*), пул стремительно заполнялся.
    
- Поскольку стандартный GC не очищал область PermGen на регулярной основе, строкам было просто некуда «уходить». Это массово приводило к фатальной ошибке **`OutOfMemoryError: PermGen space`**.

# 🟢 Что изменилось начиная с Java 8?
- **Смена локации:** `String Pool` окончательно перенесли в общую кучу (**Heap**), внутри которой непрерывно и полноценно функционирует **обычный Garbage Collector**.
    
- **Автоматическое управление:** Теперь все ненужные строки, на которые в коде программы больше никто не ссылается, своевременно и беспрепятственно удаляются силами GC, что практически свело к нулю риск подобных утечек памяти.

### 📝 Краткий вывод
- **До Java 8:** Строки в пуле жили «намертво» и очищались только вместе с выгрузкой породивших их классов (если это вообще было технически возможно).
    
- **В Java 8+:** Строки в пуле являются рядовыми объектами кучи и регулярно очищаются обычным сборщиком мусора по стандартным правилам трассировки.

## 🔹 Изменяемые строки: `StringBuilder` и `StringBuffer`
Когда нужно часто модифицировать текст (склеивать строки в цикле, удалять подстроки), обычный `String` порождает терабайты мусора в куче, так как при любой операции `+` создается новый объект `String`. Для решения этой проблемы используются изменяемые (Mutable) классы-буферы.

### 1. `StringBuilder`
Класс, представляющий изменяемую последовательность символов.
- **Потокобезопасность:** **Не потокобезопасен.** Метод модификации данных (`append`, `insert`) не синхронизированы.
    
- **Производительность:** Очень быстрый. За счет отсутствия блокировок работает быстрее, чем `StringBuffer`.
    
- **Применение:** Рекомендуется использовать во всех однопоточных сценариях (внутри методов, при сборке SQL-запросов, логов и т.д.).

### 2. `StringBuffer`
Исторический класс, аналог `StringBuilder`, появившийся еще в Java 1.0.
- **Потокобезопасность:** **Потокобезопасен.** Все основные методы модификации синхронизированы с помощью ключевого слова `synchronized`.
    
- **Производительность:** Медленный. Накладные расходы на синхронизацию потоков сильно снижают скорость работы.
    
- **Применение:** В современных приложениях используется крайне редко, так как безопаснее и эффективнее использовать `StringBuilder` локально внутри потока, либо применять специализированные concurrent-коллекции.

### Суммарная таблица для ревью:

|**Критерий**|**String**|**StringBuilder**|**StringBuffer**|
|---|---|---|---|
|**Изменяемость**|Immutable (Неизменяемый)|Mutable (Изменяемый)|Mutable (Изменяемый)|
|**Потокобезопасность**|Да (Thread-Safe)|Нет|Да (Thread-Safe)|
|**Скорость работы**|Медленный при конкатенации|Очень быстрый|Медленный (из-за `synchronized`)|
|**Где хранятся объекты**|Пул строк / Куча (Heap)|Куча (Heap)|Куча (Heap)|
