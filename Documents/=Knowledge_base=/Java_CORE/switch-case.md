Тут есть два разных контекста, и их важно разделить:

---

## 1. В старом `switch` (до Java 14) — `break` ОБЯЗАТЕЛЕН
```java
// ❌ Старый стиль (C-style fall-through)
switch (payment) {
    case CreditCardPayment cc:
        System.out.println("Прокатываем карту");
        break; // БЕЗ BREAK - провалится в следующий case!
    case PayPalPayment pp:
        System.out.println("Перенаправляем на PayPal");
        break;
    default:
        System.out.println("Неизвестный платеж");
}
```
**Почему нужен `break`?**  
Потому что в старом синтаксисе после выполнения `case` выполнение **проваливается (fall-through)** в следующий `case`, если не поставить `break`. Это legacy-поведение из C/C++, и оно было источником тонны багов.

---
## 2. В новом `switch` (Java 14+, особенно Java 17+ с pattern matching) — `break` **НЕ НУЖЕН**

В примере используется **новый синтаксис `switch` с `->`** (*стрелочная нотация*), который появился в *Java 14* как preview и стал стандартом в *Java 17*.
```java
// ✅ Новый стиль (без fall-through!)
switch (payment) {
    case CreditCardPayment cc -> System.out.println("Прокатываем карту");
    case PayPalPayment pp     -> System.out.println("Перенаправляем на PayPal");
    case CryptoPayment crypto -> System.out.println("Ждем подтверждения");
    // break НЕ НУЖЕН!
}
```

**Что происходит:**
	
- Стрелка `->` означает, что выполняется **только один блок** для выбранного `case`.
    
- **Fall-through отсутствует** по умолчанию.
    
- Не нужен `break`, т.к. после выполнения блока `switch` завершается автоматически.

---
## 3. А что если нужно выполнить **несколько строк** кода?
```java
switch (payment) {
    case CreditCardPayment cc -> {
        // Несколько строк кода
        System.out.println("Прокатываем карту");
        processCard(cc);
        logPayment(cc);
        // break НЕ НУЖЕН даже в блоке!
    }
    case PayPalPayment pp -> {
        System.out.println("Перенаправляем на PayPal");
        redirectToPayPal(pp);
    }
}
```

---
## 4. А есть еще **`yield`** (*для возврата значения)*

Если `switch` используется как **выражение** (*возвращает значение*), используется `yield` вместо `break`:
```java
// switch как выражение (Java 14+)
String status = switch (payment) {

    case CreditCardPayment cc -> {
        validateCard(cc);
        yield "Обработка карты"; // возврат значения
    }
    
    case PayPalPayment pp -> "Обработка PayPal"; // короткая форма
    
    case CryptoPayment crypto -> {
        // несколько строк
        waitForBlockchain();
        yield "Крипто-обработка";
    }
};
```

Здесь `yield` заменяет `return` внутри `switch`-выражения.

## 📊 Сравнительная таблица

| Синтаксис                                     | `break` нужен?               | Fall-through? | Возврат значения         |
| --------------------------------------------- | ---------------------------- | ------------- | ------------------------ |
| **Старый `switch` <br>(с `:`)**               | ✅ ДА                         | ✅ Есть        | ❌ Нет (только оператор)  |
| **Новый `switch` <br>(с `->`)**               | ❌ НЕТ                        | ❌ Нет         | ❌ Нет <br>(оператор)     |
| **`switch`-выражение <br>(с `->` + `yield`)** | ❌ НЕТ                        | ❌ Нет         | ✅ ДА <br>(через `yield`) |
| **`switch`-выражение <br>(с `:` + `yield`)**  | ❌ НЕТ (но <br>можно `yield`) | ❌ Нет         | ✅ ДА <br>(через `yield`) |

---
## 🎯 Почему в примере с запечатанными классами <br>нет `break`?

Потому что я использовал **современный, правильный синтаксис**, который:
1. Безопаснее (нет случайного fall-through)
2. Чище читается
3. Лучше интегрируется с **pattern matching** (который пришел в Java 21)

---
## ⚠️ Важно для собеседования

Если тебя спросят: _"Почему в новом switch не нужен break?"_

Ответь так:
> _"В Java начиная с 14 версии появилась стрелочная нотация `->` в операторе `switch`. Она работает по принципу 'один case — один блок', без проваливания (fall-through), поэтому `break` больше не нужен. Для возврата значения из `switch`-выражения используется `yield`. Это сделало код более безопасным и читаемым, особенно в связке с pattern matching, где мы перебираем иерархию запечатанных классов."_

---
## 📝 Дополнительный бонус: что такое **fall-through**?

Это когда выполнение идет из одного `case` в следующий, если нет `break`:
```java
// ❌ ОПАСНО! (старый синтаксис)
switch (day) {
    case MONDAY:
        System.out.println("Понедельник");
        // НЕТ BREAK → проваливается во вторник!
    case TUESDAY:
        System.out.println("Вторник");
        break;
}
// Вывод:
// Понедельник
// Вторник
```

Именно из-за таких багов в Java 14 и ввели `->` и `yield`, чтобы `break` ушел в прошлое (почти).
