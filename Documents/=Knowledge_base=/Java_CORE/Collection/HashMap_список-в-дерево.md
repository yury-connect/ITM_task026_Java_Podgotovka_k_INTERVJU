Механизм перевода односвязанного списка в красно-черное дерево

---

начиная с Java 8, если в одной корзине (bucket) `HashMap` набирается **8 элементов** (и общий размер таблицы больше 64), этот бакет превращается из связного списка в красно-черное дерево. Изначально это дерево состоит из специальных узлов `TreeNode`.

Но ведь внутри дерева элементы _обязаны_ как-то сравниваться и сортироваться, а у объектов в `HashMap` (`например, твоего кастомного класса User`) методов `Comparable` или внешних компараторов может и не быть.

Вот как изощренно и красиво инженеры Sun/Oracle решили эту дилемму. Под капотом включается **четырехуровневый алгоритм спасения**.

## 🏎️ Алгоритм сравнения элементов в дереве HashMap

Когда мапе нужно сравнить текущий ключ `A` с ключом `B` внутри дерева, она запускает следующий конвейер проверок (из исходного кода `java.util.HashMap`):

### Шаг 1: Проверка по хэш-кодам (Самая быстрая)

Сначала мапа сравнивает их `hash` (который был вычислен при попадании в карту).

- Если `A.hash < B.hash`, объект `A` идет налево в дерево.
    
- Если `A.hash > B.hash`, объект `A` идет направо.
    
- **А что если хэши одинаковые?** (Ведь дерево и создается именно тогда, когда у многих элементов совпали хэш-коды — коллизия). Тогда мапа переходит к Шагу 2.
    

### Шаг 2: Попытка использовать `Comparable` (Если он всё-таки есть)

Мапа с помощью рефлексии проверяет: _«А вдруг класс ключа `A` реализует интерфейс `Comparable`?»_. Если реализует, она проверяет, совпадает ли класс объекта `B` с классом объекта `A`. Если всё совпало, она просто вызывает привычный метод: `A.compareTo(B)`.

- **А если класс НЕ реализует `Comparable`?** (Твой случай). Тогда мапа переходит к Шагу 3.
    

### Шаг 3: Сравнение по имени классов (Если объекты разных типов)

Если в одну корзину по воле случая попали объекты совершенно разных классов (например, твоя кастомная `StringWrapper` и какой-нибудь `User`), мапа берет строковые имена их классов (`A.getClass().getName()`) и сравнивает их как обычные строки через `String.compareTo()`.

- **А если класс один и тот же?** (У нас в коллизии сидят 10 объектов одного твоего класса `User`). Тогда имена классов одинаковые, и этот шаг не дает ответа. Включается "тяжелая артиллерия" — Шаг 4.
    

### Шаг 4: Метод «последней надежды» — `tieBreakOrder()`

Если ни хэши, ни типы, ни интерфейсы не помогли определить, кто больше, а кто меньше, мапа вызывает внутренний метод-судью, который называется `tieBreakOrder(Object a, Object b)`.

Этот метод использует нативную функцию самой виртуальной машины Java:

Java

```
System.identityHashCode(obj)
```

**Identity Hash Code** — это тот самый первозданный хэш-код, который выдает JVM по умолчанию для любого объекта, если у него вообще не переопределен метод `hashCode()` (вычисляется на основе случайных чисел рантайма и уникален для каждого физического объекта в памяти).

Поскольку два разных объекта в один момент времени не могут иметь одинаковый `identityHashCode` в рамках одной JVM, этот метод **гарантированно** возвращает уникальное число для каждого ключа.

- Если `System.identityHashCode(A) < System.identityHashCode(B)`, то `A` признается "меньшим" и уходит налево.
    
- Иначе — направо.
    

## 💡 Архитектурный итог

Мапка поступает хитро: она не падает с ошибкой `ClassCastException`, как сделал бы `TreeMap`. Вместо этого она **на ходу строит суррогатную логику сравнения**.

Она искусственно выстраивает объекты в красно-черное дерево, используя сначала их хэши, затем типы, а в крайнем случае — их системные адреса-идентификаторы (`identityHashCode`). Дерево получается рабочим, уникальность не нарушается, а скорость поиска из-за коллизий по-прежнему остается высокой!
