# ЧАСТЬ 1. Высокоуровневая карта процессов `HashMap` (Кратко)

Работа `HashMap` базируется на массиве бакетов (`Node<K,V>[] table`), индексы которых вычисляются на основе хэш-кода ключа.

```
                  [ Вызов put(key, value) или get(key) ]
                                    │
                                    ▼
                     [ Этап 1. Хэширование ключа ]
                   (key.hashCode() + XOR-shift сглаживание)
                                    │
                                    ▼
                [ Этап 2. Вычисление индекса бакета ]
                       (index = hash & (length - 1))
                                    │
                                    ▼
            [ Этап 3. Адресация в массиве и анализ бакета ]
                                    │
        ┌───────────────────────────┼───────────────────────────┐
        ▼                           ▼                           ▼
  [ Бакет пуст ]             [ Бакет: Список ]          [ Бакет: Дерево ]
  (Добавление /              (Перебор связей            (Спуск по дереву:
   Экшн за O(1))              O(N) ──► ресайз/           O(log N) через
                              трификация)                tieBreakOrder)
```

### Сводная матрица граничных условий и констант:
	
- **Начальная емкость (Default Capacity):** 16 ячеек массива (всегда степень двойки: $2^n$).
    
- **Коэффициент загрузки (Load Factor):** 0.75 (баланс между памятью и скоростью поиска).
    
- **Порог трификации (`TREEIFY_THRESHOLD`):** $\ge 8$ элементов в одной цепочке.
    
- **Порог деградации дерева (`UNTREEIFY_THRESHOLD`):** $\le 6$ элементов в бакете.
    
- **Минимум для деревьев (`MIN_TREEIFY_CAPACITY`):** 64 бакета во всей мапе.

# ЧАСТЬ 2. Низкоуровневая анатомия механизмов `HashMap` (Детально)

## 🛠️ Шаг 1. Сглаживание хэш-кода (The Hashing Step)

Когда в мапу передается ключ, JVM не использует значение `key.hashCode()` в сыром виде. Оно пропускается через внутреннюю функцию `hash(Object key)`.

### ── Что происходит под капотом:
	
1. Если `key == null`, мапа жестко возвращает хэш `0` (поэтому в `HashMap` всегда можно положить один `null`-ключ, и он всегда будет жить в бакете с индексом `0`).
    
2. Если ключ существует, вызывается его нативный метод `key.hashCode()`.
    
3. Полученный 32-битный `int` сдвигается без знака на 16 бит вправо (`>>> 16`) и подмешивается к исходному значению через оператор исключающего ИЛИ (`^`).

```Java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

### ── Возможные вариации и причины:

- **Зачем нужен сдвиг (XOR-shift)?** Массивы мапы на старте маленькие (размер 16). При вычислении индекса учитываются только самые нижние биты хэша. Если у разных ключей хэш-коды будут отличаться только в верхних битах (например, `0xFFFF0001` и `0x00A10001`), без сдвига они гарантированно ударятся в одну и ту же ячейку, порождая коллизии. Подмешивание верхних битов к нижним обеспечивает равномерное распределение элементов.
    
- **Результат:** Полученный сглаженный хэш сохраняется в поле `final int hash` создаваемого узла, чтобы процессор больше никогда не тратил ресурсы на его повторный перерасчет.

## 📐 Шаг 2. Вычисление физического индекса ячейки

Мапе необходимо преобразовать сглаженный хэш-код (который может быть огромным числом, в том числе отрицательным) в легальный индекс внутри границ текущего массива.

### ── Что происходит под капотом:

Вместо дорогой для CPU операции деления по модулю (`hash % table.length`), инженеры применили операцию побитового умножения (AND):

$$\text{index} = \text{hash} \ \& \ (\text{table.length} - 1)$$

### ── Возможные вариации и причины:

- **Почему длина массива строго степень 2 ($2^n$)?** Только при длине в виде степени двойки (например, 16), число `length - 1` (15) в двоичной системе превращается в идеальную битовую маску из единиц (`00001111`). Побитовое И с такой маской мгновенно зануляет все верхние биты хэша, оставляя только число, не превышающее размер массива. Это работает на уровне транзисторов процессора за 1 такт.
    
- **Если массив расширился (Resize):** Длина увеличивается в два раза (например, 16 ──► 32). Битовая маска расширяется (`1111` ──► `11111`). При пересчете индекса у элемента есть ровно два варианта: он либо останется по старому индексу `index`, либо улетит по адресу `index + oldCapacity`.

## 🧱 Шаг 3. Анализ структуры бакета и логика ветвления

Процессор обращается к ячейке `table[index]`. Это обычный массив Java, хранящий ссылки. Здесь возможны три сценария развития событий:

### ── Вариация А: Ячейка пуста (`null`)
	
- **При вызове `get()`:** Мапа сразу возвращает `null`.
    
- **При вызове `put()`:** Создается объект `Node<K,V>`, ссылка на него записывается в ячейку массива. Операция завершается за идеальные $O(1)$.

### ── Вариация Б: В ячейке связный список (`Node`)
	
Произошла коллизия — хэши разных ключей указали на один бакет. Ячейка ссылается на голову односвязного списка.

- **При вызове `get()`:** Мапа начинает итерироваться по ссылкам `next`. Для каждого узла она проверяет:
    $$\text{if } (\text{node.hash} == \text{hash} \ \&\& \ (\text{node.key} == \text{key} \ || \ \text{key.equals}(\text{node.key})))$$
    
    Если совпадение найдено — возвращает значение.
    
- **При вызове `put()`:** Мапа бежит до конца списка. Если по пути находит эквивалентный ключ — перезаписывает значение. Если дошла до хвоста и ключа нет — прикрепляет новый `Node` в самый конец.

> **Важно (Триггер Ресайза/Трификации):** В момент добавления мапа считает длину списка. Если она достигла **8 элементов**, мапа останавливает выполнение и смотрит на общий размер всей таблицы (`table.length`).
> 
> - Если емкость всей мапы **меньше 64**, перевод в дерево отменяется. Мапа вызывает метод `resize()`, увеличивая массив в 2 раза, надеясь рассеять коллизию.
>     
> - Если емкость мапы **больше или равна 64**, запускается метод `treeifyBin()`. Список в этой ячейке уничтожается, и на его месте возводится красно-черное дерево.

### ── Вариация В: В ячейке красно-черное дерево (`TreeNode`)

Если бакет уже трифицирован, ячейка массива хранит ссылку на корень дерева (`TreeNode`). Поиск и вставка начинают работать со скоростью $O(\log N)$.

Поскольку дереву для балансировки нужно строго знать, какой объект "больше", а какой "меньше", включается **четырехуровневый алгоритм сравнения ключей**:

1. **Компарация по внутренним хэшам:** Мапа сравнивает сохраненные сглаженные `hash` (из Шага 1). Если они разные, направление движения определено (меньший хэш — влево, больший — вправо). Если хэши совпали, идем глубже.
    
2. **Проверка на `Comparable`:** Через рефлексию проверяется, реализует ли ключ интерфейс `Comparable`. Если да, вызывается метод `A.compareTo(B)`. Если интерфейса нет или он вернул 0 (а объекты разные по `equals`), идем на шаг 3.
    
3. **Сравнение по именам классов:** Если типы объектов разные, мапа сравнивает строки их полных имен классов (`A.getClass().getName().compareTo(...)`). Если класс один и тот же, применяется финальный арбитраж.
    
4. **Метод последней надежды — `tieBreakOrder()`:** Мапа сравнивает объекты по их **`System.identityHashCode(obj)`** (нативный системный хэш-код JVM, привязанный к уникальности объекта в памяти). Это гарантирует уникальное распределение элементов "влево/вправо" по дереву, исключая зависание алгоритма.

## 📈 Шаг 4. Изменение размеров (Resize) и обратная деградация

Каждая операция `put` завершается проверкой: не пора ли увеличивать мапу глобально?

### ── Механика Ресайза:

Если общее количество элементов в карте (`size`) превысило порог `threshold` (Capacity × Load Factor), создается новый массив `Node[]` увеличенного в 2 раза размера. Все элементы из старого массива переносятся в новый.

### ── Судьба существующих деревьев при ресайзе:

Когда мапа переносит элементы из старого бакета-дерева в новый массив, это дерево расщепляется методом `split()` на две независимые части (согласно новой битовой маске индекса).

- Мапа подсчитывает количество элементов, доставшихся каждой из двух новых половин.
    
- Если в какой-то из половин количество элементов падает до **`UNTREEIFY_THRESHOLD` ($\le 6$ элементов)**, красно-черное дерево в этой новой ячейке схлопывается (деградирует) обратно в простой односвязный список.

### ── Потеря исходного порядка элементов:

При обратном превращении дерева в список **первоначальный порядок добавления элементов не восстанавливается**.

Внутри узлов `TreeNode` параллельно древесным ссылкам (`left`, `right`) поддерживаются двусвязные ссылки `next` и `prev` для ускорения итерации. Но в процессе балансировки и поворотов красно-черного дерева эти ссылки `next/prev` перестраиваются под текущую топологию дерева. При деградации мапа просто превращает эту измененную древесную структуру связей в обычный список.

## 🚨 Шаг 5. Массовое удаление и утечки памяти

Если из мапы, раздутой до 100 000 элементов, удалить 99 995 объектов через метод `remove()`, рантайм поведет себя следующим образом:

1. **Освобождение памяти данных:** Объекты ключей, значений и оберток `Node/TreeNode` теряют ссылки и успешно уничтожаются Сборщиком Мусора (GC).
    
2. **Массив остается раздутым:** Внутренний массив `Node[] table` **никогда автоматически не уменьшается** и не имеет встроенного метода сжатия (аналога `trimToSize()` из `ArrayList`).
    
3. **Деградация скорости итерации:** Если запустить цикл `for-each` по такой опустошенной мапе, Java будет вынуждена делать пустые проверки `if (table[i] == null)` по всем десяткам тысяч пустых ячеек массива, тратя ресурсы CPU на ровном месте.
    
4. **Решение проблемы:** Единственный способ очистить память массива — вручную перелить элементы в новую мапу через конструктор копирования: `Map<K,V> cleanMap = new HashMap<>(oldMap);`.
