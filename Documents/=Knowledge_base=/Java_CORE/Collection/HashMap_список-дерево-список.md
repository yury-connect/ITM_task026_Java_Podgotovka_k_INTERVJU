Механизм перевода односвязанного списка в красно-черное дерево

---
# Механизм трификации (Treeification) в `HashMap` и обратная деградация

Когда в одной корзине (bucket) `HashMap` скапливается много элементов (коллизия), мапа перестраивает внутреннюю структуру хранения, чтобы защитить производительность от деградации до $O(N)$.

## 📊 1. Триггеры и константы перестройки (В цифрах)
Переходы между связным списком и красно-черным деревом жестко контролируются тремя константами в исходном коде `java.util.HashMap`:
	
- **`TREEIFY_THRESHOLD = 8`** — порог превращения списка в дерево. Если при добавлении (`put`) нового элемента длина связного списка в конкретном бакете достигает **8 элементов**, мапа инициирует процесс трификации.
    
- **`UNTREEIFY_THRESHOLD = 6`** — порог обратного превращения дерева в список. Если при удалении (`remove`) или ресайзе количество узлов в дереве падает до **6 элементов**, структура бакета сжимается обратно в односвязный список.
    
- **`MIN_TREEIFY_CAPACITY = 64`** — минимальный размер всей таблицы. **Это критическое условие!** Превращение списка в дерево внутри бакета возможно _только тогда_, когда общее количество бакетов во всей мапе **равно 64 или более**.

> **Разрыв люфта (Почему 8 и 6, а не 7 и 7):** Разница между порогами сделана умышленно. Если бы оба порога были равны 7, то при частых, поочередных операциях добавления и удаления элемента на границе («дребезг» размера 7 ──► 8 ──► 7) мапа непрерывно и ресурсоемко перестраивала бы структуру туда-обратно, критически просаживая CPU.

## 🛠️ 2. Механизм перевода списка в дерево и алгоритмы сравнения
Когда в бакете наступает триггер `TREEIFY_THRESHOLD` (8 элементов) и общая емкость мапы $\ge 64$, обычные узлы списка `Node<K,V>` заменяются на массивные узлы дерева `TreeNode<K,V>`.

Поскольку красно-черное дерево требует строгого упорядочивания элементов (налево — меньшие, направо — большие), а у кастомных ключей разработчика может не быть интерфейса `Comparable`, `HashMap` запускает **четырехуровневый алгоритм сравнения «последней надежды»**:

```
[Сравнение двух ключей A и B внутри дерева HashMap]
                     │
                     ▼
    (Шаг 1: Сравнение по hash)
       ──► A.hash != B.hash ──► [Определили сторону: влево/вправо]
                     │ (если хэши равны)
                     ▼
    (Шаг 2: Попытка вызвать Comparable)
       ──► A implements Comparable ──► [Вызов A.compareTo(B)]
                     │ (если Comparable нет)
                     ▼
	(Шаг 3: Сравнение по имени классов)
       ──► Сравнение строк через ClassName.compareTo()
                     │ (если один и тот же класс)
                     ▼
	(Шаг 4: Метод tieBreakOrder)
       ──► Сравнение через System.identityHashCode(obj)
```

1. **Шаг 1: Проверка по хэш-кодам.** Сначала мапа сравнивает вычисленные хэши ключей. Если `A.hash < B.hash`, объект `A` уходит налево, если больше — направо. Если хэши равны (прямая коллизия), мапа идет глубже.
   >**хэш - полученный внутри мапки** (то есть финальный хэш, который `HashMap` рассчитала и сохранила в поле `hash` самого узла). Примерно так:
   ```Java
static final int hash(Object key) {
	int h;
	return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
   ```
    
2. **Шаг 2: Попытка использовать `Comparable`.** Мапа через рефлексию проверяет, реализует ли класс ключа `A` интерфейс `Comparable`. Если да, и класс объекта `B` такой же, вызывается метод `A.compareTo(B)`. Если нет, включается шаг 3.
    
3. **Шаг 3: Сравнение по имени классов.** Если в коллизию попали объекты разных типов (например, `StringWrapper` и `User`), мапа берет строковые имена их классов (`A.getClass().getName()`) и сравнивает их как обычные строки. Если класс один и тот же, включается финальный судья.
    
4. **Шаг 4: Метод последней надежды — `tieBreakOrder()`.** Мапа вызывает внутреннюю функцию, которая сравнивает объекты по их **`System.identityHashCode(obj)`**. Это первозданный хэш-код JVM по умолчанию, вычисляемый на основе рантайм-идентификатора объекта. Так как два физически разных объекта в памяти в один момент времени не могут иметь одинаковый `identityHashCode`, этот шаг гарантированно и уникально ранжирует элементы, распределяя их по дереву.    

## 🔄 3. Обратная деградация дерева в список и сохранение порядка
Что происходит, когда элементы начинают удаляться, и дерево сжимается до порога `UNTREEIFY_THRESHOLD` (6 элементов)?

### Будет ли сохранен исходный порядок элементов после цикла «Список ──► Дерево ──► Список»?

> **Ответ:** **Нет, исходный порядок элементов гарантированно нарушится.** #### Почему порядок теряется:
> 
> Узлы дерева `TreeNode<K,V>` в `HashMap` являются гибридными структурами. Они расширяют класс `LinkedHashMap.Entry` и одновременно содержат в себе:

- Ссылки для дерева: `left`, `right`, `parent`, `red/black` (цвет).
    
- Ссылки для списка: `next` и `prev` (двусвязный список).
    

Да-да, пока бакет является деревом, мапа _параллельно_ поддерживает внутри него структуру **двусвязного списка** для облегчения обхода. Однако при трификации и балансировке красно-черного дерева (поворотах узлов) ссылки `next` и `prev` постоянно перестраиваются в соответствии с внутренней топологией дерева и результатами алгоритма `tieBreakOrder`.

Когда дерево вырождается обратно при достижении размера 6, мапа просто берет эту текущую древесную последовательность связей `next` и превращает её обратно в простой односвязный список `Node<K,V>`. Изначальный порядок, в котором ты добавлял элементы в карту до трификации, безвозвратно теряется.

## 📈 4. Увеличение количества бакетов (Resize) против трификации

Превращение коллизионного бакета в дерево — это локальная мера защиты. Глобальное же управление памятью в мапе осуществляется через **Ресайз (Resize)** — увеличение количества бакетов в 2 раза.

### При каких условиях увеличивается количество бакетов?

Общее количество бакетов увеличивается в два раза при выполнении одного из условий:

1. **Превышение Load Factor:** Когда общее количество элементов в мапе (`size`) превышает порог:
    
    $$\text{Threshold} = \text{Capacity} \times \text{Load Factor}$$
    
    _(По умолчанию: для мапы размера 16 с Load Factor 0.75 ресайз до 32 бакетов произойдет на 13-м элементе)._
    
2. **Защита от преждевременного перевода в дерево:** Если в одном бакете накопилось 8 элементов, но общий размер таблицы (`capacity`) **меньше 64**, мапа **ОТКАЗЫВАЕТСЯ переводить бакет в дерево**. Вместо этого она принудительно вызывает метод `resize()`, увеличивая число бакетов в 2 раза и перераспределяя (перехэшируя) элементы. Мапа надеется, что при увеличении числа бакетов коллизия рассосется сама собой.
    

### Как ресайз влияет на существующие деревья?

Когда происходит ресайз, абсолютно каждое существующее в мапе дерево расщепляется на две части (так как емкость бакетов увеличилась вдвое, часть элементов по новому битовому индексу останется в старом бакете `index`, а часть улетит в бакет `index + oldCapacity`).

В процессе этого расщепления (`split`) мапа подсчитывает, сколько элементов осталось в каждой из двух половин. Если в какой-то из половин осталось **6 или меньше элементов**, это поддерево тут же автоматически деградирует обратно в односвязный список (`untreeify`).

## 📝 Финальное резюме (Шпаргалка для собеседования)

- Список переходит в дерево, если в бакете **$\ge 8$ элементов** И общая емкость мапы **$\ge 64$**.
    
- Если в бакете $\ge 8$ элементов, но общая емкость **$< 64$**, дерева не будет — вместо этого мапа сделает глобальный **`resize()`**.
    
- Дерево превращается обратно в список, если в нем осталось **$\le 6$ элементов** (происходит при удалении или при расщеплении бакетов во время ресайза).
    
- Порядок элементов при обратном превращении в список **не сохраняется**, так как связи перестраивались в угоду балансировке красно-черного дерева.
    
- Поиск без `Comparable` возможен благодаря методу **`tieBreakOrder()`**, который ранжирует объекты по системному `System.identityHashCode(obj)`.
