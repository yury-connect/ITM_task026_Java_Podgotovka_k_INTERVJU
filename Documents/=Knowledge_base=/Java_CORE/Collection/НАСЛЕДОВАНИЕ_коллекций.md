## 🔍 Какие коллекции можно наследовать

**Почти все стандартные коллекции НЕ являются `final`**, поэтому от них можно наследоваться:

| ✅Можно наследовать✅ | ❌Нельзя (final)❌          |
| ------------------- | ------------------------- |
| `ArrayList`         | `String`                  |
| `LinkedList`        | `Integer`                 |
| `HashSet`           | `LocalDate`               |
| `TreeSet`           | `Arrays` (утилитный)      |
| `HashMap`           | `Collections` (утилитный) |
| `TreeMap`           | `System`                  |
| `ConcurrentHashMap` |                           |
| `Vector` (устарел)  |                           |
| `Stack` (устарел)   |                           |
```java
// ✅ Это компилируется и работает
public class MyArrayList<E> extends ArrayList<E> { ... }
public class MyHashMap<K,V> extends HashMap<K,V> { ... }
```

### ❌ Почему наследовать коллекции **НЕ** РЕКОМЕНДУЕТСЯ?

#### 1. **Нарушение инкапсуляции**
```java
public class LoggingList<E> extends ArrayList<E> {
    @Override
    public boolean add(E e) {
        System.out.println("Adding: " + e);
        return super.add(e);
    }
}
```
**Проблема:** Если кто-то вызовет `addAll()`, внутри неё может не вызываться `add()` (зависит от реализации JDK). Ваше логирование сработает не всегда.

#### 2. **Хрупкость реализации (Fragile Base Class)**
```java
// В Java 8 addAll() вызывал add()
// В Java 11+ реализация может измениться
// Ваш переопределённый метод может перестать вызываться!
```

#### 3. **Нарушение контрактов**
```java
public class MyMap extends HashMap<String, String> {
    @Override
    public String put(String key, String value) {
        // ваша логика
        return super.put(key, value);
    }
}
// Кто-то вызывает:
map.putAll(otherMap); // Может не вызвать ваш put()
```
#### 4. **Проблемы с equals() и hashCode()**

При наследовании легко нарушить контракт `equals()`/`hashCode()`, если добавить новые поля.
