# Двоичное дерево поиска / Binary Search Tree (BST)

**Двоичное дерево поиска (BST)** — это модифицированное бинарное дерево, в котором на уровне самой структуры данных наложено строгое математическое свойство упорядоченности. Это свойство превращает дерево из обычного иерархического контейнера в мощный инструмент для быстрого поиска, вставки и удаления элементов.

---
## 📜 1. Фундаментальное свойство BST
Для любого узла $X$ в двоичном дереве поиска выполняются следующие правила:
	
- Значение любого узла в **левом поддереве** элемента $X$ строго **меньше** значения самого узла $X$.
    
- Значение любого узла в **правом поддереве** элемента $X$ строго **больше** значения самого узла $X$.
    
- Оба поддерева (левое и правое) также обязаны являться двоичными деревьями поиска.

> **Важно:** По умолчанию классическое BST не допускает хранения дубликатов (ключи должны быть уникальными). Если дубликаты необходимы по бизнес-логике, их либо складывают в специальный список внутри самого узла, либо жестко договариваются считать дубликаты «строго большими» (и отправлять направо) или «строго меньшими» (налево).

---
## 🏎️ 2. Основные операции и их сложность
Благодаря свойству упорядоченности, поиск элементов в BST напоминает алгоритм бинарного поиска по массиву: на каждом шаге мы отбрасываем ровно половину вариантов.

### 🔍 А. Поиск элемента (Search)
Мы сравниваем искомый ключ $K$ с текущим узлом.
	
1. Если $K == \text{узел.value}$ — элемент найден.
    
2. Если $K < \text{узел.value}$ — мы рекурсивно или в цикле уходим в **левого** потомка.
    
3. Если $K > \text{узел.value}$ — мы уходим в **правого** потомка.
    
4. Если мы дошли до ссылки `null` — элемента в дереве нет.

### 📥 Б. Вставка элемента (Insertion)
Новый элемент всегда вставляется в дерево в качестве **нового листа**. Мы бежим по дереву точно так же, как при поиске, пока не упремся в свободное место (`null`), куда и прикрепляем новый узел.

### 🗑️ В. Удаление элемента (Deletion) — _Самая сложная операция_
При удалении узла $X$ возможны три сценария, в зависимости от его окружения:
	
1. **Узел является листом (нет детей):** Мы просто зануляем ссылку на него у его родителя.
    
2. **Узел имеет только одного ребенка (левого или правого):** Мы удаляем узел $X$, а его родителя напрямую связываем с этим единственным ребенком (узел «пропускается»).
    
3. **Узел имеет двух детей:** Чтобы не разрушить дерево, узел $X$ нельзя просто удалить. Его значение нужно заменить на значение **преемника** (самого близкого по значению элемента). Преемником может быть:
	    
    - Либо _самый правый_ элемент в его левом поддереве (максимум слева).
        
    - Либо _самый левый_ элемент в его правом поддереве (минимум справа).
        
        После подмены преемник (который гарантированно имеет не более одного ребенка) удаляется по правилу 1 или 2.

## 📊 3. Временная сложность и проблема деградации
В идеальном, сбалансированном состоянии высота BST составляет $\log_2 N$. Соответственно, операции поиска, вставки и удаления выполняются за великолепное время $O(\log N)$.

### Временная сложность операций:

|**Операция**|**В среднем (Сбалансированное дерево)**|**В худшем случае (Вырожденное дерево)**|
|---|---|---|
|**Поиск (Search)**|$O(\log N)$|$O(N)$|
|**Вставка (Insert)**|$O(\log N)$|$O(N)$|
|**Удаление (Delete)**|$O(\log N)$|$O(N)$|

### 🚨 Проблема вырождения (Худший случай)
Обычное двоичное дерево поиска крайне чувствительно к порядку входных данных. Если добавлять элементы в дерево в строго отсортированном порядке (например: 10, 20, 30, 40, 50), каждый новый элемент будет уходить строго направо.

Дерево **выродится в линейный односвязный список**, высота дерева станет равной $N$, а скорость поиска рухнет до $O(N)$.

> **Как это решается на практике:** Чтобы избежать деградации, в продакшене используют **самобалансирующиеся деревья поиска** (AVL-деревья или Красно-черные деревья). Они автоматически применяют операции «поворотов» ветвей при вставках и удалениях, насильно удерживая высоту дерева в рамках $O(\log N)$.

## 🏃 4. Магия центрированного обхода (In-order Traversal)
Как мы помним из базовых бинарных деревьев, центрированный обход в глубину следует правилу: `Лево ──► Корень ──► Право`.

Если запустить этот обход на двоичном дереве поиска, то за счет его структуры мы **гарантированно посетим все узлы в строго отсортированном по возрастанию порядке**. Это позволяет выполнять сортировку данных (Tree Sort) со сложностью $O(N \log N)$.
```Java
// Метод выведет элементы BST в идеальном порядке: от меньшего к большему
public void printInOrder(TreeNode node) {
    if (node != null) {
        printInOrder(node.left);       // Идем в левое поддереве (там все, что меньше)
        System.out.print(node.value);  // Обрабатываем текущий корень
        printInOrder(node.right);      // Идем в правое поддерево (там все, что больше)
    }
}
```

## ⚖️ 5. Плюсы и минусы BST

#### **Преимущества:**
- **Динамический размер:** В отличие от массивов, не требует перевыделения памяти и копирования данных при расширении.
    
- **Быстрый поиск и модификация:** При должной балансировке работает кардинально быстрее списков и массивов на больших объемах данных.
    
- **Встроенная сортировка:** Данные всегда поддерживаются в упорядоченном состоянии «из коробки», облегчая поиск диапазонов (минимумов, максимумов).

#### **Недостатки:**
- **Нестабильность без балансировки:** Обычное BST не гарантирует скорость $O(\log N)$ и легко деградирует в список при плохих входных данных.
    
- **Сложное удаление:** Алгоритм удаления узлов с двумя детьми требует многоходовых перестановок и перелинковки ссылок.
    
- **Расход памяти:** Повышенные затраты на хранение указателей `left` и `right` для каждого элемента.
