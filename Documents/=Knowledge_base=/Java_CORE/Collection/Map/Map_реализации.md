# 🗺️ Комплексный гид по реализациям интерфейса `Map` в Java

Семейство `Map` в Java разделено на три фундаментальные архитектурные группы: общего назначения, специализированные и потокобезопасные (конкурентные).

---
## 🟥 ГРУППА 1: Коллекции общего назначения

### 1. `HashMap` (Стандартный дефолт)
Базовая хэш-таблица, обеспечивающая максимально быструю работу за счет прямого доступа к памяти.

- **Внутреннее устройство:** Массив бакетов (`Node<K,V>[] table`), индекс ячейки вычисляется как `hash & (length - 1)`. При коллизиях бакет растет как _односвязный список_. Начиная с Java 8, если длина списка $\ge 8$ элементов, а общая емкость мапы $\ge 64$, бакет перестраивается (_трифицируется_) в _красно-черное дерево_. При уменьшении элементов до $\le 6$ дерево деградирует обратно в список.
    
- **Порядок элементов:** Абсолютно хаотичный. Порядок итерации зависит исключительно от хэш-кодов ключей и может измениться после изменения размера массива (`resize`).
    
- **Потокобезопасность:** ❌ Нет. При конкурентной модификации выбрасывает `ConcurrentModificationException` (механизм fail-fast).
    
- **Специфика на собеседовании:** Потребуют расписать алгоритм «последней надежды» `tieBreakOrder()` (сравнение по `System.identityHashCode()`), используемый в деревьях мапы, если у кастомных ключей нет интерфейса `Comparable`.

### 2. `LinkedHashMap` (Хэш-таблица с предсказуемым порядком)
Гибрид хэш-таблицы и двусвязного списка, созданный для сохранения порядка элементов.

- **Внутреннее устройство:** Расширяет `HashMap`. Её узлы (`LinkedHashMap.Entry`) содержат дополнительные ссылки `before` и `after`. Поверх стандартного массива бакетов натягивается **единый глобальный двусвязный список**, проходящий через все элементы мапы.
    
- **Порядок элементов:** 1. _Insertion-order (По умолчанию):_ Порядок строго соответствует хронологии добавления элементов через `put()`.
    
    2. _Access-order (Включается флагом в конструкторе):_ Порядок перестраивается при чтении. Вызов `get(key)` перемещает этот элемент в самый конец списка как «самый свежий».
    
- **Потокобезопасность:** ❌ Нет.
    
- **Специфика на собеседовании:** Спросят про паттерн **LRU-Cache (Least Recently Used)**. Расскажи, что переопределив защищенный метод `removeEldestEntry()`, можно заставить мапу автоматически удалять элементы, к которым дольше всего не обращались, при превышении лимита размера.

### 3. `TreeMap` (Сортирующее дерево)
Реализация интерфейса `NavigableMap`, поддерживающая ключи в строго отсортированном состоянии.

- **Внутреннее устройство:** Физически представляет собой **Красно-черное дерево** (самобалансирующееся двоичное дерево поиска). Операции поиска, вставки и удаления занимают стабильное время $O(\log N)$.
    
- **Порядок элементов:** Строго отсортированный по ключам. Либо естественный порядок (_Natural Ordering_ через интерфейс `Comparable`), либо кастомный порядок (заданный через объект `Comparator` в конструкторе).
    
- **Потокобезопасность:** ❌ Нет.
    
- **Специфика на собеседовании:** 1. **Ловушка с `null`:** Ключ `null` запрещен (вызовет `NullPointerException`), так как мапа обязана вызвать метод сравнения.
    
    2. **Игнорирование `equals()`:** Мапа проверяет уникальность ключей **исключительно** через `compareTo()` или `compare()`. Если метод вернул `0`, элемент признается дубликатом и перезаписывается, даже если `equals()` возвращает `false`.

## 🟨 ГРУППА 2: Специализированные мапы

### 4. `WeakHashMap` (Мапа с защитой от утечек памяти)
Хэш-таблица, оптимизированная под автоматическое кэширование временных данных.

- **Внутреннее устройство:** Ключи мапы оборачиваются в **слабые ссылки (`WeakReference`)**. Мапа работает в связке с системной очередью `ReferenceQueue`.
    
- **Порядок элементов:** Хаотичный (наследует логику хэш-таблиц).
    
- **Потокобезопасность:** ❌ Нет.
    
- **Специфика на собеседовании:** Как только в приложении исчезает последняя сильная ссылка на объект-ключ, Сборщик Мусора (GC) при первом же цикле удаляет этот объект из памяти. `WeakHashMap` замечает это через `ReferenceQueue` и автоматически вычищает соответствующее значение, предотвращая утечки памяти (`OutOfMemoryError`). Идеальна для кэшей.

### 5. `IdentityHashMap` (Нарушитель контрактов Java)
Специализированная мапа для системных нужд, игнорирующая базовую логику сравнения объектов.

- **Внутреннее устройство:** Внутри нет бакетов и цепочек `Node`. Под капотом лежит один плоский массив `Object[] table`. Ключ и значение хранятся в соседних ячейках (`table[i]` — ключ, `table[i+1]` — значение). Коллизии разрешаются методом _линейного исследования_ (запись в следующую свободную ячейку).
    
- **Специфика на собеседовании:** Главный подвох — мапа **категорически игнорирует метод `equals()` и хэш-коды**. Равенство ключей проверяется исключительно оператором **строгого сравнения ссылок (`==`)**. Два объекта `new String("key")` и `new String("key")` внутри этой мапы будут жить как два совершенно разных ключа. Применяется при сериализации или глубоком копировании графов объектов.

### 6. `EnumMap` (Высокоскоростная микросхема)
Экстремально быстрая мапа, ключами которой могут быть только элементы одного конкретного перечисления (`Enum`).

- **Внутреннее устройство:** Полное отсутствие хэш-функций и коллизий. Внутри лежит **простой массив Java (`Object[] vals`)**, длина которого строго равна количеству элементов в `Enum`.
    
- **Порядок элементов:** Соответствует порядку объявления констант в самом `Enum`.
    
- **Специфика на собеседовании:** Мапа работает со скоростью чистых массивов за $O(1)$. При вызове `put(MyEnum.RED, value)` мапа берет порядковый номер константы (`MyEnum.RED.ordinal()`) и использует его напрямую как индекс массива. Занимает минимум памяти и работает быстрее, чем `HashMap`.

## 🟦 ГРУППА 3: Высоконагруженные конкурентные (Многопоточные) мапы

### 7. `ConcurrentHashMap` (Индустриальный стандарт многопоточности)
Высокопроизводительная потокобезопасная хэш-таблица, заменившая устаревшую `Hashtable`.

- **Внутреннее устройство (Java 8+):** Построена на массиве бакетов. Для безопасной вставки в пустую ячейку используются **CAS-операции** (Compare-And-Swap) на уровне процессора без блокировки потока. Если в ячейке уже есть коллизия (цепочка или дерево), поток блокирует **только голову (первый узел) этого конкретного бакета** с помощью легковесного монитора `synchronized`.
    
- **Порядок элементов:** Хаотичный.
    
- **Потокобезопасность:** Да. Чтение (`get()`) происходит **полностью без блокировок** за счет использования ключевого слова `volatile` у полей узлов. Мапа позволяет множеству потоков параллельно писать в разные бакеты, не мешая друг другу.
    
- **Специфика на собеседовании:** Обязательно упомяни, что итератор этой мапы имеет поведение **weakly consistent (согласованный в общих чертах)**. Он не выбрасывает `ConcurrentModificationException`, отражает состояние мапы на момент создания итератора и может (но не гарантирует) заметить изменения, внесенные другими потоками во время обхода.

### 8. `ConcurrentSkipListMap` (Многопоточный сортировщик)
Потокобезопасный аналог `TreeMap`, реализующий интерфейс `ConcurrentNavigableMap`.

- **Внутреннее устройство:** Реализована на базе структуры данных **Skip List (Список с пропусками)**. Вместо сложной балансировки дерева (которая в многопоточной среде требовала бы полной блокировки всей структуры ради поворота ветвей), здесь используется многоуровневый связный список. Нижний уровень содержит все элементы отсортированными, а верхние уровни служат «экспресс-дорожками» для быстрого перепрыгивания через узлы.
    
- **Порядок элементов:** Строго отсортированный.
    
- **Потокобезопасность:** Да. Все изменения связей между узлами на разных уровнях выполняются атомарно с помощью неблокирующих CAS-инструкций. Скорость поиска составляет $O(\log N)$.
