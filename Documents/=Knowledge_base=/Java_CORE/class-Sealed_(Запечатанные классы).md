# **Sealed (*Запечатанные классы*)**

**Sealed Classes (Запечатанные классы)** — это важнейшая фича Java (ставшая стабильной в Java 17), которая позволяет разработчику официально заявить: _«Я создаю класс-родитель, но я **строго ограничиваю** круг тех, кто имеет право от него наследоваться»_.

До появления запечатанных классов в Java было только две крайности управления наследованием:

1. Класс открыт для всех (`public class`) — любой программист в проекте или внешней библиотеке может от него унаследоваться.
    
2. Класс закрыт для всех (`final class`) — наследоваться нельзя вообще никому.

`Sealed` классы — это элегантная золотая середина. Ты открываешь наследование, но только для «своих».

### 🛠️ Как это пишется в коде?

Представь, что мы пишем модуль для обработки платежей. У нас есть базовый класс `Payment` и мы хотим, чтобы в системе существовало строго три его наследника: `CreditCardPayment`, `PayPalPayment` и `CryptoPayment`. Никаких других видов платежей никто самовольно добавить не должен.
```Java
// 1. Помечаем родительский класс как sealed и перечисляем разрешенных наследников
public sealed class Payment permits CreditCardPayment, PayPalPayment, CryptoPayment {
    private final double amount;

    public Payment(double amount) {
        this.amount = amount;
    }
}
```

Теперь мы обязаны объявить наших наследников. Но здесь Java ставит жесткое условие: **каждый разрешенный наследник обязан явно заявить, как он закрывает цепочку наследования дальше**.

Для этого у наследников есть ровно три пути (модификатора):

#### Вариант А: Наследник тоже закрыт (`final`) — _Самый частый выбор_
```Java
// Больше от CreditCardPayment наследоваться нельзя вообще никому
public final class CreditCardPayment extends Payment {
    public CreditCardPayment(double amount) { super(amount); }
}
```

#### Вариант Б: Наследник тоже запечатан (`sealed`) — _Продолжаем контролировать ветку_
```Java
// Мы разрешаем наследовать PayPalPayment, но снова строго по списку
public sealed class PayPalPayment extends Payment permits PersonalPayPal, BusinessPayPal {
    public PayPalPayment(double amount) { super(amount); }
}
```

#### Вариант В: Наследник открывается (`non-sealed`) — _Свобода для этой ветки_
```Java
// Мы снимаем ограничения. От CryptoPayment может наследоваться кто угодно
public non-sealed class CryptoPayment extends Payment {
    public CryptoPayment(double amount) { super(amount); }
}
```

### 🚨 Главные правила и ограничения компиляции

Чтобы код скомпилировался, Java выдвигает к `sealed` структурам три железных требования:

1. **Локализация:** Все разрешенные наследники (`permits`) должны находиться **в том же модуле** (если используются Java-модули) или **в том же пакете (package)**, что и сам запечатанный класс. Ты не можешь запечатать класс в своем коде и разрешить наследоваться от него классу из другой библиотеки.
    
2. **Прямое наследование:** Наследники, указанные в секции `permits`, обязаны напрямую расширять (`extends` или `implements`) запечатанный класс/интерфейс.
    
3. **Автосокращение пакета:** Если запечатанный класс и все его наследники объявлены **внутри одного и того же `.java` файла** (как вложенные классы), секцию `permits` можно вообще опустить! Компилятор сам догадается, кто наследники.

### 🎯 Главный профит: Зачем это нужно на практике?

Помимо очевидной защиты архитектуры от «кривых рук», запечатанные классы открывают фантастические возможности для **Pattern Matching** (сопоставления с паттернами), начиная с Java 21 в операторе `switch`.

Так как компилятор на 100% знает весь список наследников, тебе больше не нужно писать блок `default` в `switch`! Компилятор сам проверит, все ли варианты ты обработал:
```Java
public void processPayment(Payment payment) {
    // SWITCH сам понимает, что вариантов всего 3!
    switch (payment) {
        case CreditCardPayment cc -> System.out.println(
	        "Прокатываем карту");
        case PayPalPayment pp     -> System.out.println(
	        "Перенаправляем на сайт PayPal");
        case CryptoPayment crypto -> System.out.println(
	        "Ждем подтверждения транзакции в блокчейне");
        // Блок default НЕ НУЖЕН. 
        // Если ты удалишь один из case, код просто не скомпилируется!
    }
}
```

Если завтра твоя команда решит добавить новый тип платежа, вы добавите его в `permits`, и компилятор **сразу же подсветит ошибки** во всех `switch` по всему проекту, напоминая, что этот новый платеж тоже нужно обработать. Это полностью исключает случайные баги в продакшене.
