## 17. Хранение данных и *Log Compaction*

#### **Как Kafka пишет на диск и зачем нужно хранение по ключу?**

- **Sequential Write:** Kafka пишет данные строго последовательно в конец файла (append-only). Это позволяет использовать пропускную способность диска на 100%, так как отсутствуют задержки на поиск секторов (seek time), что делает HDD почти такими же быстрыми, как SSD для этой задачи.
    
- **Page Cache:** Kafka не кэширует данные в куче (JVM Heap), а делегирует это операционной системе. Данные пишутся в Page Cache и асинхронно сбрасываются на диск. При чтении, если данные еще в кэше, обращения к диску не происходит вовсе.
    
- **Log Compaction (`cleanup.policy=compact`):** Механизм, при котором Kafka удаляет старые записи с тем же ключом, оставляя только **последнее актуальное значение**.
    
    - _Пример:_ Хранение текущего баланса пользователя или статуса заказа. Нам не нужна вся история изменений за год, достаточно знать финальное состояние.

## 18. Политики очистки и сегменты

#### **Зачем лог разбивается на файлы и как работают лимиты?**

Партиция — это не один огромный файл, а набор **сегментов** (по умолчанию по 1 ГБ).

- **Зачем нужны сегменты:** Очистка данных (удаление или компакция) происходит целыми сегментами. Это дешевая операция для ОС (удаление файла), в отличие от удаления строк внутри файла.
    
- **Политики очистки (`delete`):**    
    - `retention.ms`: Удалять сегменты, где все записи старше указанного времени (*например, 7 дней*).
        
    - `retention.bytes`: Удалять старые сегменты, если общий размер партиции превысил лимит.
    
- **Активный сегмент:** Текущий файл, в который идет запись. Он никогда не удаляется и не компактируется, пока не будет "закрыт" по достижении лимита размера или времени.

## 19. `ZooKeeper` vs `KRaft`

#### **Почему *Kafka* перешла на собственное управление метаданными?**

С версии 3.3 (*и окончательно в 4.0*) Kafka перешла на протокол **KRaft**.

- **Проблема ZooKeeper:** Был "бутылочным горлышком". При падении контроллера или большом количестве партиций (*более 100к*) синхронизация метаданных занимала слишком много времени. Это требовало отдельного кластера и навыков его настройки.
    
- **Преимущества KRaft:**    
    - **Масштабируемость:** Поддержка миллионов партиций.
        
    - **Быстрое восстановление:** Метаданные хранятся в самой Kafka в специальном логе. Новый контроллер мгновенно получает актуальное состояние.
        
    - **Простота:** Больше не нужно разворачивать и мониторить отдельный стек (ZooKeeper). Кластер становится монолитным.

## 20. Низкоуровневые оптимизации: `Zero-copy`

#### **Как *Kafka* достигает такой огромной пропускной способности?**

В обычном приложении данные копируются 4 раза (Disk → Kernel → App Buffer → Socket Buffer → NIC). Kafka использует системный вызов **`sendfile`** (в Java — `FileChannel.transferTo()`), реализующий принцип **Zero-copy**:

1. Данные читаются из диска в **Kernel Page Cache**.
    
2. Ядро ОС копирует данные напрямую из Page Cache в буфер сетевой карты (NIC).
    
3. **User Space (JVM)** вообще не участвует в пересылке. CPU не тратит циклы на копирование байтов туда-обратно. Это позволяет Kafka выдавать скорость, ограниченную только пропускной способностью сети.

## 21. Экосистема (*Streams, Connect, Registry*)

#### **Когда использовать специализированные инструменты?**

- **Kafka Streams:** Библиотека для сложной обработки. Используйте её, если нужно:
    
    - Сделать **Join** двух топиков (например, заказы и пользователи).
        
    - Сделать агрегацию (сумма продаж за час в скользящем окне).
        
    - Хранить промежуточное состояние в **KTable** (локальная БД RocksDB под капотом).
        
    - _В отличие от `@KafkaListener`, Streams берет на себя управление состоянием и масштабирование агрегатов._
    
- **Schema Registry (Avro/Protobuf):**
    
    - **Зачем:** Гарантирует, что Producer не пришлет данные в формате, который Consumer не сможет прочитать.
        
    - **Как работает:** Producer отправляет ID схемы, а Consumer скачивает схему по этому ID из Registry.
        
    - **Совместимость:** Позволяет добавлять новые поля в данные так, чтобы старые версии сервисов не "падали" (Backward/Forward Compatibility).
    
- **Kafka Connect:** Готовые коннекторы для синхронизации.
    
    - _Пример:_ "Переливать все изменения из БД Postgres в Elasticsearch без написания кода".
