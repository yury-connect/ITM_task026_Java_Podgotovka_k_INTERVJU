# 📮 Семантики доставки данных в **`Kafka`**

---
## 🎯 Что такое семантика доставки?
**Определение:** Гарантии того, что сообщение будет доставлено от Producer до Consumer **ровно один раз**, несмотря на сбои.

## 📊 Три типа семантики

|Тип|Producer → Broker|Broker → Consumer|Общий итог|
|---|---|---|---|
|**At-most-once**|≤ 1 раз|≤ 1 раз|**Максимум 1 раз** (потери)|
|**At-least-once**|≥ 1 раз|≥ 1 раз|**Минимум 1 раз** (дубли)|
|**Exactly-once**|= 1 раз|= 1 раз|**Ровно 1 раз** (идеал)|

---
## 1️⃣ **At-most-once** (*Не более одного раза*)

### 🔧 Настройки
```properties
# Producer
acks=0
enable.idempotence=false

# Consumer
enable.auto.commit=true
auto.commit.interval.ms=1000
```

### 🔄 Как работает
```text
Producer → [ОТПРАВИЛ] → Broker
           (не ждёт подтверждения)
    
Consumer → [ЧИТАЕТ] → [КОММИТИТ OFFSET] → [ОБРАБАТЫВАЕТ]
           (коммит до обработки!)
```

### ⚠️ Риски

|Что может случиться|Результат|
|---|---|
|Producer не получил ack|Сообщение **потеряно**|
|Consumer упал после коммита|Сообщение **потеряно**|

### ✅ Плюс / ❌ Минус

|(+) Скорость|(-) Потеря данных|
|---|---|
|Максимальная производительность|Некритичные метрики, логи|

---
## 2️⃣ **At-least-once** (*Не менее одного раза*)

### 🔧 Настройки
```properties
# Producer
acks=all
enable.idempotence=false
retries=Integer.MAX_VALUE

# Consumer
enable.auto.commit=false
```

### 🔄 Как работает
```text
Producer → [ОТПРАВИЛ] → ждёт ack от всех реплик → [OK]
           (если нет ack → ретрай)
    
Consumer → [ЧИТАЕТ] → [ОБРАБАТЫВАЕТ] → [КОММИТИТ OFFSET]
           (коммит ПОСЛЕ обработки!)
```

### ⚠️ Риски

|Что может случиться|Результат|
|---|---|
|Producer отправил, но ack не пришёл → ретрай|**ДУБЛИКАТ**|
|Consumer обработал, но упал до коммита|**ДУБЛИКАТ** при перезапуске|

### ✅ Плюс / ❌ Минус

|(+) Нет потерь|(-) Дубликаты|
|---|---|
|Критичные данные (платежи, заказы)|Нужна идемпотентность на consumer|

---
## 3️⃣ **Exactly-once** (*Ровно один раз*)

### 🔧 Настройки
```properties
# Producer
acks=all
enable.idempotence=true
max.in.flight.requests.per.connection=1

# Consumer
enable.auto.commit=false
isolation.level=read_committed
```

### 🔄 Как работает
```text
Producer:
1. Включает идемпотентность → каждое сообщение имеет PID + sequence number
2. Broker дубликаты отсекает

Consumer + Transaction:
1. Читает сообщение
2. Обрабатывает
3. Коммитит offset В ТОЙ ЖЕ ТРАНЗАКЦИИ, что и вывод результата
```

### 🛡️ Механизмы Exactly-once

|Компонент|Механизм|
|---|---|
|**Producer**|Идемпотентность (`enable.idempotence=true`)|
|**Producer**|Транзакции (`initTransaction()`, `sendOffsetsToTransaction()`)|
|**Consumer**|`read_committed` (не видит незакоммиченные)|
|**Consumer Group**|Кооперация с Producer через `transactional.id`|

### ✅ Плюс / ❌ Минус

|(+) Нет дублей, нет потерь|(-) Медленнее, сложнее|
|---|---|
|Финансы, критические транзакции|~20-30% потеря производительности|

---
## 📊 Итоговая таблица

|Семантика|Потери|Дубли|Скорость|Сложность|Когда использовать|
|---|---|---|---|---|---|
|**At-most-once**|✅ Да|❌ Нет|🚀 Макс|📘 Низкая|Логи, метрики, мониторинг|
|**At-least-once**|❌ Нет|✅ Да|⚡ Высокая|📙 Средняя|**Большинство кейсов**|
|**Exactly-once**|❌ Нет|❌ Нет|🐢 -30%|📕 Высокая|Финансы, платёжные системы|

## 🔧 Как выбрать для вашего проекта
```java
// Notification Service (как пример)
// Потери или дубли уведомлений НЕ критичны
// → At-least-once достаточно!

@KafkaListener(
    topics = "delivery.notify-event",
    properties = {
        "enable.auto.commit=false",      // ручной коммит
        "isolation.level=read_committed" // читаем только закоммиченные
    }
)

public void consume(NotificationEvent event) {
    // обработали
    // коммитим вручную ПОСЛЕ успеха
    ack.acknowledge();
}
```

## 🧠 Шпаргалка

| Семантика         | Запомнить                                                |
| ----------------- | -------------------------------------------------------- |
| **At-most-once**  | "Отправил и забыл" — быстро, но **теряем**               |
| **At-least-once** | "Повторяй пока не получится" — надёжно, но **с дублями** |
| **Exactly-once**  | "Транзакции + идемпотентность" — идеально, но **дорого** |

## 📝 Итог одной фразой

> **At-most-once** — потеря данных допустима.  
> **At-least-once** — дубликаты допустимы (нужна идемпотентность).  
> **Exactly-once** — ничего не теряем и не дублируем (но медленно).
