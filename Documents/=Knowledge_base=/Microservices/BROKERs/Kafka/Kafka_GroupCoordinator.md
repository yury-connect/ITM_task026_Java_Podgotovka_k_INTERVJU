# 📦 Распределение *партиций* между *консьюмерами* в *группе* **Kafka**

**Основная цель механизмов:** Обеспечение параллельной обработки данных, равномерное распределение нагрузки (балансировка) и строгое сохранение порядка сообщений.

### 👑 Роль `Group Coordinator`
**Group Coordinator** — это конкретный **брокер** (broker) на стороне Kafka, который выполняет следующие функции:
	
- Отслеживает актуальный состав консьюмер-группы (consumer group).
    
- Запускает и координирует процесс перераспределения партиций (rebalance).
    
- Контролирует текущее состояние группы.

### ⚙️ [Стратегии распределения партиций](Kafka_Ребалансировка_PartitionAssignor)
Конкретный алгоритм назначения выбирается **на стороне консьюмера**. Существует несколько базовых стратегий:
	
1. **Range:** Распределяет партиции блоками по диапазонам (по порядку).
    
2. **RoundRobin:** Распределяет партиции циклически по кругу (веером).
    
3. **Sticky:** Пытается максимально сохранить за консьюмерами их старые партиции, чтобы минимизировать лишние движения данных при ребалансировке (rebalance).

### 📊 Пример работы (4 партиции и 2 консьюмера)
При стандартном сбалансированном сценарии партиции разделятся поровну:
	
- **Consumer A** $\rightarrow$ Partition 0, 1
    
- **Consumer B** $\rightarrow$ Partition 2, 3

### 🛑 Жесткие системные ограничения
При любом алгоритме распределения Kafka всегда строго соблюдает три правила:
	
- **Один к одному:** 
  Одна конкретная партиция может быть назначена **только 1** консьюмеру из группы.
    
- **Параллелизм:** 
  Один консьюмер может без проблем читать сразу несколько партиций.
    
- **Гарантия порядка:** 
  Порядок обработки сообщ-й гарант-но сохраняется **только внутри** одной партиции.
