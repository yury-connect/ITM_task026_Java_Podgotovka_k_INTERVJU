# 📡 Атрибут `acks` в `Kafka Producer`

---
## 🎯 Что такое `acks`

**Определение:** Параметр, который определяет, **сколько брокеров должны подтвердить** получение сообщения, прежде чем `Producer` сочтёт отправку успешной.

**Ключевой вопрос:** Гарантия сохранности vs Скорость отправки.

---

## 📊 Три возможных значения

|Значение|Название|Что делает|Гарантия|Скорость|
|---|---|---|---|---|
|**`acks=0`**|Fire-and-forget|Producer не ждёт подтверждения|**Нулевая**|🚀 **Максимальная**|
|**`acks=1`**|Leader only|Ждёт подтверждения только от лидера|**Низкая**|⚡ **Высокая**|
|**`acks=all`** (или `-1`)|All replicas|Ждёт подтверждения от лидера + всех ISR-реплик|**Максимальная**|🐢 **Низкая**|

---
## 1️⃣ `acks=0` — Fire-and-forget

### Как работает
```text
Producer → [ОТПРАВИЛ] → Broker
           (не ждёт ответа, идёт дальше)
```

### ⚠️ Риски

|Сценарий|Результат|
|---|---|
|Брокер упал до получения|Сообщение **потеряно навсегда**|
|Сетевой сбой|Producer **не узнает** об ошибке|

### ✅ Когда использовать
- Логи, метрики, трейсинг    
- Большие объёмы некритичных данных    
- Максимальная производительность любой ценой    
```properties
acks=0
# Скорость: ~1 млн сообщений/сек (зависит от сети)
```

---
## 2️⃣ `acks=1` — Leader only

### Как работает
```text
Producer → [ОТПРАВИЛ] → Leader Broker → [ОК]
                         ↓
                    Follower (могут не успеть)
```

### ⚠️ Риски

|Сценарий|Результат|
|---|---|
|Лидер получил, подтвердил, но умер до репликации|❌ **Потеря данных**|
|Реплики не успели синхронизироваться|Новый лидер не имеет этого сообщения|

### ✅ Когда использовать
- Стандартный выбор для большинства приложений    
- Компромисс между скоростью и надёжностью    
```properties
acks=1
# Скорость: ~100K сообщений/сек
# Риск: потеря при отказе лидера
```

---
## 3️⃣ `acks=all` (или `acks=-1`) — All replicas

### Как работает
```text
Producer → [ОТПРАВИЛ] → Leader Broker → ждёт...
                         ↓           ↓
                    Follower 1    Follower 2
                         ↓           ↓
                    подтвердили   подтвердили
                         ↓
                    Leader → [ОК] → Producer
```

### ⚠️ Важные нюансы

|Доп. параметр|Влияние|
|---|---|
|`min.insync.replicas`|Минимум реплик, которые должны подтвердить (обычно 2)|
|`replication.factor`|Общее количество реплик (обычно 3)|

**Формула надёжности:**
```text
Если replication.factor = 3
и min.insync.replicas = 2

→ сообщение НЕ потеряется, даже если 1 брокер упадёт
```

### ✅ Когда использовать
- Финансовые транзакции, платежи    
- Любые критичные данные    
- Приоритет надёжности над скоростью    
```properties
acks=all
min.insync.replicas=2
# Скорость: ~10K-20K сообщений/сек
# Надёжность: максимальная
```

---
## 📊 Сравнительная таблица

|Характеристика|`acks=0`|`acks=1`|`acks=all`|
|---|---|---|---|
|**Потеря сообщения**|✅ Да (при любом сбое)|⚠️ При падении лидера|❌ Нет (при правильной конфигурации)|
|**Задержка (latency)**|Минимальная|Низкая|Высокая|
|**Пропускная способность**|Максимальная|Высокая|Средняя|
|**Нагрузка на брокер**|Минимальная|Средняя|Высокая|
|**Рекомендация для продакшена**|❌ Нет|✅ Да (80% кейсов)|✅ Да (20% кейсов — финансы)|

---
## 🔗 Связь с другими параметрами

|Параметр|Как связан с `acks`|
|---|---|
|`retries`|При `acks=all` нужно ставить `Integer.MAX_VALUE`|
|`enable.idempotence`|Для exactly-once требует `acks=all`|
|`max.in.flight.requests.per.connection`|Для exactly-once требует `=1`|
|`min.insync.replicas`|Работает в паре с `acks=all`|
|`replication.factor`|Должен быть ≥ `min.insync.replicas`|

---
## 🧠 Выбор значения для вашего проекта
```properties
# Для Notification Service (учебный проект)
# Потеря уведомления — не критично
# → выбираем acks=1

spring.kafka.producer.acks=1
```

```properties
# Для Payment Service (финансы)
# Потеря невозможна
# → выбираем acks=all + min.insync.replicas=2

spring.kafka.producer.acks=all
spring.kafka.producer.retries=10
```

---
## 📝 Итог одной фразой

> **`acks=0`** — быстро, но теряем  
> **`acks=1`** — золотая середина (стандарт)  
> **`acks=all`** — надёжно, но медленно (финансы)
