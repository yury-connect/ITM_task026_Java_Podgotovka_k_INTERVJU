# Как спринг работает с транзакциями? Расскажите про аннотацию `@Transactional`.

---
## 🧩 Ключевые компоненты *Spring*-транзакций
🔽 `@Transactional`, `PlatformTransactionManager`, `AOP Proxy` 🔽

| Компонент                           | Описание                                                                                                                |
| :---------------------------------- | :---------------------------------------------------------------------------------------------------------------------- |
| `@Transactional`                    | **Помечает** методы или классы, метаданные которых преобразуются в транзакционную логику.                               |
| `PlatformTransactionManager`        | Центральный интерфейс, управляющий **физическим** открытием, коммитом и откатом транзакций.                             |
| **AOP Proxy**                       | Промежуточный объект-обертка, создаваемый Spring для **перехвата внешних вызовов** и внедрения транзакционного аспекта. |
| `TransactionSynchronizationManager` | Инфраструктурный класс, хранящий ресурсы (соединение, `EntityManager`) в `ThreadLocal` для **текущего** потока.         |

---
## 🛠️ Как Spring управляет транзакциями?
🔽Spring обеспечивает управление транзакциями ч-з `PlatformTransactionManager` 🔽

Управление транзакциями строится вокруг интерфейса `PlatformTransactionManager`, который **абстрагирует** логику взаимодействия с конкретными технологиями доступа к данным (`DataSourceTransactionManager` для JDBC, `JpaTransactionManager` для JPA/Hibernate).  

Транзакционная модель Spring гарантирует соблюдение принципов **ACID** (`Атомарность`, `Согласованность`, `Изоляция`, `Долговечность`) на уровне бизнес-логики.

## 🌟 Что происходит при вызове метода с `@Transactional`?
🔽 Механизм работы по шагам 🔽
```TEXT
[Внешний вызов] 
		│ 
		▼ 
┌───────────────────────────────────────────────────┐ 
│ AOP Proxy │ (Перехватывает управление)            │
└───────┬───────────────────────────────────────────┘ 
		│ 
		▼ 
┌───────────────────────────────────────────────────┐ 
│ TransactionInterceptor │ ──► [TransactionManager] │
│(Проверяет propagation, открывает DB Conn)         │
└───────┬───────────────────────────────────────────┘ 
		│ 
		▼ 
┌───────────────────────────────────────────────────┐ 
│ Целевой метод бина │ ──► [EntityManager Proxy]    │
│(Берет текущую сессию из ThreadLocal)              │
└───────────────────────────────────────────────────┘
```
1. **Перехват вызова через прокси**: 
* Внешний клиент вызывает метод бина. Запрос попадает в созданный Spring **AOP Proxy** объект. 
* `TransactionInterceptor` перехватывает управление и извлекает настройки аннотации `@Transactional`. 
  
2. **Работа TransactionManager**: 
* Интерцептор делегирует управление `PlatformTransactionManager`. 
* Менеджер проверяет, выполняется ли транзакция в текущем потоке (`ThreadLocal`) и сверяется с параметром `propagation`. 
* Если нужно открыть новую транзакцию: из пула запрашивается физическое соединение с БД, у него выставляется `autoCommit = false`, создается `EntityManager`/`Session` и привязывается к `ThreadLocal`. 

3. **Выполнение бизнес-логики и работа EntityManager proxy**: 
* Вызывается реальный метод сервиса. 
* Когда слой DAO обращается к `EntityManager`, фактически вызов идет к **EntityManager proxy** (обычно `SharedEntityManagerInvocationHandler`). Этот прокси на лету достает из `ThreadLocal` активную сессию текущего потока и выполняет SQL-команды. 
  
4. **Завершение и очистка**: 
* **Успешное завершение**: `TransactionInterceptor` перехватывает возврат из метода и дает команду менеджеру закоммитить изменения (`connection.commit()`). 
* **Исключение**: Если метод выбросил `RuntimeException` или `Error`, инициируется откат (`connection.rollback()`). 
* Ресурсы (`ThreadLocal`, `Session`) очищаются, а соединение возвращается в пул.

## 🎯 Физические vs Логические транзакции
🔽 В чем разница в контексте Spring 🔽

* **Физическая транзакция**: Реальное транзакционное соединение на уровне драйвера базы данных (JDBC `Connection`, операции `begin`/`commit`/`rollback`). 
  
* **Логическая транзакция**: Границы методов, помеченных аннотацией `@Transactional`. Несколько вложенных логических транзакций могут выполняться в рамках одной физической. 
  
> **Пример с `Propagation.REQUIRES_NEW`**: Если метод с `REQUIRES_NEW` вызывается из другого транзакционного метода, внешняя *логическая* транзакция приостанавливается, и под нее создается вторая отдельная *физическая* транзакция (открывается второе физическое соединение с БД).

## ⚙️ Атрибуты аннотации `@Transactional`
🔽 propagation, isolation, rollbackFor, noRollbackFor, readOnly, timeout 🔽

| Атрибут         | Описание                                                                              | Значение по умолчанию                        |
| :-------------- | :------------------------------------------------------------------------------------ | :------------------------------------------- |
| `propagation`   | Настройка распространения (поведение при наличии или отсутствии активной транзакции). | `Propagation.REQUIRED`                       |
| `isolation`     | Уровень изолированности транзакции в БД.                                              | `Isolation.DEFAULT` (Берется из настроек БД) |
| `rollbackFor`   | Массив классов исключений, при которых транзакция **обязана** откатиться.             | `RuntimeException.class`, `Error.class`      |
| `noRollbackFor` | Массив классов исключений, которые **не должны** приводить к откату.                  | Пусто (`{}`)                                 |
| `readOnly`      | Флаг "Только для чтения". Оптимизирует работу JPA (отключает dirty checking).         | `false`                                      |
| `timeout`       | Максимальное время выполнения транзакции в секундах до принудительного отката.        | -1 (Таймаут подсистемы БД)                   |

## 🚧 Типы распространения транзакций (`Propagation`)
🔽 Сводная таблица поведения 🔽

| Propagation           | Поведение, <BR>если транзакция ЕСТЬ                                | Поведение, <BR>если транзакции НЕТ                          |
| :-------------------- | :----------------------------------------------------------------- | :---------------------------------------------------------- |
| `REQUIRED` <BR>✅      | Выполняется в рамках текущей транзакции.                           | Создает новую транзакцию.                                   |
| `REQUIRES_NEW` <br>✨  | Приостанавливает текущую, создает **новую физическую** транзакцию. | Создает новую транзакцию.                                   |
| `NESTED` <BR>📍       | Создает `Savepoint` внутри текущей физической транзакции.          | Создает новую транзакцию.                                   |
| `MANDATORY` <BR>🚨    | Выполняется в рамках текущей транзакции.                           | Выбрасывает `IllegalTransactionStateException`.             |
| `SUPPORTS` <BR>🔄     | Выполняется в рамках текущей транзакции.                           | Выполняется без транзакции (в не-транзакционном контексте). |
| `NOT_SUPPORTED` <BR>❌ | Приостанавливает текущую, выполняется без транзакции.              | Выполняется без транзакции.                                 |
| `NEVER` <BR>⛔️        | Выбрасывает `IllegalTransactionStateException`.                    | Выполняется без транзакции.                                 |

### Важные нюансы применения: 
* `REQUIRED` — выбор по умолчанию для 95% задач. 
* `REQUIRES_NEW` — незаменим для изолированных операций, которые должны сохраниться в любом случае (например, аудит, логирование ошибок, списание комиссии). 
* `NESTED` — требует поддержки точек сохранения (`Savepoints`) на уровне драйвера БД и диалекта (работает в PostgreSQL, Oracle; не поддерживается во многих NoSQL). Позволяет откатить только вложенный кусок логики, не прерывая внешнюю транзакцию.

## 🔒 Уровни изоляции (`Isolation`)
🔽 Управление аномалиями данных 🔽

| Уровень              | Описание                                                                        | Защита от аномалий                       |
| :------------------- | :------------------------------------------------------------------------------ | :--------------------------------------- |
| **DEFAULT**          | Берет настройки подключенной БД (в PostgreSQL/Oracle это `READ_COMMITTED`).     | Зависит от СУБД                          |
| **READ_UNCOMMITTED** | Допускает чтение незафиксированных изменений ("грязное чтение"). Самый быстрый. | Нет защиты                               |
| **READ_COMMITTED**   | Запрещает чтение "грязных" данных. Возможны неповторяющиеся чтения.             | Грязное чтение ❌                         |
| **REPEATABLE_READ**  | Гарантирует, что повторный запрос вернет те же строки. Возможны фантомы.        | Неповторяющееся чтение ❌                 |
| **SERIALIZABLE**     | Полная имитация последовательного выполнения. Исключает любые аномалии.         | Все аномалии ❌ (Высокий риск блокировок) |

## 🚧 Особенности вызовов методов внутри класса (Self-call)
🔽 Влияние архитектуры AOP Proxy на внутренние вызовы 🔽

| Сценарий вызова в рамках ОДНОГО класса | Результат | Причина |
| :--- | :--- | :--- |
| Метод с `@Transactional` ──► Метод с `@Transactional(propagation = REQUIRES_NEW)` | ⚠️ **Новая транзакция НЕ создается**, используется текущая. | Вызов идет через ключевое слово `this`, минуя прокси-обертку. Настройки `propagation` игнорируются. |
| Метод БЕЗ `@Transactional` ──► Метод с `@Transactional` | ⚠️ **Транзакция НЕ создается вообще**. | Внешний вызов пришел на не-транзакционный метод. Внутренний вызов прошел в обход прокси. |
| Вызов между РАЗНЫМИ классами (`@Transactional` ──► `@Transactional`) | ✅ **Всё работает корректно** согласно настройкам `propagation`. | Вызов идет через ссылку на другой бин, управление честно перехватывается прокси-объектом. |

### Способы обхода ограничения (Решения):
1. **Вынесение логики**: Перенести вложенный метод в другой Spring-бин (лучшее архитектурное решение).
2. **Self-injection (Внедрение самого себя)**: Внедрить ссылку на собственный бин через контекст (обязательно с аннотацией `@Lazy`, чтобы избежать циклической зависимости `Circular Dependency`).

```java
@Service
public class OrderService {

    @Autowired
    @Lazy  // Ленивая инициализация прокси для предотвращения циклической зависимости
    private OrderService self; 

    public void processOrder() {
        // Вызов через self принудительно направляет запрос через AOP Proxy
        self.updateInventory(); 
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void updateInventory() {
        // Эта логика гарантированно выполнится в новой физической транзакции
    }
}
```

## ⚠️ Критические подводные камни и частые ошибки

### 1. Проверяемые исключения (Checked Exceptions) и неожиданный коммит
По умолчанию Spring откатывает транзакции **только при возникновении `RuntimeException` и `Error`**. Если твой метод выбрасывает проверяемое исключение (например, `IOException`, `SQLException` или твой кастомный бизнес-эксепшн, наследуемый напрямую от `Exception`), транзакция **успешно закоммитится**, несмотря на ошибку!

- **Решение**: Всегда явно указывай класс верхнего уровня в атрибуте: `@Transactional(rollbackFor = Exception.class)`.

### 2. Использование аннотации на не-public методах
Spring AOP прокси по умолчанию перехватывает методы только с модификатором доступа `public`. Если повесить `@Transactional` на `protected`, `private` или package-private метод, компилятор не выдаст ошибку, но транзакционная логика будет полностью проигнорирована.

### 3. Ошибка `LazyInitializationException` вне транзакции
Попытка обратиться к полям с ленивой загрузкой (`FetchType.LAZY`) после того, как метод с `@Transactional` завершил работу, приведет к падению приложения. Контекст транзакции уже закрыт, и сессия Hibernate уничтожена.

- **Решение**: Загружать коллекции до закрытия транзакции (через `JOIN FETCH` / `EntityGraph`) или держать транзакцию открытой на протяжении всей цепочки обработки данных.

### 4. Неправильный выбор TransactionManager в гетерогенных системах
Если в приложении одновременно используется чистый JDBC/MyBatis и JPA/Hibernate, дефолтный `DataSourceTransactionManager` не сможет синхронизировать кэш первого уровня Hibernate.

- **Решение**: При использовании JPA/Hibernate в конфигурации всегда должен быть объявлен и использоваться `JpaTransactionManager`. Он умеет корректно управлять нативным JDBC-соединением «под капотом».

### 5. Блокировки ресурсов из-за длинных транзакций
Включение в транзакционный метод долгих операций (запросы к сторонним внешним API, тяжелая генерация файлов, отправка писем) держит физическое соединение с базой данных занятым. Это быстро приводит к исчерпанию пула соединений (`Connection Pool Starvation`).

- **Решение**: В транзакции должен находиться только минимально необходимый блок кода, работающий непосредственно с БД. Внешние интеграции следует выносить за рамки `@Transactional`.

## ⚙️ Настройка конфигурации менеджера транзакций
В современных приложениях на **Spring Boot** управление транзакциями (`@EnableTransactionManagement`) и автоконфигурация менеджера транзакций включены **автоматически**, если в `classpath` обнаружены стартеры `spring-data-*` или `spring-tx`.

В случае ручной настройки (чистый Spring) конфигурация выглядит так:
```Java
@Configuration
@EnableTransactionManagement // Включает обработку аспектов @Transactional
public class TransactionConfig {

    @Bean
    public PlatformTransactionManager transactionManager(
	    EntityManagerFactory emf) {
        // Идеальный выбор для проектов с JPA / Hibernate
        return new JpaTransactionManager(emf);
    }
}
```
