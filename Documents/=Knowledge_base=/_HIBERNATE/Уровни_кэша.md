###  Hibernate: Уровни кэша (*супер-кратко для собеса*)

**1. L1 (*First-Level Cache*)**
- **Где:** Внутри **сессии (`Session`)**.    
- **Включен:** **Всегда**, отключить нельзя.    
- **Суть:** Хранит сущности по `ID`. Две одинаковые `get()` в одной сессии = **один SQL** (второй раз берет из кэша).    
- **Живет:** Пока жива сессия (обычно до конца транзакции).    

**2. L2 (*Second-Level Cache*)**
- **Где:** Внутри **фабрики сессий (`SessionFactory`)**.    
- **Включен:** **Нет**, нужно настраивать (EhCache, Hazelcast, Redis).    
- **Суть:** Хранит сущности по `ID`. Доступен **для всех сессий** приложения (общий).    
- **Стратегии:** `READ_ONLY`, `READ_WRITE` (самый частый), `NONSTRICT_READ_WRITE`.    

**3. Query Cache (*Кэш запросов*)**
- **Где:** Отдельно от L2, но тоже на уровне `SessionFactory`.    
- **Включен:** **Нет**, нужно включать отдельно (`use_query_cache: true`).    
- **Суть:** Кэширует не сущности, а **списки ID**, полученные в результате `JPQL`/`SQL` запроса. Работает только **в паре с L2**.    
- **Нюанс:** Если хоть одна сущность изменилась — весь кэш запроса слетает.

---
### 🎯 Итоговая таблица (*для зубрежки*)

| Уровень         | Привязка                          | Вкл. по умолч. | Хранит           | Стратегии              |
| --------------- | --------------------------------- | -------------- | ---------------- | ---------------------- |
| **L1**          | `Session` <br>(Одна сессия)       | ✅ **Да**       | Сущности по `ID` | Нет                    |
| **L2**          | `SessionFactory` <br>(Все сессии) | ❌ **Нет**      | Сущности по `ID` | `READ_WRITE` <br>и др. |
| **Query Cache** | `SessionFactory`                  | ❌ **Нет**      | `ID` из запросов | Зависит <br>от L2      |

---
### 💡 "Золотое правило" для собеса:

> _"L1 спасает от дублей в рамках одного запроса. L2 спасает от дублей между разными запросами/пользователями. Query Cache спасает от повторного выполнения тяжелых JPQL-запросов, но хранит только ID, поэтому его бесполезно включать без L2."_
