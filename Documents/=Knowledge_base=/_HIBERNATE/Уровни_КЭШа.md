###  Hibernate: Уровни кэша (*супер-кратко для собеса*)

> [!IMPORTANT]
> КРАТКО
### **1. L1 (*First-Level Cache*)**
*Обязательный, привязан к сессии (`Session`), работает всегда, нельзя отключить.*
- **Где:** Внутри **сессии (`Session`)**.    
- **Включен:** **Всегда**, отключить нельзя.    
- **Суть:** Хранит сущности по `ID`. 
  Две одинаковые `get()` в одной сессии = **один SQL** (*второй раз берет из кэша*).    
- **Живет:** Пока жива сессия (*обычно до конца транзакции*).    

### **2. L2 (*Second-Level Cache*)**
*Опциональный, привязан к `SessionFactory`, разделяется между всеми сессиями, настраивается.*
- **Где:** Внутри **фабрики сессий (`SessionFactory`)**.    
- **Включен:** **Нет**, нужно настраивать (*EhCache, Hazelcast, Redis*).    
- **Суть:** Хранит сущности по `ID`. 
  Доступен **для всех сессий** приложения (*общий*).    
- **Стратегии:** `READ_ONLY`, `READ_WRITE` (*самый частый*), `NONSTRICT_READ_WRITE`.    

### **3. Query Cache (*Кэш запросов*)**
*Отдельный механизм, кэширует результаты запросов (обычно исп-ся вместе с L2).*
- **Где:** Отдельно от L2, но тоже на уровне `SessionFactory`.    
- **Включен:** **Нет**, нужно включать отдельно (`use_query_cache: true`).    
- **Суть:** Кэширует не сущности, а **списки ID**, 
  полученные в результате `JPQL`/`SQL` запроса. Работает только **в паре с L2**.    
- **Нюанс:** Если хоть одна сущность изменилась — весь кэш запроса слетает.

---
### 🎯 Итоговая таблица (*для зубрежки*)

| Уровень         | Привязка                            | Вкл. по умолч. | Хранит           | Стратегии              |
| --------------- | ----------------------------------- | -------------- | ---------------- | ---------------------- |
| **L1**          | `Session` <br>(*Одна сессия*)       | ✅ **Да**       | Сущности по `ID` | Нет                    |
| **L2**          | `SessionFactory` <br>(*Все сессии*) | ❌ **Нет**      | Сущности по `ID` | `READ_WRITE` <br>и др. |
| **Query Cache** | `SessionFactory`                    | ❌ **Нет**      | `ID` из запросов | Зависит <br>от L2      |

---
### 💡 "Золотое правило" для собеса:

> _- L1 спасает от дублей в рамках одного запроса. 
> - L2 спасает от дублей между разными запросами/пользователями. 
> - Query Cache спасает от повторного выполнения тяжелых JPQL-запросов, но хранит только ID, поэтому его бесполезно включать без L2._


## ⚠️ Подводные камни (*для собеседования*)

### ❌ Проблема 1: **Грязное чтение (*Stale Data*)**
Если один сервер обновляет данные через L2 кэш, а другой сервер (с другой JVM) обновляет их напрямую в БД — кэш может **устареть**.  
**Решение:** Использовать распределённый кэш (Redis) или ограничить время жизни кэша (`timeToLive`).

### ❌ Проблема 2: **L2 кэш + коллекции = проблема**
Если в сущности есть коллекция (`@OneToMany`, `@ManyToMany`), и ты кэшируешь родителя — *Hibernate* **не кэширует коллекции** по умолчанию. Нужно использовать `@Cache` на коллекции отдельно.

### ❌ Проблема 3: **Кэш запросов + L2 кэш = лишняя нагрузка**
*Query Cache* хранит только **ID** сущностей, а сами сущности подгружаются из **L2** кэша. Если **L2** выключен — **Query Cache** работает, но каждая сущность всё равно грузится из БД по `ID` (*что может быть даже хуже, чем без кэша*).

## 📊 Детальный разбор

### 1️⃣ **First-Level Cache (L1) — Кэш уровня сессии**

|Свойство|Описание|
|---|---|
|**Привязка**|К объекту `Session` (одна сессия = один кэш)|
|**Включен**|**Всегда**, нельзя отключить|
|**Время жизни**|Пока жива `Session` (обычно — одна транзакция/запрос)|
|**Что хранит**|Загруженные сущности **по ID** (первичному ключу)|
|**Кто видит**|Только текущая сессия (другие сессии не видят)|
|**Сброс**|`session.evict(entity)`, `session.clear()`, `session.close()`|

**Как это работает:**
```java
// Первый запрос — загрузка из БД
User user1 = session.get(User.class, 1L); // SELECT в БД
// Второй запрос — данные берутся из L1 кэша (SQL НЕ отправляется!)
User user2 = session.get(User.class, 1L); // БЕЗ SELECT!
```

**Важно:** L1 кэш работает **только в пределах одной сессии**. Если ты закроешь сессию и откроешь новую — данные будут загружены заново.

---
### 2️⃣ **Second-Level Cache (L2) — Кэш уровня фабрики сессий**

|Свойство|Описание|
|---|---|
|**Привязка**|К `SessionFactory` (одна фабрика = один кэш)|
|**Включен**|**Опционально**, требует настройки (по умолчанию **выключен**)|
|**Время жизни**|Пока жива `SessionFactory` (обычно — всё время работы приложения)|
|**Что хранит**|Сущности **по ID** (кеширует целые объекты или их поля)|
|**Кто видит**|**Все сессии** фабрики (общий кэш)|
|**Стратегии**|`READ_ONLY`, `NONSTRICT_READ_WRITE`, `READ_WRITE`, `TRANSACTIONAL`|

#### **Как настроить:**

**Шаг 1:** Добавить зависимость (например, EhCache):
```xml
<dependency>
    <groupId>org.hibernate.orm</groupId>
    <artifactId>hibernate-ehcache</artifactId>
</dependency>
```

**Шаг 2:** Включить L2 кэш в `application.yml`:
```yaml
spring:
  jpa:
    properties:
      hibernate:
        cache:
          use_second_level_cache: true
          region:
            factory_class: org.hibernate.cache.jcache.JCacheRegionFactory
        javax:
          cache:
            provider: org.ehcache.jsr107.EhcacheCachingProvider
```

**Шаг 3:** Аннотировать сущность:
```java
@Entity
@Cacheable // Включает кэширование для этой сущности
@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class User {
    @Id
    private Long id;
    // ...
}
```

#### **Стратегии кэширования L2:**

| Стратегия                  | Описание                                                        | Когда использовать                   |
| -------------------------- | --------------------------------------------------------------- | ------------------------------------ |
| **`READ_ONLY`**            | Только чтение, никогда не обновляется                           | Справочники, статические данные      |
| **`NONSTRICT_READ_WRITE`** | Обновления допускаются, но возможна небольшая несогласованность | Редко меняющиеся данные              |
| **`READ_WRITE`**           | Строгая согласованность (использует мягкие блокировки)          | **Стандарт для большинства случаев** |
| **`TRANSACTIONAL`**        | Полная поддержка транзакций (для JTA)                           | Распределенные транзакции            |

---
### 3️⃣ **Query Cache — Кэш запросов**

|Свойство|Описание|
|---|---|
|**Привязка**|К `SessionFactory`, но **отдельный** от L2|
|**Что хранит**|**Результаты запросов** (не сами сущности)|
|**Зависит от L2**|**Да!** Если включен Query Cache, но не включен L2 — запросы кэшируются, но сущности всё равно грузятся по ID из БД|
|**Инвалидация**|При изменении любой сущности, затронутой запросом|

**Как включить:**
```yaml
spring:
  jpa:
    properties:
      hibernate:
        cache:
          use_query_cache: true
```

**Использование в коде:**
```java
// Первый запрос — выполняется в БД
List<User> users1 = session.createQuery("FROM User WHERE age > 18", User.class)
    .setCacheable(true)  // 🎯 Включаем кэширование для этого запроса
    .list();
    
// Второй запрос — берется из кэша запросов (SQL НЕ отправляется!)
List<User> users2 = session.createQuery("FROM User WHERE age > 18", User.class)
    .setCacheable(true)
    .list();
```
