Если в классической теории SQL (ANSI/ISO) описаны 4 уровня изоляции, то **Oracle** подходит к этому вопросу максимально прагматично и из коробки поддерживает **всего 3 уровня изоляции** (*из которых активно используются фактически **два***).

Давай разберем, как устроен этот механизм в Oracle, почему там нет режима `READ UNCOMMITTED` и как он ведет себя под нагрузкой.

---
### 1️⃣ `READ COMMITTED` (*По умолчанию*)

Это дефолтный уровень изоляции для всех сессий в Oracle.
	
- **Как работает:** Каждая инструкция (`SELECT`, `UPDATE` и т.д.) внутри транзакции видит **только те данные, которые были зафиксированы (`COMMIT`) ДО начала выполнения этой конкретной инструкции**.
    
- **Особенность Oracle:** Oracle никогда не заставляет читающие транзакции ждать пишущие! Благодаря архитектуре мультиверсионности (MVCC) и сегментам отката (`Undo Tablespace`), `SELECT` читает «старую» закомиченную версию данных из Undo. Писатели не блокируют читателей, а читатели не блокируют писателей.
    
- **Защита:** Защищает от «Грязного чтения» (Dirty Read).
    
- **Аномалии:** Возможны «Неповторяемое чтение» (Non-Repeatable Read) и «Фантомное чтение» (Phantom Read). Если внутри одной транзакции выполнить один и тот же `SELECT` дважды, а между ними другая сессия сделает `COMMIT`, данные в ответах будут разными.

---
### 2️⃣ `SERIALIZABLE`

Максимально строгий уровень, обеспечивающий полную изоляцию транзакции.
	
- **Как работает:** Вся транзакция в целом видит снимок данных (Snapshot), сделанный **на момент старта самой транзакции**, а не отдельного SQL-запроса. Все запросы внутри одной транзакции видят абсолютно одинаковую картину мира.
    
- **Защита:** Защищает от _грязного_, _неповторяемого_ и _фантомного_ чтения.
    
- **Опасность (*Ошибки ORA-08177*):** Если транзакция в режиме `SERIALIZABLE` попытается обновить (`UPDATE`/`DELETE`) строку, которую с момента начала этой транзакции успела изменить и закомитить другая параллельная транзакция, Oracle выбросит исключение:
`ORA-08177: can't serialize access for this transaction`

Приложение должно быть готово перехватить эту ошибку и повторить всю транзакцию заново.

---
### 3️⃣ `READ ONLY`

Специфичный для Oracle режим, предназначенный для формирования сложных аналитических отчетов.
	
- **Как работает:** Абсолютно аналогичен `SERIALIZABLE` с той лишь разницей, что внутри этой транзакции **запрещены любые изменения данных** (`INSERT`, `UPDATE`, `DELETE` вызовут ошибку).
    
- **Зачем нужен:** Позволяет длинному отчету (например, выписке за год) выполнять десятки `SELECT`-запросов к разным таблицам и гарантированно видеть согласованные данные на момент запуска отчета, абсолютно не мешая другим пользователям обновлять базу.

---
## ❓ А где же `READ UNCOMMITTED` (Грязное чтение)?

В Oracle БД уровня **`READ UNCOMMITTED` физически не существует**.

Если ты попытаешься выполнить команду:
```SQL
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
```
Oracle выдаст синтаксическую ошибку.

**Почему его нет?**

В базах вроде `MS SQL Server` грязное чтение придумали для производительности — чтобы `SELECT` не ждал блокировок от `UPDATE`. Но в Oracle благодаря блокам `Undo` читатели **и так никогда не ждут блокировок писателей**. В грязном чтении в Oracle просто нет никакого технического смысла — база и без него отдает согласованные закомиченные данные на высокой скорости!

### 💡 Резюме для собеседования

Если на интервью спросят про уровни изоляции в Oracle:

> _«В отличие от стандарта ANSI SQL, Oracle поддерживает всего три режима: `READ COMMITTED` (по умолчанию), `SERIALIZABLE` и кастомный `READ ONLY` для отчетов. Режима `READ UNCOMMITTED` в Oracle нет, так как благодаря архитекторе Undo/MVCC чтение закомиченных данных происходит без блокировок строк»._
