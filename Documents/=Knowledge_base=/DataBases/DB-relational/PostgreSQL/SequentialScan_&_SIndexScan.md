В PostgreSQL существует несколько способов прочитать данные из таблицы. Выбор метода (скана) — это главное решение, которое принимает оптимизатор, оценивая стоимость каждой операции.

---
### 1. Sequential Scan (Seq Scan) <BR>— Полное сканирование таблицы

**Что это:** Чтение всех страниц (блоков) таблицы последовательно **от начала до конца**, одной за другой. Для каждой строки проверяется условие `WHERE`.

**Когда используется:**
- Когда в таблице мало строк (например, < 1000) — дешевле прочитать все, чем тратить время на поиск по индексу.
    
- Когда условие `WHERE` неселективно (должно вернуть > 5-10% строк таблицы) — индекс создаст слишком много случайных обращений к диску, а последовательное чтение быстрее.
    
- Когда нет подходящего индекса для условия.

**Пример:**
```sql
EXPLAIN SELECT * FROM users WHERE age > 18;
-- Если 80% пользователей старше 18, оптимизатор выберет Seq Scan.
-- Результат: Seq Scan on users (cost=0.00..1500.00 rows=80000 width=100)
```

- **Плюсы:** Отлично работает на HDD (*минимальное движение головки*), хорошо читает большие объемы (*полная таблица*).  
- **Минусы:** Читает всю таблицу, даже если нужно всего 2 строки. Очень медленно для больших таблиц при точечных запросах.

---
### 2. Index Scan <BR>— Сканирование по индексу

**Что это:** Сначала находится нужная запись в `B-Tree` индексе (*быстрый поиск, O(log N)*), затем, получив физический адрес строки (*TID: номер страницы и позицию в ней*), *PostgreSQL* идет в таблицу (*Heap*) и вычитывает саму строку, чтобы проверить видимость (*MVCC*) и получить остальные столбцы.

**Когда используется:** Когда условие `WHERE` очень селективно (возвращает **мало** строк, например, **< 5%** таблицы) и есть подходящий индекс.

**Пример:**
```sql
EXPLAIN SELECT * FROM users WHERE id = 12345;
-- Если id - PRIMARY KEY.
-- Результат: Index Scan using users_pkey on users (cost=0.29..8.30 rows=1 width=100)
```

- **Плюсы:** Очень быстрый для точечных запросов, задействует только нужные строки.  
- **Минусы:** При большом количестве возвращаемых строк становится катастрофой, так как каждое обращение к таблице — это случайный I/O (особенно медленно на HDD). Это называется **"Random I/O Problem"**.

---
### 3. Index-Only Scan <BR>— Только индекс (*Самый **быстрый**)*

**Что это:** Особый случай Index Scan. Если все столбцы, которые нужны в `SELECT`, уже есть внутри индекса (благодаря `INCLUDE` или простому составному индексу), PostgreSQL **вообще не идет в таблицу**. Видимость строк проверяется по специальной Visibility Map (VM).

**Когда используется:** Когда создан "покрывающий индекс" (Covering Index) и страницы таблицы помечены как полностью видимые (all-visible).

**Пример:**
```sql
-- Создаем покрывающий индекс
CREATE INDEX users_cover_idx ON users (id) INCLUDE (name, email);
-- Запрос
EXPLAIN SELECT id, name, email FROM users WHERE id = 12345;
-- Результат: Index-Only Scan using users_cover_idx on users (cost=0.29..4.30 rows=1)
-- Heap Fetches: 0 (это идеал!)
```

**Плюсы:** Максимальная скорость, так как читается только компактный индекс, таблица не трогается вообще.

---
### 4. Bitmap Scan <BR>— Комбинированный (*для нескольких индексов*)

**Что это:** Гибридный метод. Он используется, когда нужно выбрать много строк, но условия используют несколько индексов или один индекс, возвращающий много записей.

**Как работает:**

1. Сканирует индекс(ы) и создает в памяти битовую карту (Bitmap) всех страниц (блоков), которые содержат подходящие строки.
    
2. Сортирует эту карту по номеру страницы (убирая дубликаты).
    
3. Затем идет в таблицу (Heap) и читает **только эти страницы** последовательно (один раз каждую), извлекая все подходящие строки.

**Пример:**
```sql
-- Есть два отдельных индекса: на age и на city
EXPLAIN SELECT * FROM users WHERE age = 25 OR city = 'Moscow';
-- Результат: 
-- Bitmap Heap Scan on users
--   -> BitmapOr (или BitmapAnd)
--       -> Index Scan using idx_age (возвращает много TID)
--       -> Index Scan using idx_city (возвращает много TID)
```

**Плюсы:** Решает проблему случайного I/O (упорядочивает чтение страниц), позволяет комбинировать несколько индексов.  
**Минусы:** Все равно читает всю страницу целиком (если на странице 100 строк, а подходит 1, он прочитает все 100).

---
### 5. Tid Scan <BR>— Сканирование по физическому адресу

**Что это:** Самый прямой и быстрый доступ. PostgreSQL читает строки напрямую по их физическому адресу в таблице (`ctid`).

**Когда используется:** Обычно неявно, внутри подзапросов или при работе с `IN` из индекса. Реже используется вручную.

**Пример:**

sql

EXPLAIN SELECT * FROM users WHERE ctid = '(10, 5)';
-- Результат: Tid Scan on users (cost=0.00..4.01 rows=1)

**Плюсы:** Мгновенное чтение, минуя индекс.

---
### 6. Sample Scan <BR>— Сканирование выборки (TABLESAMPLE)

**Что это:** Используется не для поиска, а для получения статистической выборки. Читает заданный процент строк (или заданное количество страниц) случайным или систематическим образом.

**Пример:**
```sql
SELECT * FROM users TABLESAMPLE SYSTEM(10); -- Читает 10% случайных страниц
```

**Плюсы:** Сверхбыстрый для получения приблизительных статистических данных.

### 📊 Сводная таблица сравнения

| Тип скана           | Где читает данные              | Скорость                         | Когда выбирается                                                          |
| ------------------- | ------------------------------ | -------------------------------- | ------------------------------------------------------------------------- |
| **Seq Scan**        | Вся таблица                    | Низкая (для больших таблиц)      | Нет индекса или условие неселективно (>10% строк)                         |
| **Index Scan**      | Индекс <BR>+ таблица           | Средняя (зависит от Random I/O)  | Условие селективно (<5% строк)                                            |
| **Index-Only Scan** | Только индекс                  | **Максимальная**                 | Индекс покрывает все столбцы в `SELECT`                                   |
| **Bitmap Scan**     | Индекс(ы) + сортировка страниц | Хорошая (минимальный Random I/O) | Используется несколько индексов или индекс возвращает много строк (5-30%) |
| **Tid Scan**        | Таблица по адресу              | Мгновенная                       | Доступ по системному столбцу `ctid`                                       |

---
### 🎯 Практический совет для продакшена

Если вы видите в `EXPLAIN ANALYZE` медленный `Seq Scan` на большой таблице — создайте индекс. Если видите `Index Scan` с огромными цифрами `Heap Fetches` (при включенном `track_io_timing`) — добавьте `INCLUDE` или перепишите запрос на покрывающий индекс, чтобы превратить его в `Index-Only Scan`. Это дает прирост скорости в 2-5 раз без изменения логики.
