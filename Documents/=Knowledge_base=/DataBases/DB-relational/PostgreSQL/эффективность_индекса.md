**ВОПРОС**: какими свойствами должна обладать таблица, чтобы индекс был максимально эффективен *(большой размер таблицы + больше операций чтения)* 

Чтобы индекс был максимально эффективен для **большой таблицы** с **преобладанием операций чтения** (OLAP или тяжелый OLTP), он должен обеспечивать **максимальную селективность**, **минимальную стоимость доступа** и **покрытие запроса**.

Ниже — структурированный чек-лист свойств таблицы и данных, которые делают индекс сверхэффективным.

---
### 1. Высокая **кардинальность** столбца (***Уникальность***)

Индекс эффективен, когда условие `WHERE` отсекает **малую долю строк** (высокая селективность). Для столбцов с низкой кардинальностью (например, `status` с 3 значениями) индекс бесполезен — оптимизатор выберет Seq Scan.
	
- **Требование:** Столбец должен содержать множество уникальных значений (или хотя бы десятки тысяч различных вариантов).
    
- **Пример:** Индекс на `user_id` (уникальный) отличный. Индекс на `gender` (2 значения) — бесполезен для чтения.

---
### 2. Сильная корреляция с **физическим порядком** (***Clustering***)

Это самое важное свойство для чтения. **Корреляция** (значение в `pg_stats.correlation`) показывает, насколько порядок значений в столбце совпадает с физическим порядком строк в таблице (ctid).
	
- **Корреляция = 1.0:** Значения идут строго **по порядку**. *Index Scan* будет читать страницы последовательно, почти как Seq Scan. Это **очень быстро**.
    
- **Корреляция ~ 0:** Значения разбросаны **хаотично**. *Index Scan* будет прыгать по всей таблице (Random I/O), читая одну страницу за раз. Это **медленно** (в 100–1000 раз медленнее для HDD).

**Как добиться:** Периодически выполнять `CLUSTER table_name USING index_name;` (пересортировывает таблицу по индексу). Либо использовать `BRIN`-индексы для данных, которые физически сортируются по времени (например, `created_at`).

---
### 3. "Покрывающий индекс" (Covering Index) — INCLUDE

Чтобы индекс был максимально эффективен для чтения, он должен полностью "***покрывать***" запрос. Это значит, что все необходимые столбцы есть **внутри самого индекса**, и <u>не нужно обращаться к таблице (Heap) вообще</u>. Это называется **Index-Only Scan**.
	
- **Свойство:** Используйте синтаксис `CREATE INDEX ... INCLUDE (col2, col3)`.
    
- **Пример:**
```sql
-- Плохо: SELECT name, email FROM users WHERE id = 5; (нужен поход в таблицу)
-- Хорошо (покрывающий индекс):
CREATE INDEX users_read_idx ON users (id) INCLUDE (name, email);
SELECT name, email FROM users WHERE id = 5; -- Только индекс, таблица не читается!
```

---
### 4. "Разреженность" индекса (BRIN для монотонных данных)

Если таблица огромная (миллиарды строк) и данные вставляются во временном порядке (например, `created_at`), то **B-Tree** индекс будет слишком большим (займет сотни ГБ). Для чтения эффективнее использовать **BRIN** (Block Range Index).
	
- **Свойство:** Индекс хранит не каждую строку, а мин/макс значение для каждого блока страниц (обычно 128 страниц).
    
- **Условие:** Данные должны быть сильно коррелированы с `ctid`.
    
- **Пример:** `CREATE INDEX logs_time_idx ON logs USING BRIN (created_at);` — индекс будет в 1000 раз меньше B-Tree, а скорость поиска по диапазону будет почти такой же высокой.

---
### 5. Минимальная **ширина** строки (*Узкие индексы*)

Чем меньше байт занимает индексная запись, тем больше записей помещается на одну страницу памяти (буферный кэш). Это ускоряет сканирование в разы.
	
- **Свойство:** Индексируйте `INTEGER` (4 байта) или `BIGINT` (8 байт), но избегайте `TEXT` или `VARCHAR(255)`, если только это не покрывающий индекс.
    
- **Исключение:** Для текста используйте `hash`-индекс или частичный индекс (`WHERE`), если это возможно.

---
### 6. *Частичные* индексы (***Partial Index***) для часто фильтруемых данных

Если вы часто ищете только по одному статусу (например, `WHERE status = 'active'`), индекс по всей таблице будет содержать много "мертвых" записей. Создайте индекс только для нужных строк.
	
- **Свойство:** `WHERE` в определении индекса.
    
- **Пример:** `CREATE INDEX users_active_idx ON users (last_login) WHERE status = 'active';` Такой индекс будет маленьким и сверхбыстрым для активных пользователей.

---
### 📊 Сводная таблица свойств для максимальной эффективности

|Свойство таблицы/данных|Почему это важно|Как проверить/добиться|
|---|---|---|
|**Высокая селективность** (уникальность)|Индекс используется (Seq Scan не выбирается)|`COUNT(DISTINCT col) / COUNT(*)` > 0.1|
|**Высокая корреляция** (~0.9 – 1.0)|Index Scan читает данные последовательно (быстрый I/O)|`SELECT correlation FROM pg_stats WHERE tablename='table';`|
|**Покрытие запроса (INCLUDE)**|Доступ только к индексу (Index-Only Scan)|Анализировать запросы `EXPLAIN` на наличие "Heap Fetches: 0"|
|**Монотонный рост (для BRIN)**|Индекс сверхмал (разреженный)|Данные вставляются с возрастающим `id` или `created_at`|
|**Узкий тип данных**|Больше записей в кэше|Использовать `INT` вместо `TEXT`, `UUID` вместо `VARCHAR`|
|**Частичный индекс (`WHERE`)**|Индекс не содержит "мусорные" строки|Если 90% строк `deleted=True`, индекс только по `deleted=False`|

---

### 🚀 Идеальный пример для чтения

Представьте таблицу `orders` с 1 млрд записей. 90% запросов — `SELECT order_id, total, created_at FROM orders WHERE customer_id = ? ORDER BY created_at DESC LIMIT 10;`

**Идеальный индекс:**
```sql
CREATE INDEX orders_perfect_idx ON orders (customer_id, created_at DESC) 
INCLUDE (order_id, total);
```

**Почему он идеален:**
1. `customer_id` — высокая селективность.
2. `created_at DESC` + `customer_id` — идеальная сортировка (Composite).
3. `INCLUDE` — покрывает все поля запроса (Index-Only Scan, таблица не трогается).
4. За счет `customer_id` в начале ключа, строки для одного клиента физически сгруппированы (если делать `CLUSTER`), что дает 100% корреляцию.

---

**Главный вывод для чтения:**  
Ваша цель — превратить любой `SELECT` в **Index-Only Scan** (только индекс, без таблицы). Для этого индексы должны быть "узкими" (мало столбцов в ключе), "широкими" (много столбцов в `INCLUDE`) и "сортированными" (высокая корреляция с `ctid`).
