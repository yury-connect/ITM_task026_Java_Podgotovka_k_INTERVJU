# 🔒 Оптимистическая блокировка в PostgreSQL

---
### 📌 Определение
**Оптимистическая блокировка** — это **не встроенный механизм СУБД**, а **паттерн проектирования**, реализуемый на уровне приложения. Она предполагает, что конфликты между транзакциями случаются редко, поэтому блокировки строк **не захватываются**, а проверка версии данных выполняется **перед сохранением**.

---
### ⚙️ Как работает (Схема)
```text
1. Чтение данных + запоминание версии (version или xmin)
2. ... время на размышления / бизнес-логика ...
3. Обновление с проверкой версии в WHERE
4. Если rows_affected == 0 → конфликт → повтор (retry)
```

---
### 📝 Пример реализации

**Чтение:**
```sql
SELECT id, name, price, version FROM products WHERE id = 1;
-- version = 5
```

**Обновление:**
```sql
UPDATE products
SET price = 12.00, version = version + 1
WHERE id = 1 AND version = 5;
-- Если UPDATE затронул 0 строк → кто-то нас опередил!
```

---
### 🆚 Отличие от пессимистической (ключевое!)

| Характеристика                 | Оптимистическая               | Пессимистическая <br>(`FOR UPDATE`) |
| ------------------------------ | ----------------------------- | ----------------------------------- |
| **Блокировка строки**          | ❌ Нет (строка не блокируется) | ✅ Да (строка блокируется сразу)     |
| **Когда проверяется конфликт** | При сохранении (в конце)      | При чтении (в начале)               |
| **Реализация**                 | `version` + `WHERE`           | `SELECT ... FOR UPDATE`             |
| **Где живет логика**           | В коде приложения             | В СУБД                              |

---
### 🧠 Где хранить версию?

|Вариант|Плюсы|Минусы|
|---|---|---|
|**Явное поле `version`**|Контролируемо, понятно|Нужно добавлять колонку|
|**Системное поле `xmin`**|Не нужно менять схему|Неудобно для отладки, зависит от MVCC|

---
### 🎯 Когда использовать?

#### ✅ **Да, если:**
- Много чтения, мало записи.
- Конфликты редки (*например, правка профиля*).
- Нужна высокая производительность (н*ет накладных расходов на блокировки*).
- Долгая бизнес-логика между чтением и записью (*например, корзина покупок*).

#### ❌ **Нет, если:**
- Высокая конкуренция за одну строку (*например, остатки товара на складе*).
- Конфликты происходят часто → много откатов → падает производительность.
- Операция должна гарантированно завершиться с первого раза.

---
### 💡 Реализация в **Spring / Hibernate**
```java
@Entity
public class Product {

    @Version
    private Long version; // Hibernate сам проверяет версию
}
```

*Hibernate* автоматически добавляет `version` в `WHERE` при обновлении и бросает `OptimisticLockException` при конфликте.

### 📊 Сводная таблица

|Критерий|Оптимистическая блокировка|
|---|---|
|**Суть**|"Конфликтов нет, буду проверять в конце"|
|**Использует `FOR UPDATE`?**|❌ Нет|
|**Блокирует строки?**|❌ Нет|
|**Где реализована**|В приложении (версионирование)|
|**Аномалии при конфликте**|Ошибка + повтор транзакции|
|**Популярна в**|Веб-приложениях, микросервисах|
|**Гарантирует консистентность?**|Да, при правильной реализации|

---
### 🔥 Золотая фраза для собеседования

> _"Оптимистическая блокировка в PostgreSQL — это не отдельный режим, а паттерн. Мы не блокируем строки, а добавляем поле `version` и при обновлении проверяем, что версия не изменилась. Если затронуто 0 строк — кто-то нас опередил, и мы повторяем транзакцию."_
