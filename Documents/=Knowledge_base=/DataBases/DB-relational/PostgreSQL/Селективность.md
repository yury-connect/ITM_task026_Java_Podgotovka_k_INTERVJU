**Селективность** (*Selectivity*) — это одна из **ключевых** метрик, которую использует оптимизатор PostgreSQL (и других СУБД) для оценки стоимости выполнения запроса и выбора плана.

Простыми словами: **Селективность — это доля строк в таблице, которые, по оценке оптимизатора, вернет условие `WHERE` (или другое условие соединения).**

Значение селективности лежит в диапазоне от **0.0 до 1.0**:
- **0.0** — условие не вернет ни одной строки (***очень хорошо** для индексов*).
- **1.0** — условие вернет все строки таблицы (*индекс **бесполезен**, нужен `Seq Scan`*).

---
### 1. Как селективность влияет на план запроса?

Оптимизатор умножает **Селективность** на **Общее количество строк в таблице (RelTuples)**, чтобы получить **Кардинальность (Cardinality)** — оценку количества возвращаемых строк.
	
- **Высокая селективность (*низкое значение, например, `0.001`*):** Оптимизатор думает, что вернется <u>мало строк</u> (*1 из 1000*). Скорее всего, он выберет **Index Scan** (*поиск по индексу*), так как это **быстро**.
    
- **Низкая селективность (*высокое значение, например, `0.8`*):** Оптимизатор думает, что вернется <u>много строк</u> (*800 из 1000*). Он выберет **Sequential Scan** (*полное сканирование таблицы*), так как читать всю таблицу последовательно дешевле, чем скакать по индексу туда-сюда.

---
### 2. Как *PostgreSQL* оценивает селективность?

У *PostgreSQL* нет "*идеального знания*" данных. Он строит оценки на основе **статистики**, собранной командой `ANALYZE` (*или автоматическим процессом `autovacuum`*). Эта статистика хранится в системном каталоге `pg_statistic` (*и в удобочитаемом виде в `pg_stats`*).

Основные источники для оценки:

|Источник|Для чего используется|Где хранится|
|---|---|---|
|**NULL-фракция**|Доля строк со значением `NULL` в столбце.|`pg_stats.null_frac`|
|**Гистограмма**|Распределение значений для **неравенств** (`>`, `<`, `BETWEEN`).|`pg_stats.histogram_bounds`|
|**Самые частые значения (MCV)**|Список значений, которые встречаются чаще всего, и их частоты. Используется для **равенства** (`=`).|`pg_stats.most_common_vals` и `pg_stats.most_common_freqs`|
|**Корреляция**|Связь между физическим порядком строк (ctid) и значением столбца. Влияет на стоимость Index Scan.|`pg_stats.correlation`|

### 3. Примеры вычисления селективности

Допустим, у нас есть таблица `users` с `100 000` строк.

#### Пример 1: Равенство (`WHERE id = 12345`)
- Это первичный ключ (`PRIMARY KEY`). Статистика говорит, что уникальных значений столько же, сколько строк.
- Селективность = `1 / количество уникальных значений` ≈ `1 / 100000` = **0.00001**.
- Кардинальность = `100000 * 0.00001` ≈ **1 строка**. Будет использован `Index Scan`.

#### Пример 2: Частое значение (MCV) для равенства
- `WHERE status = 'active'`.
- Статистика показывает, что 80% строк имеют статус 'active'.
- `pg_stats.most_common_freqs` для значения 'active' = **0.8**.
- Селективность = **0.8** (берется прямо из частоты).
- Кардинальность = `100000 * 0.8` = **80 000 строк**. Будет использован `Seq Scan`.

#### Пример 3: Неравенство (`WHERE salary > 50000`)
- `WHERE salary > 50000`.
- Оптимизатор смотрит в гистограмму (*которая разбивает диапазон зарплат на корзины*). Он видит, что значение 50 000 находится в 3-й корзине из 10.
- Приблизительная оценка доли строк справа = `1 - (позиция в гистограмме)` = `1 - 0.3` = **0.7**.
- Селективность ≈ **0.7**.
- Кардинальность = `100000 * 0.7` = **70 000 строк**.

---
### 4. Что такое "комбинированная *селективность" (для нескольких условий*)?

Если у вас `WHERE a = 1 AND b = 2`, оптимизатор перемножает селективности (*при условии независимости столбцов, хотя это и не всегда так*):
	
- Селективность для `a = 1` = 0.1 (10% строк).
    
- Селективность для `b = 2` = 0.2 (20% строк).
    
- Итоговая селективность = `0.1 * 0.2` = **0.02** (2% строк).

### 5. Почему оценки бывают ошибочными? (*Проблемы*)

|Проблема|Почему возникает|Последствия|
|---|---|---|
|**Устаревшая статистика**|Не запускался `ANALYZE` после массовых вставок/обновлений.|Оптимизатор думает, что в таблице 100 строк, а там уже 1 млн. План будет ужасным.|
|**Корреляция данных**|Значения в столбце физически отсортированы так же, как и в индексе.|`Index Scan` будет стоить дешевле, чем оценивает стандартная формула.|
|**Независимость столбцов**|PostgreSQL считает, что значения в разных столбцах не связаны.|Если у вас 50% мужчин и 50% программистов, оптимизатор ожидает 25% программистов-мужчин. Но если все программисты — мужчины, реальная селективность будет 50%, а не 25%, и план может ошибиться. (Для этого есть расширение `CREATE STATISTICS`).|

---
### 📊 Итоговая памятка
- **Селективность** (от 0 до 1) — это **оценка** доли строк, которую вернет условие.
- **Высокая селективность** (число близко к 0) → **мало строк** → хорошо для **Index Scan**.
- **Низкая селективность** (число близко к 1) → **много строк** → лучше **Seq Scan**.
- **Источник данных** для оценки — статистика из `pg_stats`, собранная `ANALYZE`.
- **Главный враг** — устаревшая статистика и корреляция столбцов (*для сложных условий*).
