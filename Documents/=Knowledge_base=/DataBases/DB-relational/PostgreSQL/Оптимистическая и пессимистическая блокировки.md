**Оптимистическая** и **пессимистическая** блокировки — это не типы блокировок PostgreSQL (как `FOR UPDATE`), а **стратегии управления конфликтами**, которые определяют, _когда_ и _как_ мы проверяем целостность данных.

Давайте разберем их структурированно, с примерами, а затем покажу, как это связано с реальными типами блокировок в PostgreSQL.

---
### 1. **Пессимистическая** блокировка (*Pessimistic Locking*)

**Философия:** "Конфликты неизбежны. Я лучше заблокирую данные сейчас, пока не закончу с ними, чтобы никто не мог их изменить."

**Как работает:** Транзакция захватывает блокировку на строку (или таблицу) в момент чтения и держит её до своего завершения (`COMMIT` или `ROLLBACK`). Другие транзакции, пытающиеся изменить эту строку, будут заблокированы (ждать) или получат ошибку.

**В PostgreSQL реализуется через:** `SELECT ... FOR UPDATE` (эксклюзивная блокировка) или `SELECT ... FOR SHARE` (разделяемая блокировка).

**Пример (бронирование билета):**
```sql
BEGIN;
-- Захватываем строку в монопольное владение
SELECT * FROM tickets WHERE id = 123 AND status = 'available' FOR UPDATE;
-- Проверяем статус (другие транзакции сейчас не могут его изменить)
UPDATE tickets SET status = 'booked', booked_by = 'user_42' WHERE id = 123;
COMMIT; -- Блокировка снимается
```

**Плюсы:** Гарантирует консистентность при любом уровне конкуренции. Простота логики (*нет проверок версий*).  
**Минусы:**
	
- Снижает производительность (*транзакции ждут друг друга*).
    
- Риск **взаимоблокировок (*Deadlock*)**, когда две транзакции ждут друг друга.
    
- Плохо масштабируется при высокой нагрузке на запись.

**Когда использовать:** Высокая конкуренция за одни и те же данные (например, остатки товара на складе, когда 1000 человек покупают последний iPhone). Либо когда операция требует строгой последовательности.

### 2. **Оптимистическая** блокировка (*Optimistic Locking*)

**Философия:** "Конфликты случаются редко. Я не буду блокировать данные, а просто проверю перед сохранением, не изменил ли их кто-то другой."

**Как работает:** Транзакция читает данные без блокировок. Перед сохранением изменений она проверяет, не изменилась ли версия данных с момента чтения. Если изменилась — транзакция откатывается и повторяется.

**В PostgreSQL реализуется:**

- Через прикладной механизм (версионирование).
    
- Через MVCC (PostgreSQL сам использует эту философию для `REPEATABLE READ` и `SERIALIZABLE`).
    

**Самый частый паттерн реализации (через версию):**  
Добавляем в таблицу столбец `version`.

sql

CREATE TABLE users (
    id INT PRIMARY KEY,
    name TEXT,
    balance INT,
    version INT DEFAULT 1
);

**Пример (перевод денег):**

sql

-- Транзакция 1 начинает
BEGIN;
-- Читаем данные с версией
SELECT balance, version FROM users WHERE id = 1; -- version = 5
-- ... делаем расчеты (в это время Транзакция 2 могла изменить данные)
-- Пытаемся обновить, но проверяем, что version не изменился
UPDATE users 
SET balance = 900, version = version + 1 
WHERE id = 1 AND version = 5;
-- Если UPDATE вернул 0 строк (никто не обновился) -> значит, данные изменились!
-- Делаем ROLLBACK и повторяем всю транзакцию заново.
COMMIT;

**Более короткий путь в PostgreSQL:** Использовать уровень изоляции `SERIALIZABLE`. PostgreSQL сам отследит конфликты и выдаст ошибку `serialization_failure`.

**Плюсы:** Высокая производительность при низкой конкуренции (нет ожиданий блокировок). Хорошо масштабируется горизонтально.  
**Минусы:**

- При высокой конкуренции приводит к частым откатам (транзакции повторяются).
    
- Требует написания логики повтора (retry-цикла) в коде приложения.
    
- Не гарантирует успеха с первого раза.
    

**Когда использовать:** Низкая вероятность конфликтов (например, правка профиля пользователя), чтение преобладает над записью, или в распределенных системах (микросервисы).

### 3. Сравнительная таблица

|Критерий|Пессимистическая блокировка|Оптимистическая блокировка|
|---|---|---|
|**Главная идея**|"Лучше перестраховаться и заблокировать"|"Будем решать проблемы по факту"|
|**Когда захватывается блокировка**|При чтении (`SELECT FOR UPDATE`)|При сохранении (`UPDATE ... WHERE version = ?`)|
|**Время удержания блокировки**|От `SELECT` до `COMMIT` (долго)|Только на время выполнения `UPDATE` (мгновенно)|
|**Риск конфликтов**|Управляется блокировками (транзакции ждут)|Управляется откатами (транзакции перезапускаются)|
|**Производительность**|Падает при высокой нагрузке (ожидания)|Падает при высокой нагрузке (откаты)|
|**Подходит для**|Высокая конкуренция (бронирование, остатки)|Низкая конкуренция (редактирование профиля)|
|**Реализация в PostgreSQL**|`FOR UPDATE`, `FOR SHARE`|`version` столбец или `SERIALIZABLE` уровень|

### 4. Связь с типами блокировок PostgreSQL

Теперь, когда мы разобрали стратегии, важно понять, что **типы блокировок PostgreSQL** (мы обсуждали их ранее) — это инструменты для реализации **пессимистического** подхода.

- `FOR UPDATE` — эксклюзивная блокировка строки (пессимистичный подход).
    
- `FOR SHARE` — разделяемая блокировка (другие читать могут, менять — нет).
    

Оптимистический подход **не использует** эти команды. Он полагается на MVCC и сравнение версий на уровне приложения.

### 5. Жизненный пример: Система бронирования отелей

Представьте, что 2 человека пытаются забронировать последний номер в отеле.

**Пессимистичный подход (через `FOR UPDATE`):**

- Клиент А: `SELECT * FROM rooms WHERE id=10 FOR UPDATE;` (заблокировал строку).
    
- Клиент Б: пытается сделать то же самое, но **ждет**, пока А не завершит транзакцию.
    
- Клиент А: `UPDATE rooms SET booked=true ...; COMMIT;`
    
- Клиент Б: просыпается, видит, что номер занят, и получает ошибку.
    

**Оптимистичный подход (через `version`):**

- Клиент А и Б читают номер одновременно (без блокировок): `version = 1`.
    
- Клиент А: `UPDATE rooms SET booked=true, version=2 WHERE id=10 AND version=1;` — успешно (1 строка обновлена).
    
- Клиент Б: `UPDATE rooms SET booked=true, version=2 WHERE id=10 AND version=1;` — обновляет **0 строк** (так как version уже стал 2). Клиент Б получает ошибку и перезапускает транзакцию.

### 🏆 Итог: что и когда выбирать

1. **Если ваша система — это банк, билеты или остатки товара** (высокая конкуренция, критично к целостности) — используйте **пессимистические** блокировки (`FOR UPDATE`). Это надежнее и проще в логике.
    
2. **Если ваша система — социальная сеть, каталог товаров или длительные бизнес-процессы** (низкая конкуренция, много чтения) — используйте **оптимистические** блокировки (версионирование или `SERIALIZABLE`). Это дает лучший перформанс.
    
3. **Если вы пишете микросервис и не можете использовать транзакции БД** — только оптимистический подход через версии и паттерн Saga.
    

В реальных проектах часто используют гибридный подход: на основные "горячие" записи (остатки) ставят `FOR UPDATE`, а на остальные — версионирование.
