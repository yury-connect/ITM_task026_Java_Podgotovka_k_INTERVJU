Покрывающий индекс (Covering Index) - одна из самых мощных оптимизаций для операций чтения в PostgreSQL.

## 1. Определение

**Покрывающий индекс** — это индекс, который содержит **все столбцы**, необходимые для выполнения запроса, прямо внутри своей структуры. Благодаря этому PostgreSQL может вернуть данные, **вообще не обращаясь к таблице** (Heap), что дает колоссальный выигрыш в производительности.

Такой доступ называется **Index-Only Scan**.

---
## 2. Как это работает (внутренняя механика)

#### Обычный **`Index Scan`**:
	
1. Находим запись в B-Tree индексе.
    
2. Получаем из неё `TID` (физический адрес строки: номер страницы + позиция).
    
3. Идем в таблицу (Heap) по этому адресу, читаем страницу в память.
    
4. Извлекаем оттуда нужные столбцы.
    
5. **Проверяем видимость строки** (MVCC) — видна ли она нашей транзакции.

#### **Покрывающий индекс (Index-Only Scan):**
	
1. Находим запись в B-Tree индексе.
    
2. **Все нужные столбцы уже есть в этой же записи индекса.**
    
3. Проверяем видимость строки по **Visibility Map** (отдельная bitmap-структура, которая говорит, что все строки на данной странице видны всем транзакциям).
    
4. Если страница помечена как "all-visible" — **возвращаем данные сразу, не трогая таблицу**.

**Ключевой нюанс:** Если страница **не** помечена как all-visible (есть недавние изменения), PostgreSQL всё равно может сделать Index-Only Scan, но ему придется проверить видимость по таблице (это будут `Heap Fetches`). Хорошо, когда `Heap Fetches: 0`.

---
## 3. Синтаксис создания (INCLUDE)

В PostgreSQL покрывающий индекс создается с помощью ключевого слова **`INCLUDE`** (появилось с версии 11).
```sql
CREATE INDEX index_name ON table_name (key_columns) INCLUDE (include_columns);
```

**Правила:**
- **`key_columns`** — столбцы, по которым строится B-Tree (по ним идет поиск и сортировка).
    
- **`include_columns`** — столбцы, которые просто хранятся в листьях индекса, но **не участвуют** в поиске и не влияют на уникальность.

---
## 4. Пример: До и После

Допустим, у нас есть таблица `orders` с миллионом записей.
```sql
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    customer_id INT NOT NULL,
    total DECIMAL(10,2),
    created_at TIMESTAMP,
    status TEXT
);
```

### Пример запроса (*очень частый в приложении*):
```sql
SELECT id, total, status 
FROM orders 
WHERE customer_id = 12345;
```

### ❌ Плохой вариант (без покрывающего индекса):
```sql
CREATE INDEX orders_customer_idx ON orders (customer_id);
```

**Что произойдет:** Index Scan найдет 50 записей, затем для **каждой** из них пойдет в таблицу, чтобы прочитать `total` и `status`. Это 50 случайных обращений к диску (Random I/O). При 1000 запросов в секунду — диск умрет.

### ✅ Хороший вариант (покрывающий индекс):
```sql
CREATE INDEX orders_customer_covering_idx 
ON orders (customer_id) 
INCLUDE (total, status);
```

**Что произойдет:** Index-Only Scan найдет те же 50 записей в индексе, но все данные (`id`, `total`, `status`) уже там. **Обращений к таблице нет.**

---
## 5. Почему покрывающий индекс быстрее?

|Характеристика|Обычный Index Scan|Index-Only Scan (покрывающий)|
|---|---|---|
|**Чтение индекса**|Да|Да|
|**Чтение таблицы (Heap)**|Да (для каждой строки)|**Нет** (если страница all-visible)|
|**Случайный I/O**|Есть (много)|**Нет**|
|**Размер индекса**|Меньше (только ключи)|Больше (ключи + INCLUDE)|
|**Скорость**|Базовая|**В 2–10 раз быстрее**|

---
## 6. Важные нюансы и ограничения

### 6.1. Столбцы в `INCLUDE` не участвуют в поиске
Если вы напишете `WHERE total > 100` и поместите `total` в `INCLUDE`, индекс **не поможет**, так как поиск идет только по ключевым столбцам. Для фильтрации `total` его нужно добавить в ключевую часть (не в `INCLUDE`).

### 6.2. Размер индекса
Покрывающий индекс физически больше обычного, так как он хранит копии данных. Чем больше столбцов в `INCLUDE`, тем больше места на диске и в памяти (буферный кэш). Нужно соблюдать баланс.

### 6.3. Видимость (Visibility Map)
Если таблица часто обновляется, страницы могут не быть "all-visible". Тогда `Index-Only Scan` всё равно будет ходить в таблицу для проверки видимости. Но это всё равно дешевле, чем полный Index Scan, потому что читается только `visibility map` (очень маленькая структура).

### 6.4. Обновление индекса при изменении данных
При каждом `UPDATE` или `INSERT` PostgreSQL должен обновлять не только ключевые столбцы, но и все столбцы в `INCLUDE`. Это замедляет запись. Поэтому покрывающие индексы лучше всего работают в системах с **преобладанием чтения** (OLAP / отчеты) или для стабильных, редко обновляемых данных.

---
## 7. Когда использовать покрывающий индекс?

#### **Используйте, если:**
	
- Есть частый, "тяжелый" запрос с фиксированным набором столбцов.
    
- Таблица большая (чтобы Random I/O было критично).
    
- Данные редко обновляются (чтобы не платить цену за поддержку индекса).
    
- Вы точно знаете, что запрос будет выполняться сотни/тысячи раз в секунду.

#### **Не используйте, если:**
	
- Запрос возвращает все столбцы таблицы (`SELECT *`).
    
- Таблица маленькая (Seq Scan будет быстрее).
    
- Данные очень часто обновляются (оверхед на запись перевесит выгоду).
    
- Индекс становится слишком большим и не помещается в память.

---
## 8. Диагностика — как проверить

Чтобы убедиться, что индекс работает правильно, используйте `EXPLAIN (ANALYZE, BUFFERS)`:
```sql
EXPLAIN (ANALYZE, BUFFERS) 
SELECT id, total, status FROM orders WHERE customer_id = 12345;
```

**Идеальный результат:**
```text
Index-Only Scan using orders_customer_covering_idx on orders
  (cost=0.42..8.44 rows=50 width=42)
  Heap Fetches: 0   <-- ВОТ ЭТО КЛЮЧЕВОЙ ПОКАЗАТЕЛЬ!
  Buffers: shared hit=3   <-- Данные взяты только из кэша (индекс)
```

Если `Heap Fetches` больше нуля, это означает, что PostgreSQL не смог проверить видимость по `visibility map` и ходил в таблицу. В таком случае стоит выполнить `VACUUM` для таблицы, чтобы обновить карту видимости.

## 📊 Итог: Сравнение стратегий

|Критерий|Seq Scan|Index Scan|**Index-Only Scan**|
|---|---|---|---|
|**Читает таблицу?**|Да (всю)|Да (выборочно)|**Нет**|
|**Читает индекс?**|Нет|Да|Да|
|**Скорость**|Низкая для больших таблиц|Средняя|**Максимальная**|
|**Стоимость записи**|Низкая|Средняя|**Высокая** (индекс больше)|
|**Идеально для**|Маленьких таблиц|Селективных запросов|**Частых, точных SELECT**|

#### **Золотое правило:**
	
1. Сначала напишите логику.
    
2. Найдите самые медленные запросы через `pg_stat_statements`.
    
3. Посмотрите их `EXPLAIN`.
    
4. Если видите `Index Scan` с большим числом `Heap Blocks`, создайте покрывающий индекс через `INCLUDE`. Это дает прирост в производительности до 5-10 раз без изменения кода приложения.
