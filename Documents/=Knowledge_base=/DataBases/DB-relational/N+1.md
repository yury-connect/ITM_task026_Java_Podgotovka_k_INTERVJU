# 📦 Проблема **N+1** в JPA/Hibernate

---
## 🎯 Что такое N+1

**Когда:** Загружаете 1 родительскую сущность + делаете N отдельных запросов к БД для загрузки каждого связанного ребёнка.

**Пример:**
```java
// 1 запрос → получаем 10 пользователей
List<User> users = userRepository.findAll();
// 10 запросов → для КАЖДОГО пользователя идём в БД за заказами
for (User user : users) {
    user.getOrders();  // N отдельных запросов!
}
```

**Итого:** `1 + N` запросов вместо **одного**.

---
## ✅ Как решить **N+1** 

### 1️⃣ **JOIN FETCH** (рекомендуется)
```java
@Query("SELECT u FROM User u JOIN FETCH u.orders")
List<User> findAllWithOrders();
```

```java
@Query("SELECT u FROM User u JOIN FETCH u.orders o")
List<User> findAllWithOrders();
```
**Плюсы:** Один запрос, всё загружается сразу.  
**Минусы:** Для больших данных может быть проблема с памятью.

---
### 2️⃣ **@EntityGraph**
```java
@EntityGraph(attributePaths = "orders")
@Query("SELECT u FROM User u")
List<User> findAllWithOrders();
```

```java
@EntityGraph(attributePaths = {"orders", "profile"})
User findById(Long id);
```
**Плюсы:** Гибкость, можно задавать несколько полей.  
**Минусы:** Нужно контролировать, чтобы не перегружать память.

---
### 3️⃣ **Batch Fetching**
```yaml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
```

```java
@BatchSize(size = 100)
@OneToMany(mappedBy = "user")
private List<Order> orders;
```
**Как работает:** Загружает связанные сущности пачками (не по одному). Вместо `N` запросов → `N / batch_size` запросов.

**Плюсы:** Минимизирует количество запросов без изменения кода.  
**Минусы:** Не решает полностью проблему, только уменьшает.

---
### 4️⃣ **DTO + Join**
```java
@Query("""
       SELECT new com.example.UserOrderDto(u.id, u.name, o.id, o.total)
       FROM User u
       JOIN u.orders o
       """)
List<UserOrderDto> findAllWithOrders();
```
**Плюсы:** Только нужные данные, один запрос.  
**Минусы:** Жёсткая привязка к DTO.

---
### 5️⃣ **Hibernate.initialize()** (костыль)
```java
User user = userRepository.findById(id);
Hibernate.initialize(user.getOrders());
```
**Плюсы:** Простой.  
**Минусы:** Всё равно делает второй запрос (но контролируемый).

---
## 📊 Сравнение способов

| Способ                | Запросов    | Память     | Сложность    |
| --------------------- | ----------- | ---------- | ------------ |
| **Без решения (N+1)** | `1 + N`     | 🔴 Высокая | ❌ Плохо      |
| **JOIN FETCH**        | `1`         | 🔴 Высокая | ✅ Просто     |
| **@EntityGraph**      | `1`         | 🔴 Высокая | ✅ Просто     |
| **Batch Fetching**    | `1 + N/100` | 🟡 Средняя | 🟢 Прозрачно |
| **DTO**               | `1`         | 🟢 Низкая  | 🟡 Средне    |

---
## 🎯 Как выбрать

|Сценарий|Решение|
|---|---|
|**Много данных, нужны все поля**|`JOIN FETCH` или `@EntityGraph`|
|**Много данных, только часть полей**|**DTO + Join**|
|**Оптимизация без переписывания запросов**|**Batch Fetching**|
|**Одна сущность с несколькими коллекциями**|`@EntityGraph`|
|**Чтобы не загружать лишнего**|**DTO**|

---
## ✅ Итог одной фразой

> **N+1 — это когда грузишь родителей, а потом дёргаешь БД за каждого ребёнка.**
> 
> **Решается через `JOIN FETCH`, `@EntityGraph`, `Batch Fetching` или DTO.**
