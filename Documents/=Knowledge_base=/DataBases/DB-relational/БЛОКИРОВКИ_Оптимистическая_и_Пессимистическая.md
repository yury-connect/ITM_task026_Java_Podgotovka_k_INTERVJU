Оптимистическая блокировка применяется в высококонкурентных системах с преобладанием чтений. Она не блокирует строки на уровне СУБД. В JPA реализуется аннотацией `@Version` над целочисленным полем сущности. На уровне чистого SQL при обновлении Spring Data генерирует запрос: `UPDATE cargo SET status = :status, version = :version + 1 WHERE id = :id AND version = :current_version`. Если параллельная транзакция успела изменить строку и инкрементировать версию, СУБД вернёт 0 изменённых строк, и приложение выбросит `OptimisticLockException`. Пессимистическая блокировка применяется, когда конфликты часты и цена ошибки высока (например, списание баланса или резерв ячейки логистики). На уровне СУБД она удерживает эксклюзивные замки. `PESSIMISTIC_READ` транслируется в SQL как `SELECT ... FOR SHARE`, позволяя другим только читать строку, но запрещая модификацию. `PESSIMISTIC_WRITE` транслируется в `SELECT ... FOR UPDATE`, полностью блокируя строку как для параллельного чтения с блокировкой, так и для любой модификации до завершения текущей транзакции.
