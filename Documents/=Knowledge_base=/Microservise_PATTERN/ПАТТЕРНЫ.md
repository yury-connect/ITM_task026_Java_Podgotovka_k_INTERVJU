## 1. Категория: <br>**Декомпозиция** (*Проектирование сервисов*)
_Паттерны о том, как правильно нарезать монолит на независимые кусочки._

1. **Decomposition by Bounded Context (по ограниченным контекстам):** Разделение сервисов строго по границам предметных областей (DDD). _Самый важный паттерн для старта архитектуры._
    
2. **Decomposition by Business Capability (по бизнес-возможностям):** Нарезка сервисов на основе организационной структуры и того, что бизнес делает (например, «Управление заказами», «Доставка»).
    
3. **Strangler Fig / Strangler (Душитель):** Постепенная миграция с монолита на микросервисы путем «выкусывания» функционала и замены его микросервисами за фасадом. _Главный паттерн миграции._

---
## 2. Категория: <br>**Интеграция и Доступ** (*API*)

_Паттерны о том, как клиенты взаимодействуют с микросервисами._

1. **API Gateway (Шлюз API):** Единая точка входа для всех внешних клиентов. Берет на себя маршрутизацию, аутентификацию, логирование и лимитирование запросов.
    
2. **BFF (Backend for Frontend):** Создание отдельных API-шлюзов под разные типы клиентов (свой шлюз для мобилок, свой — для Web, свой — для сторонних интеграций).
    
3. **Service Mesh (Сеть сервисов):** Инфраструктурный слой (например, Istio) для управления коммуникацией _внутри_ кластера между самими микросервисами (Service-to-Service).

---
## 3. Категория: <br>**Управление данными** (*Data Management*)

_Паттерны решения главной проблемы микросервисов — распределенных данных._

1. **[Database per Service](_Database_per_Service) (База данных на сервис):** У каждого микросервиса своя личная БД. Никто другой не имеет права лазить туда напрямую. _Базовое правило микросервисов._
    
2. **Saga (Сага):** Реализация распределенных транзакций через цепочку локальных транзакций. Бывает оркестрованная (через центральный сервис) и хореографическая (через события).
    
3. **CQRS (Разделение ответственности на чтение и запись):** Разделение приложения на две разные базы/модели: одна оптимизирована только для записи (OLTP), другая — только для быстрого чтения (OLAP).
    
4. **Event Sourcing (Источник событий):** Состояние сущности не перезаписывается, а хранится как последовательность неизменяемых событий (логов) с возможностью их «перемотки».
    
5. **Shared Database (Общая БД):** Антипаттерн (или паттерн переходного периода), когда несколько сервисов ходят в одну БД. Сильно связывает сервисы.

## 4. Категория: <br>**Наблюдаемость** (*Observability*)

_Паттерны, помогающие понять, что прямо сейчас ломается в продакшене._

1. **Distributed Tracing (Распределенная трассировка):** Сквозной мониторинг пути запроса через все микросервисы с помощью уникального `Trace ID` (инструменты вроде Zipkin, Jaeger).
    
2. **Log Aggregation (Агрегация логов):** Сбор логов со всех контейнеров и сервисов в одно централизованное хранилище (стек ELK/PLG) для удобного поиска.
    
3. **Health Check API:** Каждый сервис обязан иметь эндпоинт (например, `/health`), по которому оркестратор (Kubernetes) понимает, живое приложение или его пора перезапустить.

---
## 5. Категория: <br>**Надежность** и **Отказоустойчивость** (Resilience)

_Паттерны, защищающие систему от эффекта домино, когда один упавший сервис роняет всё остальное._

1. **Circuit Breaker (Предохранитель):** Логический выключатель. Если вызываемый сервис начинает сыпать ошибками, предохранитель «размыкается» и временно перестает слать туда запросы, отдавая дефолтный ответ (fallback).
    
2. **Retry (Повторный вызов):** Автоматический повтор запроса при кратковременных сетевых сбоях (желательно с экспоненциальной задержкой — Exponential Backoff).
    
3. **Bulkhead (Перегородки):** Изоляция ресурсов (например, пулов потоков) под разные задачи. Если забьется пул потоков для отправки писем, это не должно заблокировать обработку платежей.

---
## 6. Категория: <br>**Кросс-сервисное взаимодействие** (Communication)

_Паттерны организации общения между узлами._

1. **Messaging / Asynchronous Communication:** Асинхронное общение через брокеры сообщений (Kafka, RabbitMQ) для максимальной отвязки (decoupling) сервисов друг от друга.
    
2. **Remote Procedure Invocation (RPI):** Синхронное общение «запрос-ответ» (gRPC, REST), когда сервису критически важно получить ответ здесь и сейчас.
    
3. **Transactional Outbox:** Потокобезопасная отправка событий. Сервис пишет бизнес-данные и событие в свою БД в рамках _одной_ локальной транзакции, а специальный релей (Debezium) уже пересылает событие в Kafka.
