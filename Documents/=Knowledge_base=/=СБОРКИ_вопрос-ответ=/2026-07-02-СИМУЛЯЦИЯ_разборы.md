## **1. `equals()` и `hashCode()`**

**Полный ответ:**
> Контракт equals() и hashCode() определён в Object. Главное правило: **если два объекта равны по equals(), то их hashCode() должны возвращать одинаковое значение**. Обратное неверно.

> **Правила equals()** (должны соблюдаться):
> 
> - **Рефлексивность**: `x.equals(x) == true`
> - **Симметричность**: `x.equals(y) == y.equals(x)`
> - **Транзитивность**: если `x.equals(y)` и `y.equals(z)`, то `x.equals(z)`
> - **Consistency**: вызовы возвращают один результат (при неизменном состоянии объектов)
> - `x.equals(null) == false`

> При переопределении equals() **обязательно** переопределяем hashCode(). Иначе объекты, которые равны, могут попадать в разные buckets в HashMap/HashSet, что нарушит работу коллекции.

**Современный способ**:
```Java
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof User user)) return false;
    return id == user.id && Objects.equals(email, user.email);
}

@Override
public int hashCode() {
    return Objects.hash(id, email);
}
```

> **Важные нюансы**:
> 
> - В JPA/Hibernate: equals()/hashCode() лучше делать **только по бизнес-ключу** (natural id), а не по ID базы (генерируемому).
> - Records в Java 16+ генерируют корректно автоматически.

**Follow-up вопросы, которые любят спрашивать:**

- Что будет, если не переопределить hashCode() при переопределённом equals()?
- Можно ли использовать == вместо equals() для объектов?

### **2. String `pool` и `intern()**`

**Глубокий ответ:**

> String pool (String Constant Pool) — область в Heap (с Java 7+), где хранятся строковые литералы и строки, добавленные через intern().

> Механизм:
> 
> - Все строковые литералы ("hello") автоматически попадают в пул.
> - `new String("hello")` создаёт объект в обычном heap, но ссылается на пул.
> - `String.intern()` — если строка уже есть в пуле, возвращает ссылку на неё, иначе добавляет и возвращает.

> **Почему это важно:**
> 
> - Экономия памяти.
> - Возможность использовать `==` для сравнения вместо `equals()` (быстрее), но **только** для строк из пула.

> **Проблемы:**
> 
> - До *Java 7* пул находился в PermGen → OutOfMemory.
> - Слишком частый `intern()` на больших объёмах может привести к заполнению пула.

### **3. `final`, `finally`, `finalize()**`

> - **final** — модификатор:
>     - переменная (примитив/ссылка) — нельзя изменить значение/переприсвоить.
>     - метод — нельзя переопределить.
>     - класс — нельзя наследовать (String, LocalDateTime и др.).

> - **finally** — блок, выполняющийся **почти всегда** (кроме System.exit(), падения JVM, смерти потока).

> - **finalize()** — устаревший (deprecated). Вызывается GC **перед** сборкой. Проблемы: нет гарантий вызова, задержки GC, исключения в finalize() игнорируются. Замена — Cleaner (Java 9+) или try-with-resources.

### **4. Records (Java 16+)**

> Records — это **nominal tuples** (именованные кортежи), предназначенные для хранения неизменяемых данных.

> Автоматически генерируется:
> 
> - Compact canonical constructor
> - Private final fields
> - Accessor methods (name(), age() вместо getName())
> - equals(), hashCode(), toString()
> - Component methods

> Можно:
> 
> - Добавлять static поля/методы
> - Дополнительные конструкторы
> - Реализовывать интерфейсы

> Нельзя:
> 
> - Наследоваться
> - Иметь mutable поля
> - Переопределять accessor'ы (можно, но не рекомендуется)

### **5. `Sealed Classes` (Java 17)**

> Позволяют **явно** ограничивать иерархию наследования.

```Java
public sealed interface Shape permits Circle, Rectangle, Square { }
public final class Circle implements Shape { ... }
```

> Преимущества:
> 
> - Exhaustive pattern matching в switch (Java 21+)
> - Лучшая безопасность и моделирование домена
> - Компилятор знает все возможные подтипы

### **6. Virtual Threads (Project Loom)**

> Это **революция** в concurrency-модели Java.

> Отличия от Platform Threads:
> 
> - Легковесные (несколько КБ вместо 1+ МБ)
> - Управляются JVM, а не ОС
> - Можно создавать **миллионы** потоков
> - Код остаётся императивным (blocking calls не блокируют carrier thread)

> Применение: высоконагруженные I/O-bound сервисы (REST, базы, внешние API).

```Java
Thread.startVirtualThread(() -> { ... });
// или ExecutorService с newVirtualThreadPerTaskExecutor()
```

### **7. Garbage Collectors (ZGC / Shenandoah)**

> **G1** (по умолчанию) — баланс между throughput и latency.

> **ZGC** (с Java 15 production-ready):
> 
> - Pause time < 10ms даже на терабайтах heap
> - Concurrent marking, relocation
> - Colored pointers + load barriers

> **Shenandoah**:
> 
> - Аналогично ZGC, но из Red Hat
> - Часто чуть лучше throughput

> Когда использовать: низкие latency-требования (финтех, трейдинг, user-facing сервисы).

### **8. `ConcurrentHashMap` vs `Collections.synchronizedMap()`**

> Collections.synchronizedMap() — грубая синхронизация (блокирует весь map).

> ConcurrentHashMap:
> 
> - Lock striping (сегменты)
> - Concurrent reads
> - Высокая производительность при многих потоках
> - Методы: computeIfAbsent, merge, compute
> - Итераторы weakly consistent (не бросают ConcurrentModificationException)

### **9. Жизненный цикл Spring Bean**

**Полный порядок:**

1. Instantiation (BeanFactory создаёт объект)
2. Populate properties (внедрение зависимостей)
3. BeanPostProcessor.postProcessBeforeInitialization() (например, @Autowired уже сработал)
4. @PostConstruct / InitializingBean.afterPropertiesSet() / custom init-method
5. BeanPostProcessor.postProcessAfterInitialization() (AOP proxies, @Async и т.д.)
6. Bean ready
7. Destruction: @PreDestroy / DisposableBean.destroy()

**10. @Component, @Service, @Repository**
