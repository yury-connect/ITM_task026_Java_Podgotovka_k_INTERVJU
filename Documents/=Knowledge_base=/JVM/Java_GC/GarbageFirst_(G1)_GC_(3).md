## 🗑 G1 GC (Garbage First)
GC <u>по умолчанию</u> в современных версиях Java Virtual Machine.

---
# 🥇 Главная идея
**Heap** делится не на 2 области (Young/Old), а на **много одинаковых регионов**.
👉 G1 собирает **самые «мусорные» регионы сначала** (Garbage First).

---
# 🧩 Архитектура
### Heap → набор регионов (обычно `1`–`32`MB)
> Типы регионов:
> - Eden    
> - Survivor    
> - Old    
> - Humongous *(Очень крупные объекты// обычно >50% региона в G1)*    

---
# 🔄 Как работает

### 1️⃣ Young GC
> - Чистится Eden    
> - Короткие Stop-the-World паузы    
> - Быстро    
### 2️⃣ Concurrent Mark
> - Параллельно помечает живые объекты в Old    
> - Минимальные паузы    
### 3️⃣ Mixed GC
> - Чистит часть Old + Young    
> - Выбирает регионы с максимумом мусора    

---
# ⏱ Главная цель

👉 **Предсказуемые паузы**
Настраивается через:
**`-XX:MaxGCPauseMillis=200`**
> G1 пытается уложиться в заданный *target* *(Доп. нагрузка на CPU из-за работы GC)*.

---
# ⚙ Плюсы
> ✅ Предсказуемый **latency**  *(Время отклика системы //`запрос` → `ответ`)*
> ✅ Хорош для больших **heap** (>4–8GB)  
> ✅ Параллельная и конкурентная работа  
> ✅ Меньше **full GC** *(Полная сборка всего heap// `Young` + `Old`)*

---
# ⚠ Минусы
> ❌ Сложная внутренняя логика  
> ❌ Немного выше CPU overhead *(Доп. нагрузка на CPU из-за работы GC)*  
> ❌ Humongous объекты могут фрагментировать heap

---
# 🧠 Когда использовать

| Сценарий       | Подходит? |
| -------------- | --------- |
| Микросервисы   | Да        |
| Большой heap   | Да        |
| Низкая latency | Да        |
| Real-time      | Нет       |

---
# 🧠 Как запомнить

**CMS** → минимальные паузы  
**Parallel** → максимальный throughput  
**G1** → баланс + предсказуемость
